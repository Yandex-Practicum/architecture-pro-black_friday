**Основные сложности**
1. Нужен поиск по товарной категории и по диапазону цен.
2. При оформлении заказов нужно быстро проверять и обновлять товарные остатки. Если остатков недостаточно - создание заказа отклоняется.
3. Большой объём заказа может быть проблемой - пользователь в одном заказе может набрать большое количество разных товарных позиций. Поскольку при оформлении заказа необходимо знать актуальные товарные позиции на момент оформления и обновлять их так, чтобы при создании других заказов от других пользователей были всегда согласованные данные, это приводит к необходимости транзакций. В кластерной системе транзакции становятся распределёнными и несут в себе соответствующие издержки.
4. Неясность в отношении бизнес-возможности онлайн-магазина:  должен ли онлайн-магазин при отсутствии товара в географической зоне пользователя показывать ему наличие товаров из других географических зон и инициировать доставку с другого склада или же показывать пользователю только наличие товаров и возможности их доставки в его географической зоне?

**Результат обсуждения с бизнесом**:
1. По вопросу с доставкой из других городов бизнес принимает решение ограничиваться только географической зоной пользователя. То есть, если пользователь из Казани, то проверять все товарные остатки нужно на складе в Казани. Из других городов доставка не оформляется. Это существенно упрощает поиск товаров, создание заказов и работу с товарными остатками.
2. Предполагается, что 95% процентов пользователей в основном заказывают по 1-2 товара, нет необходимости затрачивать усилия на особую обработку больших заказов. Это согласовывается с требованиями по времени обработки заказов (95-ый перцентиль времени запроса создания заказа).

#### Работа с пользовательскими и гостевыми корзинами
Корзины разделяются на гостевые и пользовательские. Пользовательские корзины жёстко привязаны к пользователю, у пользователя всегда только одна корзина. Это необходимо, чтобы пользователь мог несколько раз авторизовываться и всегда видеть одну и ту же корзину после авторизации.
Гостевые корзины создаются при создании новой веб-сессии, когда пользователь заходит на сайт онлайн-магазина. Гостевые корзины также жеёстко привязаны, но уже к web-сессии, таким образом, пользователь может создавать множество корзин, создавая разные web-сесиии.

По корзинам необходимо хранить следующую информацию:

| Атрибут                         | Описание                                                                                                                                                                                                                                |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| № гостевой сессии               | Уникальный цифробуквенный номер гостевой сессии. Гостевая сессия создаётся, если пользователь первый раз заходит на сайт онлайн-магазине (или когда заходит после очистки данных браузера - cookie, локального хранилища, и так далее). |
| № пользователя                  | Номер пользователя. Если пользователь авторизовался на сайте, то корзине с указанной гостевой сессией присваивается еще номер авторизованного пользователя.                                                                             |
| Состояние корзины               | У корзины могут быть следующие состояния: **активная**; **заказанная**; **заброшенная**.                                                                                                                                                |
| Дата и время создания корзины   | Момент создания корзины.                                                                                                                                                                                                                |
| Дата и время изменения корзины. | Момент последнего изменения корзины (добавления, удаления товара).                                                                                                                                                                      |
| Время удаления корзины          | Для автоматической очистки.                                                                                                                                                                                                             |
| Список товаров                  | Список товаров, добавленных в корзину.                                                                                                                                                                                                  |
**Основные сложности**:
* Пользовательская корзина должна быть одна: все авторизованные сессии у пользователя видят одну и ту же корзину.
* У гостевых сессий также может быть только одна активная корзина.
* Гостевая и пользовательская корзины отличаются по способу доступа - гостевая должна быть получена по номеру web-сессии, а пользовательская - по номеру пользователя. Мощность множества номеров web-сессий больше мощности множества номеров пользователей.
* Плохая прогнозируемость потока событий, создающих гостевые корзины (пользователи могут чистить куки, заходить с новых браузеров, открывать сайт в вкладках "Инкогнито", и так далее).

#### Работа с заказами
По заказам необходимо хранить следующую информацию:

| Атрибут                        | Описание                                                                                                                                                                    |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| № заказа                       | Обозначает уникальный заказ в системе.                                                                                                                                      |
| № клиента                      | Номер пользователя.                                                                                                                                                         |
| Дата и время оформления заказа | Дата и время оформления заказа.                                                                                                                                             |
| Список заказанных товаров      | Список заказанных товаров содержит следующие данные:<ul><li>Ссылка на товарную позицию</li><li>Стоимость товарной позиции на момент заказа</li><li>Количество</li></ul><br> |
| Статус заказа                  | У заказа могут быть следующие статусы: создан, в обработке, доставлен.                                                                                                      |
| Сумма заказа                   | Общая сумма заказа в валюте пользователя (рубли).                                                                                                                           |
| Геозона                        | Геозона пользователя, создавшего заказ.                                                                                                                                     |
**Основные сложности**:
1. При оформлении заказов нужно быстро проверять и обновлять товарные остатки. Если остатков недостаточно - создание заказа отклоняется. При низкой локальности хранения товарных остатков (информация по товарным остаткам равномерно распределена по кластеру СУБД), обработка заказа приводит к кластерным запросам, где по каждому товару необходимо делать запросы с проверкой и изменением товарного остатка в определённый сегмент кластера СУБД. При высокой локальности хранения товарных остатков есть риск перекоса трафика на определённые сегменты кластера СУБД.
2. Необходимость использования транзакций в распределённом кластере.

**Приблизительная оценка количества заказов**:
1. Активная аудитория онлайн-магазина оценивается примерно в 50 миллионов человек.
2. 80% активных пользователей генерируют потоки заказов, каждый из которых частотой примерно 1 заказ в 6 месяцев (или 2 заказа в год).
3. 20% активных пользователей генерируют потоки заказов, каждый из которых частотой примерно 1 заказ в месяц (или 12 заказов в год).
4. Общий поток заказов в год ожидается примерно следующий: `50 000 000 * (0.8 * 2 + 0.2 * 12) = 200 000 000` заказов в год.
5. Заказы должны храниться в оперативном хранилище год, затем данные по заказам переносятся на архивную систему хранения.
6. Первоначальное планирование объёма хранения заказов рассчитывается на 2 года.
7. Итого в оперативном хранилище должно храниться до 400 000 000 документов по заказам.
### Решение

#### Схема коллекции carts

| ---------------------------------- | ---------------------- | ------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **№ корзины**                      | **\_id**               | 12 байт             | ObjectId             | Уникальный идентификатор корзины. Суррогатный ключ.                                                                                                                                                                                                                                                                            |
| **№ владельца корзины**            | **owner\_id**          | 64 байта            | string (uuid \| int) | Номер владельца корзины. В случае, если владельцем корзины является неавторизованный пользователь, то в качестве номера владельца корзины в данном поле выступает номер сессии. В случае, когда владельцем корзины является авторизованный пользователь, то в этом поле значением является номер авторизованного пользователя. |

| Поля индекса           | Параметры индекса                            | Описание                                                                                                                                                                          |
| ---------------------- | -------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `hashed(owner_id)`     | `hashed`                                     | Индекс для обеспечения доступа к гостевым и пользовательским корзинам по номерам пользователей и гостевых сессий. Также этот индекс является индексом для сегментирования данных. |
| `owner_id`<br>`status` | `filterBy(status=active)`<br>`unique = true` | Частичный индекс для обеспечения правила "1 активная корзина на 1 владельца".                                                                                                     |
**Реализация доступа к корзинам**:
1. Специальный атрибут `owner_key` играет роль общего ключа для пользовательских и гостевых корзин. Также он является основным ключом для сегментирования данных.
2. Связь 1 к 1 корзины к владельцу (пользователю или web-сесиии) обеспечивается с помощью уникального составного индекса по полям `{owner_key, status}`, где `status` равен `active`. 

**Приблизительная оценка объёма данных**:
1. Общее количество пользователей: `50 000 000`.
2. Предположительная оценка количества неавторизованных пользователе (web-сессий): `150 000 000`. То есть, каждый третий пользователь из 150 млн. регистрируется и авторизуется в итоге на сайте.
3. Среднее количество товаров  в корзинах: 4 позиции.
4. Средний размер документа по одной корзине: `12 + 64 + 64 + 8 + 8 + 16 + 16 + 16 + 4 * (12 + 8 + 16) = 348`. Округляем до `512` с учётом возможной служебной информации.
5. Ожидаемый объём данных по активным корзинам: `200 000 000 * 512 = 102 400 000 000 ~ 95 Гб.

**Обоснование выбора ключа для сегментирования**:
1. Множества значений `session_id` и `user_id` не пересекаются, так как изначально у них разные типы - для `user_id` используется тип `integer`, для `session_id` используется строковое представление UUID, соответственно, мощность множества значений `owner_key` составляет сумму мощностей множеств `session_id` и `user_id`.
2. Множество значений ключа `owner_key` обладает большой мощностью и низкой частотой распределения значений.

**Приблизительная оценка объёма данных**:
1. Предположим, что заказ содержит в среднем 3 позиции. Опираясь на приведённую выше схему, оценим средний размер документа: `12 + 8 + 16 + 8 + 16 + 4 * (12 + 8) + 128 = 268 `байт. Округлим до `382` байт, с учётом возможной служебной информации.
2. Полный объём данных за 2 года составит: `268 * 200 000 000 * 2 = 99 Гб`. 

**Обоснование выбора индекса сегментирования**:
1. С точки зрения пользователей, необходимо быстро получать историю заказов. Наиболее оптимальным вариантом сегментирования данных будет такой, при котором данные по одному пользователю будут сгруппированы в пределах одного сегмента СУБД.
2. **Мощность множества значений**: `50 000 000`.
3. **Частота запросов по ключу**: ожидается, что каждый пользователь в рамках своего взаимодействия с онлайн-магазином будет запрашивать статус текущего заказа, а также просматривать историю заказов. Поэтому выбор ключа `user_id` приводит к наиболее равномерному распределению вычислительной нагрузки.

**Обоснование выбора индекса сегментирования**:
1. Данный индекс позволяет равномерно распределять данные по сегментам СУБД, поскольку мощность множества его значений высокая (68 000 000), и при этом частота его значений низкая, постоянная.
2. Для борьбы с нагрузкой, создаваемой кластерными запросами, можно использовать следующие средства:
	1. Кешировать результаты поиска товаров по категориям и ценам. Фильтрацию по ценам товаров можно ограничить путём предоставления пользователю диапазонов цен, чтобы уменьшить множество ключей кеширования. Это позволит существенно снизить поток запросов на поиск товаров в СУБД.
	2. Большинство заказов содержит в среднем до 4 позиций - это до 4-х запросов в различные сегменты кластера, что вполне укладывается в требования по времени обработки заказа - до 5 секунд.

### **Альтернативы**
1. Разделение коллекции **carts** на две схожие по схеме, но различные по дисциплине доступа: **guest\_carts** и **user\_carts**.
	1. **Плюсы**:
		1. Простота доступа к данным по разным типам корзин. Логика для обеспечения доступа к гостевой корзине явно показывает обращение к коллекции **guest\_carts**, логика обращения доступа к пользовательской корзине явно показывает обращение к коллекции **user\_carts**.
		2. Упрощение правил обработки корзин: у авторизованных пользователей всегда всего одна корзина.
	2. **Минусы**:
		1. Дублирование схем.
		2. Необходимость работы с несколькими коллекциями.
	3. **Обоснование отказа от данного решения**: требовалась общая простота схемы данных, т.к. пользовательские и гостевые корзины с точки зрения набора атрибутов являлись одной и той же сущности.
2. Хранение товарных остатков по каждой геозоне вложенным документом в структуре информации по товару, где ключ - геозона, значение - остаток.
	1. **Плюсы**:
		1. Высокая локальность данных - все остатки на складах по конкретному товару лежат в документе по этому же товару.
	2. **Минусы**:
		1. При большом потоке заказов на конкретный товар из разных географических зон необходимо обновлять товарные остатки в одном и том же документе, что приводит к необходимости блокировок, а соответственно, замедлению обработки потока заказов.
	3. **Обоснование отказа от данного решения**: замедление обработки потока заказов из-за блокировок в одном документе.
