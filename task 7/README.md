## Коллекция products
Шард‑ключ: category
Стратегия шардирования: диапазонное шардирование (range-based)

Почему подходят под особенности коллекций и операций:
- Товары группируются по категориям («Электроника», «Аудиотехника» и т. д.), что соответствует частым операциям поиска по категориям.
- Запросы с фильтрацией по цене обычно идут в связке с категорией (category + price), поэтому данные по одной категории будут локализованы в одном/нескольких шардах.
- Обновления остатков происходят в рамках конкретного товара, а не всей категории, поэтому конфликт записей минимален.

## Коллекция orders
Шард‑ключ: user_id
Стратегия шардирования: хеш‑шардирование (hash-based)

Почему подходят под особенности коллекций и операций:
- Основные операции — поиск истории заказов по user_id и отображение статуса конкретного заказа.
- Хеш‑шардирование равномерно распределяет заказы по шардам, избегая «горячих точек» (например, если один пользователь делает много заказов).
- Запросы по user_id будут направляться в один шард, обеспечивая высокую скорость чтения.
- Геозона (geo_zone) не подходит как единственный ключ, так как заказы из одной геозоны могут принадлежать разным пользователям, что снизит локальность данных.

## Коллекция carts
Шард‑ключ: составной ключ {"user_id": 1, "session_id": 1}
Стратегия шардирования: диапазонное шардирование (range-based)

Почему подходят под особенности коллекций и операций:
- Основные операции: получение активной корзины по user_id или session_id.

Составной ключ позволяет:
- Локализовать все корзины одного пользователя (user_id) в одном шарде (быстрое получение истории корзин).
- Группировать гостевые корзины по session_id для эффективного поиска.
- Статус active часто используется в фильтрах, но сам по себе не обеспечивает равномерного распределения.
- TTL‑очистка (expires_at) не требует шардирования по времени, так как выполняется фоново и не критична к производительности.
