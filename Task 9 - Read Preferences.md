# Настройка чтения с реплик и консистентность

## 1. Коллекция `products`

| Операция | Read Preference | Обоснование |
|----------|----------------|-------------|
| Описание товара на странице продукта | `secondaryPreferred` | Данные о товаре (название, категория, атрибуты) меняются редко. Чтение с secondary снижает нагрузку на primary. Устаревшие на 1–2 сек данные не критичны — пользователь не заметит |
| Поиск товаров по категории и фильтрация по цене | `secondaryPreferred` | Каталожные запросы тяжёлые и частые. Secondary справится, даже если цена обновилась секунду назад — пользователь увидит актуальную цену на странице товара |
| Проверка остатка при добавлении в корзину | `primary` | **Критично.** Остатки меняются часто (при каждой покупке). Чтение устаревшего остатка → риск продажи недоступного товара. Только primary гарантирует актуальные данные |
| Списание остатка при оформлении заказа | `primary` | Операция записи — всегда через primary. Используется `findOneAndUpdate` с проверкой `stock >= quantity` для атомарного списания |

**Допустимая задержка репликации:** до **2 секунд** для каталожных операций. Карточки товаров и результаты поиска обновляются плюс кешируются в Redis (TTL 60 сек), поэтому задержка реплики несущественна. Для остатков задержка недопустима — читаем с primary.

## 2. Коллекция `orders`

| Операция | Read Preference | Обоснование |
|----------|----------------|-------------|
| Отображение статуса заказа (для покупателя) | `primary` | **Критично.** Пользователь только что оплатил заказ и сразу проверяет статус. Чтение с secondary может показать устаревший статус (например, «ожидает оплаты» вместо «оплачен»), что вызовет обращение в поддержку |
| История заказов пользователя (список) | `secondaryPreferred` | Исторические заказы не меняются. Даже если последний заказ появится с задержкой в 1–2 сек — это допустимо. Тяжёлые запросы с сортировкой по дате лучше отдать secondary |
| Аналитика и отчёты (внутренние) | `secondary` | Аналитические запросы тяжёлые и длинные. Их нужно изолировать от primary, чтобы не влиять на пользовательские операции. Задержка в несколько секунд для отчётов допустима |
| Создание заказа | `primary` | Запись — всегда primary. Используется транзакция (создание заказа + списание остатков) |

**Допустимая задержка репликации:** до **2 секунд** для истории заказов, до **10 секунд** для аналитики. Для текущего статуса заказа — задержка недопустима.

## 3. Коллекция `carts`

| Операция | Read Preference | Обоснование |
|----------|----------------|-------------|
| Получение активной корзины (`status: "active"`) | `primary` | **Критично.** Корзина — рабочий документ, изменяется при каждом действии пользователя. Чтение устаревшей версии → потеря только что добавленного товара, дублирование позиций, ошибки при слиянии гостевой корзины |
| Добавление/удаление товара в корзине | `primary` | Запись — всегда primary. Операция `updateOne` с `$push` / `$pull` по массиву `items` |
| Слияние гостевой корзины в пользовательскую | `primary` | **Критично.** Операция затрагивает две корзины: чтение гостевой + обновление пользовательской + отметка гостевой как `abandoned`. Любое чтение устаревших данных приведёт к потере товаров |
| Очистка брошенных корзин (фоновый процесс) | `secondary` | Фоновый batch-процесс по выборке корзин с `expires_at < now`. Задержка не критична — корзина удалится чуть позже |

**Допустимая задержка репликации:** для фоновой очистки — до **30 секунд**. Все пользовательские операции с корзиной — только primary, задержка недопустима.

## Настройка maxStalenessSeconds

Для операций на secondary ограничиваем допустимый лаг репликации, чтобы не читать слишком устаревшие данные:

```javascript
// Каталог товаров — лаг не более 5 сек
db.products.find({ category: "electronics" }).readPref("secondaryPreferred", [], { maxStalenessSeconds: 5 })

// История заказов — лаг не более 5 сек
db.orders.find({ customer_id: "cust_123" }).readPref("secondaryPreferred", [], { maxStalenessSeconds: 5 })

// Аналитика — лаг до 30 сек допустим
db.orders.aggregate([...]).readPref("secondary", [], { maxStalenessSeconds: 30 })
```

## Принципы выбора

1. **Primary** — когда данные меняются часто и устаревшее чтение влечёт бизнес-риск (продажа отсутствующего товара, потеря товара из корзины, неверный статус заказа)
2. **secondaryPreferred** — когда данные меняются редко или задержка в 1–2 сек не заметна пользователю (каталог, история)
3. **secondary** — для тяжёлых фоновых/аналитических операций, которые нужно изолировать от primary
