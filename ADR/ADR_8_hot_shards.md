## Контекст

В рамках проекта онлайн-магазина "Мобильный мир" используется MongoDB в шардированной конфигурации для хранения данных о товарах, заказах и корзинах. Согласно предыдущему архитектурному решению (ADR_7_sharding.md), были выбраны стратегии шардирования: хэш по `_id` для коллекции `products`, диапазонное по `user_id` и `created_at` для `orders`, и по `user_id/session_id` для `carts`.

Наблюдается, что 70% запросов приходится на товары из категории "Электроника", что создает риск возникновения "горячих" шардов (hot shards) — шардов, на которых сконцентрирована чрезмерно высокая нагрузка по сравнению с другими. Это может привести к снижению общей производительности системы, увеличению задержек и узким местам в работе приложения.

## Проблема

Необходимо разработать архитектурное решение для:
1.  Раннего выявления "горячих" шардов.
2.  Эффективного устранения дисбаланса нагрузки.
3.  Обеспечения устойчивости системы к пикам нагрузки на популярные категории товаров, в первую очередь "Электроника".

Решение должно быть совместимо с существующей схемой шардирования и не нарушать производительность ключевых бизнес-операций.

## Решение

### 1. Метрики для мониторинга состояния шардов

Для комплексного мониторинга и выявления "горячих" шардов необходимо собирать и анализировать следующие метрики на уровне каждого шарда:

*   **Производительность и нагрузка:**
    *   `ops/sec`: Количество операций чтения и записи в секунду. Аномально высокое значение по сравнению со средним по кластеру — прямой признак перегрузки.
    *   `queryExecutor.scanned`: Количество просканированных документов. Высокое значение может указывать на неэффективные запросы или отсутствие индексов.
    *   `queryExecutor.scannedObjects`: Количество просканированных объектов.
    *   `network.bytesIn` / `network.bytesOut`: Объём входящего и исходящего сетевого трафика.

*   **Ресурсы системы:**
    *   `cpu.user` / `cpu.system`: Использование CPU процессом `mongod`.
    *   `extra_info.page_faults`: Количество page faults. Резкий рост может указывать на нехватку оперативной памяти (RAM) для рабочего набора данных (working set).
    *   `mem.resident`: Объём физической памяти, используемой процессом `mongod`.
    *   `mem.virtual`: Объём виртуальной памяти.

*   **Задержки (Latency):**
    *   `opLatencies.reads.latency` / `opLatencies.writes.latency` / `opLatencies.commands.latency`: Средняя и 95-я перцентиль задержки операций. Постоянно высокие задержки — ключевой индикатор проблемы.

*   **Размер данных:**
    *   `dataSize`: Общий размер данных в коллекциях на шарде.
    *   `numObjects`: Количество документов.
    *   `storageSize`: Используемое дисковое пространство.

*   **Состояние балансировщика:**
    *   `balancer.numchunksmoved`: Количество перемещенных чанков за определенный период. Низкая активность при явном дисбалансе может указывать на проблемы с балансировщиком.
    *   `balancer.state`: Текущее состояние балансировщика (включен/выключен).

**Источники метрик:**
*   Команда `sh.status()` — предоставляет общий обзор распределения чанков по шардам.
*   Команда `db.serverStatus()` — детальная статистика по каждому `mongod` процессу.
*   Агрегационный пайплайн с `$collStats` — для получения статистики по конкретным коллекциям.
*   Интеграция с системами мониторинга (например, Prometheus + Grafana) для сбора, визуализации и настройки алертов.

### 2. Механизмы выявления "горячих" шардов

Выявление должно быть автоматизировано и основано на анализе собранных метрик:

1.  **Пороговые алерты:** Настройка алертов в системе мониторинга при превышении критических значений (например, CPU > 85%, средняя задержка чтения > 50ms, ops/sec > 2x среднего по кластеру).
2.  **Анализ дисбаланса данных:** Регулярный анализ вывода `sh.status()` для выявления шардов, на которых сконцентрировано значительно больше чанков или данных, чем на других.
3.  **Анализ производительности:** Сравнение метрик производительности (ops/sec, latency) каждого шарда со средними значениями по кластеру. Шард, который постоянно находится в топ-1 по нагрузке, является кандидатом на "горячий".

### 3. Механизмы устранения дисбаланса и автоматического перераспределения

Для устранения дисбаланса и предотвращения его в будущем предлагается использовать комбинацию встроенных механизмов MongoDB и архитектурных решений:

#### a) Оптимизация и настройка встроенного балансировщика MongoDB

MongoDB имеет встроенный балансировщик, который автоматически перемещает чанки между шардами для выравнивания распределения данных.

*   **Настройка окон балансировки (Balancer Windows):** Настроить балансировщик на работу в определенные периоды низкой нагрузки (например, с 2:00 до 5:00 по МСК), чтобы минимизировать влияние на производительность в пиковые часы.
    ```javascript
    // Установка окна балансировки
    sh.setBalancerState(true)
    sh.setBalancerWindow({ start: "02:00", stop: "05:00" })
    ```
*   **Throttling:** При необходимости можно ограничить скорость перемещения чанков с помощью `balancerBytesPerSecond`, чтобы не перегружать сеть.
    ```javascript
    // Ограничение скорости перемещения
    sh.setBalancerByteLimit(104857600) // 100MB/s
    ```

#### b) Использование тегов зон (Zone Sharding) для управления распределением

Это наиболее эффективный способ борьбы с "горячими" шардами для популярных категорий.

1.  **Создание зоны для "горячих" данных:** Создать отдельную зону, например, `hot_zone`, и назначить ей один или несколько выделенных, более мощных шардов (например, `shard03`, `shard04`).
2.  **Привязка диапазона шард-ключа к зоне:** Для коллекции `products` можно попробовать использовать составной шард-ключ `{ category: 1, _id: "hashed" }` и привязать диапазон `{ category: "Электроника" }` к зоне `hot_zone`.
    ```javascript
    // Создание зоны
    sh.addShardToZone("shard03", "hot_zone")
    sh.addShardToZone("shard04", "hot_zone")

    // Привязка диапазона ключей к зоне
    sh.addTagRange("somedb.products", { category: "Электроника", _id: MinKey }, { category: "Электроника", _id: MaxKey }, "hot_zone")
    ```
    **Важно:** Это требует изменения шард-ключа, что является дорогостоящей операцией. Альтернативой может быть использование `zone` для управления распределением чанков на уровне уже существующего хэш-ключа, но это менее точный контроль.

#### c) Реактивное перераспределение с использованием Change Streams

Для более гибкого и реактивного управления можно реализовать внешний контроллер, который будет:
1.  Мониторить метрики в реальном времени (через Prometheus или напрямую через `$changeStream` на коллекции `config` или `$collStats`).
2.  При обнаружении "горячего" шарда автоматически вызывать команду `moveChunk` для перемещения чанков с перегруженного шарда на менее нагруженные.
    ```javascript
    // Пример команды для ручного перемещения чанка
    sh.moveChunk("somedb.products", { _id: ObjectId("...") }, "shard02")
    ```
    Такой контроллер может использовать более сложные алгоритмы балансировки, чем встроенный балансировщик.

## Выводы

Предложен комплексный подход к выявлению и устранению "горячих" шардов, сочетающий:
1.  **Наблюдаемость:** Комплексный сбор метрик для раннего обнаружения проблем.
2.  **Автоматизацию:** Использование встроенного балансировщика MongoDB с настройкой окон и throttling.
3.  **Архитектурные решения:** Применение тегов зон (zone sharding) для изоляции высоконагруженных данных на выделенные ресурсы.
4.  **Гибкость:** Возможность реализации внешнего реактивного контроллера для сложных сценариев.

Это решение обеспечит стабильную производительность шардированного кластера MongoDB даже при значительных пиках нагрузки на отдельные категории товаров.
