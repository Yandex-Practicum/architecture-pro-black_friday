## Контекст

Онлайн-магазин «Мобильный мир» масштабировался и теперь включает множество категорий товаров. Бэкенд состоит из микросервисов, а данные хранятся в MongoDB в трёх основных коллекциях: `products`, `orders` и `carts`. Рост объёма данных и нагрузки требует перехода к шардированной архитектуре для обеспечения производительности, масштабируемости и отказоустойчивости.

## Проблема

Необходимо спроектировать схемы коллекций и выбрать оптимальные стратегии шардирования, чтобы:
- Обеспечить равномерное распределение данных и нагрузки по шардам.
- Минимизировать количество операций между шардами (scatter-gather).
- Поддерживать высокую производительность для ключевых операций: создание заказов, обновление остатков, поиск по пользователю и т.д.

## Решение

### 1. Схемы коллекций

#### Коллекция `products`

**Описание:** Хранит информацию о товарах.

**Схема:**
```json
{
  "_id": ObjectId, // Уникальный идентификатор товара
  "name": String, // Наименование
  "category": String, // Категория товара (например, "Электроника", "Книги")
  "price": Number, // Цена
  "stock": {
    "MSK": Number, // Остаток в Москве
    "SPB": Number, // Остаток в Санкт-Петербурге
    "EKB": Number, // Остаток в Екатеринбурге
    // ... другие геозоны
  },
  "attributes": {
    "color": String, // Цвет
    "size": String  // Размер
    // ... другие атрибуты
  }
}
```

**Ключевые операции:**
- Частые обновления остатков (`stock`) при покупках.
- Поиск по категориям и ценам.
- Чтение описания товара.

#### Коллекция `orders`

**Описание:** Хранит заказы клиентов.

**Схема:**
```json
{
  "_id": ObjectId, // Уникальный идентификатор заказа
  "user_id": ObjectId, // Идентификатор клиента
  "created_at": ISODate, // Дата и время оформления заказа
  "items": [
    {
      "product_id": ObjectId,
      "quantity": Number,
      "price": Number // Цена на момент заказа
    }
  ],
  "status": String, // Статус заказа ("pending", "shipped", "delivered")
  "total_amount": Number, // Общая сумма заказа
  "geo_zone": String // Геозона заказа (например, "MSK")
}
```

**Ключевые операции:**
- Создание нового заказа.
- Поиск истории заказов по `user_id`.
- Отображение статуса заказа по `_id`.

#### Коллекция `carts`

**Описание:** Хранит текущие корзины пользователей и гостей.

**Схема:**
```json
{
  "_id": ObjectId, // Уникальный идентификатор корзины
  "user_id": ObjectId, // Идентификатор пользователя (если авторизован)
  "session_id": String, // Идентификатор сессии (для гостей)
  "items": [
    {
      "product_id": ObjectId,
      "quantity": Number
    }
  ],
  "status": String, // Статус ("active", "ordered", "abandoned")
  "created_at": ISODate,
  "updated_at": ISODate,
  "expires_at": ISODate // TTL индекс для автоматической очистки
}
```

**Ключевые операции:**
- Создание и получение активной корзины по `user_id` или `session_id`.
- Добавление/удаление товаров.
- Слияние гостевой корзины в пользовательскую.
- Отметка корзины как заказанной.

### 2. Стратегии шардирования и выбор шард-ключей

#### Коллекция `products`

**Стратегия:** **Хэш-шардирование (Hash Sharding)**

**Шард-ключ:** `{ _id: "hashed" }`

**Обоснование:**
- **Равномерное распределение:** Товары добавляются со временем, и их идентификаторы `_id` (обычно ObjectId) имеют временной порядок. Прямое шардирование по `_id` создаст "горячие" шарды, так как все новые товары будут попадать в один шард. Хэширование `_id` гарантирует равномерное распределение по всем шардам.
- **Частые обновления:** Операции обновления остатков (`stock`) происходят по `product_id`. Поскольку `product_id` является частью `_id`, запросы на обновление будут направлены напрямую к одному шарду, минимизируя scatter-gather.
- **Поиск по категориям:** Поиск по `category` или `price` может потребовать scatter-gather, но это допустимо, так как такие запросы менее часты, чем обновления остатков.

**Альтернатива с составным ключом:**
Возможно рассмотреть составной шард-ключ `{ category: 1, _id: "hashed" }`. Это могло бы улучшить производительность для запросов, фильтрующих по категории, так как документы одной категории будут группироваться на меньшем количестве шардов. Однако, это создает риски:
  - **Неравномерное распределение:** Если одна категория (например, "Электроника") значительно больше других, она может перегрузить один или несколько шардов.
  - **Сложность управления:** Увеличивается сложность балансировки данных между шардами.
  - **Ограниченная выгода:** Основная операция — обновление остатков по `product_id` — не выигрывает от этого ключа, так как `product_id` (часть `_id`) уже эффективно направляет запросы.

**Вывод:** Стандартный хэш-ключ `{ _id: "hashed" }` остаётся предпочтительным выбором, так как он обеспечивает максимальную устойчивость к скученности данных и оптимален для самой частой операции — обновления остатков. Составной ключ с категорией не оправдывает потенциальных рисков неравномерного распределения.

**Команда для включения шардирования:**
```javascript
sh.shardCollection("somedb.products", { "_id": "hashed" })
```

#### Коллекция `orders`

**Стратегия:** **Диапазонное шардирование (Range-based Sharding)**

**Шард-ключ:** `{ user_id: 1, created_at: -1 }`

**Обоснование:**
- **Горячие точки по пользователю:** Основная операция — получение истории заказов по `user_id`. Группировка всех заказов одного пользователя вместе (co-location) позволяет выполнять этот запрос на одном шарде.
- **Сортировка по времени:** Добавление `created_at` в ключ обеспечивает, что заказы одного пользователя будут отсортированы по времени, что упрощает пагинацию и поиск последних заказов.
- **Создание заказов:** Новые заказы создаются с новым `created_at`. Поскольку `user_id` распределён, новые заказы разных пользователей будут распределяться по разным шардам, предотвращая перегрузку одного шарда.

**Команда для включения шардирования:**
```javascript
sh.shardCollection("somedb.orders", { "user_id": 1, "created_at": -1 })
```

#### Коллекция `carts`

**Стратегия:** **Диапазонное шардирование (Range-based Sharding)**

**Шард-ключ:** `{ user_id: 1, session_id: 1 }`

**Обоснование:**
- **Для авторизованных пользователей:** `user_id` будет уникальным, и `session_id` может быть опущен или использоваться как дополнительный дескриптор. Запросы по `user_id` будут эффективны.
- **Для гостевых пользователей:** `user_id` будет `null`, но `session_id` будет уникальным для каждой сессии. Это гарантирует, что гостевые корзины будут равномерно распределены по шардам, так как каждый уникальный `session_id` будет создавать уникальное значение составного ключа.

Этот подход решает проблему "горячего" шарда для гостей и сохраняет возможность эффективного доступа к корзинам по `user_id` для авторизованных пользователей.

**Команда для включения шардирования:**
```javascript
sh.shardCollection("somedb.carts", { "user_id": 1, "session_id": 1 })
```

## Выводы

Предложенные стратегии шардирования оптимизированы под паттерны доступа к данным:
- `products`: Хэш-шардирование по `_id` для равномерного распределения и эффективных обновлений.
- `orders`: Диапазонное шардирование по `{ user_id, created_at }` для эффективного доступа к истории заказов.
- `carts`: Диапазонное шардирование по `{ user_id, session_id }` для быстрого доступа к активным корзинам.

Это решение обеспечит масштабируемость и высокую производительность для растущего онлайн-магазина.
