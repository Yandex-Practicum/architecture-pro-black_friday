## Контекст

Онлайн-магазин "Мобильный мир" в настоящее время использует MongoDB в шардированной конфигурации. Несмотря на успешное решение проблем с масштабированием и "горячими" шардами (ADR_7, ADR_8), наблюдается сложность в управлении полным перераспределением данных при изменении количества шардов. Для достижения более высокой отказоустойчивости, упрощения масштабирования и обеспечения равномерного распределения данных без дорогостоящих операций перебалансировки, рассматривается миграция на Apache Cassandra.

## Требования

1.  Определить критически важные данные (заказы, товары, корзины, история заказов, сессии) и обосновать целесообразность применения Cassandra для них.
2.  Разработать концептуальную модель данных для выбранных сущностей, определить partition key и кластерные ключи, обосновать выбор с точки зрения равномерного распределения и избежания 'горячих' партиций.
3.  Определить стратегии обеспечения целостности данных (Hinted Handoff, Read Repair, Anti-Entropy Repair), назначить их сущностям и обосновать выбор с учетом компромиссов между задержкой и согласованностью.

## Решение

### 1. Обоснование применения Cassandra

Apache Cassandra является распределенной колоночной СУБД, идеально подходящей для сценариев, требующих высокой доступности, линейного масштабирования и устойчивости к сбоям. Ниже приведено обоснование для каждой сущности:

| Сущность | Обоснование применения Cassandra |
|---|---|
| **products** | Высокая частота чтения (просмотр каталога, поиск). Cassandra обеспечивает очень низкую задержку при чтении. Данные редко обновляются, что соответствует write-once, read-many (WORM) паттерну. |
| **orders** | Критически важны высокая доступность и отказоустойчивость. Cassandra не имеет single point of failure, что гарантирует возможность оформления заказов даже при выходе из строя одного или нескольких узлов. |
| **carts** | Активные корзины требуют быстрого доступа для чтения и записи. Cassandra обеспечивает предсказуемую производительность при высокой нагрузке. Данные имеют ограниченный срок жизни (TTL), что эффективно поддерживается в Cassandra. |
| **sessions** | Сессии пользователей — это данные с высокой частотой записи и чтения, которые должны быть доступны с минимальной задержкой. Cassandra идеально подходит для хранения сессий благодаря своей производительности и отказоустойчивости. |

**Общие преимущества Cassandra перед MongoDB в данном контексте:**
- **Линейное масштабирование:** Добавление нового узла автоматически инициирует равномерное перераспределение данных (с помощью vnodes) без необходимости ручного управления чанками и балансировщиком.
- **Высокая доступность:** Все узлы равноправны, отсутствие master-узла устраняет single point of failure.
- **Упрощенное управление:** Механизмы vnodes и gossip-протокол значительно упрощают администрирование кластера по сравнению с управлением шардами и реплика-сетами в MongoDB.

### 2. Концептуальная модель данных

В Cassandra данные моделируются под конкретные запросы. Ниже представлены таблицы для ключевых операций.

#### Таблица `products_by_category`

**Назначение:** Эффективный поиск товаров по категории.

```cql
CREATE TABLE products_by_category (
    category text,
    product_id uuid,
    name text,
    price decimal,
    stock map<text, int>, // Например, {"MSK": 10, "SPB": 5}
    attributes map<text, text>,
    PRIMARY KEY (category, product_id)
) WITH CLUSTERING ORDER BY (product_id ASC);
```

- **Partition Key:** `category` — Группирует все товары одной категории на одном или нескольких узлах, что оптимизирует запросы по категории.
- **Clustering Key:** `product_id` — Обеспечивает уникальность и позволяет эффективно получать конкретный товар.
- **Обоснование:** Несмотря на риск "горячей" партиции для популярной категории (например, "Электроника"), это компенсируется использованием vnodes, которые автоматически распределяют данные по нескольким физическим узлам. Для запросов по имени товара можно создать отдельную таблицу `products_by_name`.

#### Таблица `orders_by_user`

**Назначение:** Получение истории заказов пользователя.

```cql
CREATE TABLE orders_by_user (
    user_id uuid,
    order_id uuid,
    created_at timestamp,
    status text,
    total_amount decimal,
    geo_zone text,
    items list<frozen<item>>, // Предполагается определение UDT `item`
    PRIMARY KEY (user_id, created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
```

- **Partition Key:** `user_id` — Гарантирует, что все заказы одного пользователя хранятся на одном узле, что позволяет выполнять запросы истории заказов без scatter-gather.
- **Clustering Keys:** `created_at DESC, order_id ASC` — Позволяет получать заказы в порядке убывания даты (сначала новые) и избегать коллизий при одинаковом времени.
- **Обоснование:** Это прямое соответствие паттерну доступа. Создание нового заказа будет направлено на партицию пользователя, что эффективно.

#### Таблица `carts_by_user`

**Назначение:** Хранение активной корзины пользователя.

```cql
CREATE TABLE carts_by_user (
    user_id uuid,
    session_id text,
    items list<frozen<item>>,
    status text,
    created_at timestamp,
    updated_at timestamp,
    expires_at timestamp,
    PRIMARY KEY (user_id)
) WITH default_time_to_live = 86400; // 24 часа
```

- **Partition Key:** `user_id` — Для авторизованных пользователей корзина привязана к их ID.
- **Обоснование:** Простая и эффективная модель для быстрого доступа. Для гостевых пользователей можно использовать отдельную таблицу `carts_by_session` с `session_id` в качестве partition key.

#### Таблица `sessions_by_id`

**Назначение:** Хранение данных сессии пользователя.

```cql
CREATE TABLE sessions_by_id (
    session_id text PRIMARY KEY,
    user_id uuid,
    data map<text, text>,
    created_at timestamp,
    expires_at timestamp
) WITH default_time_to_live = 3600; // 1 час
```

- **Partition Key:** `session_id` — Уникальный идентификатор сессии, обеспечивает равномерное распределение.
- **Обоснование:** Прямой доступ по ID сессии с автоматическим удалением устаревших записей.

### 3. Стратегии обеспечения целостности данных

Cassandra использует несколько механизмов для обеспечения согласованности и восстановления после сбоев.

| Механизм | Назначение | Применение к сущностям | Обоснование |
|---|---|---|---|
| **Hinted Handoff** | Временное хранение записей, если целевой узел недоступен. Запись отправляется позже, когда узел восстановится. | **Все сущности** | Обеспечивает высокую доступность для операций записи. Даже если один узел упал, приложение может продолжать работать. Критично для `orders` и `carts`. |
| **Read Repair** | Автоматическое выявление и исправление расхождений данных при чтении. Когда клиент читает данные с нескольких реплик, Cassandra сравнивает версии и обновляет устаревшие реплики. | **products**, **orders** | Подходит для данных, которые читаются чаще, чем пишутся. Обеспечивает постепенное восстановление согласованности без дополнительных затрат. |
| **Anti-Entropy Repair** (nodetool repair) | Периодическая фоновая операция, которая сравниет хеши данных (Merkle trees) на всех репликах и синхронизирует расхождения. | **Все сущности** | Необходим для долгосрочной целостности данных, особенно для редко читаемых данных (например, старые заказы), которые могут не затрагиваться Read Repair. Должен выполняться регулярно (например, еженедельно). |

**Выбор уровня согласованности (Consistency Level):**
- **`products` (чтение):** `ONE` — Низкая задержка для просмотра каталога.
- **`products` (запись):** `QUORUM` — Гарантия, что обновление цены/наличия зафиксировано на большинстве реплик.
- **`orders`, `carts` (чтение/запись):** `QUORUM` — Баланс между доступностью и согласованностью. Гарантирует, что пользователь видит актуальное состояние.
- **`sessions` (чтение/запись):** `ONE` — Приоритет — низкая задержка и высокая доступность.
