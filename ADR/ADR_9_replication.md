## Контекст

В рамках проекта онлайн-магазина "Мобильный мир" используется шардированный кластер MongoDB с реплика-сетами для обеспечения отказоустойчивости и масштабируемости. Наблюдается высокая нагрузка на данные о товарах, особенно в категории "Электроника", что создает риск перегрузки primary-нод. Для эффективного распределения нагрузки необходимо оптимизировать политики чтения, максимально используя secondary-реплики там, где это допустимо по требованиям консистентности.

## Требования

1. Для коллекций `products`, `orders` и `carts` определить операции чтения и назначить им политику чтения (primary / secondaryPreferred).
2. Указать допустимую задержку репликации (в секундах) для операций, читающих с secondary.
3. Обосновать выбор политики для каждой операции, опираясь на:
   - Требования к консистентности данных.
   - Частоту обновлений данных.
   - Бизнес-логику (риск продажи недоступного товара, отображение устаревшего статуса заказа пользователю).
4. Представить результат в виде таблицы с колонками: Коллекция, Операция чтения, Политика чтения, Допустимая задержка репликации (сек), Обоснование.

## Решение

| Коллекция | Операция чтения | Политика чтения | Допустимая задержка репликации (сек) | Обоснование |
|---|---|---|---|---|
| **products** | Получение списка товаров (каталог, поиск) | secondaryPreferred | 30 | Высокая нагрузка на категорию "Электроника" (ADR_8). Небольшая устаревшая информация о товаре (например, цена) допустима для большинства пользователей. Чтение с secondary снижает нагрузку на primary. |
| **products** | Получение детальной информации о товаре | secondaryPreferred | 10 | Пользователь ожидает актуальную информацию. Задержка до 10 секунд приемлема, так как частые обновления цены/наличия маловероятны. |
| **products** | Проверка наличия товара перед добавлением в корзину | primary | 0 | Критически важно избежать продажи недоступного товара. Должна использоваться самая актуальная информация. |
| **orders** | Получение списка заказов пользователя | secondaryPreferred | 60 | История заказов редко обновляется. Пользователь может не заметить изменение статуса нового заказа в течение минуты. |
| **orders** | Получение детальной информации о конкретном заказе | primary | 0 | Пользователь ожидает видеть самый последний статус заказа (например, "доставлен"). Устаревшая информация может вызвать недовольство. |
| **carts** | Получение содержимого корзины | primary | 0 | Корзина активно изменяется пользователем (добавление/удаление товаров). Отображение устаревшего состояния приведет к плохому UX и возможной потере данных. |

## Резюме

Ключевые принципы, использованные при принятии решений:

1. **Критичность данных**: Для операций, где актуальность данных критична (наличие товара, детали заказа, корзина), выбрана политика `primary`, гарантирующая чтение самой свежей информации.
2. **Опыт пользователя (UX)**: В операциях, где задержка не влияет на UX (например, просмотр каталога), используется `secondaryPreferred` для распределения нагрузки.
3. **Нагрузка на систему**: Для высоконагруженных операций (например, получение списка товаров) предпочтение отдается чтению с реплик, чтобы снизить нагрузку на primary-ноду.
4. **Частота обновлений**: Для данных, которые редко меняются (история заказов), допустима большая задержка репликации.

Таким образом, баланс между производительностью и консистентностью достигнут за счет дифференцированного подхода к настройке политик чтения в зависимости от бизнес-требований каждой операции.
