### **Название задачи: Выявление и устранение «горячих» шардов в MongoDB**  
### **Автор: Торопов Андрей**
### **Дата: 04.04.2026**

## Контекст

В интернет-магазине «Мобильный мир» после роста ассортимента и нагрузки возникла проблема: около 70% запросов приходится на товары категории «Электроника».

В результате один из шардов MongoDB оказался перегружен (hot shard), что приводит к:
- увеличению времени ответа;
- росту нагрузки на CPU и память;
- ухудшению общей производительности системы.

Причина — неравномерное распределение нагрузки: популярные категории создают непропорциональную нагрузку на отдельные шарды.

---

## Цель

Разработать стратегию:
- выявления «горячих» шардов;
- автоматического перераспределения данных;
- предотвращения подобных ситуаций в будущем.

---

## Метрики мониторинга

### 1. Нагрузка на шарды
- CPU utilization;
- использование памяти;
- количество соединений;
- read/write QPS;
- latency (p95/p99).

### 2. Распределение данных
- количество чанков на шард;
- объём данных на шард;
- доля данных коллекции.

### 3. Распределение запросов
- доля запросов по шардам;
- доля запросов к категориям;
- доля single-shard vs multi-shard запросов;
- частота доступа к диапазонам shard key.

---

## Пороговые значения

### Warning
- один шард > 40% нагрузки;
- latency > 2x от среднего;
- перекос данных > 25%.

### Critical
- один шард > 60% нагрузки;
- CPU > 85%;
- балансировка не помогает.

---

## Механизмы устранения

### 1. Балансировка
```js
sh.startBalancer()
```

### 2. Ручное перемещение
```js
sh.moveChunk("mobile_world.products", { category: "electronics" }, "shard02")
```

### 3. Улучшение shard key
```js
{ category: 1, _id: "hashed" }
```

### 4. Перешардирование
```js
sh.reshardCollection("mobile_world.products", {
  key: { category: 1, _id: "hashed" }
})
```

### 5. Zone sharding
- выделение отдельных шардов под горячие категории

---

## Автоматическая стратегия

1. Мониторинг каждые 5 минут  
2. При превышении порогов:
   - проверка балансировщика  
3. Если проблема сохраняется:
   - анализ запросов  
   - выявление hot range  
4. Применение:
   - балансировка  
   - moveChunk  
   - refine / reshard  
5. Для постоянных hot-данных:
   - использование зон  

---

## Риски

- нагрузка при миграции данных;
- сложность resharding;
- неверный shard key;
- усложнение архитектуры из-за зон.

---

## Итог

Используется комбинация:
- мониторинга;
- балансировки;
- адаптации shard key;
- перераспределения данных.

---

## Вывод

Hot shard — это проблема распределения нагрузки.  
Решение — наблюдаемость + адаптивное шардирование.
