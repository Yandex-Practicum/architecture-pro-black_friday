## Контекст

Интернет-магазин «Мобильный мир» существенно вырос: помимо аксессуаров для смартфонов, платформа продаёт электронику, аудиотовары, бытовую технику и другие категории. Нагрузка на MongoDB увеличилась, поэтому требуется спроектировать схемы коллекций и выбрать стратегию шардирования для основных сущностей: `orders`, `products`, `carts`.

В системе есть три ключевые коллекции:

### 1. `orders`
Хранит оформленные заказы:
- уникальный идентификатор заказа;
- идентификатор клиента;
- дату и время оформления;
- список товаров и цены на момент покупки;
- статус заказа;
- общую сумму;
- геозону заказа.

Основные операции:
- быстрое создание заказов с одновременным списанием остатков;
- поиск истории заказов конкретного пользователя;
- отображение статуса заказа.

### 2. `products`
Хранит сведения о товарах:
- уникальный идентификатор товара;
- наименование;
- категорию;
- цену;
- остатки по геозонам;
- дополнительные атрибуты.

Основные операции:
- частые обновления остатков при покупках;
- поиск товаров по категориям;
- фильтрация по диапазону цен;
- получение карточки товара.

### 3. `carts`
Хранит активные и неактивные корзины пользователей и гостей:
- `_id`;
- `user_id` или `session_id`;
- массив `items`;
- статус корзины;
- даты создания и обновления;
- `expires_at` для TTL-очистки.

Основные операции:
- создание корзины;
- получение активной корзины по `user_id` или `session_id`;
- добавление, замена и удаление товаров;
- слияние гостевой корзины в пользовательскую;
- отметка корзины как заказанной.

---

## Требования к шардированию

При выборе shard key учитываются следующие критерии:

- равномерное распределение данных между шардами;
- минимизация hot spot при вставках и обновлениях;
- максимальное число target-запросов вместо scatter-gather;
- соответствие основным шаблонам доступа к данным;
- стабильность shard key во времени.

## Решение

# 1. Коллекция `orders`

## Предлагаемая схема

```json
{
  "_id": "ObjectId(...)",
  "customer_id": "user_12345",
  "created_at": "ISODate(\"2026-04-04T10:15:00Z\")",
  "status": "created",
  "geo_zone": "msk",
  "items": [
    {
      "product_id": "prd_1001",
      "name": "Смартфон X",
      "price": 49990,
      "quantity": 1
    },
    {
      "product_id": "prd_2050",
      "name": "Наушники Y",
      "price": 7990,
      "quantity": 2
    }
  ],
  "total_amount": 65970
}
```

## Кандидаты в shard key
- `customer_id`
- `geo_zone`
- `created_at`

## Выбранный shard key
```js
{ customer_id: "hashed" }
```

## Обоснование выбора

Для `orders` основной сценарий чтения — история заказов конкретного пользователя. Шардирование по `customer_id` позволяет маршрутизировать такие запросы в один шард.

Использование **hashed shard key** даёт:
- хорошее распределение заказов между шардами;
- отсутствие перегрева одного шарда при массовом создании заказов;
- предсказуемое масштабирование при росте числа клиентов.

Шардирование по `created_at` не подходит, потому что это монотонно растущее поле: новые заказы концентрировались бы в одном диапазоне и создавали hot shard.

Шардирование по `geo_zone` возможно только как дополнительная оптимизация для zone sharding, но не как основной shard key, так как история заказов пользователя важнее географической локальности.

## Индексы

```js
db.orders.createIndex({ customer_id: 1, created_at: -1 })
db.orders.createIndex({ status: 1, created_at: -1 })
db.orders.createIndex({ _id: 1 })
```

## Примеры команд MongoDB

```js
use mobile_world

db.orders.createIndex({ customer_id: "hashed" })

sh.shardCollection("mobile_world.orders", { customer_id: "hashed" })
```

## Плюсы
- быстрый поиск истории заказов пользователя;
- равномерное распределение записей;
- хорошая масштабируемость при росте числа клиентов.

## Минусы и риски
- запросы без `customer_id` могут превращаться в scatter-gather;
- получение заказа только по нестандартным полям без shard key потребует дополнительных индексов.

# 2. Коллекция `products`

## Предлагаемая схема

```json
{
  "_id": "prd_1001",
  "name": "Смартфон X",
  "category": "electronics",
  "price": 49990,
  "stock_by_geo": {
    "msk": 50,
    "ekb": 20,
    "kgd": 30
  },
  "attributes": {
    "color": "black",
    "memory": "256GB"
  }
}
```

## Кандидаты в shard key
- `_id`
- `category`
- `price`
- составной ключ `{ category, _id }`

## Выбранный shard key
```js
{ category: 1, _id: "hashed" }
```

## Обоснование выбора

Для `products` есть два разных шаблона доступа:
1. карточка товара по `product_id`;
2. каталог с фильтрацией по категории и диапазону цен.

Использовать `price` как shard key нецелесообразно:
- цена может изменяться;
- диапазоны цен распределяются неравномерно;
- при изменении shard key возможны дополнительные миграции чанков.

Использование составного ключа `{ category: 1, _id: "hashed" }` позволяет:
- логически группировать товары по категории;
- сохранять хорошее распределение внутри категории;
- избегать перекоса данных;
- поддерживать масштабирование каталога.

Фильтрация по цене реализуется отдельным индексом, а не через shard key.

## Индексы

```js
db.products.createIndex({ category: 1, price: 1 })
db.products.createIndex({ name: 1 })
db.products.createIndex({ "attributes.color": 1 })
db.products.createIndex({ _id: 1 })
```

## Примеры команд MongoDB

```js
use mobile_world

db.products.createIndex({ category: 1, _id: "hashed" })

sh.shardCollection("mobile_world.products", { category: 1, _id: "hashed" })
```

## Плюсы
- хорошее распределение данных внутри товарных категорий;
- эффективная работа каталога;
- снижение риска hot shard.

## Минусы и риски
- запросы только по диапазону цен без категории будут менее эффективны;
- при сильном перекосе по одной сверхпопулярной категории возможно неравномерное распределение нагрузки.

# 3. Коллекция `carts`

## Предлагаемая схема

```json
{
  "_id": "ObjectId(...)",
  "owner_key": "user:user_12345",
  "user_id": "user_12345",
  "session_id": null,
  "status": "active",
  "items": [
    {
      "product_id": "prd_1001",
      "quantity": 2,
      "price": 49990
    }
  ],
  "created_at": "ISODate(\"2026-04-04T09:00:00Z\")",
  "updated_at": "ISODate(\"2026-04-04T10:00:00Z\")",
  "expires_at": "ISODate(\"2026-04-20T00:00:00Z\")"
}
```

Для гостевой корзины:

```json
{
  "_id": "ObjectId(...)",
  "owner_key": "session:sess_98765",
  "user_id": null,
  "session_id": "sess_98765",
  "status": "active",
  "items": [
    {
      "product_id": "prd_2050",
      "quantity": 1,
      "price": 7990
    }
  ],
  "created_at": "ISODate(\"2026-04-04T09:10:00Z\")",
  "updated_at": "ISODate(\"2026-04-04T09:15:00Z\")",
  "expires_at": "ISODate(\"2026-04-10T00:00:00Z\")"
}
```

## Кандидаты в shard key
- `user_id`
- `session_id`
- `_id`
- унифицированный ключ владельца корзины

## Выбранный shard key
```js
{ owner_key: "hashed" }
```

## Обоснование выбора

У коллекции `carts` два способа доступа:
- по `user_id` для авторизованного пользователя;
- по `session_id` для гостя.

Если выбрать только `user_id`, гостевые корзины будут обрабатываться менее эффективно. Если выбрать только `session_id`, пострадают пользовательские корзины.

Поэтому вводится вычисляемое поле:
- `owner_key = "user:<user_id>"` для пользователя;
- `owner_key = "session:<session_id>"` для гостя.

Это позволяет:
- унифицировать доступ к активной корзине;
- использовать один shard key для обоих сценариев;
- упростить получение и обновление корзины;
- улучшить распределение данных между шардами.

## Индексы

```js
db.carts.createIndex({ owner_key: 1, status: 1 })
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
db.carts.createIndex(
  { user_id: 1, status: 1 },
  { partialFilterExpression: { user_id: { $exists: true, $ne: null } } }
)
db.carts.createIndex(
  { session_id: 1, status: 1 },
  { partialFilterExpression: { session_id: { $exists: true, $ne: null } } }
)
```

## Примеры команд MongoDB

```js
use mobile_world

db.carts.createIndex({ owner_key: "hashed" })
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })

sh.shardCollection("mobile_world.carts", { owner_key: "hashed" })
```

## Примеры операций

### Получение активной корзины пользователя
```js
db.carts.findOne({
  owner_key: "user:user_12345",
  status: "active"
})
```

### Получение активной гостевой корзины
```js
db.carts.findOne({
  owner_key: "session:sess_98765",
  status: "active"
})
```

### Пометка корзины как заказанной
```js
db.carts.updateOne(
  {
    owner_key: "user:user_12345",
    status: "active"
  },
  {
    $set: {
      status: "ordered",
      updated_at: new Date()
    }
  }
)
```

### Слияние гостевой корзины в пользовательскую
1. Прочитать гостевую корзину по `owner_key = "session:<session_id>"`.
2. Найти или создать пользовательскую корзину по `owner_key = "user:<user_id>"`.
3. Перенести `items`.
4. Пометить гостевую корзину как `abandoned`.

## Плюсы
- единый механизм доступа и для пользователей, и для гостей;
- равномерное распределение корзин;
- удобная поддержка сценария merge после логина;
- автоматическая очистка старых корзин через TTL.

## Минусы и риски
- требуется дополнительное вычисляемое поле `owner_key`;
- merge корзин остаётся бизнес-операцией на уровне приложения, а не MongoDB.

---

## Альтернативы

### `orders`
- `{ geo_zone: 1 }` — подходит хуже, потому что история заказов пользователя является более важным сценарием.
- `{ created_at: 1 }` — создаёт hot shard из-за монотонного роста.

### `products`
- `{ _id: "hashed" }` — даёт отличное распределение и быстрый доступ к карточке товара, но хуже учитывает каталог по категориям.
- `{ price: 1 }` — не рекомендуется из-за изменяемости цены и возможной неравномерности диапазонов.

### `carts`
- `{ user_id: "hashed" }` — подходит только для авторизованных пользователей.
- `{ session_id: "hashed" }` — подходит только для гостей.
- `{ _id: "hashed" }` — хорошо распределяет данные, но не помогает маршрутизации основных бизнес-запросов.

---

## Итоговое решение

Для системы интернет-магазина выбраны следующие стратегии шардирования:

- **`orders`** → `{ customer_id: "hashed" }`
- **`products`** → `{ category: 1, _id: "hashed" }`
- **`carts`** → `{ owner_key: "hashed" }`

Эти ключи лучше всего соответствуют шаблонам доступа:
- заказы читаются по пользователю;
- товары фильтруются по категории и читаются по идентификатору;
- корзины должны одинаково эффективно работать и для пользователей, и для гостей.

Такой подход обеспечивает:
- горизонтальное масштабирование;
- более равномерное распределение нагрузки;
- снижение числа scatter-gather запросов по ключевым сценариям;
- поддержку роста ассортимента и числа пользователей.

## Вывод

Выбранная стратегия шардирования учитывает реальные бизнес-операции интернет-магазина и ориентирована не только на хранение данных, но и на шаблоны их чтения и обновления. Основной принцип решения — шардировать коллекции по тем полям, которые чаще всего участвуют в целевых запросах и при этом обеспечивают хорошую кардинальность и баланс нагрузки.
