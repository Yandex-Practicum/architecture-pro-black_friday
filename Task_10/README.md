# Миграция на Cassandra

## Критически важные данные интернет-магазина
Из частей данных интернет-магазина наиболее критически важными с точки зрения целостности и скорости обработки являются:

- **Товары**: Эти данные относительно статичны и обновляются редко. Важно иметь быстрое чтение и минимальную задержку при доступе клиентов к карточкам товаров. Здесь применима база данных типа key-value или документная база данных, поскольку необходимость в частых обновлениях отсутствует.

- **Заказы**: Данные о заказах содержат информацию о составе заказа, доставке и статусах заказов. Эти данные активно записываются и читаются системой. Нужно учитывать частые изменения статуса заказа и синхронизацию между несколькими источниками информации. Эти данные критически важны для бизнеса, так как от них зависит выполнение заказов и удовлетворенность клиентов.

- **Складские остатки товаров**: Информация об остатках товаров является важной для интернет-магазина. Запасы регулярно обновляются, и любые сбои могут привести к негативному опыту покупок. Операциям нужен высокий уровень надежности и устойчивости к сбоям, так как от этого зависит корректное отображение и доступность товаров для покупателей.

- **Платежные транзакции**: Информация о платежах является наиболее чувствительными данными с высокими требованиями к целостности и согласованности. Каждая операция должна завершаться атомарно («все или ничего»), без риска потери денег покупателя или магазина. Требуется поддержка моделей ACID (атомарность, согласованность, изолированность, долговечность)

- **Пользовательские сессии**: Данные о пользовательских сессиях могут содержать информацию о предпочтениях и активности пользователей на платформе. Эти данные важны для персонализации предложений и маркетинговых кампаний. Важна быстрая запись и стабильная работа даже при высоком трафике.

## Переход на Cassandra

Применение Cassandra имеет смысл для следующих данных интернет-магазина:

- Пользовательские сессии
- Платежные транзакции
- Складские остатки товаров
- Стутусы заказов и доставка

Общими чертами данных, для которых применение Cassandra имеет смысл, являются высокая скорость записи и чтения, масштабируемость, геораспределенность (в случае, если интернет-магазин имеет множество точек продажи) и необходимость обеспечения высокой доступности данных. Кроме того, Cassandra обладает механизмами обеспечения целостности данных, что важно для критически важных данных, таких как заказы, остатки товаров и платежные транзакции.

## Концептуальная модель для критически важных сущностей

Ниже представлена концептуальная модель данных, разработанная с учетом требований к производительности, масштабированию и целостности данных для критически важных сущностей интернет-магазина.  Рассмотрим четыре ключевых сущности: Пользовательские сессии, Платежные транзакции, Складские остатки товаров и Статусы заказов и доставка.

### 1. Пользовательские сессии (`sessions`)

Эта сущность содержит информацию о текущих сеансах пользователей на сайте, включая их идентификационные данные и временные метки последних действий.

#### Таблица sessions

|Атрибут| Тип данных|
|:--|:--|
|session_id| UUID|
|user_id| Integer|
|last_activity_at| Timestamp|
|device_type| String|
|ip_address| String|

#### Partition Key и Кластерный ключ

**Partition Key**: session_id (UUID)
**Clustering Keys**: last_activity_at, device_type

Выбор ключа:

- Уникальные значения ключей обеспечивают равномерное распределение данных.
Частый доступ осуществляется по session_id.
- Дополнительные поля сортировки позволяют фильтровать сессии по последнему действию и типу устройства.


### 2. Платежные транзакции (`transactions`)

Сущность содержит информацию обо всех операциях оплаты, совершенных пользователями, включая суммы, успешность транзакций и идентификаторы заказов.

#### Таблица transactions

|Атрибут| Тип данных|
|:--|:--|
|transaction_id| UUID|
|order_id| Integer|
|amount| Decimal|
|payment_status| Enum|
|created_at| Timestamp|

#### Partition Key и Кластерный ключ

**Partition Key**: transaction_id (UUID)
**Clustering Keys**: order_id, created_at

Выбор ключа:

- Учет уникальности транзакций гарантирует отсутствие горячих партиций.
- Использование композитного ключа улучшает производительность поисковых запросов по связанным полям.


### 3. Складские остатки товаров (`stock`)

Содержит информацию о наличии товаров на складе, включая доступные количества и местоположение запасов.

#### Таблица stock

|Атрибут| Тип данных|
|:--|:--|
|product_id| Integer|
|city_id| Integer|
|quantity_available| Integer|
|updated_at| Timestamp|

#### Partition Key и Кластерный ключ

**Partition Key**: product_id
**Clustering Keys**: city_id, updated_at

Выбор ключа:

- Основная нагрузка приходится на операции чтения по продуктам.
- Разделение данных по продуктам и городам предотвращает возникновение перегруженности одной конкретной секции данных.


### 4. Статусы заказов и доставка (`orders`)

Данная таблица включает информацию о каждом заказе, включая его сумму, доставку и текущий статус.

#### Таблица orders

|Атрибут| Тип данных|
|:--|:--|
|order_id| Integer|
|customer_id| Integer|
|total_amount| Decimal|
|delivery_status| Enum|
|shipped_at| Timestamp|

#### Partition Key и Кластерный ключ

**Partition Key**: order_id
**Clustering Keys**: customer_id, shipped_at

Выбор ключа:

- Ключевое значение должно поддерживать уникальные номера заказов.
- Кластеризация по покупателю и датам отправки помогает повысить производительность при анализе покупательской активности.


## Выбор модели с точки зрения «горячих» партиций и минимизации влияния шардинга

При выборе структуры таблиц были учтены следующие факторы:

- Предотвращение горячей партиции: Использование уникальных значений (например, UUID) в качестве первичного ключа способствует равномерному распределению данных и предотвращает создание крупных секций данных ("hot partitions").
- Минимизация влияния шардинга: Composite keys (композитные ключи) помогают снизить влияние изменений топологии кластера путем оптимизации доступа к данным. Композитные ключи позволяют производить быструю фильтрацию и упорядочивание записей.
- Оптимизация запросов: Введение clustered columns (кластерных столбцов) значительно ускоряет процесс выполнения запросов и уменьшает нагрузку на сервер.

Таким образом, предложенная структура данных обеспечит эффективное управление критическими ресурсами интернет-магазина в условиях экстремальной нагрузки, сохраняя целостность данных и предотвращая деградацию производительности.

### Примеры создания таблиц в Cassandra

```sql
CREATE KEYSPACE IF NOT EXISTS ecommerce WITH REPLICATION = {'class': 'SimpleStrategy', 'replication_factor' : 3};

USE ecommerce;

-- Создание таблицы для пользовательских сессий (sessions)
CREATE TABLE sessions (
    session_id UUID PRIMARY KEY,
    user_id INT,
    last_activity_at TIMESTAMP,
    device_type TEXT,
    ip_address TEXT
) WITH CLUSTERING ORDER BY (last_activity_at DESC);

-- Создание таблицы для платежных транзакций (transactions)
CREATE TABLE transactions (
    transaction_id UUID PRIMARY KEY,
    order_id INT,
    amount DECIMAL,
    payment_status TEXT,
    created_at TIMESTAMP
) WITH CLUSTERING ORDER BY (order_id ASC, created_at ASC);

-- Создание таблицы для складских остатков товаров (stock)
CREATE TABLE stock (
    product_id INT,
    city_id INT,
    quantity_available INT,
    updated_at TIMESTAMP,
    PRIMARY KEY ((product_id), city_id)
) WITH CLUSTERING ORDER BY (city_id ASC, updated_at DESC);

-- Создание таблицы для статусов заказов и доставки (orders)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    total_amount DECIMAL,
    delivery_status TEXT,
    shipped_at TIMESTAMP
) WITH CLUSTERING ORDER BY (customer_id ASC, shipped_at ASC);
```

## Стратегии обеспечения целостности данных в Cassandra

ля различных сущностей интернет-магазина рекомендуется применять соответствующие стратегии, учитывая компромисс между временем реакции (latency) и гарантией согласованности.

### 1. Пользовательские сессии (`sessions`)

   - Оптимальна стратегия: **Hinted Handoff**.
   - Причина выбора: Временные промежутки между изменениями небольшие, кратковременные падения доступности не оказывают существенного влияния на качество данных. Время реагирования важнее уровня гарантированной согласованности.

### 2. Платёжные транзакции (`transactions`)

   - Оптимальны стратегии: **Read Repair** и **Anti-Entropy Repair**.
   - Причина выбора: Необходимо минимизировать риск несоответствия данных и обеспечить максимальную точность транзакций. Готовность принять повышенную задержку при чтении ради точности данных.

### 3. Складские остатки товаров (`stock`)

   - Оптимальна стратегия: **Anti-Entropy Repair**.
   - Причина выбора: Регулярное сравнение и ремонт данных необходимы для предотвращения ситуаций дефицита товаров вследствие неполадок репликации. Задержка менее критична по сравнению с точностью наличия товаров.

### 4. Заказы и доставка (`orders`)

   - Оптимальна стратегия: **Read Repair**.
   - Причина выбора: Важна постоянная доступность актуальных данных о состоянии заказов. Приоритезация своевременного обнаружения и коррекции устаревших сведений.
