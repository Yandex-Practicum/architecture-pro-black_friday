# Задание 8

## 1. Горячие шарды: метрики и устранение дисбаланса (MongoDB)

Коллекция `products` шардирована; нагрузка неравномерна: категория "Электроника" даёт порядка 70% обращений. Перегружается тот шард, где в терминах шард-ключа и чанков сосредоточена "электроника" (префикс range-ключа, узкий диапазон, jumbo-чанк, зона с одной тяжёлой нодой). Нужны метрики, чтобы видеть перекос, и механизмы, которые перекладывают данные или снимают давление; популярные категории всегда дают непропорциональный трафик относительно равномерного `hashed` по id.

## 2. Метрики

### 2.1. Кластер и чанки (mongos, config при необходимости)

- `sh.status()` / `db.printShardingStatus()` — шарды, `chunks` по `ns`, состояние `balancer`.
- `db.<coll>.getShardDistribution()` — доля документов/оценка байт по шарду; сильный перекос = кандидат в дисбаланс данных.
- Jumbo в `sh.status` / `config.chunks` — jumbo балансером не двигается, пока не `split` — вечный перекос.
- `config.chunks` — `min`/`max`/`shard` для range-ключа: схлопнулся ли диапазон на одном шарде.
- `$collStats` / `dbStats` на шарде — объём, при росте в одной реплике без роста в других = перекос по носителю.

### 2.2. План и долгие операции

- `db.currentOp()` с фильтрами (аккуратно, нагрузка) — долгие `find`/`aggregate` к одному шарду.
- `setProfilingLevel` + `system.profile` — медленные запросы, фильтр по категории, план, индекс.
- `serverStatus` на `mongod` — `opcounters`, `wiredTiger.cache` / давление кэша, `locks` (по версии).

`mongostat` / `mongotop` на время сбора, экспорт в Prometheus-совместимые дашборды — по желанию.

### 2.3. Балансер

- `sh.getBalancerState()` / `sh.setBalancerState` — балансер обязан быть в рабочем состоянии, иначе чанки не мигрируют.
- Окно балансировки, политика — в `config` на config replica set

Логи `mongod` config servers: события миграции, `aborted`/`failed` — сигналы, что дисбаланс не лечится (jumbo, сеть, `maxSize`).

### 2.4. Репликация шарда

- `rs.printSecondaryReplicationInfo()` / метрика lag `secondary` к `primary` — на горячем шарде растут и запись, и `lag` у вторичек, если read не разгружен.

### 2.5. Инфраструктура

- На хосте с `mongod`: `CPU`, `iowait`, `disk` usage/IOPS, `network`, `RSS`

### 2.6. Приложение

- RPS и latency с разметкой по `category` (лог, метка в запросе) — подтвердить долю "Электроника".
- Hit/miss кэша (Redis) по витрине категории — низкий hit ускоряет деградацию Mongo.

## 3. Два типа проблемы

1. Данные разложены неравномерно — много крупных чанков/диапазонов на одном `shard` → смотрим балансер, jumbo, зоны, при необходимости смена ключа / `reshardCollection`.
2. Паттерн запроса косит в одну зону/категорию, при равной раскладке диска — пики CPU на одной реплике → `reshard` на более нейтральный ключ, кэш, чтение со вторичек с допустимой `readConcern`, `rate limit`, толще нода для зоны "Э".

## 4. Меры

### 4.1. Балансер и jumbo

Включить балансер, снять jumbo через `split`, снова дать балансеру сходиться.

```js
// mongos
sh.getBalancerState();
// sh.setBalancerState(true);
// окно балансера — config.settings, см. док. версии
```

### 4.2. Zone sharding

Зоны привязывают диапазоны `shard key` к именованным `shard` — можно выделить "Электронику" на физически сильные ноды. После `addShardToZone` и `updateZoneKeyRange` балансер переносит сегменты (синтаксис `min`/`max` — от фактического `shard key` и версии). Команда `moveRange` (замена устаревшего `moveChunk`) — точечный перенос диапазона.

```js
sh.addShardToZone("Electronics-RS", "ZONE_E");
// sh.updateZoneKeyRange("shop.products", <minKey>, <maxKey>, "ZONE_E");
// точные границы — под ваш range/hashed+prefix-ключ, см. док.
```

### 4.3. `reshardCollection` 

https://www.mongodb.com/docs/manual/core/sharding-reshard-a-collection/

Смена `shard key` без остановки сервиса по мощности и регламенту. Типичный путь с категорийным смещением: `hashed` по `_id` / по отдельному `sku_id` с высокой кардинальностью, списки "по категории" — вторичные индексы и мирное scatter-чтение.

```js
db.adminCommand({ reshardCollection: "shop.products", key: { _id: "hashed" } });
```

### 4.4. Jumbo: явный чек-лист

- Найти: `sh.status`, `config.chunks` с `jumbo: true`.
- Разбить, снова включить/дождаться балансера.

### 4.5. Снижение нагрузки не через "переезд" чанков

- Кэш листинга/карточек с TTL, инвалидация при смене остатка.
- Горизонталь `mongos` / приложения; `read` на `secondary` для выдач, где допустимо отставание.
- Масштаб по вертикали `mongod` в той зоне, куда `Balancer` вынес "Электронику" по `Zone`.

## 5. Что автоперераспределяется в MongoDB

- `Balancer` (по умолчанию) — перенос чанков/диапазонов по правилу, пока `jumbo` не мешает и `balancer` не выключен.
- `Zone` + тот же балансер — смещение диапазонов после смены карты зон, не в реальном времени как "автоскейлер приложения".
- `reshardCollection` — оркестрируемая однократная операция (окно, мониторинг, план), не cron.
- `addShard` + зоны + ожидание `Balancer` — полуручной шаг "добавили мощности и переназначили зоны".
- Кэш/реплики — не перекладывают `chunks`, снижают RPS/нагрев; то же пакет мер вместе с `Balancer`/`reshard`.

Полноценного "HPA по шардам" из коробки нет; полезны алерты + runbook: при перекосе N% чанков / роста CPU пира → `split`/`включить balancer`/`тикет на reshard+zone`/`addShard`.

## 6. Почему "70% + Электроника"

- В range-ключе, где `category` первой компонентой, "Электроника" = один или несколько тяжёлых чанков → либо `Zone` на мощные ноды, либо `reshard` на `hashed` + индексы витрины.
- В `hashed` по `_id` с равной коллекцией: по диску дисперсия обычно нормальна; 70% запросов при том же `shard key` — горячий трафик (категория, топ-товары) — кэш, вторички, `rate limit`, `zone`+железо, при необходимости отдельный read-пул.

## 7. Метрики в дашборде / алерты

Дашборд: по каждой реплике шарда — `opcounters`, `CPU`/`iowait`, `disk`, `repl` lag, по кластеру — доля `chunks`/`est.data` из `getShardDistribution`, `jumbo count=0` цель, `balancer enabled`, логи миграций.

Сигналы: один шард > ~40–50% `chunks` или данных по `products` при равных ожиданиях; `CPU` > ~80% long vs медиана пира; `balancer=off` неожиданно; jumbo>0; `replication` lag p95>порог.

## 8. Чек-лист команд для расследовании «горячего» шарда

```js
// mongos
sh.status(); //общая картина: шарды, чанки, включён ли балансер, миграции.
db.products.getShardDistribution(); // сколько примерно документов/данных на каком шарде
sh.getBalancerState(); //включён ли балансер
```

замедленные запросы (профайлинг)
```js
// замедлить, потом off
// db.setProfilingLevel(1, { slowms: 80 });
// db.setProfilingLevel(0);
```
