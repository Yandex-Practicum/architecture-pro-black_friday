# Архитектурный документ: Проектирование и оптимизация базы данных интернет-магазина

## 1. Проектирование схем коллекций для шардирования данных

### 1.1 Коллекция `products`

#### Схема коллекции

```javascript
{
  "_id": ObjectId,                // Уникальный идентификатор товара
  "name": String,                  // Наименование товара
  "category": String,              // Категория товара (индексировано)
  "price": Number,                 // Цена товара (индексировано)
  "attributes": {                   // Дополнительные атрибуты
    "color": String,                // Цвет (опционально)
    "size": String,                 // Размер (опционально)
    // другие характеристики
  },
  "stock": {                        // Остатки по геозонам
    "msk": Number,                   // Москва
    "spb": Number,                   // Санкт-Петербург
    "ekb": Number,                   // Екатеринбург
    "kld": Number,                   // Калининград
    // другие геозоны
  },
  "created_at": ISODate,            // Дата создания записи
  "updated_at": ISODate             // Дата последнего обновления
}
```

#### Индексы

```javascript
// Для поиска по категориям и фильтрации по цене
db.products.createIndex({ category: 1, price: 1 })
```

#### Стратегия шардирования

| Параметр | Значение |
|----------|----------|
| Шард-ключ | `{ category: 1, _id: 1 }` |
| Стратегия | Ranged sharding с составным ключом |

#### Обоснование выбора

**Почему `category` в составе шард-ключа:**
- Основные операции включают поиск товаров по категориям
- Данные естественно группируются по категориям
- Запросы по категории направляются только в один шард
- Категорий достаточно для равномерного распределения

**Почему составной ключ с `_id`:**
- Обеспечивает уникальность внутри категории
- Позволяет эффективно распределять новые товары
- Предотвращает создание "горячих точек" при вставке

**Операции, которые выигрывают:**

| Операция | Как работает с шард-ключом |
|----------|---------------------------|
| Поиск по категориям | Targetted query (один шард) |
| Поиск товара по ID | Broadcast query (если категория неизвестна) |
| Обновление остатков | Targetted query (если известна категория) |

### 1.2 Коллекция `orders`

#### Схема коллекции

```javascript
{
  "_id": ObjectId,                // Уникальный идентификатор заказа
  "user_id": Number,               // Идентификатор клиента (индексировано)
  "order_date": ISODate,           // Дата и время оформления заказа
  "geo_zone": String,              // Геозона заказа (например, "msk", "spb")
  "status": String,                 // Статус заказа: "new", "paid", "shipped", "delivered", "cancelled"
  "items": [                        // Список заказанных товаров
    {
      "product_id": ObjectId,
      "name": String,               // Наименование товара на момент заказа
      "price": Number,               // Цена на момент заказа
      "quantity": Number,
      "category": String             // Категория товара
    }
  ],
  "total_amount": Number,           // Общая сумма заказа
  "created_at": ISODate,            // Дата создания записи
  "updated_at": ISODate             // Дата последнего обновления
}
```

#### Индексы

```javascript
// Для поиска истории заказов пользователя
db.orders.createIndex({ user_id: 1, order_date: -1 })

// Для поиска по статусу (фоновые процессы)
db.orders.createIndex({ status: 1, order_date: 1 })
```

#### Стратегия шардирования

| Параметр | Значение |
|----------|----------|
| Шард-ключ | `{ user_id: 1, _id: 1 }` |
| Стратегия | Hashed sharding (на `user_id`) |

#### Обоснование выбора

**Почему `user_id` в качестве шард-ключа:**
- Основная операция: поиск истории заказов конкретного пользователя
- Все заказы одного пользователя попадают в один шард
- Можно выполнять сортировку по дате внутри шарда
- Равномерное распределение при хэшировании

**Почему составной ключ с `_id`:**
- Гарантирует уникальность
- Позволяет эффективно распределять при хэшировании

**Операции, которые выигрывают:**

| Операция | Как работает с шард-ключом |
|----------|---------------------------|
| История заказов пользователя | Targetted query (один шард) |
| Создание заказа | Targetted query (по user_id) |
| Обновление статуса | Targetted query (по _id + user_id) |

### 1.3 Коллекция `carts`

#### Схема коллекции

```javascript
{
  "_id": ObjectId,                // Уникальный идентификатор корзины
  "user_id": Number,               // Идентификатор пользователя (для авторизованных)
  "session_id": String,            // Идентификатор сессии (для гостей)
  "status": String,                 // Статус: "active", "ordered", "abandoned"
  "items": [                        // Список товаров в корзине
    {
      "product_id": ObjectId,
      "quantity": Number,
      "added_at": ISODate           // Время добавления
    }
  ],
  "created_at": ISODate,            // Дата создания корзины
  "updated_at": ISODate,            // Дата последнего обновления
  "expires_at": ISODate              // Время автоматического удаления (TTL)
}
```

#### Индексы

```javascript
// Для поиска активной корзины пользователя
db.carts.createIndex(
  { user_id: 1, status: 1 },
  { partialFilterExpression: { status: "active" } }
)

// Для поиска активной корзины гостя
db.carts.createIndex(
  { session_id: 1, status: 1 },
  { partialFilterExpression: { status: "active" } }
)

// TTL индекс для автоматической очистки
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
```

#### Стратегия шардирования

| Параметр | Значение |
|----------|----------|
| Шард-ключ | `{ user_id: 1, session_id: 1, _id: 1 }` |
| Стратегия | Hashed sharding с составным ключом |

#### Обоснование выбора

**Проблема:** Корзины могут принадлежать либо пользователям (`user_id`), либо гостям (`session_id`).

**Решение:** Составной шард-ключ `{ user_id: 1, session_id: 1, _id: 1 }`.

**Логика распределения:**
- Для авторизованных пользователей: `user_id` определяет шард
- Для гостей: `session_id` определяет шард
- `_id` гарантирует уникальность

**Операции, которые выигрывают:**

| Операция | Как работает с шард-ключом |
|----------|---------------------------|
| Получение корзины пользователя | Targetted query (по user_id + status) |
| Получение корзины гостя | Targetted query (по session_id + status) |
| Добавление товара | Targetted query |
| Слияние корзин | Targetted query для обеих корзин |

---

### 1.4 Сводная таблица

| Коллекция | Шард-ключ | Стратегия | Обоснование |
|-----------|-----------|-----------|-------------|
| products | `{ category: 1, _id: 1 }` | Ranged | Группировка по категориям для эффективного поиска |
| orders | `{ user_id: 1, _id: 1 }` | Hashed | Все заказы пользователя в одном шарде |
| carts | `{ user_id: 1, session_id: 1, _id: 1 }` | Hashed | Поддержка двух типов идентификаторов |

### 1.5 Примеры команд MongoDB

```javascript
// Включение шардирования для базы данных
sh.enableSharding("mobile_mir")

// Шардирование коллекции products
sh.shardCollection("mobile_mir.products", { "category": 1, "_id": 1 })

// Шардирование коллекции orders
sh.shardCollection("mobile_mir.orders", { "user_id": "hashed", "_id": 1 })

// Шардирование коллекции carts
sh.shardCollection("mobile_mir.carts", { "user_id": 1, "session_id": 1, "_id": 1 })
```

## 2. Выявление и устранение «горячих» шардов

### 2.1 Метрики мониторинга

| Метрика | Команда | Описание | Порог срабатывания |
|---------|---------|----------|---------------------|
| Распределение данных | `sh.status()` | Количество чанков на шард | Разброс > 20% |
| Размер данных | `db.stats()` | Объём данных на шарде | Разброс > 30% |
| Операции в секунду | `serverStatus.opcounters` | Нагрузка на шард | Разброс > 40% |
| Время ответа | `serverStatus.opLatencies` | Задержки операций | > 100 мс |
| Популярность категорий | Агрегация | % запросов по категориям | Дисбаланс > 50% |

### 2.2 Команды для сбора метрик

```javascript
// Статистика по шардам
db.adminCommand({ listShards: 1 })

// Распределение чанков
db.getSiblingDB("config").chunks.aggregate([
  { $group: { _id: "$shard", count: { $sum: 1 } } }
])

// Популярность категорий
db.products.aggregate([
  { $group: { _id: "$category", count: { $sum: 1 } } }
])
```

### 2.3 Механизмы устранения дисбаланса

#### Ручное перераспределение чанков

```javascript
// Перемещение чанка с "горячего" шарда
db.adminCommand({
  moveChunk: "mobile_mir.products",
  find: { category: "electronics", _id: ObjectId("...") },
  to: "shard2"
})

// Балансировка конкретной коллекции
sh.startBalancer()
sh.enableBalancing("mobile_mir.products")
```

#### Настройка окна балансировки

```javascript
// Ограничение времени работы балансировщика
use config
db.settings.updateOne(
  { _id: "balancer" },
  { $set: { 
      activeWindow: { start: "02:00", stop: "06:00" },
      mode: "full" 
  } },
  { upsert: true }
)
```

#### Изменение шард-ключа для products

**Проблема:** Все товары категории "electronics" в одном шарде (70% запросов)

**Решение:** Составной ключ с ценой для распределения

```javascript
// Новый шард-ключ
sh.shardCollection("mobile_mir.products", { "category": 1, "price": 1, "_id": 1 })
```

## 3. Настройка чтения с реплик и консистентность

### 3.1 Анализ операций и требований

#### Коллекция `products`

| Операция чтения | Частота | Требования к консистентности | Риск при устаревших данных |
|-----------------|---------|------------------------------|----------------------------|
| Поиск товаров по категориям | Очень высокая | Низкая | Устаревший ассортимент |
| Описание товара | Высокая | Средняя | Устаревшая цена |
| Проверка остатков | Высокая | Высокая | Продажа отсутствующего товара |

#### Коллекция `orders`

| Операция чтения | Частота | Требования к консистентности | Риск при устаревших данных |
|-----------------|---------|------------------------------|----------------------------|
| История заказов | Средняя | Низкая | Не видит последний заказ |
| Статус заказа | Высокая | Высокая | Неверный статус |

#### Коллекция `carts`

| Операция чтения | Частота | Требования к консистентности | Риск при устаревших данных |
|-----------------|---------|------------------------------|----------------------------|
| Получение корзины | Очень высокая | Очень высокая | Неактуальный состав |

### 3.2 Допустимая задержка репликации

| Коллекция | Тип операций | Допустимая задержка | Обоснование |
|-----------|--------------|---------------------|-------------|
| products | Поиск и фильтрация | 5-10 секунд | Некритично для пользователя |
| products | Описание товара | 1-2 секунды | Цена должна быть актуальной |
| products | Проверка остатков | < 500 мс | Предотвращение overselling |
| orders | История заказов | 5-10 секунд | Некритично |
| orders | Статус заказа | < 1 секунды | Пользователь отслеживает статус |
| carts | Все операции | < 100 мс | Строгая консистентность |

### 3.3 Стратегия чтения: Primary vs Secondary

| Коллекция | Операция | Источник чтения | Обоснование |
|-----------|----------|-----------------|-------------|
| products | Поиск по категориям | **SECONDARY** | Высокая нагрузка, низкие требования |
| products | Описание товара | **SECONDARY** | Допустима небольшая задержка |
| products | Проверка остатков | **PRIMARY** | Критично для бизнес-логики |
| orders | История заказов | **SECONDARY** | Некритично |
| orders | Статус заказа | **PRIMARY** | Актуальность критична |
| carts | Все операции | **PRIMARY** | Строгая консистентность |

### 3.4 Итоговые настройки

| Коллекция | Операция | Read Preference | Read Concern | Допустимая задержка |
|-----------|----------|-----------------|--------------|---------------------|
| products | Поиск по категориям | `secondary` | `local` | 5-10 сек |
| products | Описание товара | `secondary` | `local` | 1-2 сек |
| products | Проверка остатков | `primary` | `linearizable` | < 500 мс |
| orders | История заказов | `secondary` | `local` | 5-10 сек |
| orders | Статус заказа | `primary` | `linearizable` | < 1 сек |
| carts | Все операции | `primary` | `linearizable` | < 100 мс |

### 3.5 Примеры команд

```javascript
// Поиск товаров с secondary
db.products.find({ category: "electronics" }).readPref("secondary")

// Проверка остатков с primary
db.products.findOne({ _id: ObjectId("...") }).readPref("primary")

// Статус заказа с primary
db.orders.findOne({ _id: ObjectId("...") }).readPref("primary")

// История заказов с secondary
db.orders.find({ user_id: 12345 }).readPref("secondary").sort({ order_date: -1 })

// Корзина с primary
db.carts.findOne({ user_id: 12345, status: "active" }).readPref("primary")
```

## 4. Миграция на Cassandra

### 4.1 Анализ критических данных

| Сущность | Объём | Частота записи | Частота чтения | Требования к целостности | Критичность |
|----------|-------|-----------------|-----------------|--------------------------|-------------|
| Заказы (orders) | Высокий | 50k/сек | Высокая | Абсолютная | **Критическая** |
| Корзины (carts) | Средний | Очень высокая | Очень высокая | Высокая | **Критическая** |
| Сессии (sessions) | Очень высокий | Очень высокая | Высокая | Низкая | **Критическая** |
| Товары (products) | Средний | Низкая | Очень высокая | Средняя | **Некритическая** |

### 4.2 Модель данных для Cassandra

#### Таблица `orders_by_user`

```sql
CREATE TABLE orders_by_user (
    user_id UUID,
    order_id UUID,
    order_date TIMESTAMP,
    status TEXT,
    geo_zone TEXT,
    total_amount DECIMAL,
    items LIST<FROZEN<order_item>>,
    PRIMARY KEY ((user_id), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC, order_id ASC);
```

**Обоснование:** Partition Key `user_id` — все заказы пользователя в одной партиции. Clustering Key `order_date` + `order_id` — сортировка по дате.

#### Таблица `orders_by_status`

```sql
CREATE TABLE orders_by_status (
    status TEXT,
    order_date TIMESTAMP,
    order_id UUID,
    user_id UUID,
    geo_zone TEXT,
    total_amount DECIMAL,
    PRIMARY KEY ((status), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC, order_id ASC);
```

**Обоснование:** Для мониторинга и фоновых процессов. Partition Key `status` может создавать горячие партиции, требуется bucket-стратегия.

#### Таблица `carts_by_user`

```sql
CREATE TABLE carts_by_user (
    user_id UUID,
    cart_id UUID,
    status TEXT,
    items MAP<UUID, INT>,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    expires_at TIMESTAMP,
    PRIMARY KEY ((user_id), status, updated_at)
) WITH CLUSTERING ORDER BY (status ASC, updated_at DESC);
```

**Обоснование:** Все корзины пользователя в одной партиции, быстрый поиск активной корзины.

#### Таблица `carts_by_session`

```sql
CREATE TABLE carts_by_session (
    session_id TEXT,
    cart_id UUID,
    items MAP<UUID, INT>,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    expires_at TIMESTAMP,
    PRIMARY KEY ((session_id), updated_at)
) WITH CLUSTERING ORDER BY (updated_at DESC);
```

**Обоснование:** Для гостевых корзин, автоматическое удаление через TTL.

#### Таблица `sessions`

```sql
CREATE TABLE sessions (
    session_id TEXT,
    user_id UUID,
    created_at TIMESTAMP,
    expires_at TIMESTAMP,
    data TEXT,
    PRIMARY KEY ((session_id))
) WITH default_time_to_live = 86400; -- 24 часа
```

**Обоснование:** Прямой доступ по ID сессии, автоматическое удаление через TTL.

### 4.3 Стратегии распределения и "горячие" партиции

| Таблица | Partition Key | Риск горячей партиции | Решение |
|---------|---------------|----------------------|---------|
| orders_by_user | `user_id` | Низкий | Хэширование |
| orders_by_status | `status` | **Высокий** (статус "new") | Композитный ключ + Bucketing |
| carts_by_user | `user_id` | Низкий | Хэширование |
| carts_by_session | `session_id` | Низкий | Хэширование |
| sessions | `session_id` | Низкий | Хэширование |

#### Решение для горячей партиции `orders_by_status`

```sql
-- Добавляем bucket для равномерного распределения
CREATE TABLE orders_by_status (
    status_bucket TEXT, -- status + date_bucket
    order_date TIMESTAMP,
    order_id UUID,
    user_id UUID,
    status TEXT,
    geo_zone TEXT,
    total_amount DECIMAL,
    PRIMARY KEY ((status_bucket), order_date, order_id)
);

-- Вставка с bucket
INSERT INTO orders_by_status (
    status_bucket, order_date, order_id, user_id, status, geo_zone, total_amount
) VALUES (

    '2026-03-11 10:30:00',
    uuid(),
    123e4567-e89b-12d3-a456-426614174000,
    'new',
    'msk',
    29990.00
);
```

### 4.4 Стратегии обеспечения целостности данных

| Стратегия | Механизм | Влияние на latency | Гарантия целостности |
|-----------|----------|-------------------|---------------------|
| Hinted Handoff | Запись на другой узел при недоступности | Низкое | Средняя |
| Read Repair | Исправление при чтении | Среднее | Высокая |
| Anti-Entropy Repair | Периодическая сверка | Нет (фоновый) | Полная |

#### Выбор стратегий по сущностям

**Таблица `orders_by_user` (критичные заказы)**

| Стратегия | Применение | Обоснование |
|-----------|------------|-------------|
| Hinted Handoff | ✅ Всегда включено | Минимальная задержка записи |
| Read Repair | ✅ QUORUM | Гарантия актуальности статуса |
| Anti-Entropy Repair | ✅ Еженедельно | Полная синхронизация в фоне |

**Таблица `carts_by_user` (корзины)**

| Стратегия | Применение | Обоснование |
|-----------|------------|-------------|
| Hinted Handoff | ✅ Всегда | Высокая скорость записи |
| Read Repair | ✅ ONE | Достаточно слабой согласованности |
| Anti-Entropy Repair | ✅ Ежедневно | Корзины с TTL требуют синхронизации |

**Таблица `sessions` (сессии)**

| Стратегия | Применение | Обоснование |
|-----------|------------|-------------|
| Hinted Handoff | ✅ Всегда | Критична скорость записи |
| Read Repair | ❌ Отключена | Сессии не требуют строгой целостности |
| Anti-Entropy Repair | ❌ Не требуется | TTL автоматически удаляет |

### 4.5 Итоговые настройки

| Сущность | Write Consistency | Read Consistency | Repair Strategy |
|----------|------------------|------------------|-----------------|
| orders_by_user | `QUORUM` | `LOCAL_QUORUM` | Hinted + Read + Weekly AE |
| orders_by_status | `QUORUM` | `ONE` | Hinted + Read + Weekly AE |
| carts_by_user | `ONE` | `LOCAL_QUORUM` | Hinted + Read + Daily AE |
| carts_by_session | `ONE` | `LOCAL_QUORUM` | Hinted + Read + Daily AE |
| sessions | `ONE` | `ONE` | Hinted только |

### 4.6 Преимущества перед MongoDB

| Критерий | MongoDB | Cassandra | Преимущество |
|----------|---------|-----------|---------------|
| Масштабирование | Range-based + полное перераспределение | Consistent hashing + виртуальные узлы | ✅ Без просадок |
| Репликация | Master-slave | Leaderless | ✅ Нет единой точки отказа |
| Запись | Журнал операций | Commit log + memtable | ✅ Выше скорость |
| Геораспределение | Сложно | Нативно | ✅ Multi-DC поддержка |
| TTL | Нет | Встроенный | ✅ Автоочистка сессий |

## 5. Заключение

Предложенные архитектурные решения обеспечивают:

1. **Эффективное распределение данных** через оптимальные шард-ключи для MongoDB
2. **Мониторинг и устранение** "горячих" шардов с набором метрик и автоматическими механизмами балансировки
3. **Баланс между консистентностью и производительностью** через настройки read preference
4. **Линейную масштабируемость** при миграции критических сущностей на Cassandra
5. **Автоматическую очистку** устаревших данных через TTL для сессий и корзин

Разработанные схемы коллекций и стратегии шардирования полностью соответствуют бизнес-требованиям интернет-магазина и обеспечивают отказоустойчивость при высоких нагрузках до 50 000 запросов в секунду.
