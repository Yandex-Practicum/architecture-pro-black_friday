### <a name="_uanumrh8zrui"></a>**Дата:** 04.04.2026 г.

«Мобильный мир» хранит информацию о заказах, товарах и корзинах в трёх коллекциях в MongoDB.

## Проектирование схем коллекций для шардирования данных

### Orders
Коллекция orders включает заказы клиентов и содержит атрибуты:
- Уникальный идентификатор заказа;
- Идентификатор клиента;
- Дату и время оформления заказа;
- Список заказанных товаров и их цену;
- Статус заказа;
- Общую сумму заказа;
- Геозону заказа.

Коллекцию можно представить так:
```
orders: {
  _id: ObjectId,                    // Уникальный идентификатор заказа
  order_id: string,                 // Бизнес-идентификатор (индекс)
  user_id: string,                  // Идентификатор клиента
  order_date: Date,                 // Дата и время оформления
  items: [                          // Список товаров
    {
      product_id: string,
      product_name: string,
      quantity: number,
      price: number,
      category: string
      ...
    }
  ],
  status: string,                   // возможные статусы "pending", "paid", "shipped", "delivered", "cancelled"
  total_amount: number,             // Общая сумма
  geo_zone: string,                 // Геозона (Москва, СПБ, и т.д.)
  shipping_address: object,         // Адрес доставки
  payment_method: string,           // Способ оплаты
  created_at: Date,
  updated_at: Date
  deleted_at: Date                  // если поддерживается мягкое удаление
}
```

Можно добавить индексы:
```
db.orders.createIndex({ "user_id": 1, "order_date": -1 })
db.orders.createIndex({ "order_id": 1 })
db.orders.createIndex({ "geo_zone": 1, "order_date": 1 })
```

Основные операции:
- Быстрое создание заказов с одновременным списанием остатков.
- Поиск истории заказов конкретного пользователя.
- Отображение статуса заказа.

#### Шард-ключ и стратегия:
Тип шардирования: Хэшированный
Шард-ключ: user_id
```
sh.shardCollection("mobilny_mir.orders", { "user_id": "hashed" })
```

#### Обоснование
- Равномерное распределение нагрузки:
  - user_id с хэшированием обеспечивает равномерное распределение заказов по всем шардам
  - Избегает "горячих" шардов (например, при шардировании по геозоне)
- Поддержка основных операций:
  - Поиск истории заказов пользователя: запрос { user_id: "..." } попадает в один шард (эффективно)
  - Отображение статуса заказа: по _id или user_id быстро находит документ
  - Создание заказа: распределяется равномерно

- Горизонтальное масштабирование:
  - При росте числа пользователей новые шарды получают равномерную нагрузку
  - Хэширование предотвращает дисбаланс при неравномерном распределении пользователей
- Альтернативы и почему они не подходят:
  - order_date: создаст "горячий" шард для последних заказов
  - geo_zone: приведет к дисбалансу (Москва vs другие города)
  - status: слишком мало уникальных значений


### Products
Коллекция products хранит сведения о товарах и включает такие атрибуты:
- Уникальный идентификатор товара;
- Наименование;
- Категорию товара;
- Цену;
- Остаток товара в каждой геозоне;
Дополнительные атрибуты (цвет, размер).

Коллекцию можно представить так:
```
products: {
  _id: ObjectId,                    // Уникальный идентификатор
  product_id: string,               // SKU артикул
  name: string,                     // Наименование
  category: string,                 // Категория (Электроника, Книги, и т.д.)
  price: number,                    // Цена
  stock: [                          // Остатки по геозонам
    {
      geo_zone: string,
      quantity: number
    }
  ],
  attributes: {                     // Доп. атрибуты
    color: string,
    size: string,
    brand: string,
    manufacturer: string
  },
  description: string,              // Описание товара
  created_at: Date,
  updated_at: Date
  deleted_at: Date                 // если поддерживается мягкое удаление
}
```

Можно добавить индексы:
```
db.products.createIndex({ "category": 1, "price": 1 })
db.products.createIndex({ "product_id": 1 })
db.products.createIndex({ "name": "text" })
db.products.createIndex({ "stock.geo_zone": 1, "stock.quantity": 1 })
```
Мы так же можем добавить индекс для поиска дорогих товаров (сортировка цен по-убыванию)
```
db.products.createIndex({ "category": 1, "price": -1 })
```


Основные операции:
- Частые обновления остатков при покупках.
- Поиск товаров по категориям и фильтрация по диапазону цен.
- Описание товара на странице продукта.

#### Шард-ключ и стратегия:
1. По категории
Тип шардирования: Хэшированный
Шард-ключ: category
```
sh.shardCollection("mobilny_mir.products", { "category": "hashed" })
```

#### Обоснование
- Поддержка основных операций:
  - Поиск товаров по категориям: запросы по category будут точечными (попадают в один шард)
  - Фильтрация по диапазону цен: требует индекса { category: 1, price: 1 }, но может стать broadcast-операцией
- Равномерное распределение:
  - Хэширование по категориям распределяет товары равномерно
  - Избегает проблемы "горячих" категорий
- Недостатки:
  - Поиск по диапазону цен без указания категории - broadcast
  - Требует составных индексов для эффективной фильтрации

2. Диапазонный по ценам
Тип шардирования: Диапазонный
Шард-ключ: price
```
sh.shardCollection("mobilny_mir.products", { "price": 1 })
```

#### Обоснование
- Поддержка поиска по диапазону цен:
  - Основная операция: "фильтрация по диапазону цен"
  - Запросы типа { price: { $gte: 100, $lte: 500 } } будут точечными
  - Может быть дополнен индексом { price: 1, category: 1 }
- Равномерное распределение:
  - Цены распределены относительно равномерно
  - Новые товары попадают в разные ценовые диапазоны
- Недостатки:
  - Поиск по категории становится broadcast-операцией
  - "Горячие" точки в популярных ценовых диапазонах (например, бюджетные товары 1000-3000 руб.)

3. Составной ключ
Тип шардирования: Составной диапазонный
Шард-ключ: category, price
```
sh.shardCollection("mobilny_mir.products", { "category": 1, "price": 1 })
```

#### Обоснование
- Поддержка операций из задания:
  - "Поиск товаров по категориям и фильтрация по диапазону цен" - идеально!
  - Запрос { category: "Электроника", price: { $gte: 10000, $lte: 50000 } } будет точечным
  - Данные группируются сначала по категории, затем по цене
- Преимущества для бизнес-логики:
  - Часто пользователи ищут товары в конкретной категории с ценовым фильтром
  - Все товары одной категории на одном шарде (или нескольких, если категория большая)
  - Возможность использовать составные индексы
Лучше остановиться на варианте 3, т.к. необходим поиск товаров по категориям и фильтрация по диапазону цен


### Carts
Коллекция carts хранит данные о текущих корзинах (как гостевых, так и пользовательских) и включает атрибуты:
- Уникальный идентификатор корзины (_id);
- Идентификатор пользователя (user_id) и session_id для гостей;
- Список товаров (items): массив документов { product_id, quantity };
- Статус корзины (status): "active" | "ordered" | "abandoned";
- Дату и время создания (created_at);
- Дату и время последнего обновления (updated_at);
- Время удаления (TTL) (expires_at) — для автоматической очистки старых корзин.

Коллекцию можно представить так:
```
carts: {
  _id: ObjectId,                    // Уникальный идентификатор корзины
  user_id: string,                  // Идентификатор пользователя (для авторизованных)
  session_id: string,               // ID сессии (для гостей)
  items: [                          // Список товаров
    {
      product_id: string,
      product_name: string,
      quantity: number,
      price: number
      ...
    }
  ],
  status: string,                   // возможные статусы "active", "ordered", "abandoned"
  created_at: Date,
  updated_at: Date,
  expires_at: Date,                 // TTL индекс (например, +30 дней)
  total_items: number,              // Общее количество товаров (денормализовано)
  total_amount: number              // Общая сумма (денормализовано)
}
```

Можно добавить индексы:
```
db.carts.createIndex({ "user_id": 1, "status": 1 }, { partialFilterExpression: { status: "active" } })
db.carts.createIndex({ "session_id": 1, "status": 1 }, { partialFilterExpression: { status: "active" } })
db.carts.createIndex({ "expires_at": 1 }, { expireAfterSeconds: 0 })
db.carts.createIndex({ "updated_at": 1 })
```
partialFilterExpression - это частичный индекс (partial index), который индексирует только те документы, которые соответствуют указанному условию фильтрации
Из-за этого:
* Индекс занимает меньше места на диске
* Операции записи для неактивных корзин быстрее

Индекс
```
db.carts.createIndex({ "expires_at": 1 }, { expireAfterSeconds: 0 })
```
означает, что документы будут удалены немедленно в момент, когда текущее время станет больше или равно значению expires_at 

Основные операции:
- Создание корзины, когда заходит гость или новый пользователь.
- Получение текущей корзины по фильтру { session_id, status:"active" } или { user_id, status:"active" }.
- Добавление или замена товара в корзине.
- Удаление товара из корзины.
- Слияние гостевой корзины в пользовательскую, если пользователь залогинится:
  - прочитать гостевую { session_id, status:"active" };
  - добавить её items в корзину { user_id, status:"active" };
  - отметить гостевую как abandoned.
- Отметка корзины как заказанной.

#### Шард-ключ и стратегия:
Тип шардирования: Составной хэшированный
Шард-ключ: user_id, session_id
```
sh.shardCollection("mobilny_mir.carts", { "user_id": "hashed", "session_id": "hashed" })
```

#### Обоснование
- Уникальность запросов:
  - Авторизованные пользователи: запрос по user_id
  - Гости: запрос по session_id
  - Составной ключ позволяет эффективно обрабатывать оба типа запросов
- Равномерное распределение:
  - Хэширование user_id и session_id распределяет корзины равномерно
  - Избегает "горячих" шардов (популярные пользователи)
- Поддержка основных операций:
  - Получение текущей корзины: прямое попадание в шард по user_id или session_id
  - Создание/обновление корзины: точечные операции
  - Слияние корзин: два запроса, но оба точечные
- Специфика корзин:
  - Корзины имеют короткий жизненный цикл
  - TTL автоматически удаляет старые корзины
  - Хэширование обеспечивает равномерное удаление


## Выявление и устранение «горячих» шардов

Из-за категории «Электроника» произошла перегрузка одного из шардов MongoDB, так как 70% запросов приходилось именно на эти товары.
Поэтому сейчас нужно разработать стратегию, как выявлять и устранять такие «горячие» шарды, а ещё предложить метрики мониторинга,
чтобы в будущем можно было предотвращать такие ситуации.

Цель: Разработать систему мониторинга и автоматического перераспределения данных для предотвращения «горячих» шардов.

Ключевые метрики

|       **Метрика**        | **Описание**                                                                      | Комментарий                                                                                                    |
|:------------------------:|:----------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------|
| QPS (Queries Per Second) | Количество запросов в секунду на шард                                             | Если один шард обрабатывает 60-80% всех запросов - это hot shard<br/>(mongostat, db.serverStatus().opcounters) |
|    Chunk Distribution    | Количество чанков на шард, размер чанков                                          | нагрузка может оказаться неравномерной<br/>sh.status()                                                         |
|        Data Size         | Объем данных на шард (GB)                                                         | плохо, если один шард сильно больше остальных                                                                  |
|        CPU Usage         | Загрузка CPU на узле шарда                                                        | Hot shard часто упирается в CPU и начинает тормозить все запросы                                               |
|     Latency запросов     | Время выполнения операций (p50 / p95 / p99 latency)                               | db.currentOp(), profiler (system.profile)                                                                      |
|     Chunk migrations     | Количество миграций, время миграций, ошибки                                       | плохо, если постоянные миграции, но без успеха                                                                 |
|    Query patterns        | Какие поля используются в фильтрах, частота запросов по значениям                 | profiler, logs                                                                                                 |

Команды для сбора метрик

Распределение чанков по шардам
```
db.getSiblingDB("config").chunks.aggregate([
  { $group: { _id: "$shard", count: { $sum: 1 } } }
])
```
Размер данных на шард
```
db.getSiblingDB("config").shards.forEach(function(shard) {
  printjson(db.adminCommand({ shardStatistics: 1, shardName: shard._id }))
})
```


Настройка алертов (Prometheus + Grafana)
```
# prometheus-alerts.yml
groups:
  - name: mongodb_sharding
    rules:
      - alert: ShardImbalance
        expr: max(chunk_count) / min(chunk_count) > 1.5
        for: 10m
        annotations:
          summary: "Дисбаланс распределения чанков"
          
      - alert: HotShardCPU
        expr: mongodb_cpu_usage > 80
        for: 15m
        annotations:
          summary: "Высокая CPU нагрузка на шард"
          
      - alert: JumboChunkDetected
        expr: chunk_size_mb > 64
        for: 5m
        annotations:
          summary: "Обнаружен jumbo чанк"
```


Механизмы устранения дисбаланса

Настройка автоматического балансировщика
```
// Включение балансировщика
sh.startBalancer()

// Настройка окна балансировки (вне часов пик)
use config
db.settings.update(
  { _id: "balancer" },
  { 
    $set: { 
      activeWindow: { start: "02:00", stop: "06:00" },
      stopped: false
    } 
  },
  { upsert: true }
)

// Установка порога миграции (разница в количестве чанков)
db.settings.update(
  { _id: "balancer" },
  { $set: { migrateThreshold: 8 } }
)
```

Zoned Sharding для популярных категорий
Zoned Sharding позволяет закрепить определенные диапазоны данных за конкретными шардами
```
// 1. Создание зон для разных уровней нагрузки
sh.addShardToZone("shard1", "HIGH_DEMAND")
sh.addShardToZone("shard2", "HIGH_DEMAND")
sh.addShardToZone("shard3", "LOW_DEMAND")

// 2. Распределение категории «Электроника» по ценовым диапазонам
sh.updateZoneKeyRange(
  "mobilny_mir.products",
  { category: "Электроника", price: MinKey },
  { category: "Электроника", price: 50000 },
  "HIGH_DEMAND"
)

sh.updateZoneKeyRange(
  "mobilny_mir.products",
  { category: "Электроника", price: 50000 },
  { category: "Электроника", price: 150000 },
  "HIGH_DEMAND"
)

sh.updateZoneKeyRange(
  "mobilny_mir.products",
  { category: "Электроника", price: 150000 },
  { category: "Электроника", price: MaxKey },
  "LOW_DEMAND"
)

// 3. Размещение других категорий
sh.updateZoneKeyRange(
  "mobilny_mir.products",
  { category: "Книги", price: MinKey },
  { category: "Книги", price: MaxKey },
  "LOW_DEMAND"
)
```

Принудительный сплит больших чанков
```
// Поиск чанков, превышающих размер
db.getSiblingDB("config").chunks.find({
  ns: "mobilny_mir.products",
  $where: "this.dataSize > 64 * 1024 * 1024"
})

// Ручной сплит по значению
sh.splitAt(
  "mobilny_mir.products",
  { category: "Электроника", price: 75000 }
)

// Сплит по соотношению (50/50)
sh.splitFind(
  "mobilny_mir.products",
  { category: "Электроника", price: 50000 }
)
```


В данном случае применимы следующие автоматические механизмы перераспределения данных в MongoDB

1. Встроенная автоматика MongoDB
   Balancer
   Это автоматический процесс, который:
- перемещает chunks между shard’ами
- выравнивает распределение данных
  Проверка состояния
```
sh.getBalancerState()

sh.isBalancerRunning()
```

2. Управление chunk’ами
   Изменение размера
```
use config
db.settings.save({
  _id: "chunksize",
  value: 32
})
```
Когда уменьшать:
- hot shard
- большие chunks
- медленные миграции
3. Resharding
   изменить shard key без даунтайма
   К примеру:
```
{ category: 1 }
```
меняется на
```
{ category: 1, product_id: 1 }
```
Команда:
```
sh.reshardCollection(
  "db.products",
  { category: 1, product_id: 1 }
)
```

В данном случае:
- Zone Sharding не перераспределит нагрузку
- Hashed Sharding не поддержит range-запросы


## Настройка чтения с реплик и консистентность

### Коллекция products

|                  **Операция**                  | **Чтение с** | **Обоснование**                                     |
|:----------------------------------------------:|:-------------|-----------------------------------------------------|
|            Просмотр карточки товара            | Secondary    | Не критично увидеть остатки с задержкой 1-2 секунды |
|           Поиск и фильтрация товаров           | Secondary    | Высокая нагрузка, допустима небольшая задержка      |
|   Просмотр остатков при добавлении в корзину   | Primary      | Критично: может привести к overselling              |
|   Проверка остатков перед оформлением заказа   | Primary      | Критично: последняя проверка перед оплатой          |
| Административная панель (управление товарами)  | Primary      | Требует актуальных данных                           |

Допустимая задержка репликации: 1-5 секунд
Обоснование:
- Пользователь не заметит разницу в остатках за 1-5 секунд
- Цена и описание меняются редко, задержка 5 секунд безопасна
- При высокой нагрузке (хиты продаж) secondary реплики снижают нагрузку на primary
- Критические проверки остатков идут на primary


### Коллекция orders

|               **Операция**                | **Чтение с** | **Обоснование**                                            |
|:-----------------------------------------:|:-------------|------------------------------------------------------------|
|       История заказов пользователя        | Secondary    | Не критично увидеть последний заказ с задержкой 3-5 секунд |
|    Отображение статуса текущего заказа    | Primary      | Пользователь ожидает актуальный статус                     |
|     Просмотр заказа в личном кабинете     | Secondary    | Старые заказы не меняются                                  |
|            Аналитика и отчеты             | Secondary    | Допустима задержка, не влияет на пользователей             |
| Обновление статуса заказа (после оплаты)  | Primary      | Операция записи                                            

Допустимая задержка репликации: 5-10 секунд
Обоснование:
- Пользователь реже проверяет историю старых заказов
- История заказов — это неизменяемые данные после завершения
- Статус текущего заказа критичен (ожидание доставки, оплата)
- Задержка 5-10 секунд не критична для просмотра истории


### Коллекция carts

|         **Операция**          | **Чтение с** | **Обоснование**                                            |
|:-----------------------------:|:-------------|------------------------------------------------------------|
|  Отображение текущей корзины  | Primary      | Не критично увидеть последний заказ с задержкой 3-5 секунд |
|  Добавление товара в корзину  | Primary      | Пользователь ожидает актуальный статус                     |
| Обновление количества товара  | Primary      | Старые заказы не меняются                                  |
|  Удаление товара из корзины   | Primary      | Допустима задержка, не влияет на пользователей             |
|   Слияние гостевой корзины    | Primary      | Операция записи                                            
| Аналитика корзин (статистика) | Secondary    | Допустима задержка                                         |
| Очистка старых корзин (TTL)   | Secondary    | Фоновый процесс                                            |

Допустимая задержка репликации: 10 секунд
Обоснование:
- Корзина — это живой документ, часто обновляемый
- Пользователь ожидает мгновенного отображения изменений
- Ошибка в отображении корзины ведет к потере продаж



## Миграция на Cassandra: модель данных, стратегии репликации и шардирования

### Обоснование выбора:
Cassandra подходит для:
- Заказы и история заказов — высокая нагрузка записи, слабые требования к ACID
- Корзины — временные данные, высокая скорость записи/чтения
- Сессии — высокая нагрузка, допустима потеря отдельных сессий
- Отзывы — редкая запись, частое чтение

Cassandra НЕ подходит для:
- Остатки товаров — требуется строгая консистенция (CAS операции медленные в Cassandra)
- Товары (каталог) — лучше оставить в MongoDB или использовать Redis как кеш

### Модель данных для Cassandra

1. Сущность: Заказы (orders)
```
-- Таблица заказов с поддержкой разных запросов
CREATE TABLE IF NOT EXISTS mobile_world.orders_by_user (
    -- PARTITION KEY: user_id (обеспечивает равномерное распределение)
    -- CLUSTERING KEY: order_date DESC (последние заказы первые)
    user_id UUID,
    order_date TIMESTAMP,
    order_id UUID,
    status TEXT,                    -- pending, paid, shipped, delivered
    total_amount DECIMAL,
    geo_zone TEXT,
    items TEXT,                     -- JSON или UDT
    shipping_address TEXT,
    payment_method TEXT,
    updated_at TIMESTAMP,
    PRIMARY KEY ((user_id), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC, order_id ASC)
   AND compaction = { 'class' : 'TimeWindowCompactionStrategy', 'compaction_window_size' : 1, 'compaction_window_unit' : 'DAYS' }
   AND default_time_to_live = 2592000  -- 30 дней для истории заказов
   AND gc_grace_seconds = 86400;        -- 24 часа для удаления

-- Дополнительная таблица для поиска по статусу (анти-паттерн, но необходим)
CREATE TABLE mobile_world.orders_by_status (
    status TEXT,
    order_date TIMESTAMP,
    order_id UUID,
    user_id UUID,
    total_amount DECIMAL,
    PRIMARY KEY ((status), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC);

-- Таблица для конкретного заказа (быстрый доступ)
CREATE TABLE mobile_world.order_details (
    order_id UUID PRIMARY KEY,
    user_id UUID,
    order_date TIMESTAMP,
    status TEXT,
    total_amount DECIMAL,
    items TEXT,
    shipping_address TEXT
);
```

Обоснование ключей:
- PARTITION KEY user_id: Равномерное распределение заказов по пользователям, предотвращает "горячие" партиции
- CLUSTERING KEY order_date: Эффективный запрос истории заказов в хронологическом порядке
- Denormalization: Данные дублированы для разных паттернов запросов (CQRS)

2. Сущность: Корзины (carts)
```
-- Активные корзины (основная таблица)
CREATE TABLE mobile_world.active_carts (
    -- session_id для гостей, user_id для авторизованных
    cart_id TEXT,                    -- 'session:XXX' или 'user:XXX'
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    items TEXT,                      -- JSON массив товаров
    total_items INT,
    total_amount DECIMAL,
    status TEXT,                     -- active, abandoned
    expires_at TIMESTAMP,
    PRIMARY KEY ((cart_id))
) WITH compaction = { 'class' : 'LeveledCompactionStrategy' }
   AND default_time_to_live = 604800  -- 7 дней для корзин
   AND gc_grace_seconds = 86400;
```

Обоснование ключей:
- PARTITION KEY cart_id: Уникальный идентификатор корзины (гость или пользователь)
- Отсутствие CLUSTERING KEY: Корзина читается/пишется целиком, не нужна сортировка
- TTL 7 дней: Автоматическое удаление старых корзин


### Стратегии обеспечения целостности данных
Заказы (orders):
Стратегии:
- Hinted Handoff: ВКЛ (приоритет доступности)
- Read Repair: ВЫКЛ (слишком частая запись)
- Anti-Entropy: Ежедневно в 3:00
  Уровень:
- WRITE: LOCAL_QUORUM (2 из 3)
- READ: ONE (допустима слабая консистенция для истории)
  Обоснование: Заказы создаются быстро, допустима небольшая задержка репликации.

Корзины (carts):
Стратегии:
- Hinted Handoff: ВКЛ
- Read Repair: ВКЛ (20% запросов)
- Anti-Entropy: Еженедельно
  Consistency Level:
- WRITE: ANY (максимальная доступность)
- READ: ONE (корзина может быть восстановлена)
  Обоснование: Корзины временны, потеря некритична, важнее скорость
