## Общий контекст и цели

### Бизнес-контекст
Онлайн-магазин «Мобильный мир» расширил ассортимент с аксессуаров для смартфонов до электроники, аудио- и бытовой техники. Рост нагрузки на базу данных требует горизонтального масштабирования через шардирование MongoDB.

### Технические цели
1. **Производительность**: Обеспечить low-latency доступ к данным для критических операций
2. **Масштабируемость**: Поддержка линейного роста объема данных и нагрузки
3. **Доступность**: Минимизировать downtime и обеспечить отказоустойчивость
4. **Операционная эффективность**: Упростить администрирование и мониторинг

### Критические операции
- `orders`: История заказов пользователя (P95 < 100ms)
- `products`: Обновление остатков при покупке (P99 < 50ms)  
- `carts`: CRUD операции с активной корзиной (P95 < 50ms)

## Детализация ADR

### ADR-001: Шардирование коллекции `orders`

**Дата**: 2026-01-24  
**Статус**: Принято  
**Участники**: Архитектор БД, Lead Backend Developer, DevOps

#### Контекст
Коллекция `orders` — самая быстрорастущая (ожидается 100M+ записей). Ключевые требования:
- Мгновенный доступ к истории заказов пользователя
- Быстрое создание новых заказов
- Минимальная latency для отображения статуса

#### Варианты решения

| Вариант | Шард-ключ | Тип | Плюсы | Минусы |
|---------|-----------|-----|-------|--------|
| **Вариант A** | `_id` | Hashed | Равномерное распределение | История пользователя требует scatter-gather |
| **Вариант B** | `created_at` | Ranged | Естественный порядок, хорош для аналитики | Горячие шарды, история пользователя фрагментирована |
| **Вариант C** | `user_id` | Ranged | Локальность данных пользователя | Потенциальный дисбаланс нагрузки |

#### Решение
Выбрать **ranged шардинг по полю `user_id`**.

#### Обоснование
1. **Оптимизация частого сценария**: 80% запросов — это `db.orders.find({user_id: X})`. При шардинге по `user_id` все заказы пользователя физически находятся на одном шарде.
2. **Производительность**: Локальность данных сокращает latency запросов истории с 200ms до 20-30ms.
3. **Масштабируемость**: Новые заказы пользователя попадают на существующий шард, минимизируя миграции чанков.

#### Последствия
- **Положительные**: 
  - История заказов пользователя всегда быстрая
  - Упрощенные JOIN-like операции в рамках одного пользователя
- **Отрицательные**:
  - Требуется глобальный вторичный индекс для поиска по `_id`
  - Возможны "горячие" шарды для power-users
  - Запросы по другим полям (`geo_zone`, `created_at`) требуют scatter-gather

#### Митигирование рисков
1. Для power-users (B2B клиенты) рассмотреть префиксный шард-ключ `{user_id_prefix, user_id}`
2. Реализовать мониторинг дисбаланса чанков
3. Использовать presplitting для начального распределения

### ADR-002: Шардирование коллекции `products`

**Дата**: 2026-01-24  
**Статус**: Принято  
**Участники**: Архитектор БД, Product Manager, Backend Developer

#### Контекст
Коллекция `products` — справочник с частыми точечными обновлениями остатков. Требования:
- Атомарное обновление остатков при покупках
- Быстрый поиск по каталогу (категории, цена)
- Высокая доступность для чтения

#### Варианты решения

| Вариант | Шард-ключ | Тип | Плюсы | Минусы |
|---------|-----------|-----|-------|--------|
| **Вариант A** | `category` | Ranged | Локальность товаров одной категории | Дисбаланс (электроника vs нишевые категории) |
| **Вариант B** | `_id` | Hashed | Идеально равномерное распределение | Поиск по категории требует scatter-gather |
| **Вариант C** | `geo_zone` + `category` | Hashed | Локальность по геозоне | Усложнение схемы, редкие сценарии |

#### Решение
Выбрать **hashed шардинг по полю `_id`**.

#### Обоснование
1. **Равномерность**: Гарантирует равномерное распределение записей и нагрузки
2. **Точечные операции**: Обновление остатков `db.products.updateOne({_id: X}, ...)` всегда направляется на один шард
3. **Предсказуемость**: Простая модель, легко администрировать

#### Последствия
- **Положительные**:
  - Нет "горячих" шардов
  - Линейная масштабируемость при добавлении товаров
  - Атомарность операций обновления
- **Отрицательные**:
  - Запросы каталога (`category`, `price`) требуют scatter-gather
  - Нет географической локализации данных

#### Компенсирующие меры
1. Создать **составной индекс `{category: 1, price: 1}`** для ускорения каталога
2. Рассмотреть read-only реплики для регионализации
3. Кэширование горячих товаров на уровне приложения

### ADR-003: Шардирование коллекции `carts`

**Дата**: 2026-01-24  
**Статус**: Принято  
**Участники**: Архитектор БД, Frontend Lead, Backend Developer

#### Контекст
Коллекция `carts` содержит временные данные с высоким RPS. Особенности:
- Большой объем гостевых сессий
- Кратковременные данные (TTL 7-30 дней)
- Частые CRUD операции
- Слияние корзин при логине

#### Варианты решения

| Вариант | Шард-ключ | Тип | Плюсы | Минусы |
|---------|-----------|-----|-------|--------|
| **Вариант A** | `session_id` | Hashed | Равномерное распределение гостей | Поиск по `user_id` требует scatter-gather |
| **Вариант B** | `user_id` | Ranged | Локальность данных пользователя | Неэффективно для гостей (null значения) |
| **Вариант C** | `_id` | Hashed | Простота реализации | Нет локализации по пользователю/сессии |

#### Решение
Выбрать **hashed шардинг по полю `session_id`**.

#### Обоснование
1. **Распределение нагрузки**: Гостевые сессии — основной источник нагрузки, равномерно распределяются
2. **Критические операции**: Работа с активной корзиной (по `session_id`) всегда на одном шарде
3. **Гибкость**: Поддержка как гостевых, так и пользовательских сессий

#### Последствия
- **Положительные**:
  - Высокая производительность операций с активной корзиной
  - Автоматическая очистка устаревших данных через TTL
  - Масштабируемость при росте трафика
- **Отрицательные**:
  - Слияние корзин при логине затрагивает 2 шарда
  - Запросы по `user_id` требуют глобального индекса

#### Оптимизации
1. Создать **глобальный вторичный индекс `{user_id: 1, status: 1}`**
2. Реализовать batch-операции для слияния корзин
3. Настроить агрессивный TTL (7 дней для abandoned, 30 дней для ordered)

### ADR-004: Стратегия индексации в шардированном кластере

**Дата**: 2026-01-24  
**Статус**: Принято  
**Участники**: Архитектор БД, DevOps, Backend Developer

#### Контекст
В шардированном кластере индексы могут быть:
- **Локальными**: Создаются на каждом шарде отдельно
- **Глобальными (вторичными)**: Создаются на config servers, покрывают все шарды

#### Решение
Использовать комбинацию локальных и глобальных индексов:

| Коллекция | Индекс | Тип | Назначение |
|-----------|--------|-----|------------|
| `orders` | `{_id: 1}` | Глобальный | Поиск заказа по ID |
| `orders` | `{user_id: 1, created_at: -1}` | Локальный | Сортировка истории |
| `products` | `{category: 1, price: 1}` | Локальный | Фильтрация каталога |
| `carts` | `{user_id: 1, status: 1}` | Глобальный | Слияние корзин при логине |
| `carts` | `{expires_at: 1}` | Локальный | TTL автоматическая очистка |

#### Обоснование
1. **Глобальные индексы** для запросов, где шард-ключ неизвестен
2. **Локальные индексы** для операций в рамках одного шарда
3. **Баланс** между производительностью и overhead обслуживания

## Риски и ограничения

### Технические риски
1. **Дисбаланс чанков** в `orders` при ranged шардинге
   - **Митигация**: Регулярный мониторинг, presplitting, manual chunk migration
   
2. **Scatter-gather overload** для запросов каталога `products`
   - **Митигация**: Кэширование, read replicas, пагинация с limit
   
3. **Глобальные индексы overhead** на config servers
   - **Митигация**: Минимизировать количество, мониторинг нагрузки

### Операционные ограничения
1. **Сложность агрегаций** на всех коллекциях
   - **Рекомендация**: Выполнять в off-peak hours, использовать специальные аналитические реплики
   
2. **Бэкапы** шардированного кластера
   - **Рекомендация**: Использовать mongodump с опцией `--oplog`, тестировать восстановление

3. **Версионность схемы**
   - **Рекомендация**: Blue-green миграции, feature flags

### Бизнес-ограничения
1. **Стоимость** операций scatter-gather
   - **Компенсация**: Оптимизация запросов, использование проекций
   
2. **Latency** для кросс-шардовых транзакций
   - **Компенсация**: Локальность данных, асинхронные операции где возможно

# Архитектурное решение (ADR-008): Выявление и устранение «горячих» шардов в MongoDB

## Статус
ПРИНЯТО

## Дата
2026-01-24

## Участники
- Архитектор баз данных
- DevOps инженер
- Backend разработчик


---

## Контекст и Проблема

### Текущая ситуация
В системе онлайн-магазина «Мобильный мир» возникла критическая проблема с распределением нагрузки:

1. Концентрация запросов: 70% всех запросов к коллекции `products` приходится на товары категории «Электроника»
2. Неравномерное распределение: Текущая стратегия хешированного шардирования по `_id` не учитывает паттерны доступа
3. Перегрузка шарда: Один из шардов MongoDB постоянно перегружен

### Последствия
- Увеличение latency критических операций (P95 > 200ms)
- Риск недоступности популярных товаров  
- Снижение конверсии и потеря продаж

### Технический контекст
    `sh.shardCollection("shop.products", { _id: "hashed" })`

Проблема: товары категории "Электроника" распределяются случайно по всем шардам, создавая scatter-gather нагрузку.

---

## Рассмотренные варианты

### Вариант A: Улучшение текущей схемы
Описание: Сохранить hashed шардирование по `_id`, но добавить расширенный мониторинг и ручное перераспределение.

Плюсы:
- Минимальные изменения в схеме данных
- Нет миграции существующих данных
- Быстрое внедрение

Минусы:
- Не решает коренную проблему дисбаланса
- Ручное вмешательство = высокие операционные затраты
- Реактивный, а не проактивный подход

### Вариант B: Ranged шардирование по категории
Описание: Изменить шард-ключ на `category` или `{category, _id}`.

Плюсы:
- Локальность данных по категориям
- Упрощает балансировку

Минусы:
- Риск создания «супер-горячих» шардов для доминирующих категорий
- Все товары одной категории на одном шарде = риск перегрузки

### Вариант C: Составной ключ с ranged шардингом и зонами
Описание: Использовать составной ключ `category_order` (категория + распределительный номер) с ручным тегированием диапазонов.

Плюсы:
- Равномерное распределение горячих категорий по шардам
- Контролируемое распределение нагрузки
- Возможность тонкой настройки балансировки

Минусы:
- Сложнее в реализации
- Требует начальной настройки тегов

---

## Решение

Принят **Вариант C**: Составной ключ с ranged шардингом и зональным распределением.

### 1. Новая схема шардирования для `products`

#### 1.1 Подготовка данных
```javascript
    // Добавляем поле category_order для контролируемого распределения
    db.products.updateMany({}, [
      { $set: { 
        category_order: { 
          $concat: [
            "$category", 
            ":", 
            { 
              $toString: { 
                $mod: [
                  { $toLong: "$_id" }, 
                  1000  // Распределяем на 1000 диапазонов
                ] 
              } 
            }
          ]
        }
      }}
    ])

    // Пример значения: "electronics:423"
```

#### 1.2 Шардирование коллекции
    `sh.shardCollection("shop.products", { category_order: 1 })`

### 2. Механизм зонального распределения

#### 2.1 Настройка тегов для равномерного распределения
```javascript
    // Функция для распределения категории по шардам
    function distributeCategory(category, shards) {
      const rangesPerShard = 3; // Каждый шард получает 3 диапазона категории
      const totalRanges = 1000; // Общее количество диапазонов
      
      shards.forEach((shard, shardIndex) => {
        // Создаем уникальный тег для каждой части категории
        for (let i = 0; i < rangesPerShard; i++) {
          const rangeNum = shardIndex * rangesPerShard + i;
          const tagName = `cat_${category}_range_${rangeNum}`;
          
          sh.addShardTag(shard, tagName);
          
          // Определяем диапазон для этого тега
          const rangeSize = Math.floor(totalRanges / (shards.length * rangesPerShard));
          const start = rangeNum * rangeSize;
          const end = (rangeNum + 1) * rangeSize;
          
          sh.addTagRange("shop.products",
            { category_order: `${category}:${start}` },
            { category_order: `${category}:${end}` },
            tagName
          );
        }
      });
    }
```

#### 2.2 Распределение категории "Электроника"
```javascript
    const shards = ["shard1", "shard2", "shard3"];
    distributeCategory("electronics", shards);

    // Результат:
    // - electronics:0-111 → shard1 (тег cat_electronics_range_0)
    // - electronics:111-222 → shard1 (тег cat_electronics_range_1)
    // - electronics:222-333 → shard1 (тег cat_electronics_range_2)
    // - electronics:333-444 → shard2 (тег cat_electronics_range_3)
    // - electronics:444-555 → shard2 (тег cat_electronics_range_4)
    // - electronics:555-666 → shard2 (тег cat_electronics_range_5)
    // - electronics:666-777 → shard3 (тег cat_electronics_range_6)
    // - electronics:777-888 → shard3 (тег cat_electronics_range_7)
    // - electronics:888-999 → shard3 (тег cat_electronics_range_8)
```
### 3. Автоматическая балансировка с учетом зон

#### 3.1 Настройка интеллектуального балансировщика
```javascript
    // Конфигурация балансировщика с учетом зон
    db.adminCommand({
      configureBalanceRound: {
        enabled: true,
        maxDurationMs: 300000, // 5 минут максимум
        imbalancedThreshold: 1.5, // Дисбаланс 1.5:1
        constraints: {
          respectZones: true, // Учитывать зоны при балансировке
          maxChunksToMove: 10,
          excludeNamespaces: ["config.*", "admin.*"]
        }
      }
    })
```

#### 3.2 Автоматическое перераспределение горячих диапазонов
```javascript
    // Мониторинг и перебалансировка горячих диапазонов
    const hotRangeMonitor = setInterval(async () => {
      const hotRanges = await detectHotRanges();
      
      hotRanges.forEach(async (range) => {
        const { category, rangeStart, rangeEnd, operationsPerSec } = range;
        
        if (operationsPerSec > 1000) { // Горячий диапазон
          // Находим наименее загруженный шард
          const targetShard = await findLeastLoadedShard();
          
          // Создаем новый тег для этого диапазона
          const newTag = `hot_${category}_${Date.now()}`;
          sh.addShardTag(targetShard, newTag);
          
          // Перемещаем горячий диапазон
          sh.addTagRange("shop.products",
            { category_order: `${category}:${rangeStart}` },
            { category_order: `${category}:${rangeEnd}` },
            newTag
          );
          
          console.log(`Горячий диапазон ${category}:${rangeStart}-${rangeEnd} перемещен на ${targetShard}`);
        }
      });
    }, 60000); // Проверка каждую минуту
```

## Метрики мониторинга

### 4.2 Дашборды для оперативного контроля

Шард | Электроника | Аксессуары | Книги

## Обоснование решения

### Почему составной ключ с ranged шардингом?
1. **Контролируемое распределение**: Можем точно определить, куда попадет каждый товар
2. **Равномерность нагрузки**: Горячие категории распределяются по всем шардам
3. **Гибкость**: Можем менять распределение без изменения схемы данных
4. **Производительность**: Ranged шардинг эффективнее для запросов по диапазонам

### Как это решает проблему «горячего» шарда?
**До**: 
- Товары электроники распределены случайно
- 70% запросов идут на все шарды (scatter-gather)
- Один шард получает больше нагрузки

**После**:
- Товары электроники разделены на диапазоны
- Каждый диапазон привязан к конкретному шарду
- Нагрузка равномерно распределена между шардами
- Можно динамически перераспределять горячие диапазоны

## Контекст и Цели

### Бизнес-контекст
Онлайн-магазин «Мобильный мир» испытывает высокую нагрузку на чтение данных, особенно в пиковые часы. Необходимо распределить нагрузку чтения между primary и secondary репликами, сохраняя при этом консистентность данных.

### Технические цели
1. Снизить нагрузку на primary реплики
2. Увеличить доступность данных при сбоях
3. Обеспечить консистентность критических операций
4. Оптимизировать latency для пользователей

### Требования к консистентности
- Строгая консистентность: Всегда актуальные данные (primary)
- Eventual consistency: Допустима небольшая задержка (secondary)
- Session consistency: Консистентность в рамках сессии пользователя

---

## Стратегия чтения для коллекций

### Общие принципы
1. Read Preference в MongoDB:
   - primary - только primary (по умолчанию)
   - primaryPreferred - primary, fallback на secondary
   - secondary - только secondary
   - secondaryPreferred - secondary, fallback на primary
   - nearest - ближайшая по latency

2. Допустимая задержка репликации:
   - Для операций с secondary: ≤ 100ms (желательно ≤ 50ms)
   - Максимальный допустимый lag: 500ms
   - При lag > 1000ms: отказ от чтения с secondary

3. Мониторинг репликации:
   - replSetGetStatus для проверки lag
   - Heartbeat между репликами каждые 2 секунды
   - Oplog window: минимум 24 часа

    // Проверка lag реплик
    const checkReplicationLag = async () => {

      status.members.forEach(member => {

    // Адаптивный выбор read preference на основе lag
    function getAdaptiveReadPreference(operationType, currentLag) {
      const config = {
        products: {
          catalogSearch: { maxLag: 200, preference: 'secondaryPreferred' },
          checkStock: { maxLag: 0, preference: 'primary' }

        : 'primary';
    }

## Задание 10.1: Анализ данных для миграции

### Критерии оценки данных для Cassandra

| Критерий | Хорошо для Cassandra | Плохо для Cassandra |
|----------|---------------------|---------------------|
| **Скорость записи** | Высокая (time-series, логгирование) | Низкая (требуются ACID транзакции) |
| **Скорость чтения** | Быстрое чтение по ключу | Сложные JOIN, агрегации |
| **Масштабируемость** | Очень высокая | Требует координации между узлами |
| **Консистентность** | Eventual consistency | Строгая консистентность |
| **Структура данных** | Денормализованная, широкие строки | Нормализованная, много связей |

### Анализ сущностей магазина

#### 1. **Товары (products)**
**Характеристики**:
- Чтение: очень частое (каталог, страницы товаров)
- Запись: умеренная (обновление остатков, цен)
- Консистентность: высокая для остатков, средняя для описаний
- Размер данных: средний, стабильный рост

**Решение для Cassandra**: **ОГРАНИЧЕННО**
- Плюсы: Высокая доступность чтения
- Минусы: Нужны агрегации (фильтры, сортировка), сложные запросы
- **Рекомендация**: Оставить в MongoDB, использовать как справочник

#### 2. **Заказы (orders)**
**Характеристики**:
- Запись: высокая (создание заказов)
- Чтение: по пользователю, по ID
- Консистентность: критическая (оплата, статусы)
- Паттерн доступа: чтение истории по пользователю

**Решение для Cassandra**: **ХОРОШО ПОДХОДИТ**
- Плюсы: Write-heavy, чтение по ключу, естественная партиционизация по пользователю
- Минусы: Нужен отдельный индекс для поиска по датам
- **Рекомендация**: Перенести в Cassandra

#### 3. **Корзины (carts)**
**Характеристики**:
- Запись: очень высокая (добавление/удаление товаров)
- Чтение: по сессии/пользователю
- Консистентность: session consistency
- TTL: автоматическое удаление старых корзин

**Решение для Cassandra**: **ИДЕАЛЬНО ПОДХОДИТ**
- Плюсы: Высокая скорость записи, TTL поддержка, чтение по ключу
- Минусы: Минимальные
- **Рекомендация**: Перенести в Cassandra

#### 4. **История заказов (order_history)**
**Характеристики**:
- Запись: append-only
- Чтение: по пользователю + временной диапазон
- Консистентность: eventual consistency допустима
- Размер: быстрорастущий

**Решение для Cassandra**: **ИДЕАЛЬНО ПОДХОДИТ**
- Плюсы: Time-series данные, чтение по диапазону дат
- Минусы: Нужен композитный ключ
- **Рекомендация**: Перенести в Cassandra

#### 5. **Пользовательские сессии (user_sessions)**
**Характеристики**:
- Запись: высокая (логин, действия)
- Чтение: по session_id
- Консистентность: low consistency
- TTL: автоматическое удаление

**Решение для Cassandra**: **ИДЕАЛЬНО ПОДХОДИТ**
- Плюсы: Высокая доступность, TTL, простые запросы
- Минусы: Минимальные
- **Рекомендация**: Перенести в Cassandra

### Итоговая таблица решений

| Сущность | Перенести в Cassandra | Обоснование |
|----------|----------------------|-------------|
| **Товары (products)** | НЕТ | Сложные запросы, агрегации, лучше оставить в MongoDB |
| **Заказы (orders)** | ДА | Write-heavy, чтение по ключу, хорошая партиционизация |
| **Корзины (carts)** | ДА | Высокая скорость записи, TTL, session-based доступ |
| **История заказов** | ДА | Time-series, append-only, чтение по диапазонам |
| **Пользовательские сессии** | ДА | Высокая доступность, TTL, простые запросы |
| **Каталог товаров** | НЕТ | Сложные фильтры, сортировки, агрегации |

## Задание 10.2: Концептуальная модель данных

### Принципы проектирования для Cassandra
1. **Denormalize и дублировать данные**
2. **Design for queries** (не для хранения)
3. **Одна таблица = один запрос**
4. **Использовать композитные ключи для диапазонных запросов**

### Модель 1: Заказы (orders)

#### Таблица 1: Заказы по пользователю
```cql
    CREATE TABLE orders_by_user (
        user_id uuid,
        order_id uuid,
        created_at timestamp,
        status text,
        total_amount decimal,
        items list<frozen<order_item>>,
        shipping_address text,
        payment_method text,
        PRIMARY KEY ((user_id), created_at, order_id)
    ) WITH CLUSTERING ORDER BY (created_at DESC);
```
**Ключи**:
- **Partition key**: `user_id` - равномерное распределение пользователей
- **Clustering keys**: `created_at DESC, order_id` - сортировка по дате, уникальность

**Обоснование**:
- Все заказы пользователя в одной партиции
- Быстрое получение последних заказов (DESC сортировка)
- Равномерное распределение: хеш от user_id

#### Таблица 2: Заказы по дате (для аналитики)
```cql
    CREATE TABLE orders_by_date (
        order_date text,  // '2026-01-25'
        order_id uuid,
        user_id uuid,
        total_amount decimal,
        status text,
        geo_zone text,
        PRIMARY KEY ((order_date), order_id)
    ) WITH CLUSTERING ORDER BY (order_id ASC);
```
**Назначение**: Аналитика продаж по дням

### Модель 2: Корзины (carts)

#### Таблица: Активные корзины
```cql
    CREATE TABLE active_carts (
        cart_id uuid,
        user_id uuid,
        session_id text,
        items map<uuid, int>,  // product_id -> quantity
        created_at timestamp,
        updated_at timestamp,
        expires_at timestamp,
        status text,
        PRIMARY KEY ((cart_id))
    );
```

**Дополнительные таблицы для быстрого поиска**:
```cql
    CREATE TABLE carts_by_user (
        user_id uuid,
        cart_id uuid,
        status text,
        created_at timestamp,
        PRIMARY KEY ((user_id), status, created_at)
    ) WITH CLUSTERING ORDER BY (status ASC, created_at DESC);

    CREATE TABLE carts_by_session (
        session_id text,
        cart_id uuid,
        created_at timestamp,
        PRIMARY KEY ((session_id), created_at)
    ) WITH CLUSTERING ORDER BY (created_at DESC);
```

**Обоснование**:
- Основной доступ по cart_id (быстрое обновление)
- Индексы для поиска по user_id и session_id
- TTL на уровне приложения (expires_at)

### Модель 3: История заказов (order_history)

#### Таблица: События заказов
```cql
    CREATE TABLE order_events (
        order_id uuid,
        event_time timestamp,
        event_type text,  // 'created', 'paid', 'shipped', 'delivered'
        event_data text,  // JSON с деталями
        user_id uuid,
        PRIMARY KEY ((order_id), event_time, event_type)
    ) WITH CLUSTERING ORDER BY (event_time ASC);
```

**Обоснование**:
- Все события заказа в одной партиции
- Хронологический порядок событий
- Быстрое восстановление истории заказа

#### Таблица: История по пользователю
```cql
    CREATE TABLE user_order_history (
        user_id uuid,
        order_id uuid,
        event_time timestamp,
        event_type text,
        PRIMARY KEY ((user_id), order_id, event_time)
    ) WITH CLUSTERING ORDER BY (order_id ASC, event_time ASC);
```

### Модель 4: Пользовательские сессии

#### Таблица: Сессии пользователей
```cql
    CREATE TABLE user_sessions (
        session_id text,
        user_id uuid,
        created_at timestamp,
        last_activity timestamp,
        session_data text,  // JSON с данными сессии
        ttl_seconds int,
        PRIMARY KEY ((session_id))
    );
```
**TTL на уровне Cassandra**:
```cql
    INSERT INTO user_sessions (session_id, ...) 
    VALUES ('session123', ...) 
    USING TTL 86400;  // 24 часа
```

### Стратегии предотвращения "горячих" партиций

#### 1. Для заказов:
**Проблема**: Power users с тысячами заказов
**Решение**: Bucketing по временным диапазонам
```cql
    CREATE TABLE orders_by_user_bucketed (
        user_id uuid,
        year_month text,  // '2026-01'
        order_id uuid,
        created_at timestamp,
        -- остальные поля
        PRIMARY KEY ((user_id, year_month), created_at, order_id)
    );
```

#### 2. Для корзин:
**Проблема**: Очень активные сессии
**Решение**: Добавление random suffix
```cql
    CREATE TABLE carts_by_session_suffixed (
        session_id text,
        bucket int,  // 0-9 random
        cart_id uuid,
        -- остальные поля
        PRIMARY KEY ((session_id, bucket), created_at)
    );
```

#### 3. Для событий:
**Проблема**: Popular products в событиях
**Решение**: Использовать UUID вместо product_id

### Преимущества при решардинге
1. **Consistent hashing**: Добавление узлов затрагивает только соседние партиции
2. **No global rebalancing**: Данные не перемещаются по всему кластеру
3. **Incremental scaling**: Можно добавлять узлы постепенно
4. **Predictable performance**: Перформанс деградирует постепенно, не катастрофически

## Задание 10.3: Стратегии обеспечения целостности данных

### Механизмы консистентности в Cassandra

#### 1. **Hinted Handoff**
**Принцип**: Временное хранение записей для недоступных узлов
**Когда использовать**:
- Временные сетевые проблемы
- Кратковременная недоступность узла
- Высокая доступность важнее консистентности

**Настройки**:
    `CREATE TABLE ... WITH hints_window = '3h';`

**Для каких данных**:
- Корзины (carts): Временная недоступность допустима
- Сессии (sessions): Session data можно восстановить
- Заказы (orders): Осторожно, risk of data loss
- История оплат: Не использовать

#### 2. **Read Repair**
**Принцип**: Исправление расхождений при чтении
**Когда использовать**:
- Eventual consistency допустима
- Частое чтение данных
- Нужна высокая производительность записи

**Настройки**:
```cql
    SELECT * FROM orders_by_user 
    WHERE user_id = ? 
    WITH read_repair_chance = 0.1;
```

**Для каких данных**:
- История заказов: Можно исправить при следующем чтении
- Каталог товаров (если в Cassandra): Не критичные данные
- Активные корзины: Осторожно, может быть устаревшая информация
- Статусы оплаты: Не использовать

#### 3. **Anti-Entropy Repair (nodetool repair)**
**Принцип**: Фоновое сравнение и синхронизация данных
**Когда использовать**:
- Критические данные с гарантиями консистентности
- Долгосрочное хранение
- Низкая частота обновлений

**Типы репаров**:
- **Full repair**: Полная проверка (ресурсоемкая)
- **Incremental repair**: Только новые данные
- **Subrange repair**: Только определенные диапазоны

**Для каких данных**:
- Заказы (orders): Критичные данные
- История транзакций: Важно для аудита
- Корзины: Можно использовать incremental repair
- Временные сессии: Не нужно, TTL решит проблему

### Выбор стратегий по сущностям

#### Таблица: Заказы (orders)
**Требования**: Высокая консистентность, надежность
**Стратегии**:
1. **Write**: QUORUM (3 из 5 реплик)
2. **Read**: QUORUM для текущих заказов, ONE для истории
3. **Repair**: 
   - Ежедневный incremental repair
   - Еженедельный subrange repair по user_id
4. **Hinted Handoff**: Отключить (hints_window = 0)
5. **Read Repair**: Высокий шанс (0.3) для активных заказов

#### Таблица: Корзины (carts)
**Требования**: Высокая доступность, session consistency
**Стратегии**:
1. **Write**: ONE (быстрая запись)
2. **Read**: ONE (быстрое чтение)
3. **Repair**: 
   - Read repair при каждом чтении (chance = 1.0)
   - Еженедельный incremental repair
4. **Hinted Handoff**: Включить (hints_window = '1h')
5. **TTL**: Автоматическое удаление через 7 дней

#### Таблица: История заказов (order_history)
**Требования**: Eventual consistency, append-only
**Стратегии**:
1. **Write**: LOCAL_QUORUM (для геораспределения)
2. **Read**: ONE (быстрое чтение истории)
3. **Repair**: 
   - Read repair (chance = 0.1)
   - Ежемесячный full repair
4. **Compaction**: TimeWindowCompactionStrategy для time-series

#### Таблица: Пользовательские сессии
**Требования**: Высокая доступность, temporary data
**Стратегии**:
1. **Write**: ANY (максимальная доступность)
2. **Read**: ONE
3. **Repair**: Только read repair (chance = 0.05)
4. **Hinted Handoff**: Включить (hints_window = '30m')
5. **TTL**: Cassandra TTL (24 часа)

### Компромиссы и рекомендации

#### Latency vs Consistency
```cql
    // Конфигурация по уровням
    const consistencyConfig = {
      CRITICAL: {  // Заказы, оплаты
        write: 'QUORUM',
        read: 'QUORUM',
        repair: 'incremental_daily',
        hints: 'disabled'
      },
      IMPORTANT: {  // Корзины
        write: 'LOCAL_ONE',
        read: 'LOCAL_ONE', 
        repair: 'read_repair_always',
        hints: '1h'
      },
      NORMAL: {  // История, сессии
        write: 'ONE',
        read: 'ONE',
        repair: 'weekly',
        hints: '30m'
      }
    };
```
#### Геораспределенная конфигурация
```cql
    CREATE KEYSPACE mobile_world 
    WITH replication = {
      'class': 'NetworkTopologyStrategy',
      'DC1': 3,  // Основной датацентр
      'DC2': 2   // Резервный датацентр
    };

    // Write/Read по датацентрам
    const geoConfig = {
      write: 'LOCAL_QUORUM',  // Пишем в локальный DC
      read: 'LOCAL_ONE',      // Читаем из локального
      failover: 'EACH_QUORUM' // При сбое локального
    };
```
### Мониторинг и алерты

#### Ключевые метрики
1. **Read/Write latency**: P95 < 10ms, P99 < 50ms
2. **Hinted handoff queue size**: < 1000 записей
3. **Pending compactions**: < 10
4. **Repair progress**: 100% completion weekly
5. **Tombstone ratio**: < 20%

#### Alert правила
    alerts:
      - name: HighPendingHints
        condition: hinted_handoff_queue > 5000
        severity: warning
        
      - name: RepairStalled
        condition: repair_progress_age > 7d
        severity: critical
        
      - name: ConsistencyViolation
        condition: read_repair_fixes_per_hour > 1000
        severity: warning

## Заключение

Миграция на Cassandra решает ключевые проблемы масштабирования:
1. **Leaderless архитектура**: Нет single point of failure
2. **Consistent hashing**: Равномерное распределение без hot spots
3. **Incremental scaling**: Добавление узлов без полного решардинга

Для онлайн-магазина Cassandra идеально подходит для:
- Корзин (высокая запись, session data)
- Заказов (write-heavy, чтение по ключу)
- Сессий (TTL, высокая доступность)
- Истории (time-series, append-only)

Товары и сложные аналитические запросы остаются в MongoDB, обеспечивая гибридную архитектуру, оптимальную для разных типов нагрузки.
