# Настройка чтения с реплик (read preference) и требования к консистентности  
### Для коллекций `products`, `orders`, `carts`

Документ описывает:  
- какие операции читают с **primary**,  
- какие можно отдавать **secondary**,  
- допустимую задержку репликации,  
- обоснование выбора для бизнеса.

---

# 1. Таблица по коллекциям и операциям

## 1.1. Коллекция `products`

| Операция | Описание | Цель чтения | Допустимый лаг | Обоснование |
|----------|-----------|--------------|----------------|-------------|
| Просмотр карточки товара | `findOne({_id})`, показ описания | `SECONDARY_PREFERRED` | 3–5 сек | Описание меняется редко, устаревание не критично, можно разгружать primary |
| Список товаров + фильтры | Каталог, поиск | `SECONDARY` | 3–10 сек | Каталог тяжёлый по чтению, но не критичен по актуальности. На checkout всё перепроверяется на primary |
| Подборки топ-товаров | Агрегации | `SECONDARY` | до 1 мин | Маркетинговые блоки могут быть отложенными |
| Проверка цены/наличия при добавлении в корзину | Перед temp добавлением | `PRIMARY_PREFERRED` | ≤ 1–2 сек | Лёгкий лаг допустим, финальная проверка всё равно на primary |
| Финальная проверка остатков/цены при оформлении заказа | Checkout | `PRIMARY` | 0 сек | Критическая консистентность: нельзя продать отсутствующий товар |
| Админка: редактирование товара | CRUD + чтение | `PRIMARY` | 0 сек | Изменения должны быть видны сразу, строгая консистентность |

## 1.2. Коллекция `orders`

| Операция | Описание | Цель чтения | Допустимый лаг | Обоснование |
|----------|-----------|--------------|----------------|-------------|
| История заказов | `find({user_id})` | `SECONDARY_PREFERRED` | 3–5 сек | Появление нового заказа с задержкой в несколько секунд — допустимо |
| Просмотр деталей заказа | Детальная карточка | `PRIMARY_PREFERRED` | 1–2 сек | Пользователь может увидеть немного устаревший статус, но критичность низкая |
| Проверка актуального статуса заказа | Трекинг в реальном времени | `PRIMARY` | 0–2 сек | Для SLA и точных статусов лучше забирать primary |
| Операции оплаты / подтверждения | Платёжные переходы | `PRIMARY` | 0 сек | Строгая консистентность, нельзя ошибиться со статусом |
| BI-аналитика | Отчёты, агрегаты | `SECONDARY` | минуты–часы | Аналитика не влияет на транзакции, можно полностью увести нагрузку от primary |

---

## 1.3. Коллекция `carts`

| Операция | Описание | Цель чтения | Допустимый лаг | Обоснование |
|----------|-----------|--------------|----------------|-------------|
| Получение активной корзины пользователя | `findOne({user_id, status:'active'})` | `PRIMARY` | 0 сек | Любой лаг приводит к расхождению корзины (товары «пропадают») |
| Получение гостевой корзины | `findOne({session_id, status:'active'})` | `PRIMARY` | 0 сек | UX-критично, корзина должна отображаться моментально и корректно |
| Пересчёт корзины при checkout | Чтение всех items | `PRIMARY` | 0 сек | Строгая консистентность перед созданием заказа |
| Слияние гостевой корзины в пользовательскую | Login merge | `PRIMARY` | 0 сек | Если прочитать устаревшую корзину → потеря товаров |
| Фоновые очистки (TTL) | Batch job | `SECONDARY` | до нескольких минут | Фоновые операции не критичны, можно переносить на реплики |
| BI / отчёты по корзинам | Аналитика | `SECONDARY` | минуты–часы | Не участвует в онлайн-транзакциях |

# 2. Сводка по допустимой задержке репликации

| Тип операции | Требуемая консистентность | Допустимый лаг | Примечание |
|--------------|----------------------------|----------------|------------|
| Жизненно важные операции, влияющие на заказ | строгая | **0 сек** | checkout, остатки, цены, статусы |
| Витрина, каталог, просмотр | слабая | **1–10 сек** | не влияет на транзакции |
| Аналитика, отчёты | eventual | **минуты–часы** | можно полностью вынести на secondary |
| Фоновые задачи | eventual | **минуты** | очередь очистки, TTL, синхронизация |

# 3. Обоснование общего подхода

## Почему часть операций читают с primary?
- **Корзины (`carts`)** — самая изменяемая сущность, малейшая задержка репликации приводит к ошибкам UX и расхождению данных.
- **Оформление заказов (`orders`)** — строгая консистентность, нельзя взять неактуальные остатки или цену.
- **Обновление товарных остатков (`products`)** — важно не допустить «продажи» отсутствующего товара.

## Почему часть операций можно отдавать secondary?
- Каталог (`products`) и карточки товаров изменяются редко → secondary позволяет разгрузить primary.
- История заказов (`orders`) не критична по времени появления нового элемента.
- Аналитика всегда может быть отложенной — она не влияет на онлайн-процессы.

## Почему допустим лаг до 10 секунд для каталога?
- Цель витрины — не строгая транзакционная консистентность.
- Цены и остатки всё равно перепроверяются перед оформлением заказа.
- Разгрузка primary крайне важна при большом трафике.

# 4. Пример стратегии read preference в коде

```
# Каталог — secondary preferred
products_client = MongoClient(
    uri,
    readPreference="secondaryPreferred"
)

# Checkout, корзины — строго primary
orders_client = MongoClient(
    uri,
    readPreference="primary"
)
```

---

# 5. Итог

Разделение операций по уровням консистентности позволяет:

- сохранить UX (корзины и заказы всегда актуальны),
- разгрузить primary на тяжёлых операциях витрины,
- увести аналитику полностью на вторички,
- эффективно использовать replica set как масштабирование по чтению.
