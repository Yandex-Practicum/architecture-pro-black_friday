## Задание 7. Схемы коллекций и шард-ключи

В MongoDB шардирование выбирается на уровне коллекции через `sh.shardCollection(<ns>, <key>)`. Шард-ключ определяет, по какому полю (и каким способом — хешированно или диапазонно) данные разрезаются на чанки и распределяются по шардам.

Для выбора шард-ключа всегда смотрим на:

1. **Кардинальность** — много ли различных значений (иначе мало чанков).
2. **Частота операций по ключу** — должны ли частые запросы быть targeted (попадать на один шард).
3. **Равномерность распределения** — чтобы не возникало горячих шардов.
4. **Монотонность** — монотонно растущий ключ (timestamp, ObjectId) при ranged-шардировании создаёт hotspot на одном шарде.

### 7.1. `products`

**Схема документа:**

```json
{
  "_id": ObjectId("..."),
  "product_id": "SKU-MX-12345",
  "name": "Смартфон Mobile X 128GB",
  "category": "electronics",
  "subcategory": "smartphones",
  "price": { "amount": 49990, "currency": "RUB" },
  "stock_by_zone": {
    "msk": 50,
    "spb": 30,
    "ekb": 12
  },
  "attributes": {
    "color": "black",
    "memory_gb": 128,
    "screen_inch": 6.7
  },
  "created_at": ISODate("..."),
  "updated_at": ISODate("...")
}
```

**Основные операции:**

- Частые `update` остатков при покупках (`$inc` по `stock_by_zone.<zone>`).
- Поиск по `category` + фильтрация по `price.amount`.
- Чтение карточки товара по `product_id` (страница продукта).

**Кандидаты в шард-ключи и их оценка:**

| Кандидат                 | Кардинальность | Распределение                          | Targeted-запросы                 | Проблемы                                    |
| ------------------------ | -------------- | -------------------------------------- | -------------------------------- | ------------------------------------------- |
| `{ category: 1 }`        | низкая (~10)   | сильный перекос — «Электроника» 70%    | поиск по категории — таргет      | **горячий шард на «Электронике»**           |
| `{ _id: 1 }` (ranged)    | максимальная   | монотонный ObjectId → последние хитят 1 шард | страница товара — таргет     | hotspot на новых вставках                   |
| `{ _id: "hashed" }`      | максимальная   | **равномерно**                         | страница товара — таргет         | поиск по категории/цене — scatter-gather    |
| `{ product_id: "hashed" }` | максимальная | равномерно                             | страница товара — таргет         | scatter-gather по category/price            |

**Выбранный ключ: `{ _id: "hashed" }`.**

```javascript
sh.shardCollection("shop.products", { _id: "hashed" })
db.products.createIndex({ category: 1, "price.amount": 1 })
db.products.createIndex({ product_id: 1 }, { unique: true })
```

**Обоснование:**

- Хешированный `_id` даёт идеально равномерное распределение и снимает риск горячего шарда на категории «Электроника» — что важно, так как 70% нагрузки приходится именно на неё (см. [задание 8](#задание-8-горячие-шарды--метрики-и-устранение)).
- Частые `update` остатков идут точечно по `_id` — таргетированно, один шард.
- Чтение страницы товара (`findOne({ _id })`) — таргетировано.
- Платой остаётся scatter-gather для поиска по категории и диапазону цен, но: (а) это сценарий каталога, не критичный к latency; (б) вторичный индекс `{ category, price.amount }` локально на каждом шарде делает фильтрацию эффективной; (в) результаты обычно ограничены пагинацией.

### 7.2. `orders`

**Схема документа:**

```json
{
  "_id": ObjectId("..."),
  "user_id": "user_42",
  "created_at": ISODate("2026-05-20T14:32:00Z"),
  "items": [
    { "product_id": "SKU-MX-12345", "qty": 1, "price": 49990 },
    { "product_id": "SKU-AC-007",   "qty": 2, "price": 1990 }
  ],
  "status": "paid",
  "total": 53970,
  "geo_zone": "msk"
}
```

**Основные операции:**

- Быстрое создание заказа с одновременным списанием остатков (write-heavy).
- Поиск истории заказов конкретного пользователя.
- Отображение статуса заказа (часто — сразу после оформления).

**Кандидаты:**

| Кандидат                                 | Распределение           | История заказов user-а  | Проблемы                                          |
| ---------------------------------------- | ----------------------- | ----------------------- | ------------------------------------------------- |
| `{ _id: "hashed" }`                      | равномерное             | **scatter-gather**      | плохо для «истории заказов»                       |
| `{ user_id: "hashed" }`                  | равномерное по users    | **targeted** (1 шард)   | у супер-активных юзеров — потенциально hot key    |
| `{ user_id: 1, created_at: 1 }` (ranged) | range-запросы по дате   | targeted + сортировка   | новый user без истории — точка вставки горячая    |
| `{ created_at: 1 }`                      | монотонный → hotspot    | scatter-gather          | классический анти-паттерн                         |

**Выбранный ключ: `{ user_id: "hashed" }`.**

```javascript
sh.shardCollection("shop.orders", { user_id: "hashed" })
db.orders.createIndex({ user_id: 1, created_at: -1 })
db.orders.createIndex({ status: 1 })
```

**Обоснование:**

- Хеш по `user_id` даёт равномерное распределение заказов между шардами (пользователей сильно больше, чем заказов одного пользователя).
- Самая частая операция — «история заказов конкретного пользователя» — выполняется как targeted query на одном шарде, что критично для скорости отображения личного кабинета.
- Запись «нового заказа» — точечная вставка по `user_id`, тоже targeted.
- Отображение статуса по `_id` будет scatter-gather, но (а) встречается реже, чем «история», и (б) обычно идёт сразу после создания, когда `user_id` известен.
- Риск hot key на «жирном пользователе» (например, оптовик) есть — для прод-сценария можно расширить ключ до `{ user_id: 1, _id: 1 }`, чтобы внутри одного юзера данные разрезались дальше.

### 7.3. `carts`

**Схема документа:**

```json
{
  "_id": ObjectId("..."),
  "principal_id": "user_42",
  "user_id": "user_42",
  "session_id": null,
  "items": [
    { "product_id": "SKU-MX-12345", "qty": 1 }
  ],
  "status": "active",
  "created_at": ISODate("..."),
  "updated_at": ISODate("..."),
  "expires_at": ISODate("...")
}
```

Поле `principal_id` — синтетический идентификатор «владельца» корзины. Для авторизованного пользователя — это `user_id`, для гостя — `"guest:" + session_id`. Это позволяет иметь единый шард-ключ как для гостевых, так и для пользовательских корзин.

**Основные операции:**

- Создание корзины при заходе нового гостя/пользователя.
- Получение текущей активной корзины по `{ principal_id, status: "active" }`.
- Добавление/удаление товара.
- Слияние гостевой корзины в пользовательскую при логине.
- Отметка корзины как заказанной (`status: "ordered"`).
- TTL-очистка старых корзин по `expires_at`.

**Кандидаты:**

| Кандидат                          | Получение активной корзины | Слияние guest→user             | Распределение            |
| --------------------------------- | -------------------------- | ------------------------------ | ------------------------ |
| `{ _id: "hashed" }`               | scatter-gather             | scatter-gather (читаем guest)  | равномерно               |
| `{ user_id: "hashed" }`           | targeted для user, не работает для гостя | гость на другом шарде | гости не учитываются     |
| `{ session_id: "hashed" }`        | targeted для гостя, не для авторизованного | разные шарды        | для логиненных нет       |
| `{ principal_id: "hashed" }`      | **targeted** для обоих     | гость и user в разных шардах   | **равномерно**           |

**Выбранный ключ: `{ principal_id: "hashed" }`.**

```javascript
sh.shardCollection("shop.carts", { principal_id: "hashed" })
db.carts.createIndex({ principal_id: 1, status: 1 })
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 }) // TTL
```

**Обоснование:**

- Самая частая операция «получить активную корзину» становится targeted query по `principal_id` — это критично для latency открытия страницы корзины.
- Хеширование убирает риск горячих диапазонов (например, сессии, начинающиеся одинаково).
- Слияние гостевой корзины в пользовательскую остаётся scatter-операцией (две записи на разных шардах), но это редкий разовый сценарий и приемлемо.
- TTL-индекс по `expires_at` работает локально на каждом шарде — фоновая очистка распределена.

### 7.4. Сводная таблица

| Коллекция  | Шард-ключ                          | Тип       | Targeted-операции                      |
| ---------- | ---------------------------------- | --------- | -------------------------------------- |
| `products` | `{ _id: "hashed" }`                | hashed    | страница товара, update остатков       |
| `orders`   | `{ user_id: "hashed" }`            | hashed    | история заказов пользователя, вставка  |
| `carts`    | `{ principal_id: "hashed" }`       | hashed    | чтение/обновление активной корзины     |

## Задание 8. Горячие шарды — метрики и устранение

### 8.1. Метрики мониторинга

Для контроля распределения нагрузки между шардами нужно собирать метрики на трёх уровнях.

**A. Распределение данных и чанков (логика балансировщика):**

| Метрика                                | Команда / источник                                            | Когда тревога                                          |
| -------------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------ |
| Кол-во чанков на шард                  | `sh.status()`, `config.chunks` aggregate                      | расхождение между шардами > 8 чанков                   |
| Размер коллекции по шардам             | `db.<coll>.getShardDistribution()`                            | один шард > 1.5× медианы                               |
| Топ jumbo-чанков                       | `db.chunks.find({ jumbo: true })` в `config`                  | любые jumbo — повод к refine shard key / split         |
| Статус балансировщика                  | `sh.getBalancerState()`, `db.adminCommand({balancerStatus:1})` | `inBalancerRound: true` слишком долго — балансер залип |

**B. Нагрузочные метрики каждого узла шарда (mongod):**

| Метрика                       | Источник                                                | Норма                       |
| ----------------------------- | ------------------------------------------------------- | --------------------------- |
| Ops/sec (insert/update/query) | `db.serverStatus().opcounters`, `mongostat`             | сравнивать между шардами    |
| Latency p95/p99               | `db.serverStatus().latencies`, exporter Prometheus      | < 50 мс для большинства ops |
| WiredTiger cache pressure     | `serverStatus().wiredTiger.cache`                       | dirty bytes < 20% cache size|
| Lock %, queue depth           | `serverStatus().globalLock`, `serverStatus().opcounters` | очередь > 100 — критично    |
| Replication lag (secondaries) | `rs.printSecondaryReplicationInfo()`, oplog window      | lag < 1 сек, oplog > 24 ч   |
| CPU / RAM / Disk IOPS / Net   | host-level (node_exporter)                              | CPU < 70%, disk < 80% util  |

**C. Метрики mongos и приложения:**

| Метрика                               | Источник                                            | Зачем                                                  |
| ------------------------------------- | --------------------------------------------------- | ------------------------------------------------------ |
| Targeted vs scatter-gather queries    | `mongos` log с `slowms=0`, `explain("executionStats")` | scatter > 20% — индикатор неоптимального шард-ключа    |
| Cache hit rate в Redis (для cache layer) | `redis-cli INFO stats`                           | падение hit rate → больше нагрузка на MongoDB          |
| Распределение запросов по шардам      | `mongos` metrics из exporter                        | один шард 70% запросов = горячий                       |

**Сборка:** `mongodb_exporter` + Prometheus + Grafana дашборды; алертинг через Alertmanager (PagerDuty / Telegram).

### 8.2. Стратегии устранения дисбаланса

**Шаг 1 — диагностика.** Запросом ниже находим, какие чанки/коллекции вызывают перекос:

```javascript
db.adminCommand({ balancerStatus: 1 })
sh.status({ verbose: true })
db.products.getShardDistribution()
db.chunks.aggregate([
  { $group: { _id: "$shard", count: { $sum: 1 } } },
  { $sort: { count: -1 } }
])
```

**Шаг 2 — встроенные механизмы MongoDB.**

1. **Balancer**. По умолчанию включён, перемещает чанки между шардами. Если выключен — `sh.startBalancer()`. Для прод можно настроить окно балансировки (не в часы пик):

   ```javascript
   db.settings.update(
     { _id: "balancer" },
     { $set: { activeWindow: { start: "02:00", stop: "06:00" } } },
     { upsert: true }
   )
   ```

2. **Pre-split chunks**. Перед заливкой больших объёмов вручную разрезаем диапазоны, чтобы данные сразу легли равномерно:

   ```javascript
   sh.splitAt("shop.products", { _id: <hash> })
   sh.moveChunk("shop.products", { _id: <hash> }, "shard2")
   ```

3. **Refining shard key (MongoDB 4.4+)**. Добавляем дополнительное поле к существующему ключу — увеличивает cardinality, разрезает горячий диапазон без полного решардинга:

   ```javascript
   db.adminCommand({
     refineCollectionShardKey: "shop.orders",
     key: { user_id: "hashed", created_at: 1 }
   })
   ```

4. **Resharding (MongoDB 5.0+)**. Полная смена шард-ключа онлайн. Дорого по ресурсам — применять как «крайнее средство»:

   ```javascript
   db.adminCommand({
     reshardCollection: "shop.orders",
     key: { user_id: "hashed", _id: 1 }
   })
   ```

5. **Zone sharding (tag-aware)**. Привязка диапазонов к конкретным шардам — для геораспределения или для изоляции «горячей» категории на специально настроенных шардах с быстрым железом:

   ```javascript
   sh.addShardTag("shard1", "hot-electronics")
   sh.addTagRange(
     "shop.products",
     { category: "electronics" },
     { category: "electronics￿" },
     "hot-electronics"
   )
   ```

**Шаг 3 — архитектурные меры для категории «Электроника» (70% запросов).**

- Вынести «Электронику» в отдельную физическую коллекцию `products_electronics` с собственным шард-ключом — например, `{ subcategory: "hashed" }` (смартфоны, наушники, ноутбуки и т. д.), чтобы дробить категорию изнутри.
- Усилить кеш-слой Redis для топ-N товаров категории (по просмотрам/корзинам) — это сразу снимает первичный трафик.
- Чтения каталога направить на `secondaryPreferred` (см. [задание 9](#задание-9-read-preference-и-консистентность)) — это распределяет read-нагрузку между primary и двумя secondary внутри реплика-сета.
- Pre-warming кеша для «Чёрной пятницы» — заранее залить в Redis карточки топ-товаров.

**Шаг 4 — постфактум-проверка.** После любого изменения снова смотрим `getShardDistribution()`, метрики ops/sec и latency. Цель — разброс между шардами не более ~10%.

## Задание 9. Read preference и консистентность

В шардированном MongoDB-кластере с репликацией каждый шард — replica set из 1 primary + 2 secondary. Чтения можно отправлять на primary или secondary. Чтение с secondary разгружает primary и увеличивает пропускную способность чтений, но возвращает потенциально устаревшие данные (eventual consistency, replication lag).

Решение по каждой операции принимаем исходя из:

1. Допустима ли «устаревшая» версия данных (риск бизнеса)?
2. Нужен ли read-your-own-writes (пользователь только что записал и должен сразу увидеть результат)?
3. Можно ли мириться с replication lag в ~1–5 секунд?

### 9.1. Сводная таблица

| Коллекция  | Операция                                             | Read preference          | Read concern  | Допустимый lag |
| ---------- | ---------------------------------------------------- | ------------------------ | ------------- | -------------- |
| `products` | Карточка товара (описание, цена, атрибуты)           | `secondaryPreferred`     | `local`       | 5–10 сек       |
| `products` | Каталог: фильтр по категории + диапазон цен          | `secondaryPreferred`     | `local`       | 10–30 сек      |
| `products` | **Проверка остатка при оформлении заказа**           | **`primary`**            | **`majority`**| 0              |
| `products` | Обновление остатка (`$inc stock_by_zone`)            | `primary` (write)        | n/a           | n/a            |
| `orders`   | Создание заказа                                      | `primary` (write)        | n/a           | n/a            |
| `orders`   | Чтение только что созданного заказа (страница «спасибо»)| `primary`             | `majority`    | 0              |
| `orders`   | История заказов пользователя (личный кабинет)        | `secondaryPreferred`     | `local`       | 5 сек          |
| `orders`   | Статус «обработка / в доставке» (фоновый поллинг)    | `secondaryPreferred`     | `local`       | 5 сек          |
| `carts`    | Создание корзины / добавление товара                 | `primary` (write)        | n/a           | n/a            |
| `carts`    | **Получение активной корзины пользователя**          | **`primary`**            | **`majority`**| 0              |
| `carts`    | Слияние гостевой корзины при логине                  | `primary` (read + write) | `majority`    | 0              |
| `carts`    | Отметка корзины как `ordered`                        | `primary` (write)        | n/a           | n/a            |

### 9.2. Обоснование по коллекциям

**products.**

- *Каталог и карточка товара.* Описание, картинка, набор атрибутов меняются редко. Пользователю не страшно, если цена обновилась 8 секунд назад — отображение каталога важнее консистентности. Поэтому `secondaryPreferred`: чтения распределяются по 2 secondary, разгружая primary. Это критично для большого трафика на «Электронику».
- *Проверка остатка при оформлении.* Здесь риск бизнес-логики: если прочитать stock с secondary с лагом 3 секунды, можно «продать» товар, которого уже нет. Поэтому `primary` + `read concern majority`: читаем только то, что закоммичено на большинстве реплик. На самом этапе списания используется атомарный `findOneAndUpdate({ _id, "stock_by_zone.msk": { $gte: qty } }, { $inc: { "stock_by_zone.msk": -qty } })` — это write на primary с гарантией.

**orders.**

- *Только что созданный заказ.* После `POST /orders` пользователь редиректится на страницу «спасибо за заказ». Read-your-own-writes критичен — `primary` + `majority`.
- *История заказов и статус существующего заказа.* Старые заказы не меняются вообще; статус «в обработке» → «отправлено» меняется раз в час-сутки. Лаг в 5 секунд незаметен. `secondaryPreferred` для отображения списка заказов и поллинга статуса.

**carts.**

- *Получение активной корзины.* Самый тонкий момент. Если пользователь только что нажал «добавить в корзину» и при перезагрузке страницы корзины он увидит «корзина пуста» (потому что secondary не успел отреплицировать) — это разрушает доверие. Поэтому всегда `primary` + `majority`.
- *Слияние гостевой при логине.* Двухшаговая транзакция (read guest cart → write user cart → mark guest abandoned). Любое использование secondary внесёт inconsistency. Только `primary`.

### 9.3. Замечания по реализации

Read preference задаётся в драйвере на уровне connection string или каждой операции:

```python
# connection string-level
client = MongoClient("mongodb://mongos_router:27017/?readPreference=primary")

# per-operation
db.products.find({ "category": "electronics" }).read_preference(SecondaryPreferred())
db.products.find_one({ "_id": pid }, read_preference=Primary(), read_concern=ReadConcern("majority"))
```

Replication lag нужно мониторить (`rs.printSecondaryReplicationInfo()`); если для шарда секундный SLA нарушается, читать с secondary опасно даже для «холодных» данных — нужно либо тюнить сеть/диски, либо временно вернуть чтения на primary.

## Задание 10. Миграция на Cassandra

### 10.1. Что именно мигрируем

В Cassandra хорошо ложатся данные с такими свойствами:

- Высокая интенсивность записи и/или чтения.
- Запросы преимущественно по известному ключу (без сложных range-фильтров / JOIN / агрегаций).
- Допустима eventual consistency или нужна настраиваемая (tunable) консистентность.
- Желательна геораспределённость (multi-DC).
- Данные могут расти линейно (timeseries / append-only).

Плохо подходят:

- Сложная аналитика, ad-hoc фильтрация по нескольким полям.
- Сильно транзакционные операции с несколькими сущностями.
- Малые объёмы и низкая нагрузка (оверкилл).

Применительно к «Мобильному миру»:

| Сущность               | Мигрировать в Cassandra? | Почему                                                                                            |
| ---------------------- | ------------------------ | ------------------------------------------------------------------------------------------------- |
| `carts` (корзины)      | **Да**                   | Write-heavy, чтение по ключу `principal_id`, TTL встроенный, легко мульти-DC                      |
| История заказов (closed orders) | **Да**           | Append-only timeseries, чтение по `user_id` + дате, легко шардить, не требует транзакций          |
| User sessions          | **Да**                   | Классический KV с TTL, очень write-heavy, eventual consistency допустима                          |
| `products` (каталог)   | **Нет**                  | Нужны range-запросы по `category` + `price`, частые фильтры — Cassandra плохо это делает; остаётся в MongoDB + Redis |
| Активные `orders` (статус «новый», «оплачен», «в сборке») | **Нет** | Сильная консистентность, транзакционность со списанием остатков — оставляем в MongoDB           |
| Финансы / биллинг      | **Нет**                  | Строгая ACID-семантика — отдельная реляционная БД                                                 |

### 10.2. Концептуальная модель и ключи

В Cassandra при моделировании идём **от запроса**: одна таблица = один шаблон запроса. Дублирование данных между таблицами — нормальная практика.

#### `carts`

Запросы:
- получить активную корзину по `principal_id` (одна актуальная корзина);
- история корзин пользователя (опционально).

```cql
CREATE TABLE shop.carts (
    principal_id text,
    status       text,           -- 'active' | 'ordered' | 'abandoned'
    cart_id      timeuuid,
    items        map<text, int>, -- product_id -> quantity
    created_at   timestamp,
    updated_at   timestamp,
    expires_at   timestamp,
    PRIMARY KEY ((principal_id, status), cart_id)
) WITH CLUSTERING ORDER BY (cart_id DESC)
  AND default_time_to_live = 2592000;   -- 30 дней; per-row TTL может перезаписать

-- активная корзина:
SELECT * FROM shop.carts
WHERE principal_id = ? AND status = 'active'
LIMIT 1;
```

- **Partition key `(principal_id, status)`** — все корзины одного владельца с одним статусом лежат в одной партиции. Получение активной — 1 партиция, 1 запись (с `LIMIT 1`). Равномерное распределение, потому что `principal_id` имеет высокую кардинальность, а Cassandra хеширует токен Murmur3.
- **Clustering `cart_id` (timeuuid DESC)** — последняя по времени корзина первой в партиции.
- **TTL** — встроенный, не нужен фоновый cleanup.

#### `orders_history` (история заказов)

Запросы:
- история заказов пользователя за период (последние N, заказы за месяц);
- по `order_id` напрямую (если знаем).

```cql
-- основная таблица — оптимизирована под "история пользователя"
CREATE TABLE shop.orders_history (
    user_id      text,
    bucket       text,           -- 'YYYY-MM' — разбивка по месяцам
    created_at   timestamp,
    order_id     uuid,
    items        frozen<list<frozen<order_item>>>,
    status       text,
    total        decimal,
    geo_zone     text,
    PRIMARY KEY ((user_id, bucket), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);

-- вспомогательная — поиск по order_id (lookup table)
CREATE TABLE shop.orders_by_id (
    order_id   uuid PRIMARY KEY,
    user_id    text,
    bucket     text,
    created_at timestamp
);
```

- **Partition key `(user_id, bucket)` где `bucket = "YYYY-MM"`** — даёт двойной эффект: (а) равномерное распределение по пользователям; (б) ограничивает размер партиции — даже у самого активного пользователя один месяц редко содержит больше сотен заказов, т. е. партиция не «раздувается» (best practice — < 100 MB / < 100k строк на партицию).
- **Clustering `created_at DESC`** — типичный паттерн «последние N заказов»: первые строки в партиции = свежие заказы, range-scan дешёвый.
- **`orders_by_id`** — компактный индекс «ID → координаты в основной таблице»; нужен, потому что в Cassandra нет вторичных индексов вида «найди по `order_id` через всю таблицу» без полного сканирования.

#### `user_sessions`

```cql
CREATE TABLE shop.user_sessions (
    session_id  uuid PRIMARY KEY,
    user_id     text,
    created_at  timestamp,
    data        text,            -- произвольный сериализованный payload
    user_agent  text,
    ip          inet
) WITH default_time_to_live = 86400;   -- 24 часа
```

- **Partition key `session_id`** — UUID, идеально равномерное распределение.
- TTL очищает старые сессии без вмешательства приложения.

### 10.3. Стратегии целостности и репликации

В Cassandra репликация — leaderless: replication factor `RF=3` (на каждый DC), запись/чтение идут с настраиваемой `consistency level` (CL).

**Стратегии восстановления согласованности:**

1. **Hinted Handoff** — если реплика временно недоступна, координатор сохраняет «hint» (намёк) и отдаёт его, когда реплика вернётся. По умолчанию — 3 часа hint window. Дёшево, фоново, **подходит всем таблицам**.

2. **Read Repair** — при чтении на CL > ONE координатор сравнивает версии с реплик; если есть расхождение, фоном пишет актуальную версию на отстающую реплику. Параметр `read_repair_chance` (или async-механизм в новых версиях).

3. **Anti-Entropy Repair (`nodetool repair`)** — полный «гребень» по дереву Меркла между репликами. Дорого по IO и CPU. **Обязательный для всех таблиц**: запускать раз в `gc_grace_seconds / 2` (по умолчанию `gc_grace_seconds = 10 days`, значит — каждые 5 дней), иначе возможны воскрешения удалённых данных («zombie deletes»).

**Выбор по нашим таблицам:**

| Таблица            | Стратегия                                         | CL запись      | CL чтение       | Обоснование                                                                                          |
| ------------------ | ------------------------------------------------- | -------------- | --------------- | ---------------------------------------------------------------------------------------------------- |
| `carts`            | Hinted Handoff + Read Repair + еженедельный `nodetool repair` | `LOCAL_QUORUM` | `LOCAL_QUORUM`  | Read-your-own-writes критичен (см. задание 9). Активные корзины — небольшой volume, can afford QUORUM. |
| `orders_history`   | Hinted Handoff + еженедельный `nodetool repair`   | `LOCAL_QUORUM` | `LOCAL_ONE`     | На запись — гарантия (заказ нельзя «потерять»). На чтение — быстро, расхождение в истории заказов не страшно; Read Repair дочистит фоном. |
| `user_sessions`    | Hinted Handoff + `nodetool repair`                | `ONE`          | `ONE`           | Сессии eventual consistent, можно жить с лагом; главное — latency, поэтому минимальный CL.            |

**Replication strategy (на keyspace):**

```cql
CREATE KEYSPACE shop
  WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'dc_msk': 3,
    'dc_ekb': 3
  };
```

- `NetworkTopologyStrategy` (а не `SimpleStrategy`) обязательна для прод и multi-DC.
- RF=3 в каждом DC — типовая комбинация: переживает падение 1 узла внутри DC при CL=`LOCAL_QUORUM` (нужно 2 из 3).

**Trade-off CL vs latency:**

- `CL=ONE` — самая низкая latency, но самая слабая консистентность (можно прочитать устаревшую версию).
- `CL=LOCAL_QUORUM` — нужен ответ от большинства реплик в локальном DC; latency выше, но read-your-own-writes гарантируется в пределах DC (если writes тоже идут с `LOCAL_QUORUM`).
- `CL=EACH_QUORUM` — кворум в каждом DC; для критичных записей в multi-DC, но дорого.

**Решардинг и масштабирование.** В отличие от MongoDB, Cassandra не двигает данные крупными чанками. При добавлении ноды она получает свою долю virtual nodes (vnodes) — обычно ~256 на узел — и принимает к себе соответствующие диапазоны токенов. Перемещается ровно `1/N` данных, остальные `(N−1)/N` остаются на месте — нет «свинга» во время балансировки, что и решает исходную проблему «Чёрной пятницы».

### 10.4. Резюме

| Что переносится            | Куда           | Шард-ключ / Partition key       | Read CL         | Write CL        |
| -------------------------- | -------------- | ------------------------------- | --------------- | --------------- |
| `products`, активные orders, биллинг | **Остаётся в MongoDB** | как в [задании 7](#задание-7-схемы-коллекций-и-шард-ключи) | как в [задании 9](#задание-9-read-preference-и-консистентность) | — |
| `carts`                    | Cassandra      | `(principal_id, status)`        | `LOCAL_QUORUM`  | `LOCAL_QUORUM`  |
| `orders_history`           | Cassandra      | `(user_id, bucket)`             | `LOCAL_ONE`     | `LOCAL_QUORUM`  |
| `user_sessions`            | Cassandra      | `session_id`                    | `ONE`           | `ONE`           |

Этот гибрид сохраняет за MongoDB то, что она делает хорошо (богатые запросы по каталогу, транзакционные активные заказы), и отдаёт Cassandra те данные, где её модель (равномерный токен-ринг, leaderless, линейное масштабирование) даёт максимум выгоды.
