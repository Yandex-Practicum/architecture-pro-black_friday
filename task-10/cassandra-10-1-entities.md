# Задание 10.1. Какие данные критичны и куда в Cassandra

При росте нагрузки Mongo с разбиением по диапазонам при добавлении шардов долго перекладывала данные — в пик страдает задержка. Cassandra масштабит иначе (без «перелить сразу всё»), но нельзя ожидать одной классической распределённой SQL-транзакции на «заказ+остаток+оплата везде сразу»; нужен дизайн таблиц и согласование копий на нодах (10.3).

| Сущность | Зачем критично (по-простому) | Заметка |
|----------|-----------------------------|---------|
| `orders` | деньги, нельзя потерять/дублировать, статусы для клиента | заказ = юр. смысл |
| `products` + остаток | нельзя продать больше, чем есть | рассинхрон = претензии |
| `carts` | позиции, смена гостя/логина | потеря = недопродажа |
| История заказов | «мои заказы», споры | в основном чтение |
| Сессия | к корзине, много мелких обращений | сброс = раздражение, не «банкротство» |
| Каталог (текст, картинки) | витрина | куча чтения, мало срочных правок |
| События, клики | аналитика | часть событий можно «не досчитать» по договорённости |

Cassandra к месту: много записей и чтений, много нод, запросы по одному логичному id (пользователь, товар), несколько таблиц под разные экраны, допустима небольшая рассинхронизация копий, если заранее согласована (шкурный риск vs скорость).

Осторожно: весь сценарий «оформить сложный заказ, списать везде, банк» в одной транзакции как в классическом ядре — чаще делают отдельным сервисом и/или другой БД, а в Cassandra кладут уже готовые снимки (как в 10.2).

Логи, сессия, снимок карточки, лента «мои заказы» в Cassandra — норм. Учёт оплаты нередко в другом месте; в Cassandra — уже согласованные факты, которые туда кладёт приложение.
