# Архитектурное решение: Масштабирование и Миграция данных

В данном документе представлены решения по проектированию модели данных, стратегии шардирования и миграции на NoSQL (Cassandra) для обеспечения масштабируемости интернет-магазина "Мобильный мир".

## 1. Проектирование шардирования MongoDB (Задание 7)

Цель: Выбрать ключи и стратегии шардирования для коллекций MongoDB, чтобы обеспечить равномерное распределение нагрузки и данных.

### Коллекция `users` (Заказы)
*   **Shard Key:** `user_id`
*   **Стратегия:** **Hashed Sharding**
*   **Обоснование:**
    *   **Data Locality:** Заказы конкретного пользователя будут храниться на одном шарде. Это оптимизирует самый частый запрос — "Показать мои заказы" (`find({user_id: ...})`), так как `mongos` сможет таргетировать запрос в конкретный шард, а не опрашивать все (Scatter-Gather).
    *   **Равномерность:** Хеширование `user_id` предотвращает создание "горячих" шардов, связанных с временными метками (проблема монотонно возрастающих ключей), и обеспечивает равномерное распределение данных по кластеру.

### Коллекция `products` (Товары)
*   **Shard Key:** `product_id`
*   **Стратегия:** **Hashed Sharding**
*   **Обоснование:**
    *   **Избегание перекосов (Data Skew):** Категории товаров могут иметь разный размер (тысячи чехлов vs десятки ноутбуков). Шардирование по категории создало бы несбалансированные ("Jumbo") чанки. Хеширование по ID гарантирует идеальное распределение.
    *   **Write Distribution:** Операции обновления остатков и цен будут равномерно распределены по всем шардам.
    *   *Примечание:* Поиск по категориям будет использовать Scatter-Gather, но для Read-Heavy нагрузки это компенсируется чтением с реплик (см. Задание 9).

### Коллекция `carts` (Корзины)
*   **Shard Key:** `user_id` (или `owner_id` для поддержки гостевых сессий)
*   **Стратегия:** **Hashed Sharding**
*   **Обоснование:**
    *   Корзина — это приватный ресурс пользователя. Доступ к ней всегда происходит по идентификатору владельца. Хранение корзины на одном шарде обеспечивает максимальную скорость записи и чтения.

## 2. Устранение "горячих" шардов (Задание 8)

**Проблема:** Неравномерная нагрузка на шард из-за популярной категории товаров ("Электроника").

### Метрики мониторинга
Для выявления проблем необходимо отслеживать следующие показатели в разрезе **каждого шарда**:
1.  **Hardware:** CPU Usage, Disk IOPS. (Дисбаланс: на одном шарде 90%, на других 10%).
2.  **MongoDB Specific:**
    *   `opcounters` (Query/Insert/Update per sec) — количество операций.
    *   `currentQueue` — длина очереди операций (рост очереди = перегрузка).
    *   `jumbo_chunks` — количество неделимых больших чанков данных.

### Механизмы устранения
1.  **Chunk Splitting (Разделение):** Принудительное разделение больших чанков (`sh.splitAt()`), чтобы балансировщик мог перенести их части на другие шарды.
2.  **Refining Shard Key:** Если текущий ключ (например, категория) не позволяет дробить данные, необходимо изменить ключ шардирования на более гранулярный (например, `category_id` + `product_id` или просто `product_id` hashed).
3.  **Изоляция нагрузки (Zone Sharding):** Выделение отдельных мощных серверов под "горячие" теги/диапазоны данных и настройка правил зон (`sh.addTagRange()`), чтобы популярные товары жили на dedicated железе.

## 3. Репликация и Консистентность (Задание 9)

Выбор стратегии чтения (`Read Preference`) балансирует между актуальностью данных и скоростью.

| Коллекция | Операция | Read Preference | Обоснование |
| :--- | :--- | :--- | :--- |
| **Carts** | Просмотр корзины | **Primary** | **Критичен UX.** Пользователь должен видеть добавленный товар мгновенно. "Исчезновение" товара из-за лага репликации недопустимо. |
| **Orders** | Создание/Оплата | **Primary** | **Финансовая точность.** Нельзя допустить создание дублей или оплату несуществующего заказа. |
| **Orders** | История заказов | **PrimaryPreferred** | Пользователь ожидает увидеть только что сделанный заказ в истории. Лаг здесь нежелателен. |
| **Products** | Каталог/Листинг | **Secondary** (maxStaleness: 120s) | **Скорость и разгрузка мастера.** Если пользователь увидит старую цену на 5 минут дольше — это не критично. Важнее быстрый отклик каталога. |
| **Products** | Чекаут (Списание) | **Primary** | Проверка остатков должна быть строгой, чтобы не продать отсутствующий товар (Overselling). |

## 4. Миграция на Cassandra (Задание 10)

В связи с пиковыми нагрузками ("Черная пятница") принято решение мигрировать часть подсистем на **Apache Cassandra** (Leaderless архитектура) для обеспечения высокой доступности (High Availability) и скорости записи.

### 10.1. Выбор данных для миграции
**Кандидат:** Коллекция **`carts` (Корзины)**.
*   **Почему:**
    *   **Write-Heavy:** Огромное количество операций записи/обновления.
    *   **Availability > Consistency:** В пик нагрузке важнее, чтобы корзина открылась (даже если там не хватает одной позиции), чем чтобы сервис упал с ошибкой.
    *   **Simple Query Pattern:** Доступ всегда идет по ключу пользователя (Key-Value access).
    *   **No Transactions:** Для корзины не требуются сложные ACID транзакции.

### 10.2. Модель данных (CQL)

В Cassandra моделирование происходит "от запроса". Основной запрос: *Получить содержимое корзины пользователя*.

```sql
CREATE TABLE cart_items (
    user_id uuid,          -- Partition Key
    product_id uuid,       -- Clustering Key
    quantity int,
    price decimal,
    added_at timestamp,
    PRIMARY KEY ((user_id), product_id)
) WITH CLUSTERING ORDER BY (product_id ASC);
```

*   **Partition Key (`user_id`):** Гарантирует, что вся корзина пользователя лежит на одной физической ноде. Это обеспечивает быстрое чтение и запись.
*   **Clustering Key (`product_id`):** Позволяет быстро находить, обновлять или удалять конкретный товар в корзине без перезаписи всей партиции.

### 10.3. Стратегии консистентности

Для обеспечения надежности и целостности данных в Leaderless среде применяются:

1.  **Hinted Handoff:**
    *   *Применение:* Включено всегда.
    *   *Зачем:* Если узел-владелец корзины временно недоступен (сетевой сбой), координатор сохранит "подсказку". Как только узел вернется, данные будут дозаписаны. Это критично для сохранения товаров в корзине при сбоях.

2.  **Read Repair (Восстановление при чтении):**
    *   *Применение:* Активно используется.
    *   *Зачем:* Корзина — часто читаемый объект (при каждом обновлении страницы). Механизм Read Repair будет автоматически синхронизировать реплики при чтении, обеспечивая актуальность данных без запуска "тяжелых" фоновых процессов.

3.  **Anti-Entropy (Nodetool Repair):**
    *   *Применение:* Регулярный запуск (например, еженедельно) для синхронизации "холодных" данных (брошенных корзин), которые давно не читались.

**Consistency Level (CL):**
*   **Запись:** `QUORUM` (баланс надежности) или `ONE` (в пик нагрузки для максимальной скорости, полагаясь на Hinted Handoff).
*   **Чтение:** `QUORUM` (чтобы исключить чтение устаревших данных).
