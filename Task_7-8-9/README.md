# Шардирование

## Потенциальные кандидаты для шард-ключей

### Для коллекции "products":

- "category" - так как запросы на выборку товаров по категории могут быть частыми.
- "product_id" - уникальный идентификатор товара, который также может быть использован для равномерного распределения данных.

### Для коллекции "orders":

- "customer_id" - так как запросы на историю заказов конкретного пользователя могут быть частыми.
- "order_date" - дата оформления заказа, если запросы часто выполняются по временному интервалу.

### Для коллекции "carts":

- "user_id" - идентификатор пользователя, так как запросы на текущую корзину пользователя могут быть частыми.
- "session_id" - идентификатор сессии для гостевых корзин, если запросы на такие корзины также часто выполняются.

## Стратегии шардирования

### Коллекция "products":

Стратегия шардирования: Хеширование
Шард-ключ: "category"
При выборе стратегии хеширования по атрибуту "category" данные о товарах будут равномерно распределены по шардам в зависимости от их категории. Это позволит эффективно распределять запросы на выборку товаров по категориям и фильтрацию по диапазону цен.

### Коллекция "orders":

Стратегия шардирования: Диапазон
Шард-ключ: "customer_id"
Выбор стратегии диапазонного шардирования по атрибуту "customer_id" позволит хранить данные о заказах конкретного пользователя на одном шарде, что обеспечит быстрый доступ к истории заказов клиента.

### Коллекция "carts":

Стратегия шардирования: Хеширование
Шард-ключ: "user_id"
Хеширование по атрибуту "user_id" позволит хранить данные о корзинах конкретного пользователя на одном шарде, что облегчит доступ к текущей корзине пользователя и операциям добавления, удаления товаров.

Выбранные стратегии шардирования и шард-ключи соответствуют особенностям коллекций и операциям, которые часто выполняются в приложении. Они обеспечат эффективное распределение данных по шардам и быстрый доступ к необходимой информации.

## Примеры команд MongoDB

Создание заказа в коллекции "orders" с одновременным списанием остатков:

```mongodb
db.orders.insertOne({
    "order_id": "12345",
    "customer_id": "98765",
    "order_date": ISODate("2022-01-01T00:00:00Z"),
    "ordered_items": [
        {
            "product_id": "ABCDE",
            "quantity": 2,
            "price": 10
        },
        {
            "product_id": "FGHIJ",
            "quantity": 1,
            "price": 20
        }
    ],
    "status": "pending",
    "total_amount": 40,
    "geo_zone": "Moscow"
});

db.products.updateOne(
    { "product_id": "ABCDE" },
    { $inc: { "stock.Moscow": -2 } }
);

db.products.updateOne(
    { "product_id": "FGHIJ" },
    { $inc: { "stock.Moscow": -1 } }
);
```

Поиск истории заказов конкретного пользователя в коллекции "orders":
```mongodb
db.orders.find({ customer_id: "789" })
```

Отображение статуса заказа в коллекции "orders":
```mongodb
db.orders.find({ order_id: "123456" }, { status: 1 })
```

Обновление остатков товаров при покупках в коллекции "products":
```mongodb
db.products.updateOne(
  { product_id: "abc123" },
  { $inc: { stock.Moscow: -2 } }
)
```

Поиск товаров по категориям и фильтрация по диапазону цен в коллекции "products":
```mongodb
db.products.find({ category: "Electronics", price: { $gte: 100, $lte: 500 } })
```

Отображение описания товара на странице продукта в коллекции "products":
```mongodb
db.products.find({ product_id: "abc123" }, { name: 1, attributes: 1 })
```

Создание корзины для нового пользователя в коллекции "carts":
```mongodb
db.carts.insertOne({
    "user_id": "guest123",
    "session_id": "session456",
    "items": [],
    "status": "active",
    "created_at": ISODate(),
    "updated_at": ISODate(),
    "expires_at": ISODate("2022-12-31T23:59:59Z")
});
```

Получение текущей корзины пользователя в коллекции "carts":
```mongodb
db.carts.findOne({ "session_id": "session456", "status": "active" });
```
или 
```mongodb
db.carts.findOne({ "user_id": "guest123", "status": "active" });
```

Добавление или замена товара в корзине в коллекции "carts":
```mongodb
db.carts.updateOne(
    { "user_id": "guest123", "status": "active" },
    { $push: { "items": { "product_id": "ABCDE", "quantity": 2 } } },
    { upsert: true }
);
```

Удаление товара из корзины в коллекции "carts":
```mongodb
db.carts.updateOne(
    { "user_id": "guest123", "status": "active" },
    { $pull: { "items": { "product_id": "FGHIJ" } } }
);
```

Слияние гостевой корзины в пользовательскую, если пользователь залогинится:
```mongodb
var guestCart = db.carts.findOne({ "session_id": "session456", "status": "active" });
db.carts.updateOne(
    { "user_id": "user789", "status": "active" },
    { $push: { "items": { $each: guestCart.items } } }
);
db.carts.updateOne(
    { "session_id": "session456", "status": "active" },
    { $set: { "status": "abandoned" } }
);
```

Отметка корзины как заказанной:
```mongodb
db.carts.updateOne(
    { "user_id": "user789", "status": "active" },
    { $set: { "status": "ordered" } }
);
```

## Выявление и устранение «горячих» шардов

### Метрики мониторинга состояния шардов:

- Процент запросов к каждому шарду по категориям товаров.
- Общее количество запросов к каждому шарду.
- Статистика по времени ответа на запросы к каждому шарду.
- Количество записей в каждом шарде по категориям товаров.
- Статус использования ресурсов (память, процессор) на каждом шарде.
- Количество запросов, которые привели к перегрузке шарда.

### Механизмы автоматического перераспределения данных:

- Реализация скрипта мониторинга, который будет анализировать метрики и определять "горячие" шарды.
- При обнаружении перегруженного шарда, скрипт должен запускать механизм автоматического перераспределения данных.
- Механизм перераспределения может включать перемещение данных из перегруженного шарда на другие шарды, чтобы равномерно распределить нагрузку.
- Перераспределение данных должно происходить без прерывания работы приложения, возможно с использованием функционала MongoDB для балансировки данных.

### Предложения по предотвращению "горячих" шардов:

- Разработка стратегии шардирования, учитывающей популярность категорий товаров. Например, использование составного шард-ключа, который включает категорию товара.
- Мониторинг нагрузки и запросов на регулярной основе для оперативного выявления проблемных шардов.
- Оптимизация запросов и индексов для равномерного распределения нагрузки на шарды.
- Внедрение кэширования запросов к "горячим" шардам для снижения нагрузки.

## Примеры команд MongoDB

**Мониторинг запросов к шардам:**
```mongodb
db.currentOp(true)
```
Эта команда позволяет просмотреть текущие операции в MongoDB, включая запросы к шардам. Можно анализировать количество запросов к каждому шарду и выявлять перегруженные шарды.

**Мониторинг использования ресурсов шардов:**
```mongodb
db.serverStatus()
```
Эта команда позволяет получить информацию о состоянии сервера MongoDB, включая использование памяти, процессора и других ресурсов. Можно анализировать загрузку шардов и выявлять перегруженные шарды.

**Перераспределение данных в MongoDB:**
```mongodb
sh.moveChunk("database.collection", { shardKeyField: value }, "destinationShard")
```
Эта команда позволяет перемещать части данных (chunk) между шардами в MongoDB. Путем перемещения данных из "горячего" шарда на другие шарды можно устранить перегрузку.

**Настройка балансировки данных:**
```mongodb
sh.setBalancerState(true)
```
Эта команда включает балансировку данных в кластере MongoDB. При включенной балансировке MongoDB автоматически перемещает данные между шардами для равномерного распределения нагрузки.

**Настройка алгоритма балансировки:**
```mongodb
sh.setBalancerConfig({ _id: "balancer", activeWindow: { start: "00:00", stop: "23:59" } })
```
Эта команда позволяет настроить временное окно активности балансировки данных. Например, можно ограничить балансировку в периоды низкой нагрузки на систему.

# Настройка чтения с реплик и консистентность

| Коллекция	| Операции чтения на secondary | Операции чтения на primary	|Допустимая задержка репликации | 
|:--|:--|:--|:--|
|products|Чтение товаров по запросам, не связанным с обновлением остатков|Чтение товаров при обновлении остатков|Незначительная задержка, не более нескольких секунд|
|orders	|Чтение информации о заказах для отображения пользователю|Чтение и обновление информации о заказах|Незначительная задержка, не более нескольких секунд|
|carts|Чтение и обновление корзин пользователей|Чтение и обновление корзин пользователей|Незначительная задержка, не более нескольких секунд|	
		
## Обоснование выбора

### products

**Операции чтения на secondary:** запросы, не связанные с обновлением остатков товаров, могут использовать secondary реплику, так как они не влияют на целостность данных и не требуют моментальной консистентности.
**Операции чтения на primary:** чтение товаров при обновлении остатков должно происходить только на primary, чтобы избежать возможных конфликтов и гарантировать актуальность данных.
**Допустимая задержка репликации:** незначительная задержка в несколько секунд допустима для secondary, чтобы обеспечить обновление данных.

### orders

**Операции чтения на secondary:** чтение информации о заказах для отображения пользователя может использовать secondary, так как это не критичные данные, которые могут быть немного неактуальны.
**Операции чтения на primary:** чтение и обновление информации о заказах должны происходить только на primary для обеспечения консистентности и избежания конфликтов.
**Допустимая задержка репликации:** незначительная задержка в несколько секунд допустима для secondary.

### carts

**Операции чтения на secondary:** чтение содержимого корзин пользователей могут использовать secondary, так как это не критичные данные и небольшая задержка допустима.
**Операции чтения на primary:** чтение статуса корзин пользователей должны происходить на primary для обеспечения консистентности и избежания проблем с параллельными изменениями.
**Допустимая задержка репликации:** незначительная задержка в несколько секунд допустима для secondary.

Таким образом, операции чтения, которые не требуют мгновенной консистентности данных, могут использовать secondary реплику, с допустимой задержкой в несколько секунд, чтобы обеспечить эффективность и масштабируемость системы, при этом важные операции чтения должны происходить только на primary.
