# Задание 10. Cassandra для интернет-магазина

## 10.1. Анализ данных и обоснование применения Cassandra

### Классификация данных по критичности

| Сущность | Целостность | Скорость записи | Скорость чтения | Геораспределённость |
|----------|-------------|-----------------|-----------------|---------------------|
| Товары (остатки) | **Критическая** — продажа несуществующего товара | Высокая (каждая покупка) | Высокая (каталог) | Средняя |
| Заказы (активные) | **Критическая** — потеря заказа, двойное списание | Высокая (Black Friday) | Средняя | Средняя |
| Корзины | Средняя — потеря корзины неприятна, но не фатальна | Очень высокая (каждое действие) | Очень высокая | Высокая |
| История заказов | Низкая — данные неизменяемые | Низкая (append-only) | Средняя (по запросу) | Высокая |
| Пользовательские сессии | Низкая — можно пересоздать | Очень высокая | Очень высокая | **Критическая** |

### Где Cassandra имеет смысл

| Сущность | Подходит для Cassandra? | Обоснование |
|----------|------------------------|-------------|
| **Пользовательские сессии** | **Да** | Идеальный кейс: огромный объём записи, TTL из коробки, eventual consistency допустима (пересоздать сессию не страшно), геораспределённость критична — пользователи из разных регионов |
| **История заказов** | **Да** | Append-only данные, не изменяются после создания. Партиционирование по `customer_id` — естественный паттерн доступа. Cassandra эффективна для time-series данных с кластерной сортировкой по дате |
| **Корзины** | **Да** | Высокая частота записи/чтения, короткоживущие данные с TTL, eventual consistency допустима (пользователь увидит свой товар при следующем обновлении страницы) |
| **Товары (остатки)** | **Нет** | Списание остатков требует строгой консистентности и атомарных операций (compare-and-set). В Cassandra LWT (lightweight transactions) работают медленно и не масштабируются — MongoDB с `findOneAndUpdate` подходит лучше |
| **Заказы (активные)** | **Нет** | Создание заказа — транзакция (заказ + списание остатков + обновление корзины). Cassandra не поддерживает multi-partition транзакции. Потеря или дублирование заказа — критический бизнес-риск |

## 10.2. Концептуальная модель данных

### Пользовательские сессии

```sql
CREATE TABLE sessions (
    session_id  text,
    user_id     text,
    geo_zone    text,
    data        map<text, text>,
    created_at  timestamp,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 3600
  AND compaction = {'class': 'TimeWindowCompactionStrategy', 'compaction_window_unit': 'HOURS', 'compaction_window_size': 1};
```

**Partition key: `session_id`**
- Высокая кардинальность — каждая сессия уникальна, данные распределяются равномерно по кластеру
- Все операции — точечный доступ по `session_id` (get/set/delete), идеально ложится на одну партицию
- TTL = 1 час — автоматическая очистка, нет роста данных
- Горячие партиции невозможны: одна сессия = один пользователь = минимальная нагрузка на партицию

### История заказов

```sql
CREATE TYPE order_item (
    product_id  text,
    name        text,
    quantity    int,
    price       decimal
);

CREATE TABLE orders_by_customer (
    customer_id text,
    year_month  text,
    created_at  timestamp,
    order_id    uuid,
    items       list<frozen<order_item>>,
    status      text,
    total       decimal,
    geo_zone    text,
    PRIMARY KEY ((customer_id, year_month), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
```

**Partition key: `(customer_id, year_month)` (составной)**
- `customer_id` один давал бы неограниченный рост партиции для активных покупателей
- Добавление `year_month` (например, `"2026-05"`) ограничивает размер партиции: один клиент за месяц сделает максимум сотни заказов — партиция компактная
- Запрос «история за апрель» → одна партиция, без scatter

**Clustering key: `(created_at DESC, order_id)`**
- Сортировка по дате: последние заказы возвращаются первыми без дополнительной сортировки
- `order_id` — уникальность строки при совпадении `created_at`

**Горячие партиции:** маловероятны. Даже VIP-клиент генерирует ограниченное число заказов в месяц. При решардинге `year_month` обеспечивает гранулярность — чанки распределяются равномерно.

### Корзины

```sql
CREATE TYPE cart_item (
    product_id  text,
    quantity    int
);

CREATE TABLE carts (
    cart_id     uuid,
    user_id     text,
    session_id  text,
    status      text,
    items       list<frozen<cart_item>>,
    created_at  timestamp,
    updated_at  timestamp,
    PRIMARY KEY (cart_id)
) WITH default_time_to_live = 604800;

-- Таблица-индекс: поиск активной корзины по user_id
CREATE TABLE active_cart_by_user (
    user_id   text,
    cart_id   uuid,
    PRIMARY KEY (user_id)
);

-- Таблица-индекс: поиск активной корзины по session_id
CREATE TABLE active_cart_by_session (
    session_id  text,
    cart_id     uuid,
    PRIMARY KEY (session_id)
);
```

**Partition key: `cart_id` (UUID)**
- Максимальная равномерность распределения — UUID гарантирует уникальность и случайность
- Все CRUD-операции с корзиной по `cart_id` — одна партиция
- Lookup-таблицы (`active_cart_by_user`, `active_cart_by_session`) — денормализация в стиле Cassandra, каждая тоже с высокой кардинальностью ключа

**Горячие партиции:** невозможны — одна корзина = один пользователь, нагрузка на партицию минимальна. TTL 7 дней — объём данных стабилен.

**Решардинг:** UUID распределяется через consistent hashing равномерно. При добавлении/удалении узлов перемещается минимум данных (~1/N от общего объёма).

## 10.3. Стратегии обеспечения целостности данных

### Обзор стратегий

| Стратегия | Как работает | Влияние на latency | Когда данные станут консистентными |
|-----------|-------------|--------------------|------------------------------------|
| **Hinted Handoff** | Координатор сохраняет запись для недоступного узла и отправляет при его восстановлении | Не влияет на клиента | После восстановления узла (обычно минуты) |
| **Read Repair** | При чтении координатор сравнивает ответы реплик и обновляет устаревшие | Добавляет latency к чтению | При следующем чтении |
| **Anti-Entropy Repair** | Фоновый процесс (`nodetool repair`) сравнивает Merkle-деревья реплик | Не влияет на клиентские запросы | По расписанию (часы/дни) |

### Выбор стратегий по сущностям

#### Пользовательские сессии

| Стратегия | Применять? | Обоснование |
|-----------|-----------|-------------|
| Hinted Handoff | **Да** | Основной механизм. Сессия короткоживущая (TTL 1 час). Если узел упал, hint доставит данные при восстановлении. Если hint не успел — сессия уже протухла, потеря некритична |
| Read Repair | **Нет** | Добавляет latency на каждое чтение, а сессии читаются очень часто. Компромисс не оправдан: сессия либо есть, либо пользователь логинится заново |
| Anti-Entropy Repair | **Нет** | Бессмысленно — данные с TTL 1 час удалятся раньше, чем запустится repair |

**Consistency Level:** `CL=ONE` для записи и чтения — минимальная latency, потеря сессии не критична.

#### История заказов

| Стратегия | Применять? | Обоснование |
|-----------|-----------|-------------|
| Hinted Handoff | **Да** | Базовая защита при кратковременной недоступности узла. Заказ записывается один раз и должен сохраниться |
| Read Repair | **Да** | Заказы читаются пользователем при просмотре истории. Read Repair исправит расхождения при обращении. Дополнительная latency (единицы мс) допустима — пользователь просматривает историю нечасто |
| Anti-Entropy Repair | **Да, по расписанию** | Старые заказы могут не читаться месяцами — Read Repair не сработает. Запуск `nodetool repair` раз в неделю гарантирует, что данные не потеряются при замене узлов или после длительных сбоев |

**Consistency Level:** `CL=QUORUM` для записи (заказ не должен потеряться), `CL=ONE` для чтения (допустима eventual consistency при просмотре истории).

#### Корзины

| Стратегия | Применять? | Обоснование |
|-----------|-----------|-------------|
| Hinted Handoff | **Да** | Основной механизм. Корзина обновляется часто, hints обеспечат доставку при кратковременных сбоях |
| Read Repair | **Да** | Корзина активно читается (каждое открытие страницы). Read Repair обнаружит расхождения между репликами при чтении. Дополнительная latency минимальна (1–2 мс), но корзина будет актуальной |
| Anti-Entropy Repair | **Нет** | Корзины короткоживущие (TTL 7 дней) и постоянно читаются — Read Repair достаточен. Anti-Entropy Repair не даёт выигрыша для данных, которые и так часто проверяются при чтении |

**Consistency Level:** `CL=LOCAL_QUORUM` для записи (не потерять содержимое корзины), `CL=LOCAL_ONE` для чтения (минимальная latency, Read Repair исправит при необходимости).

### Сводная таблица

| Сущность | Hinted Handoff | Read Repair | Anti-Entropy Repair | Write CL | Read CL |
|----------|:-:|:-:|:-:|----------|---------|
| Сессии | + | — | — | ONE | ONE |
| История заказов | + | + | + (еженедельно) | QUORUM | ONE |
| Корзины | + | + | — | LOCAL_QUORUM | LOCAL_ONE |
