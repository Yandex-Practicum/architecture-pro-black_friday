Применение Cassandra имеет смысл:

- *Корзины*
Огромное количество мелких записей и обновлений. Корзины часто создаются и бросаются.

    Преимущества Cassandra:
    - TTL (Time-To-Live): Каждая строка корзины автоматически удалится через 7 дней без нагрузки на БД (в отличие от фоновых процессов удаления в Монге).
    - Скорость записи
    - Геораспределенность: Если магазин глобальный, корзина пользователя будет доступна в ближайшем к нему дата-центре с минимальной задержкой.

- *История заказов* 
    После того как заказ завершен (статус "Доставлен"), он больше не меняется. Это архивные данные, объем которых растет и не мизменяется
    
    Преимущества Cassandra:
    - Масштабируемость: Когда заказов станет миллиард,  просто добавляется новый узел в кластер Cassandra.
    -  Данные физически лежат на диске отсортированными по дате, что делает открытие страницы "История заказов" мгновенным.

Cassandra не подйодёт:

- Актуальные остатки 

    Причина в Eventually Consistent системе. Два пользователя в разных регионах могут одновременно "купить" последний товар, и из-за задержки репликации (лаг Paxos/LWT) система одобрит оба заказа.

- Управление каталогом (Product Management)

    В Cassandra для каждого фильтра придется строить отдельную таблицу.

    ## корзина

### структура данных и распределение

- Partition Key: user_id (UUID или Session ID).
- Clustering Columns: added_at (timestamp, DESC), product_id (uuid).

```sql
CREATE TABLE carts_by_user (
    user_id uuid,
    added_at timestamp,
    product_id uuid,
    quantity int,
    price_snapshot decimal,
    PRIMARY KEY ((user_id), added_at, product_id)
) WITH CLUSTERING ORDER BY (added_at DESC);
```

- Равномерное распределение: Хэш от ID пользователя разносит корзины равномерно по всем узлам кольца.

- Эффективный доступ: Все товары одной корзины физически лежат в одном разделе  на диске. Чтение всей корзины — это одна последовательная операция ввода-вывода 

- Горячие партиции: Риск минимален, так как один пользователь физически не может генерировать нагрузку, способную «положить» узел.

### Стратегии: 

**Hinted Handoff + Read Repair (при долгом отсутствии реплики)**

Для корзины критична Latency. Пользователь не должен ждать, пока база сверяет контрольные суммы на трех узлах.

При недоступности одного узла Hinted Handoff восстановит товар в корзину через секунду.А если данные разошлись, Read Repair исправит это при следующем открытии страницы корзины.

Как компромисс мы допускаем, что в малом проценте случаев пользователь может увидеть старый состав корзины на долю секунды ради правктически мгновенного отклика.

## история заказов

разделим процесс создания заказа (MongoDB/SQL) и хранение истории (Cassandra).

### структура данных и распределение

- Partition Key: user_id.
- Clustering Columns: created_at (timestamp, DESC), order_id (uuid).

```sql
CREATE TABLE orders_history_by_user (
    user_id uuid,
    created_at timestamp,
    order_id uuid,
    items_blob text, -- JSON со списком товаров из монги
    total_amount decimal,
    status text,
    PRIMARY KEY ((user_id), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

- Равномерное распределение: Аналогично корзинам, заказы распределяются по user_id. 

- Эффективный доступ: CLUSTERING ORDER BY (created_at DESC) позволяет мгновенно отдавать «последние 10 заказов» для личного кабинета без сортировки в памяти.

### Стратегии: 

**Read Repair + Anti-Entropy Repair**

Это архивные данные. Здесь важна целостность. Используем Read Repair в блокирующем режиме: перед ответом пользователю база должна убедиться, что данные на репликах совпадают.
Периодически (арз в неделю) запускаем Anti-Entropy Repair, чтобы вычистить старые "мертвые" записи и гарантировать, что ни один старый заказ не пропал.

Компромисс: Чтение истории может быть чуть медленнее, но данные будут актуальны
