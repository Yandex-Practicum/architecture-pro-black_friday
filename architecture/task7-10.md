# Задание 7:

## Описание схемы коллекций.

### Коллекция заказы:

#### Описание:

```
orders: {
  _id: ObjectId,
  user_id: "u123",
  created_at: ISODate("2026-02-25T12:34:56Z"),
  geo_zone: "MSK",
  status: "created" | "paid" | "shipped" | "delivered" | "cancelled",
  items: [
    {
      product_id: ObjectId,
      title: "Смартфон X",
      category: "electronics",
      unit_price: 49990,
      quantity: 1,
      amount: 49990
    }
  ],
  total_amount: 49990,
}
```

Кандидат на шард-ключ: `user_id`.
Стратегия кеширования: хеширование.
Ключ и стратегия выбраны из-за большого количества пользователей, что позволит равномерно распределить заказы по всем
шардам.

#### Индексы:

1. Оптимизация поиска заказов пользователя:
   ```javascript
    db.orders.createIndex({ user_id: 1, created_at: -1 })
    ```

### Товары:

#### Описание:

```
products: {
_id: ObjectId,
title: "Смартфон X",
category: "electronics",
price: 49990,
attrs: { color: "black", size: "128gb" },

stock_by_zone:{
    "EKB": 50,
    "KGD": 30,
    "MSK": 120
  },
}
```

Кандидат на шард-ключ: `category+price`.
Стратегия кеширования: диапазонная.
Ключ и стратегия выбраны из-за того, что ожидается часты поиск товаров по категориям и фильтрация по диапазону цен.
Явным недостатком будет стратегии будет возможное появление "горячих" шардов.

#### Индексы:

1. Оптимизация поиска товаров:
   ```javascript
    db.products.createIndex({ category: 1, price: 1 })
    ```

### Коллекция корзины:

#### Описание:

```
carts: {
_id: ObjectId,
user_id: "u123" | null,
session_id: "s-abc" | null,
owner_key: "u:u123" | "s:s-abc",
status: "active" | "ordered" | "abandoned",
items: [
    {
      product_id: ObjectId,
      quantity: 2, 
      added_at: ISODate(...) 
    }
],
created_at: ISODate(...),
updated_at: ISODate(...),
expires_at: ISODate(...)
}
```

Кандидат на шард-ключ: `owner_key`.
Стратегия кеширования: хеширование.
Отдельный атрибут создан для простоты расчёта ключа. Данная стратегия позволит быстро искать корзину конкретного
пользователя, а судя по описанию основная масса операций будет начинаться с поиска корзины пользователя. Также данный
шард-ключ обеспечит высокую кардинальность данных.

#### Индексы:

1. Оптимизация поиска корзины для конкретного владельца:
   ```javascript
    db.carts.createIndex({ owner_key: 1, status: 1 })
    ```
2. Оптимизация поиска устаревших корзины:
   ```javascript
    db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
    ```

# Задание 8:

## Предлагаемое решение:

### Добавляемые метрики:

1. Добавить метрики по каждому шарду: CPU/RAM/Network/Disk I/O, ops/sec (read/write) и p95/p99 latency,
   connections/queue - чтобы детектить “горячий” шард как перекос нагрузки.
2. Метрики распределения данных по products: размер данных и число чанков на шард, чтобы видеть перекос нагрузки и
   эффективность балансировки.

### Механизм автоматического перераспределения данных:

Если видим, что диапазоны делают шард горячим, мы увеличиваем дробность внутри этой категории, чтобы балансер мог
разнести нагрузку.
Что автоматизировать:

1. Обнаружение горячих промежутков (например, electronics + price [X..Y] даёт высокий QPS/latency на одном шарде)
2. Инициировать split на границах цен.
3. Форсирование перемещения чанка после разделения в менее загруженный шард.
4. Миграцию осуществлять только при низкой нагрузке (p95 latency меньше порогового значения).

# Задание 10:

### Задание 10.1:

Заказы: критично не потерять заказ и не допустить некорректных состояний (например, "оплачен, но нет позиции", "дубли
оплаты"). Статусы могут быть eventual, но запись заказа - должна быть надёжной.
Товары: критична целостность остатков в момент оформления заказа, а не описания/категории.
Корзина: важна согласованность read-your-writes для UX (пользователь сразу видит добавленное), но глобальная строгая
консистентность обычно не нужна.
История заказов: почти всегда eventual.
Пользовательские сессии: важна скорость + read-your-writes, строгая консистентность обычно не требуется.

Применение Cassandra имеет смысл для:

1. Заказы. Заказы уместно хранить в Cassandra; консистентность на критичных операциях обеспечивается выбором CL
   (например строгий кворум), а eventual допускается для отображения.
2. Товары. Их допустимо хранить в Cassandra, но только при разделении информации о товаре и остатках товаров на складах,
   т.к. консистентность информации об остатках критически важна перед оформлением заказа. Остатки в Cassandra возможны,
   но требуются специальные паттерны (резервации, идемпотентность, возможно LWT точечно).
3. История заказов. Завершённые заказы не меняются, поэтому их уместно хранить в Cassandra.
4. Корзины. Корзины (особенно гостевые) уместны в Cassandra: на них высокая нагрузка, TTL, доступ по ключу;
   консистентность - на уровне read-your-writes.
5. Сессии. Аналогично корзинам: высокая нагрузка, TTL; консистентность - на уровне read-your-writes.

### Задание 10.2:

1. Заказы. В качестве PK был выбран `id` из-за его выской кардинальности, которая обеспечит равнометное распределение по
   виртуальным нодам. В данном случает также не будет горячих нод. Добавление нод не вызовет полного reshuffle.
   Используется для:
    1. Получения заказа по `id`.
    2. Обновления статуса заказа.
    3. Идемпотентной записи.

   Модель данных:
      ```sql
      CREATE TABLE orders (
       id UUID, /* UUID v4 */
       user_id UUID,
       created_at TIMESTAMP,
       status TEXT,
       total_amount DECIMAL,
       geo_zone TEXT,

       items LIST<FROZEN<tuple<
           UUID,      /* product_id */
           INT,       /* quantity */
           DECIMAL    /* price */
    >>>   ,
       PRIMARY KEY (id)
      );
      ```

2. История заказов. В качестве PK выбран `user_id+order_month` это позволит равномерно распределить историю заказов
   одного пользователя, также он обеспечит контролируемый размер партиции и безопасную работу при 50000 prs.
   Используется для:
    1. Истории заказов пользователя.
    2. Получения последних N заказов.
    3. Пагинации.

   Модель данных:
   ```sql
   CREATE TABLE order_history_by_user (
    user_id UUID,
    order_month TEXT,        /* YYYYMM bucket */
    created_at TIMESTAMP,
    order_id UUID,
    status TEXT,
    total_amount DECIMAL,
   geo_zone TEXT,

       items LIST<FROZEN<tuple<
           UUID,      /* product_id */
           INT,       /* quantity */
           DECIMAL    /* price */
    >>>   ,

    PRIMARY KEY (
        (user_id, order_month),
        created_at,
        order_id
    )
   )
   WITH CLUSTERING ORDER BY (created_at DESC);
   ```

3. Остатки товаров. В качестве PK выбран `zone+product_id` они обеспечат высокую кардинальность данных и быстрый поиск
   остатков товара для конкретной зоны.

   Используется для ведения (получения и обновления) остатков конкретного товара в определённой зоне.

   Модель данных:
   ```sql
   CREATE TABLE stocks(
    zone: TEXT
    product_id: UUID,
    remaining: INT,
    updated_at TIMESTAMP
   )
   ```

4. Корзины. В качестве PK выбран `owner_id`, он имеет высокую кардинальность. Возникновение горячих партиций
   маловероятно, т.к. один пользователь не создать огромный QPS.
   Используется для:
    1. Получить активную корзину.
    2. Добавить/удалить товар.
    3. Обновить количество.

   Модель данных:
   ```sql
   CREATE TABLE carts (
   owner_id TEXT,          /* user_id или session_id */
   product_id UUID,
   quantity INT,
   added_at TIMESTAMP,
   updated_at TIMESTAMP,

   PRIMARY KEY ((owner_id), product_id)
   );
   ```

5. Товары. В качестве PK выбран `category+bucket`. `bucket` - целое число, которе необходимо для того, чтобы разбить
   горячую партицию электроники на N партиций. Бакет рассчитывается по формуле: `bucket = hash(product_id) % N`. N
   подбираем по ожидаемой нагрузке: 16/32/64.

   Модель данных:
   ```sql
   CREATE TABLE products_by_category_price (
   category TEXT,
   bucket SMALLINT, /* 0..N-1, например N=32 или 64 */
   price DECIMAL,
   product_id UUID,
   title TEXT,

   PRIMARY KEY (
        (category, bucket),
        price,
        product_id
    )
   )
   WITH CLUSTERING ORDER BY (price ASC);
   ```

### Задание 10.3:

Cassandra используются следующие механизмы восстановления согласованности:

1. Hinted Handoff

   Применение:
    * Корзины
    * Пользовательские сессии
    * История заказов

   Причины:
    * Допустима временная рассинхронизация
    * минимальная latency записи
2. Read Repair

   Применение:
    * Заказы
    * Остатки товаров

   Причина:
    * Чтение автоматически повышает согласованность без увеличения write latency.
3. Anti-Entropy Repair

   Применение:
    * Заказы
    * Остатки товаров

   Причина:
    * Обеспечивает долгосрочную целостность данных даже при сетевых сбоях.
