## 1. Введение

Необходимо обеспечить масштабируемость трех основных коллекций (products, orders, carts) в условиях роста нагрузки.

- **Цель**: Равномерное распределение данных и нагрузки (RW) по шардам.
- **Приоритет**: Высокая доступность на запись (оформление заказов, обновление корзин, списание остатков) и эффективность ключевых выборок (история заказов, поиск товаров).

## 2. Коллекция products 

### Схема документа

```json
{
  "_id": "ObjectId",
  "name": "String",
  "category": "String",
  "price": "Decimal128",
  "attributes": {
    "color": "String",
    "size": "String",
    "brand": "String"
  },
  "stock": [
    { "geo_zone": "msk", "quantity": 50 },
    { "geo_zone": "ekb", "quantity": 30 }
  ],
  "updated_at": "ISODate"
}
```

### Стратегия шардирования

- **Шард-ключ**: `{ _id: "hashed" }`
- **Тип**: Хэш-шардирование (Hashed Sharding).

### Обоснование

- **Равномерность записи**: 
Операция "Частые обновления остатков" является критичной. 
Если использовать Ranged Sharding по категории (например, "Электроника"), то при распродажах вся нагрузка на запись ляжет на один шард, что замедлит работу.
Хэширование по `_id` гарантирует идеальное распределение документов и нагрузки на запись по всем шардам.
- **Поиск**: Поиск товаров по категориям и фильтрам будет использовать Scatter-Gather (опрос всех шардов), но с наличием вторичного индекса `{ category: 1, price: 1 }` это допустимо, 
так как операции чтения легче масштабируются репликами, чем операции записи.

## 3. Коллекция orders

### Схема документа

```json
{
  "_id": "ObjectId",
  "user_id": "ObjectId",
  "created_at": "ISODate",
  "geo_zone": "String",
  "status": "String",
  "total_amount": "Decimal128",
  "items": [
    {
      "product_id": "ObjectId",
      "name": "String",
      "quantity": "Int",
      "price": "Decimal128"
    }
  ]
}
```

### Стратегия шардирования

- **Шард-ключ**: `{ user_id: "hashed" }`
- **Тип**: Хэш-шардирование (Hashed Sharding).

### Обоснование

- **Целевые запросы**: Одна из ключевых операций — "Поиск истории заказов конкретного пользователя". 
Если шардировать по `user_id`, все заказы конкретного человека будут лежать на одном шарде. Запрос `find({ user_id: ... })` обратится только к одному шарду, что очень эффективно.
- **Избежание горячих шардов**: Если использовать `created_at` (монотонно возрастающий ключ), все новые заказы будут писаться в последний шард, перегружая его.
Хэширование `user_id` размазывает новые заказы по кластеру равномерно.
- **Геозона**: Шардирование по `geo_zone` опасно неравномерным распределением в густо налесенных локация,
что приведет к дисбалансу данных.

## 4. Коллекция carts 

### Схема документа

```json
{
  "_id": "ObjectId",
  "user_id": "ObjectId",
  "session_id": "String",
  "status": "String",
  "created_at": "ISODate",
  "updated_at": "ISODate",
  "expires_at": "ISODate",
  "items": [
    {
      "product_id": "ObjectId",
      "quantity": "Int"
    }
  ]
}
```

### Стратегия шардирования

- **Шард-ключ**: `{ _id: "hashed" }`
- **Тип**: Хэш-шардирование (Hashed Sharding).

### Обоснование

- **Сложность идентификации**: Мы ищем корзину либо по `session_id`, либо по `user_id`. 
Выбор любого из этих полей в качестве ключа шардирования сделает поиск по другому полю "широковещательным" (broadcast query).
- **Почему не `session_id`?** При логине происходит слияние корзин. 
Если мы изменим идентификатор или структуру данных, это усложнит обновление ключа шардирования (который неизменяем).
- **Производительность записи**: Корзины — это данные с очень интенсивной записью (добавление/удаление товаров) и коротким сроком жизни.
Шардирование по `_id` (hashed) обеспечивает наилучшую пропускную способность записи.
- **TTL (Time To Live)**: Удаление старых корзин происходит фоновым процессом. Равномерное распределение данных по `_id` позволяет утилизировать ресурсы всех шардов для очистки индексов.
- **Примечание**: Для ускорения поиска корзин необходимо создать вторичные индексы: `{ session_id: 1, status: 1 }` и `{ user_id: 1, status: 1 }`.

## 5. Итоги

| Коллекция | Шард-ключ | Тип | Почему выбран                                                                                         |
| :--- | :--- | :--- |:------------------------------------------------------------------------------------------------------|
| **products** | `{ _id: "hashed" }` | Hashed | Равномерное распределение обновлений стока . Защита от хотспотов.                                     |
| **orders** | `{ user_id: "hashed" }` | Hashed | Оптимизация выборки истории заказов. Равномерная запись.                                              |
| **carts** | `{ _id: "hashed" }` | Hashed | Максимальная скорость записи. Универсальность для поиска по разным критериям через вторичные индексы. |

## 6. Примеры команд MongoDB


### Настройка Products

```javascript
sh.enableSharding("mobile_world")
db.products.createIndex({ category: 1, price: 1 })
sh.shardCollection("mobile_world.products", { _id: "hashed" })
```

### Настройка Orders

```javascript
db.orders.createIndex({ user_id: "hashed" })
db.orders.createIndex({ user_id: 1, created_at: -1 })
sh.shardCollection("mobile_world.orders", { user_id: "hashed" })
```

### Настройка Carts

```javascript
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
db.carts.createIndex({ session_id: 1, status: 1 })
db.carts.createIndex({ user_id: 1, status: 1 })
sh.shardCollection("mobile_world.carts", { _id: "hashed" })
```
