### Задание 9. Настройка чтения с реплик и консистентность

1. Таблица с рекомендациями по чтению:

|Коллекция|Операция чтения|Источник чтения|Допустимая задержка|Обоснование|
|-|-|-|-|-|
|products|Поиск товаров по категориям, фильтрация по цене, просмотр списка товаров.|secondary|1–2 секунды|Небольшая задержка допустима, так как бизнес-логика допускает небольшое несоответствие остатков (например, пользователь может увидеть товар, который уже закончился, но попытка купить его будет проверяться на primary)|
|products|Просмотр детальной страницы товара (остаток, атрибуты) при добавлении в корзину. Проверка наличия при оформлении заказа (чтобы не продать недоступный товар).|primary|0 секунд|Критична актуальность данных, чтобы не продать недоступный товар|
|orders|Просмотр истории заказов пользователя (старые заказы). Аналитика, отчёты по заказам.|secondary|5–10 секунд|История не меняется часто, допустима задержка|
|orders|Отображение текущего статуса заказа (например, «Оплачен», «Доставлен»). Любые операции, связанные с изменением статуса заказа.|primary|0 секунд|Актуальность данных критична, чтобы не отобразить устаревший статус
|carts|Просмотр содержимого корзины гостя или пользователя (если не происходит изменений).|secondary|1–2 секунды| Корзина редко меняется быстро, допустима небольшая задержка|
|carts|Добавление/удаление товаров, слияние корзин, оформление заказа. Проверка корзины перед оформлением заказа (чтобы не оформить заказ с устаревшими данными).|primary|0 секунд|Критична актуальность, чтобы не оформить заказ с устаревшими данными|

2. Общее обоснование:
  - Консистентность: Для операций, где актуальность данных критична (оформление заказа, проверка остатков), чтение только с primary. Для остальных — secondary с допустимой задержкой.
  - Частота обновлений: Чем чаще обновляются данные, тем выше риск неактуальных данных на secondary. Для товаров и корзин, которые часто меняются, актуальность важна, поэтому такие операции читаются с primary.
  - Бизнес-логика: Риск продажи недоступного товара или отображения устаревшего статуса заказа недопустим, поэтому такие операции выполняются только с primary. Для аналитики и истории — secondary допустим.
