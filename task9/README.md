# Задание 9. Настройка чтения с реплик и консистентность

Архитектурный документ для онлайн-магазина «Мобильный мир».

Для коллекций `products`, `orders`, `carts` определяем: какие чтения можно отдать на `secondary`, какие оставить на `primary`, и какой lag допустим.

**Простое правило:** если устаревшие данные могут сломать оплату, резерв или UX сразу после действия пользователя — только `primary`. Всё остальное — `secondary` / `secondaryPreferred` с `maxStalenessSeconds`.

---

## 1. Таблица операций чтения

| Коллекция | Операция | Реплика | Допустимый lag | Обоснование |
|-----------|----------|---------|----------------|-------------|
| `products` | Каталог: фильтры, сортировка, список по категории | `secondaryPreferred` | до **30 сек** | Консистентность: eventual ok. Обновления цены/описания частые, но в списке не критичны. Бизнес: не влияет на списание денег |
| `products` | Карточка товара (название, описание, фото, характеристики) | `secondaryPreferred` | до **30 сек** | Консистентность: eventual ok. Данные меняются реже остатков. Бизнес: старое описание не ломает покупку |
| `products` | Цена перед оплатой | `primary` | **0 сек** | Консистентность: strong, read-your-writes. Цена меняется в акциях. Бизнес: устаревшая цена = неверная сумма заказа |
| `products` | Остаток перед резервом / добавлением в заказ | `primary` | **0 сек** | Консистентность: strong. Остатки обновляются при каждой покупке, в распродажу — очень часто. Бизнес: риск продать недоступный товар |
| `products` | `is_active` перед покупкой | `primary` | **0 сек** | Консистентность: strong. Статус меняется редко, но последствия критичны. Бизнес: secondary может показать снятый с продажи товар |
| `products` | Рекомендации, похожие товары, витрина | `secondaryPreferred` | до **60 сек** | Консистентность: eventual ok. Обновляются фоном. Бизнес: не участвуют в checkout |
| `orders` | Список прошлых заказов | `secondaryPreferred` | до **10 сек** | Консистентность: eventual ok. Старые заказы почти не меняются. Бизнес: задержка в истории не ломает UX |
| `orders` | Детали завершённого заказа | `secondaryPreferred` | до **30 сек** | Консистентность: eventual ok. Завершённые заказы стабильны. Бизнес: snapshot в `items` не зависит от каталога |
| `orders` | Статус текущего заказа (после оплаты, отмены, смены доставки) | `primary` | **0 сек** | Консистентность: strong, read-your-writes. Статус меняется при каждом действии. Бизнес: пользователь увидит `pending` вместо `paid` / `cancelled` |
| `orders` | Подтверждение сразу после создания заказа | `primary` | **0 сек** | Консистентность: read-your-writes. Только что записали заказ. Бизнес: «заказ не создался» после успешного оформления |
| `orders` | Админский отчёт за период | `secondary` | до **60 сек** | Консистентность: eventual ok. Отчёт не в user flow. Бизнес: разгружаем primary от тяжёлых агрегаций |
| `carts` | Открытие активной корзины | `primary` | **0 сек** | Консистентность: strong. Корзина меняется при каждом клике. Бизнес: покажем удалённый товар или неверное количество |
| `carts` | Пересчёт перед checkout | `primary` | **0 сек** | Консистентность: strong. Нужны актуальные позиции, цены, скидки. Бизнес: неверная сумма или состав заказа |
| `carts` | После add / remove / изменения количества | `primary` | **0 сек** | Консистентность: read-your-writes. Пользователь только что изменил корзину. Бизнес: «товар не добавился» |
| `carts` | Брошенная корзина для маркетинга | `secondaryPreferred` | до **60 сек** | Консистентность: eventual ok. Фоновый сценарий. Бизнес: не влияет на оплату |
| `carts` | Аналитика брошенных корзин | `secondary` | до **5 мин** | Консистентность: eventual ok. Batch-отчёт. Бизнес: точность в реальном времени не нужна |

## 2. Допустимая задержка репликации

| Класс операции | Lag | Примеры |
|----------------|-----|---------|
| Критичные, влияют на оплату и UX после действия | **0 сек**, только `primary` | checkout, остатки, статус заказа, активная корзина |
| Обычные пользовательские чтения | **10–30 сек** | каталог, карточка, история заказов |
| Витринные / некритичные | **30–60 сек** | рекомендации, брошенные корзины |
| Отчёты и аналитика | **60 сек – 5 мин** | админские отчёты, аналитика корзин |

Для secondary задаём `maxStalenessSeconds` — приложение не пойдёт на сильно отстающую реплику:

```javascript
readPreference: "secondaryPreferred"
maxStalenessSeconds: 30  // значение из таблицы
```

Для операций с lag = 0:

```javascript
readPreference: "primary"
```

## 3. Обоснование по коллекциям

### `products`

Два типа данных: стабильные (название, описание, категория) и критичные (цена, `stock_by_zone`, `is_active`).

Каталог и карточки — тяжёлые read-запросы, их логично разгрузить на secondary: небольшое отставание описания или цены в списке не приводит к ошибке продажи. Но перед checkout цену, остаток и `is_active` читаем только с primary — остатки меняются при каждой покупке, в Black Friday lag даже в несколько секунд может показать товар в наличии, когда он уже закончился.

### `orders`

Завершённые заказы меняются редко — для истории достаточно eventual consistency. Текущий заказ после оплаты, отмены или смены доставки — только primary: пользователь ожидает увидеть результат своего действия, иначе покажем устаревший `pending` вместо `paid`.

### `carts`

Активная корзина постоянно меняется (add/remove/скидки) — stale read напрямую ломает checkout. Secondary только для фоновых сценариев: брошенные корзины, маркетинг, аналитика.

---

## 4. Итог

| Коллекция | Secondary | Primary |
|-----------|-----------|---------|
| `products` | каталог, карточка, рекомендации | цена, остаток, `is_active` |
| `orders` | история, завершённые, отчёты | текущий статус, сразу после create/pay/cancel |
| `carts` | брошенные, аналитика | активная корзина, checkout, после изменений |
