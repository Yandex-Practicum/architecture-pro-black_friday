# Описаие и обоснование коллекций данных

## 1. Продукты (Products)

### 1.1 Model

| Поле | Тип данных | Описание | Индексы / Особенности |
|---|---|---|---|
| _id | ObjectId | Уникальный идентификатор товара | Primary Key |
| name | String | Наименование товара | Text Index (для поиска) |
| category | String | Категория | Compound Index с ценой | 
| price | Decimal128 | Цена (точное значение для денег) | Compound Index с категорией |
| attributes | Object | Доп. атрибуты (цвет, размер) | |
| stock | Array of Objects | Ключевой элемент шардирования | См. обоснование ниже | 
| stock.zone | String | Геозона (Москва, Екатеринбург) | - |
| stock.qty | Number | Остаток в конкретной зоне | - |

### 1.2 Use cases

#### Списание остатков (при заказе)

```
db.products.updateOne(
    { _id: productId, "stock.zone": "Moscow", "stock.qty": { $gte: 1 } },
    { $inc: { "stock.$.qty": -1 } }
);
```

#### Поиск по категории и цене

```
db.products
    .find({ category: "Phones", price: { $gte: 30000, $lte: 60000 } })
    .sort({ price: 1 })
    .limit(20);
```

#### Страница товара

```
db.products.findOne({ _id: ObjectId("...") });
```


### 1.3 Шардирование и индексы

#### Шардирование

- **Стратегия шардирования**: Compound Sharding
- **Ключ шардирования**: category: 1, _id: "hashed", 
    - `category: 1` - диапазонное шардирование по категории
    - `_id: "hashed"` - хеш-шардирование по id

Такая стратегия и ключ обеспечит:
- быстрый поиск товаров по категориям, потому что config server будет хранить информацию о том, на каких шардах лежат товары с этой категорией
- быстрый поиск конкретного товара при наличии категории и id (см. пояснение ниже)

#### Индексы

- `category: 1, price: 1` - обеспечит быстрый поиск по категориям и цене (запрос будет идти только на определённые шарды по категории)
- `name: text` - обеспечит быстрый поиск по названию товара (запрос будет происходить по всем шардам, то есть со scatter-gather)

#### Проблемы

Поиск по ID идёт на все шарды (scatter-gather), так как роутер не знает о том, в каком шарде находится документ с этим ключём, потому что документы в коллекции products шардированы по category.

Как решение этой проблемы можно использовать Redis: когда на бэкенд приходит запрос продукта с определённым ID, то можно:
1. использовать кеш, чтобы достать категорию продукта
2. обратиться к БД со знанием категории продукта
3. роутер с помощью config server определяет шард, где расположен документ
4. роутер отправляет запрос на определённый шард
5. на шарде продукт быстро находится по id, так как на это поле автоматически создаётся индекс

Такая стратегия является более предпочтительной, потому что кешировать отдельный продукты по id проще, чем кешировать варианты фильтрации (по категориям и по цене) пользователей.


### 1.4 Создание коллекции

```
sh.enableSharding("shop");

db.createCollection("products");

sh.shardCollection("shop.products", { category: 1, _id: "hashed" })

db.products.createIndex({ "category": 1, "price": 1 });

db.products.createIndex({ "name": "text" });
```


# 2. Корзины (carts)

### 2.1 Model

| Поле | Тип данных | Описание | Индексы / Особенности |
|---|---|---|---|
| _id | ObjectId | Уникальный идентификатор корзины | Primary Key |
| owner_id | String | `u:<user_id>` или `s:<session_id>` | | 
| status | String | Статус корзины ("active" / "ordered" / "abandoned") | Compound Index |
| items | Array | `[{ product_id, quantity }]` | - |
| created_at | Date | Дата создания | - |
| updated_at | Date | Дата обновления | - |
| expires_at | Date | Дата истечения срока действия | TTL Index |


### 2.2 Use cases

#### Создание корзины

```
# Когда заходит гость
db.carts.insertOne({ owner_id: "s:<session_id>", status: "active", ... });

# Когда заходит пользователь
db.carts.insertOne({ owner_id: "u:<user_id>", status: "active", ...  });
```

#### Получение текущей корзины

```
# Для гостя
db.carts.findOne({ owner_id: "s:<session_id>", status: "active", ... });

# Для зарегистрированного пользователя
db.carts.findOne({ owner_id: "u:<user_id>", status: "active", ... });
```

#### Добавление или замена товара в корзине

Можно использовать поиск по user_id или session_id:
```
await db.carts.updateOne(
        {
            owner_id: "..."
            "items.product_id": ObjectId(productId)
        },
        {
            $set: {
                "items.$.quantity": quantity,           // Замена количества
                "items.$.price_snapshot": price,        // Обновление цены
                "items.$.added_at": now,
                updated_at: now,
                expires_at: new Date(now.getTime() + 7 * 24 * 60 * 60 * 1000)
            }
        }
    )
```

#### Отметка корзины как заказанной

```
# Полагаем, что делать заказы могут только зарегистрированные пользователи
db.carts.updateOne({ owner_id: "u:<user_id>" }, { $set: { status: "ordered" } });
```

### 2.3 Шардирование и индексы

#### Шардирование

- **Стратегия шардирования**: Hashed sharding
- **Ключ шардирования**: owner_id

Такая стратегия шардирование опеспечит быстрый поиск корзины для конкретного пользователя (или сессии). Все запросы корзин для конкретного пользователя всегда будут идти на один шард.

#### Индексы

- `owner_id: 1, status: 1` - составной индекс для быстрого поиска активной корзины

### 2.4 Создание коллекции

```
sh.enableSharding("shop");

db.createCollection("carts");

sh.shardCollection("shop.carts", { owner_id: "hashed" });

// Создаем TTL индекс для автоудаления старых корзин (active 7 дней, abandoned 30 дней - решается логикой приложения)
db.carts.createIndex(
    { "expires_at": 1 }, 
    { expireAfterSeconds: 0 } // Удалит документ, когда expires_at < now
);

// Составной индекс для быстрого поиска активной корзины
db.carts.createIndex({ "owner_id": 1, "status": 1 });

// Уникальный индекс: только одна активная корзина на пользователя/сессию
db.carts.createIndex(
    { "owner_id": 1 }, 
    { unique: true, partialFilterExpression: { status: "active", owner_id: { $exists: true } } }
);

```

# 3. Заказы (orders)

### 3.1 Model

| Поле | Тип данных | Описание | Индексы / Особенности |
|---|---|---|---|
| _id | ObjectId | ID заказа | - |
| user_id | ObjectId | ID покупателя | Шард-ключ |
| zone | String | Геозона доставки | Compound Index |
| created_at | Date | Дата заказа | Compound Index |
| status | String | Статус заказа ("new", "processing", "delivered") | - |
| total_amount | Decimal128 | Общая сумма | - |
| items | Array | Состав заказа с ценами на момент покупки | - |

### 3.2 Use cases

#### Быстрое создание заказа с одновременным списанием остатков:

**Проблема**:

Списание остатков (products) и создание заказа (orders) — это разные коллекции с разными шард-ключами (_id vs user_id). Это распределенная транзакция.

**Решение**:

Использовать MongoDB Transactions (Multi-Document ACID). В одной транзакции: обновляем stock в products и вставляем документ в orders. Важно: Транзакция будет медленнее обычной записи, но это плата за консистентность.

Если по мере роста магазина такое решение будет слишком нагружать БД, то можно перейти на альтернативу (Eventual Consistency): Создать заказ, поставить статус "pending_payment". В фоне слушать Change Streams и списывать остатки. При ошибке списания — отмена заказа.

#### Поиск истории заказов пользователя:

```
db.orders
    .find({ user_id: ObjectId("...") })
    .sort({ created_at: -1 })
    .limit(20);
```

#### Отображение статуса заказа

```
db.orders.findOne({ _id: orderId, user_id: userId });
```

Важно: в поиск заказа стоит всегда добавлять `user_id`, чтобы роутер БД знал, в каком шарде искать данные.

### 3.3 Шардирование и индексы

#### Шардирование

- **Стратегия шардирования**: Hashed sharding
- **Ключ шардирования**: user_id

Такая стратегия шардирование опеспечит быстрый поиск заказов для конкретного пользователя.

#### Индексы

- `user_id: 1, created_at: 1` - сортировка по дате создания для показа последних заказов

### 3.4 Создание коллекции

```
db.createCollection("orders");

sh.shardCollection("shop.orders", { user_id: "hashed" });

db.orders.createIndex({ "user_id": 1, "created_at": -1 });
```

# Проблема выявления и устранения горячих шардов

## Изменения варианты шардирования

Уже предложен оптимальный вариант шардирования для самой проблемной коллекции - продукты.

Но рассмотрим альтернативы, которые были отклонены:

### 1. Шардирование по категории

В этом случае все продукты определённой определённой категории будут относится к одному шарду.

Плюсы:
- быстрый поиск по категориям
- запросы для одной категории идут на один шард, отсутвует собирание запросов

Минусы:
- перекос в размерах шардов из-за неравномерного распределения продуктов по категориям
- риск перегрузки отдельных шардов, если какая-то категория окажется наиболее популярной

### 2. Шардирование по категории и цене

В этом случае продукты с одной категорией могут лежать на разных шардах, потому что будет введено дополнительное динамическое шардирование по цене.

Плюсы:
- быстрый поиск по категориям
- хотя запросы для одной категории идут на несколько шардов, они всё равно будут собираться быстро
- быстрые запросы по категории и диапазону цен

Минусы:
- перекос в размерах шардов из-за неравномерного распределения продуктов по категориям и цене
- необходимость перебалансировки при изменении цены продукта


## Метрики для выявления горячих шардов в Prometheus

### 1. Метрики балансировки и чанков

Можно узнать, количество чанков и распределение данных по шардам:
```
docker compose exec -T mongos_router mongosh --port 27030

use somedb;

db.helloDoc.getShardDistribution();
```

### 2. Количество операций чтения и записи на шарде

Можно на каждом шарде узнать количество операций вставки, запроса, обновления и удаления следующими командами:

```
db.serverStatus().opcounters.insert
db.serverStatus().opcounters.query
db.serverStatus().opcounters.update
db.serverStatus().opcounters.delete
```

### 3. Задержка запросов к шарду
Аналогично можно посмотреть для каждого шарда и сравнить, насколько долго выполняются операции на нем, с помощью команд:

```
db.serverStatus().opLatencies.reads.latency
db.serverStatus().opLatencies.writes.latency
```

### 4. Перебалансировка

Если обнаружится, что коллекция с продуктами неравномерно распределена, и поэтому некоторые шарды долго обрабатывают запросы, то можно попровать перебалансировку:

```
sh.setBalancerState(true)
sh.setCollectionBalancing("shop.products", true)
sh.setBalancerWindow("01:00","06:00")

sh.splitFind("shop.products", { category:"Электроника", _id:"X123" })
sh.moveChunk("shop.products", { category:"Электроника", _id:MinKey }, "shard02")
sh.mergeChunks("shop.products",
  { category:"Электроника", _id:MinKey },
  { category:"Электроника", _id:MaxKey })

```

# Настройка чтения с реплик и консистентность

| Коллекция | Операция | Реплика чтения | Допустимая задержка | Обоснование |
|---|---|---|---|---|
| products | Поиск товаров | secondary | более 5 секунд | Список товаров меняется очень часто, к тому же при оформлении заказа информация о товаре будет показана ещё несколько раз, поэтому можно допустить показ не вполне актуального списка товаров |
| products | Просмотр конкретного товара | secondary | более 5 секунд | (см. обоснование выше) |
| cart | Получение состояния корзины | primary | нет | Нужно показывать пользователю самые свежие данные, чтобы он мог оформить заказ |
| orders | Просмотр истори заказов | secondary | более 5 секунд | Это потоковая операция, которая нужна чтобы пользователь увидел общие данные, а потом, если нужно, перешёл на просмотр конкретного заказа |
| orders | Просмотр конкретного заказа | primary | нет | Нужно показывать пользователю сомые свежие данные, так как в данный момент он может принимать критические решения (отмена заказа, возврат и т.д.) |
