# Задание 8. Выявление и устранение «горячих» шардов

## набор метрик для отслеживания

- Запросы в сек: Разница между opcounters.query на шарде с «Электроникой» и остальными. 
- Shard Size Skew (Объем данных): Метрика db.collection.stats().shards. Если размер чанков (chunks) на одном шарде растет кратно быстрее, начнется большая миграцию, нагружающая диск.
- лимиты блокировки:  wiredTiger.concurrentTransactions. Если «горячий» шард исчерпал лимиты на чтение/запись, запросы начнут выстраиваться в очередь.
- Использование процессора: Если CPU на одном узле 90%, а на других 10% — это верный признак горячей категории.

## механизмы автоматического перераспределения данных

### автоматический балансировщик

создание составного ключа { category_id: 1, _id: 1 } командой refineCollectionShardKey

### перевод на хешированный ключ

сделать ключ { category_id: "hashed" }.

### зонирование на основе нагрузки

 Скрипт, который мониторит RPS (запросы в секунду) по категориям. Если категория превышает порог в 50% общей нагрузки, скрипт через sh.addTagRange() помечает её для переноса на SSD-шарды или шарды с большим количеством CPU. Балансировщик сам передвинет чанки в фоновом режиме.

 ### автоматическое добавленое шарда
  
Настройка Horizontal Pod Autoscaler (HPA) по метрике mongodb_op_counters_query_total.При достижении лимита по CPU на шарде система может автоматически инициировать добавление нового шарда в кластер. После появления нового узла встроенный Balancer MongoDB автоматически начнет переливать туда часть «горячих» данных.
