## Схемы коллекций MongoDB

- Коллекция orders

```javascript
{
   _id: ObjectId,                   // Уникальный идентификатор заказа
   customer_id: ObjectId,           // Идентификатор клиента
   order_date: Date,                // Дата и время оформления заказа
   items: [                         // Список заказанных товаров
   {
   product_id: ObjectId,
   name: String,
   price: Decimal128,               // Цена на момент заказа
   quantity: Number,   
   }
   ],
   status: String,                  // Статус заказа
   total_amount: Decimal128,        // Общая сумма заказа
   geo_zone: String,                // Геозона заказа
}

// Шард-ключ
sh.shardCollection("db.orders", {customer_id: 1})

// Дополнительные индексы
db.orders.createIndex({geo_zone: 1})
```

Шард-ключ: {customer_id}. Метод шардирования: "Хэшированное шардирование"

Альтернативный шард-ключ: {geo_zone, customer_id} - если нужна локализация по регионам

Так как основная операция поиск истории заказов, то запросы по customer_id будут локализованы на одном шарде.
Вариант шард-ключа с geo_zone позволит получать быстрее историю заказов для пользователей с шарда расположенного в их регионе, но для пользователей активно покупающих товары в разных регионах, история заказов будет собирать с разных шардов, что снизит скорость. 
Но вероятно таких пользователей немного.  


- Коллекция products

```javascript
{
  _id: ObjectId,                   // Уникальный идентификатор товара
  name: String,                    // Наименование
  category: String,                // Категория товара
  price: Decimal128,               // Текущая цена
  stock: {                         // Остатки по геозонам
    geo_zone_1: Number,
    geo_zone_2: Number,
    // ...
  },
  attributes: {                    // Дополнительные атрибуты
    color: String,
    size: String,
    // другие атрибуты
  }
}

// Шард-ключ
sh.shardCollection("db.products", {category: 1, price: 1})
```
Основные операции:
Частые обновления остатков при покупках.
Поиск товаров по категориям и фильтрация по диапазону цен.
Описание товара на странице продукта.

Шард-ключ: {category, price}. Метод шардирования: Хэшированное шардирование + Диапазонное шардирование

Так как основная операция поиск по категориям, то category помогает группировать товары одной категории на одном шарде.
Но требуется еще фильтрация по диапазону цен. Добавление _price в ключ позволит группировать товары одной категории, но разных ценовых диапазонах на разных шардах.  


- Коллекция carts

```javascript
{
  _id: ObjectId,                   // Уникальный идентификатор корзины
  user_id: ObjectId,               // Для авторизованных пользователей
  session_id: String,              // Для гостей
  items: [                         // Товары в корзине
    {
      product_id: ObjectId,
      quantity: Number,      
    }
  ],
  status: String,                  // Статус корзины "active", "ordered", "abandoned"
  created_at: Date,                // Дату и время создания 
  updated_at: Date,                // Дату и время последнего обновления
  expires_at: Date                 // Время удаления (TTL)
}


// Шард-ключ
// Вариант 1: Для пользователей
db.carts.createIndex({user_id: 1, status: active})
// Вариант 2: Для гостей
db.carts.createIndex({session_id: 1, status: active})

// TTL индекс для автоматической очистки
db.carts.createIndex({updated_at: 1}, {expireAfterSeconds: 2592000}) // 30 дней
```
Шард-ключ: {session_id, status} для гостей и {user_id, status} для авторизованных пользователей

Быстрый доступ к корзине по user_id или session_id.
Операция слияния корзин будет затрагивать максимум 2 шарда.

## Миграция на Cassandra: модель данных, стратегии репликации и шардирования

### Обоснование 
Для интернет магазина критически важными с точки зрения целостности и скорости обработки являются:
1. Товары (products) - часто читаются, редко обновляются (кроме остатков). Требуется высокая доступность.

2. Заказы (orders) - часто пишутся, часто читаются. Требуется высокая доступность и целостность.

3. Корзины (carts) - часто обновляются, требуют низкой задержки. Целостность важна, но не так критична, как для заказов.

4. Пользовательские сессии (sessions) - временные данные, требуют высокой скорости записи/чтения, целостность не критична.

Так как при изменении остатков необходима ACID транзакция, то products не подходит для Cassandra.

| **Сущность**             | **Критичность целостности** | **Критичность скорости** | **Подходит для Cassandra** | **Обоснование**                                                                           |
|:-------------------------|:---------------------------:|:------------------------:|:--------------------------:|-------------------------------------------------------------------------------------------|
| Корзины (carts)          |           Средняя           |         Высокая          |             Да             | Временные данные, eventual consistency приемлема, чтение по session_id/user_id                                        |
| Заказы (orders)          |        Очень высокая        |         Высокая          |            Нет             | Высокая скорость записи, ACID транзакции                                                  |
| История заказов          |           Высокая           |         Средняя          |             Да             | Чтение по пользователю, append-only данные                                                |
| Товары (products)        |           Высокая           |         Высокая          |            Нет             | Каталог товаров хранить остатки, а их консистентность критична. Требуются ACID транзакции |
| Пользовательские сессии  |           Низкая            |      Очень высокая       |             Да             | Ключ-значение, TTL, высокая доступность критична                                          |

### Концептуальная модель данных в Cassandra 

``` cql
CREATE TABLE carts (
    cart_id UUID,
    user_id UUID,
    session_id TEXT,
    status TEXT,  -- 'active', 'ordered', 'abandoned'
    items MAP<UUID, INT>,  -- product_id -> quantity
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    expires_at TIMESTAMP,
    PRIMARY KEY ((cart_id, user_id, session_id))
) WITH default_time_to_live = 2592000;  -- 30 дней TTL
```

Для зарегистрированных пользователей: partition key = user_id, для гостей: partition key = session_id.

``` cql
CREATE TABLE user_sessions (
    session_id TEXT,
    user_id UUID,
    session_data TEXT,  -- JSON или сериализованные данные
    created_at TIMESTAMP,
    last_activity TIMESTAMP,
    user_agent TEXT,
    ip_address INET,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400;  -- 24 часа TTL

```

Partition key: session_id

### Cтратегии беспечения целостности данных


| **Сущность**             |    **Стратегия**    | **Обоснование**                |
|:-------------------------|:-------------------:|:-------------------------------|
| Корзины (carts)          |     Read Repair     | Важна быстрая консистентность  |
| История заказов          | Anti-Entropy Repair | Данные читаются редко          |
| Пользовательские сессии  |   Hinted Handoff    | Данные временные               |
