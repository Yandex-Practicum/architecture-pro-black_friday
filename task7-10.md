## Task7

Колекции:

- orders [schemas/orders.json](schemas/orders.json)
  - В качестве стратегии шардирования будем использовать хэш-функции, в качестве ключа лучше всего подойдет clientId для равномерного распределения данных, при необходимости можно использовать геошардинг по територии заказа
- carts [schemas/carts.json](schemas/carts.json)
  - Так как данные краткосрочные (удаляются по ttl) - считаю использование шардирования не необходимым
- products [schemas/products.json](schemas/products.json)
  - Остатки лучше вынести в отдельную коллекцию для более быстрого доступа к данным и меньшего количества конфликтов при обновлении. В качестве стратегии шардирования будем использовать хэш-функции, в качестве ключа лучше всего подойдет category и productId (для остатков) для равномерного распределения данных, при необходимости можно использовать геошардинг по територии остатков


## Task8

В качестве метрик можно использовать:

- Инфраструктурные
  - Saturation
  - Utilization

- Метрики нагрузки
  - TPS
  - Latency

- Метрики распределения
  - Объем памяти
  - Коэффициент распределения по шардам

Для автоматического решардинга необходимо внедрить инфраструктурный слой, который на основе метрик будет перераспределять данные (например используя виртуальные бакеты) или в случае с mongo использовать встроенный механизм

В cassandra можно вынести:
- products (модель для чтения каталога), так как в данных сценариях допустима задержка 
  - partion key - category
- orders (модель для чтения), так как в данных сценариях допустима задержка согласованности
  - partion key - clientId
  - clustering key - cratedAt
- cart (с допущением, что возможна потеря данных при мерже гостевой и клиентской сессий)
  - partion key - userId/sessionId

Коллекции products (запись и получение остатков) и orders(запись) требуют высокой согласованности, поэтому использование cassandra нецелесообразно
