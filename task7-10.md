В качестве метрик можно использовать:

- Инфраструктурные
  - Saturation : soft_pending (количество ожидающих операций в очереди шарда) > 100 в течение 5 минут
  - Utilization : CPU на одном шарде > 80% при среднем по кластеру < 40%

- Метрики нагрузки
  - TPS : 	TPS на одном шарде > 2x от медианы по кластеру
  - Latency : Время выполнения запросов на одном шарде > 200ms при среднем 50ms

- Метрики распределения
  - Объем памяти
  - Коэффициент распределения по шардам

# Механизмы устранения перегрузки средствами MongoDB

| Механизм | Команда | Описание | Когда применять |
|----------|---------|----------|----------------|
| **Разделение горячего чанка** | `sh.splitAt("ecommerce.orders", { "clientId": "mid_value" })` | Делит перегруженный чанк на два меньших | Один чанк принимает >50% записей или превысил размер 1GB |
| **Автоматическое разделение** | `sh.splitFind("ecommerce.orders", { "clientId": "hot_client" })` | Находит чанк по документу и разделяет его | Известен конкретный клиент, создающий нагрузку |
| **Перемещение чанка** | `sh.moveChunk("ecommerce.orders", { "clientId": "hot_range" }, "shard0003")` | Переносит чанк на менее загруженный шард | После split — физически выгрузить данные с горячего шарда |
| **Включение балансировщика** | `sh.startBalancer()` | Запускает автоматическое перераспределение чанков | Данные легли неравномерно после длительной работы |
| **Настройка окна балансировки** | `db.settings.update({ "_id": "balancer" }, { "$set": { "activeWindow": { "start": "02:00", "stop": "06:00" } } })` | Ограничивает балансировку временем низкой нагрузки | Избежать влияния на пиковый трафик |
| **Отключение балансировщика** | `sh.stopBalancer()` | Останавливает перемещение чанков | Срочное ручное вмешательство или деградация производительности от балансировки |
| **Ручная балансировка** | `sh.moveChunk()` последовательно для нескольких чанков | Вручную выравнивает распределение | Автоматический балансировщик не справляется или отключен |
| **Уточнение ключа шардирования** | `sh.refineCollectionShardKey("ecommerce.orders", { "clientId": 1, "createdAt": 1 })` | Добавляет поле к ключу для более равномерного распределения | Текущий ключ создает неисправимый перекос (MongoDB 5.0+) |
| **Зонирование горячей категории** | `sh.addShardTag("shard0000", "high_performance")`<br>`sh.addTagRange("ecommerce.products", { "category": "Electronics", "productId": MinKey }, { "category": "Electronics", "productId": MaxKey }, "high_performance")` | Изолирует горячие данные на выделенных шардах | Отдельная категория (Electronics) создает 70% нагрузки |
| **Расширение зоны** | `sh.addShardTag("shard0004", "high_performance")`<br>`sh.rebalanceTags("ecommerce.products")` | Добавляет новые шарды в зону и перераспределяет чанки | Выделенных шардов стало недостаточно для горячей категории |
| **Принудительная синхронизация маршрутизации** | `db.adminCommand({ flushRouterConfig: 1 })` | Очищает кэш маршрутизации на всех mongos | После изменения конфигурации шардирования для немедленного применения |
