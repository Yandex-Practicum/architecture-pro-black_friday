# Задание 7 — Проектирование шардирования для «Мобильный мир»

Цель документа: Рекомендация по схемам коллекций `products`, `orders` и `carts` для MongoDB и выбор эффективных шард-ключей и стратегии шардирования, учитывая операции над коллекциями.

База: mobileshopdb

-----------------------------------------------------------------

1) Коллекция `products`

Cхема:

```json

{

_id: ObjectId("...")

name:  "Смартфон X",

category: "electronics",

price: 49990,

attributes: { color: "black", memory: "128GB" },

stock: { // остатки по геозонам внутри документа

"moscow": 120,

"ekb": 50,

"kaliningrad": 30

},

updated_at: ISODate()

}

```

Основные операции:

- Поиск по категории и фильтрация по диапазону цен (страница каталога).

- Частые обновления остатка при покупке.

- Получение полной информации для страницы товара.

Рекомендация по шардированию - шард по хешированному `_id` (hashed shard key).
Команда:

```js

sh.enableSharding("mobileshopdb")

sh.shardCollection("mobileshopdb.products", { _id:  "hashed" })

```
Гарантированно равномерное распределение документов по шардам, высокопроизводительные write/update, простая маршрутизация по _id.

Альтернатива: если кол-во категорий большое и равномерно распределено, то shard по `category` (range) улучшает локальность запросов на каталожные списки, но риск дисбаланса(горячие шарды) при «хитовых» категориях.

-----------------------------------------------------------------

2) Коллекция `orders`

Схема:

```json

{

_id: ObjectId(),

user_id: ObjectId("..."),

created_at: ISODate(),

items: [ { product_id: ObjectId(), price: 49990, qty: 1 }, ... ],

status: "created"  |  "paid"  |  "shipped"  |  "cancelled",

total: 123450,

geozone: "moscow"

}

```

Основные операции:

- Быстрое создание заказов с одновременным списанием остатков (транзакции/атомарные операции).

- Поиск истории заказов конкретного пользователя (user_id).

- Отображение статуса заказа (по order_id).


Рекомендация по шардированию - шард по хешированному `_id` (hashed shard key).

- Команды:

```js

sh.enableSharding("mobileshopdb")

sh.shardCollection("mobileshopdb.orders", { user_id:  "hashed" })

```

Большинство читаемых операций — получение истории конкретного пользователя; shard по `user_id`, находятся логически близко. 

Альтернатива: Для географической локальности (чтобы записи и чтения из одной геозоны были локальны), можно использовать геошардинг — это позволит маршрутизовать заказы из одной зоны на конкретные шарды. Но это усложнит балансировку при неравномерном распределении заказов по зонам.

-----------------------------------------------------------------

3) Коллекция `carts`

Примерная схема документа:

```json

{

_id: ObjectId(),

user_id: ObjectId()  |  null,

session_id: "uuid-for-guest"  |  null,

owner_id: "<user_id or session_id>", // вычисляемое поле

owner_type: "user"  |  "session",

items: [ { product_id: ObjectId(), quantity: 2 } ],

status: "active"  |  "ordered"  |  "abandoned",

created_at: ISODate(),

updated_at: ISODate(),

expires_at: ISODate()  // TTL index

}

```

Основные операции:

- Получение текущей корзины по { session_id, status:"active" } или { user_id, status:"active" }.

- Добавление/удаление товара.

- Слияние гостевой корзины в пользовательскую при логине.

Рекомендация по шардированию:

- Поле `owner_id`, которое равняется `user_id` если пользователь авторизован, иначе `session_id`.

- Shard key: `{ owner_id: "hashed" }` — хешированный owner_id.


```js

sh.enableSharding("mobileshopdb")

sh.shardCollection("mobileshopdb.carts", { owner_id:  "hashed" })

db.carts.createIndex({ owner_id:  1, status:  1 })

```

Все операции над корзиной выполняются по владельцу (user/session). Нормализованный `owner_id` позволяет иметь единую точку доступа и маршрутить все запросы на один шар для данного владельца, а hashed вариант даёт хорошее распределение по шардам.

# Задание 8 — Выявление и устранение «горячих» шардов

Цель:

Практические рекомендации по обнаружению "горячих" шардов (hot shards), набор метрик и порогов для оповещений, а также перечислить механизмы автоматического или полуавтоматического перераспределения данных и смягчения перегрузки (mitigation).

Контекст проблемы

В примере 70% запросов приходят на категорию "Электроника" — что приводит к сильной концентрации запросов и IO/CPU/lock нагрузке на шарды, где хранятся документы этой категории. Цель — иметь процесс обнаружения и набор мер, снижающих нагрузку и перераспределяющих «горячие» данные.

1. Набор метрик (что собирать)

Соберите метрики на трёх уровнях: mongod (каждый член replica set), mongos, и инфраструктура (хост/VM/контейнер). 

1.1. Метрики MongoDB (per-shard, per-node)

- операции чтения/записи в секунду

- количество операций, репликация.

- активные соединения.

- сетевой трафик.

- использование памяти.

- загрузка CPU процесса mongod.

- сколько чанков лежит на шарде и их суммарный размер.

1.2. Метрики mongos / routing

- какие коллекции получают наибольшую нагрузку.

- доля запросов, требующих опроса всех шардов.

1.3. Аппликационные метрики

- RPS по endpoint/по category (из логов приложения) — важно знать, какие категории генерируют трафик.

- latency P50/P90/P99 по endpoint.

- cache hit ratio (Redis) если кеш есть, насколько он снижает нагрузку.


2. Механизмы смягчения и перераспределения данных

2.1. Быстрые меры

- Кеширование горячих списков/страниц (Redis) и повышение TTL на кэш — уменьшит число запросов к mongos для «категории Элетроника».

- Rate-limiting / throttling по API для endpoints, генерирующих большую долю запросов.

- Самплинг/пагинация: уменьшить глубину запроса и выдавать предварительно агрегированные результаты.

2.2. Предварительное разбиение

Если «горячая» категория сконцентрирована в небольшом диапазоне ключей (при range sharding), можно заранее разрезать (split) диапазон на несколько чанков и затем переместить их по разным шардам.

Пример команд:

```js

// разбить по точке, где лежит «горячая» категория

sh.splitAt("mobileshopdb.products", { category:  "electronics", _id:  ObjectId("...") })

// посмотреть chunk, который содержит ключ

sh.find({ ns:  "mobileshopdb.products", query: { category:  "electronics" } })

// переместить chunk на другой шард

sh.moveChunk("mobileshopdb.products", { category:  "electronics" }, "shardB")

```

2.3. Изменение shard key / Resharding


```js

// новый ключ — составной: { category:1, _id: "hashed" }

db.adminCommand({

reshardCollection:  "mobileshopdb.products",

key: { category:  1, _id:  "hashed" },

unique:  false

})

```

Операция ресурсоемкая.
  

2.4  Добавление шардов / autoscaling

Автоматическое добавление новых шардов и запуск balancer'а для перераспределения чанков.

# Задание 9 — Настройка чтения с реплик и консистентность


Цель

Для коллекций `products`, `orders` и `carts` определить, какие операции чтения допустимо выполнять с `secondary` реплики, а какие должны читать только с `primary`. Указать допустимую задержку репликации (replication lag) для операций на `secondary` и обосновать выбор с точки зрения консистентности, частоты обновлений и бизнес-логики.

Краткая сводка/рекомендация

- Критические операции, связанные с модификацией состояния (checkout / списание / подтверждение оплаты / критические обновления статуса заказа), должны читать с `primary` (readPreference: "primary").
- Операции, ориентированные на отображение каталога и исторические данные (каталог товаров, страницы товаров, история заказов с некоторой задержкой) могут использовать `secondary` (readPreference: "secondaryPreferred") при условии контроля допустимой репликационной задержки и использования readConcern подходящего уровня.
- Для операций, требующих согласованности после записи (read-after-write), использовать сессии с causal consistency или читать с `primary` / readConcern="majority".

  
Таблица: 

| Коллекция | Операция чтения | Рекомендуемый target | Комментарий / обоснование |
|---|---|---|---|
| products | Страница товара (информация, описание, характеристики) | secondaryPreferred | Небольшая устарелость допустима (несколько сотен мс — 1s), для UX быстрее отдавать с replica; не влияет на наличие на складе. |
| products | Доступность / проверка перед оплатой | primary | Критично: риск продажи отсутствующего товара. Чтение должно отражать актуальное состояние (read from primary или readConcern: majority после записи). |
| products | Каталог / листинг / поиск | secondaryPreferred | Высокий RPS, устойчивая eventual consistency допустима; кеширование рекомендуется. |
| orders | Получение списка заказов пользователя (история) | secondaryPreferred | Небольшая задержка статусов допустима; пользователю достаточно P90/P99 с небольшим лагом. Для свежесозданного заказа использовать causal session / read-after-write. |
| orders | Проверка статуса во время оплаты | primary | Важна точность статуса (paid/cancelled). Чтение с primary гарантирует актуальность. |
| orders | Поиск по order_id (подтверждение покупки) | primary or majority | Если нужен абсолютный консистентный ответ сразу после записи — читать с primary или readConcern: majority. |
| carts | Получение корзины пользователя (UI) | primary | Cart — часто обновляемая сущность (add/remove). Риск рассинхронизации при показе устаревших данных — читать с primary. |
| carts | Гостевая корзина (только чтение, не во время мерджа) | secondaryPreferred | Если не выполняется merge, чтение guest-cart допустимо со вторичных с малым лагом. |

Дополнительные заметки:
- Все операции, выполняющие write+read в одном потоке (read-after-write), должны использовать causal sessions (сессии драйвера с causalConsistency=true) или читать с primary, чтобы избежать наблюдения старых данных.
- Для UI-операций, где пользователю допустима устарелость, используйте `secondaryPreferred` + readConcern="local" и, при необходимости, кеширование на уровне приложения.

Допустимая задержка сильно зависит от бизнес-требований и профиля нагрузки. Ниже рекомендации, которые можно использовать как исходные пороги:

- Стандартные (catalog, listing): lag <= 1s — приемлемо (warn при >1s, critical при >5s).

- UI страницы товара (non-stock fields): lag <= 1s — приемлемо.

- Чтение корзины UI: lag <= 200–500ms (предпочтительно) — лучше читать primary, но если читаем со secondaries, держать lag низким.

- История заказов: lag <= 2s — обычно допустимо; если требуется read-after-write — использовать causal session.

- Крайние проверки, проверки оплаты и т.д.: читаем только с primary (lag не применим — primary всегда актуален).


# Задание 10 — Миграция на Cassandra: модель данных, стратегии репликации и шардирования

10.1. Какие данные критичны и целесообразно хранить в Cassandra

- Orders — критичны по целостности, но нагрузка — очень высокая на последовательные записи (create order) во время Black Friday. Cassandra хорошо масштабируется на запись и обеспечивает высокую доступность. 
- Order history - история заказов пользователя.
- Products (каталог) — обычно read-heavy и редко обновляемый набор полей (описание, цена, атрибуты). Cassandra подходит для быстрого масштабируемого чтения.

------------------------------------------------------------

10.2. Концептуальные модели (CQL) и выбор partition / clustering keys

  

Ключевые принципы при проектировании в Cassandra:
- partition key определяет распределение данных по узлам (token-aware routing);
- clustering keys определяют порядок внутри партиции и позволяют эффективные range/limit запросы;
- избегаем естественных «горячих» партиций (один partition key, на который падает 70% запросов): применяем bucketing (sharding внутри partition) или дополнительный префикс.

Ниже — примеры схем для Orders, OrderHistory, Products, Carts, Sessions.

1) orders — запись заказа (в единой записи, write-heavy)

Цели: быстрые writes, чтение по order_id (подтверждение), чтение recent orders (по user) — hybrid, все заказы пользователя лежат в одной партиции.

2) order_events (событийная модель) — append-only для аналитики/репликации

3) products — карточка товара, read-heavy, редкие обновления
product_id как partition key равномерно распределяет товары. 

4) carts — mutable, короткоживущая сущность
5) sessions
TTL обеспечивает автоматическое очищение.


- Используем user_id or product_id как partition keys для равномерного распределения по токенам.
- Для потенциально «горячих» партиций (популярные категории, супер-юзеры) применяем bucketing (добавочный префикс) либо time-buckets.
- Cassandra при добавлении ноды не делает глобального ре-шифтинга, поэтому масштабирование менее болезненно, чем range-based sharding в Mongo.

------------------------------------------------------------

10.3. Стратегии целостности и согласованности: Hinted Handoff, Read Repair, Anti-Entropy


- Cassandra — eventually consistent by default. Гарантии согласованности достигаются выбором consistency level (CL) при запросах и применением механизмов репликации/repair.
- Баланс: lower CL (ONE) = низкая задержка, высокая доступность; higher CL (QUORUM/LOCAL_QUORUM) = сильнее согласованность, больше latency.

1) Hinted Handoff

Когда replica недоступна, координатор сохраняет hint и позднее отправляет его на восстановившийся узел. Полезно при кратковременных отрубах нод/сетей во время пиков нагрузки. Он повышает availability tanpa немедленного resync.
Подходит везде (orders, products backup, sessions), т.к. позволяет быстро принимать записи.

2) Read Repair

При чтении координатор видит различия между репликами и пытается синхронизировать их (sync on read).
Для часто читаемых сущностей (products, order status) read repair помогает держать реплики согласованными.


3) Anti-Entropy Repair (nodetool repair)

Периодическая полная синхронизация данных между узлами.
Планировать регулярные repair (например, daily/weekly, или чаще для hot ranges). Обязателен при использовании CL > ONE и при long-running clusters.
Критично для orders и order_history — запускайте repair чаще (например, nightly) и после добавления нод.
