## Задание 7. Проектирование схем коллекций для шардирования данных

### 7.1. Коллекция `orders`

#### Схема документа

```json
{
  "_id": ObjectId,
  "order_id": String,          // Уникальный идентификатор заказа
  "user_id": String,           // Идентификатор клиента
  "created_at": ISODate,       // Дата и время оформления заказа
  "items": [                   // Список заказанных товаров
    {
      "product_id": String,
      "name": String,
      "quantity": Number,
      "price": Number
    }
  ],
  "status": String,            // Статус заказа: "pending", "processing", "shipped", "delivered", "cancelled"
  "total_amount": Number,      // Общая сумма заказа
  "geo_zone": String           // Геозона заказа: "moscow", "ekaterinburg", "kaliningrad" и т.д.
}
```

#### Индексы

```javascript
db.orders.createIndex({ "user_id": 1, "created_at": -1 })  // для истории заказов пользователя
db.orders.createIndex({ "order_id": 1 }, { unique: true })  // уникальный идентификатор заказа
db.orders.createIndex({ "status": 1 })                      // для фильтрации по статусу
```

#### Выбор shard-ключа

**Shard-ключ:** `{ "user_id": "hashed" }`

**Обоснование:**
- **Равномерное распределение:** Хешированный `user_id` обеспечивает равномерное распределение данных по шардам, т.к. хеш-функция нивелирует неравномерность в распределении пользователей.
- **Поиск истории заказов:** Основная операция чтения — поиск заказов конкретного пользователя (`user_id`). При использовании `user_id` как shard-ключа все заказы пользователя попадают на один шард, что обеспечивает targeted queries (запрос идёт на один шард, а не scatter-gather).
- **Создание заказов:** Новые заказы распределяются равномерно, т.к. приходят от разных пользователей.

**Риски:**
- Если один пользователь генерирует непропорционально много заказов (например, B2B-клиент), его шард может стать «горячим». Однако для B2C-магазина это маловероятно.

**Команда шардирования:**
```javascript
sh.shardCollection("somedb.orders", { "user_id": "hashed" })
```

### 7.2. Коллекция `products`

#### Схема документа

```json
{
  "_id": ObjectId,
  "product_id": String,        // Уникальный идентификатор товара
  "name": String,              // Наименование
  "category": String,          // Категория: "electronics", "audio", "appliances", "accessories"
  "price": Number,             // Цена
  "stock": {                   // Остаток товара по геозонам
    "moscow": Number,
    "ekaterinburg": Number,
    "kaliningrad": Number
  },
  "attributes": {              // Дополнительные атрибуты
    "color": String,
    "size": String
  }
}
```

#### Индексы

```javascript
db.products.createIndex({ "product_id": 1 }, { unique: true })
db.products.createIndex({ "category": 1, "price": 1 })  // для поиска по категории и фильтрации по цене
db.products.createIndex({ "name": "text" })              // для текстового поиска
```

#### Выбор shard-ключа

**Shard-ключ:** `{ "category": 1, "_id": "hashed" }` (compound shard key)

**Обоснование:**
- **Поиск по категориям:** Основная операция чтения — поиск товаров по категории с фильтрацией по цене. Включение `category` в shard-ключ позволяет выполнять targeted queries при поиске по категории.
- **Равномерное распределение:** Хешированная компонента `_id` в compound key обеспечивает равномерное распределение внутри каждой категории, предотвращая ситуацию, когда все товары одной популярной категории (например, «Электроника») попадают на один шард.
- **Обновление остатков:** Обновления остатков распределяются равномерно, т.к. они привязаны к конкретным товарам.

**Альтернативный вариант — `{ "_id": "hashed" }`:**
Обеспечивает идеальное равномерное распределение, но все запросы по категории будут scatter-gather (на все шарды). Для магазина с большим объёмом каталога compound key предпочтительнее.

**Команда шардирования:**
```javascript
sh.shardCollection("somedb.products", { "category": 1, "_id": "hashed" })
```

### 7.3. Коллекция `carts`

#### Схема документа

```json
{
  "_id": ObjectId,
  "user_id": String,           // Идентификатор пользователя (null для гостей)
  "session_id": String,        // ID сессии для гостей
  "items": [                   // Список товаров
    {
      "product_id": String,
      "quantity": Number
    }
  ],
  "status": String,            // Статус: "active", "ordered", "abandoned"
  "created_at": ISODate,       // Дата создания
  "updated_at": ISODate,       // Дата последнего обновления
  "expires_at": ISODate        // TTL — для автоматической очистки
}
```

#### Индексы

```javascript
db.carts.createIndex({ "user_id": 1, "status": 1 })       // для получения активной корзины пользователя
db.carts.createIndex({ "session_id": 1, "status": 1 })     // для получения активной корзины гостя
db.carts.createIndex({ "expires_at": 1 }, { expireAfterSeconds: 0 })  // TTL-индекс для автоматической очистки
```

#### Выбор shard-ключа

**Shard-ключ:** `{ "_id": "hashed" }`

**Обоснование:**
- **Равномерное распределение:** Хешированный `_id` обеспечивает максимально равномерное распределение корзин по шардам.
- **Природа операций:** Большинство операций с корзинами (добавление/удаление товара, слияние) работают с конкретной корзиной по `_id`. Получение корзины по `session_id` или `user_id` — это lookup по индексу, который будет scatter-gather, но это допустимо, т.к.:
  - У каждого пользователя/сессии обычно одна активная корзина.
  - Операция поиска выполняется с фильтром по `status: "active"`, что сужает выборку.
- **Высокая частота записей:** Корзины обновляются очень часто (добавление товаров), хешированный ключ равномерно распределяет нагрузку на запись.
- **TTL-очистка:** Автоматическое удаление старых корзин через `expires_at` TTL-индекс распределяется равномерно по шардам.

**Риски:**
- Запросы `{ session_id, status: "active" }` и `{ user_id, status: "active" }` будут scatter-gather, т.к. shard-ключ не содержит эти поля. Однако с учётом того, что результат — обычно один документ, и с учётом индексов, производительность будет приемлемой.

**Команда шардирования:**
```javascript
sh.shardCollection("somedb.carts", { "_id": "hashed" })
```

## Задание 8. Выявление и устранение «горячих» шардов

### 8.1. Метрики мониторинга

#### Набор ключевых метрик

| Метрика                           | Описание                                                    | Источник                              | Пороговое значение                  |
|----------------------------------|-------------------------------------------------------------|---------------------------------------|-------------------------------------|
| **opcounters per shard**          | Количество операций (insert, query, update, delete) на шард | `db.serverStatus().opcounters`        | Разброс > 30% между шардами        |
| **chunk count per shard**         | Количество чанков на каждом шарде                           | `sh.status()`                         | Разброс > 20% между шардами        |
| **document count per shard**      | Количество документов на каждом шарде                       | `db.collection.getShardDistribution()`| Разброс > 20%                      |
| **data size per shard**           | Размер данных на каждом шарде                               | `db.collection.getShardDistribution()`| Разброс > 25%                      |
| **avg query latency per shard**   | Средняя задержка запросов на шард                           | `db.currentOp()`, profiler            | > 100ms                            |
| **disk I/O per shard**            | Использование дисковой подсистемы                           | OS-level метрики (iostat)             | > 80% утилизация                   |
| **CPU usage per shard**           | Использование процессора                                    | OS-level метрики                      | > 80%                              |
| **replication lag**               | Задержка репликации                                         | `rs.status()`                         | > 10 секунд                        |
| **connections per shard**         | Количество подключений                                      | `db.serverStatus().connections`       | > 80% от maxConns                  |
| **lock percentage**               | Процент времени под блокировками                            | `db.serverStatus().locks`             | > 5%                               |

#### Команды диагностики

**Проверка распределения данных:**
```javascript
// Общий статус шардирования
sh.status()

// Распределение данных по шардам для конкретной коллекции
db.products.getShardDistribution()

// Статистика базы данных
db.stats()
```

**Проверка нагрузки на шард:**
```javascript
// Статус сервера с счётчиками операций
db.serverStatus().opcounters

// Текущие операции
db.currentOp({ "secs_running": { "$gte": 5 } })

// Включение профайлера для медленных запросов (> 100ms)
db.setProfilingLevel(1, { slowms: 100 })
db.system.profile.find().sort({ ts: -1 }).limit(10)
```

**Проверка распределения чанков:**
```javascript
// Количество чанков на каждом шарде
use config

  { $group: { _id: "$shard", count: { $sum: 1 } } }

```

### 8.2. Механизмы автоматического перераспределения данных

#### 1. Балансировщик MongoDB (Balancer)

MongoDB имеет встроенный балансировщик, который автоматически перемещает чанки между шардами для выравнивания нагрузки.

```javascript
// Проверка статуса балансировщика
sh.getBalancerState()

// Включение балансировщика (если выключен)
sh.startBalancer()

// Настройка окна балансировки (чтобы не влиять на пиковую нагрузку)
db.settings.update(
  { _id: "balancer" },
  { $set: { activeWindow: { start: "02:00", stop: "06:00" } } },
  { upsert: true }
)
```

#### 2. Ручное перемещение чанков

Если балансировщик не справляется, можно переместить чанки вручную:

```javascript
// Перемещение конкретного чанка с горячего шарда
sh.moveChunk("somedb.products", { category: "electronics" }, "shard2")
```

#### 3. Pre-splitting (предварительное разбиение)

Для предотвращения горячих шардов при первоначальной загрузке данных:

```javascript
// Предварительное разбиение коллекции products по категориям
sh.splitAt("somedb.products", { "category": "audio" })
sh.splitAt("somedb.products", { "category": "electronics" })
sh.splitAt("somedb.products", { "category": "appliances" })
```

#### 4. Tag-Aware Sharding (зонирование)

Для управления распределением данных по конкретным шардам:

```javascript
// Назначение тегов шардам
sh.addShardTag("shard1", "popular")
sh.addShardTag("shard2", "popular")
sh.addShardTag("shard1", "regular")

// Создание зон для категорий
sh.addTagRange(
  "somedb.products",
  { "category": "electronics", "_id": MinKey },
  { "category": "electronics", "_id": MaxKey },
  "popular"  // распределить "Электронику" на оба шарда с тегом "popular"
)
```

#### 5. Изменение стратегии шардирования

Если проблема системная, может потребоваться переход с Range на Hashed sharding:

```javascript
// Для новой коллекции — использование хешированного ключа вместо range
sh.shardCollection("somedb.products_v2", { "_id": "hashed" })

// Миграция данных из старой коллекции
db.products.find().forEach(function(doc) {
  db.products_v2.insertOne(doc)
})
```

### 8.3. План действий при обнаружении горячего шарда

1. **Диагностика**: Определить, какой шард перегружен и какая коллекция/категория вызывает проблему (`getShardDistribution()`, `opcounters`).
2. **Краткосрочные меры**: Ручное перемещение чанков (`moveChunk`) с горячего шарда на менее загруженные.
3. **Среднесрочные меры**: Настройка tag-aware sharding для распределения популярных категорий по нескольким шардам.
4. **Долгосрочные меры**: Пересмотр shard-ключа, переход на compound key или hashed sharding.

## Задание 9. Настройка чтения с реплик и консистентность

### 9.1. Таблица операций чтения

#### Коллекция `orders`

| Операция                          | Read Preference      | Обоснование                                                                                         |
|----------------------------------|----------------------|-----------------------------------------------------------------------------------------------------|
| Создание заказа                   | **primary**          | Запись — всегда на primary                                                                          |
| Обновление статуса заказа         | **primary**          | Запись — всегда на primary                                                                          |
| Отображение статуса заказа        | **primary**          | Пользователь ожидает видеть актуальный статус. Отображение устаревшего статуса может вызвать повторные обращения в поддержку |
| Поиск истории заказов пользователя| **secondaryPreferred** | Допустима задержка до 5с. История не требует мгновенной актуальности, т.к. новые заказы появляются редко |
| Аналитические запросы (отчёты)    | **secondary**        | Тяжёлые запросы не должны нагружать primary. Допустима задержка до 30с                               |

#### Коллекция `products`

| Операция                          | Read Preference      | Обоснование                                                                                         |
|----------------------------------|----------------------|-----------------------------------------------------------------------------------------------------|
| Обновление остатков               | **primary**          | Запись — всегда на primary. Критически важно для предотвращения overselling                          |
| Описание товара (страница продукта)| **secondaryPreferred** | Описание и цена меняются редко. Допустима задержка до 10с                                           |
| Поиск по категориям + фильтрация  | **secondaryPreferred** | Каталог обновляется нечасто. Допустима задержка до 10с. Снимает нагрузку с primary                  |
| Проверка остатков (при оформлении) | **primary**          | Необходима актуальная информация о наличии товара. Иначе можно продать товар, которого нет на складе |

#### Коллекция `carts`

| Операция                              | Read Preference  | Обоснование                                                                                     |
|--------------------------------------|------------------|-------------------------------------------------------------------------------------------------|
| Получение активной корзины            | **primary**      | Пользователь ожидает видеть актуальное содержимое корзины                                        |
| Добавление/замена товара              | **primary**      | Запись — всегда на primary                                                                      |
| Удаление товара из корзины            | **primary**      | Запись — всегда на primary                                                                      |
| Слияние гостевой и пользовательской   | **primary**      | Критическая операция — необходимо читать и писать актуальные данные, иначе можно потерять товары  |
| Отметка корзины как заказанной        | **primary**      | Запись — всегда на primary                                                                      |
| Просмотр abandoned корзин (аналитика) | **secondary**    | Аналитическая операция, не критична по актуальности. Допустима задержка до 60с                   |

### 9.2. Допустимая задержка репликации

| Операция / Контекст                | Допустимая задержка | Обоснование                                                           |
|-----------------------------------|--------------------|--------------------------------------------------------------------- |
| Статус заказа (для пользователя)   | 0с (primary only)  | Пользователь ожидает видеть актуальный статус сразу после оплаты     |
| История заказов                    | до 5 секунд        | Новые заказы появляются нечасто, задержка не критична                 |
| Каталог товаров                    | до 10 секунд       | Цены и описания меняются редко                                       |
| Остатки (для витрины)              | до 10 секунд       | На витрине допустимо показывать слегка устаревшие остатки             |
| Остатки (при оформлении заказа)    | 0с (primary only)  | Риск overselling — нельзя продать товар, которого нет                |
| Активная корзина                   | 0с (primary only)  | Пользователь только что добавил товар и ожидает его видеть            |
| Аналитика (отчёты)                 | до 60 секунд       | Аналитические отчёты не требуют мгновенной актуальности              |

### 9.3. Обоснование выбора

**Почему большинство операций с корзинами идут на primary:**
Корзина — это объект с высокой частотой изменений (пользователь добавляет/удаляет товары за секунды). Чтение с secondary может показать устаревшее состояние (товар не добавлен, хотя пользователь только что нажал кнопку), что критически ухудшает UX.

**Почему каталог товаров можно читать с secondary:**
Каталог обновляется раз в минуты/часы (новые товары, изменение цен). Задержка в 10 секунд незаметна для пользователя. При этом перенос нагрузки на secondary значительно разгружает primary, особенно в период Black Friday, когда каталог просматривают тысячи пользователей одновременно.

**Почему проверка остатков при оформлении идёт на primary:**
Это ключевой бизнес-риск. Если прочитать устаревший остаток с secondary, можно принять заказ на товар, которого уже нет на складе. Это приводит к отменам заказов, негативным отзывам и финансовым потерям.

**Конфигурация read preference в pymongo:**

```python
from pymongo import ReadPreference

# Для каталога (чтение с secondary)
catalog_collection = db.get_collection(
    "products",
    read_preference=ReadPreference.SECONDARY_PREFERRED
)

# Для оформления заказа (только primary)
checkout_collection = db.get_collection(
    "products",
    read_preference=ReadPreference.PRIMARY
)

# С maxStalenessSeconds для ограничения задержки
from pymongo.read_preferences import SecondaryPreferred
catalog_collection = db.get_collection(
    "products",
    read_preference=SecondaryPreferred(max_staleness=10)
)
```

## Задание 10. Миграция на Cassandra

### 10.1. Определение критически важных данных

#### Анализ данных по критериям

| Сущность          | Объём записей | Частота записи  | Требования к целостности | Геораспределённость | Рекомендация           |
|------------------|--------------|-----------------|--------------------------|---------------------|------------------------|
| **Заказы (orders)**  | Высокий      | Средняя         | Высокая (транзакции)     | Да (по геозонам)    | Частично в Cassandra   |
| **Товары (products)**| Средний      | Средняя         | Высокая (остатки)        | Да (по геозонам)    | Оставить в MongoDB     |
| **Корзины (carts)**  | Очень высокий| Очень высокая   | Средняя (eventual OK)    | Нет                 | **Cassandra**          |
| **История заказов**  | Высокий      | Только append   | Средняя (immutable)      | Да                  | **Cassandra**          |
| **Пользовательские сессии** | Очень высокий | Очень высокая | Низкая (TTL, eventual OK)| Нет              | **Cassandra**          |

#### Обоснование выбора

**Данные, для которых Cassandra имеет смысл:**

1. **Корзины (carts):** Очень высокая частота записи (каждое действие пользователя — обновление корзины). Eventual consistency допустима — небольшая задержка не критична. Cassandra обеспечивает линейное масштабирование записи при добавлении узлов. Данные имеют TTL (корзины удаляются автоматически), что нативно поддерживается Cassandra.

2. **История заказов:** Append-only данные (заказы не изменяются после создания). Огромный объём данных, который постоянно растёт. Основной паттерн запросов — `SELECT * FROM orders WHERE user_id = ? ORDER BY created_at DESC`, что идеально ложится на модель Cassandra.

3. **Пользовательские сессии:** Очень высокая частота записи/чтения. Данные имеют TTL. Не требуют строгой консистентности. Leaderless-репликация Cassandra обеспечивает низкую latency.

**Данные, которые лучше оставить в MongoDB:**

1. **Товары (products):** Требуют сложных запросов (фильтрация по нескольким полям, текстовый поиск, агрегации). Обновление остатков требует atomic operations (inc/dec), что в Cassandra сложнее реализовать корректно. Cassandra не подходит для read-heavy workloads с ad-hoc queries.

2. **Активные заказы (текущие транзакции):** Требуют строгой консистентности (ACID). Необходимы атомарные обновления (изменение статуса + списание остатков). MongoDB с writeConcern: "majority" обеспечивает лучшие гарантии для транзакционных данных.

### 10.2. Концептуальная модель данных Cassandra

#### Таблица `orders_by_user` (история заказов)

```sql
CREATE TABLE orders_by_user (
    user_id       TEXT,
    order_date    TIMESTAMP,
    order_id      UUID,
    items         LIST<FROZEN<item_type>>,
    status        TEXT,
    total_amount  DECIMAL,
    geo_zone      TEXT,
    PRIMARY KEY ((user_id), order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC, order_id ASC);

CREATE TYPE item_type (
    product_id  TEXT,
    name        TEXT,
    quantity    INT,
    price       DECIMAL
);
```

**Partition key:** `user_id`
**Clustering key:** `order_date DESC, order_id ASC`

**Обоснование:**
- `user_id` как partition key — все заказы пользователя хранятся в одной партиции, что обеспечивает быстрый доступ к истории заказов.
- `order_date DESC` как clustering key — заказы автоматически сортируются по дате (новые первыми), что соответствует основному паттерну запросов.
- `order_id` в clustering key для уникальности (два заказа могут быть в одну секунду).
- **Равномерное распределение:** `user_id` обеспечивает хорошее распределение по узлам, т.к. пользователей много и они различные.
- **Горячие партиции:** Риск минимален для B2C-магазина, т.к. у каждого пользователя десятки-сотни заказов, а не миллионы.

**Пример запроса:**
```sql
-- История заказов пользователя (последние 20)
SELECT * FROM orders_by_user WHERE user_id = 'user123' LIMIT 20;

-- Заказы за период
SELECT * FROM orders_by_user
WHERE user_id = 'user123'
  AND order_date >= '2026-01-01'
  AND order_date <= '2026-02-01';
```

#### Таблица `carts_by_session` (корзины)

```sql
CREATE TABLE carts_by_session (
    session_id    TEXT,
    user_id       TEXT,
    cart_id       UUID,
    items         LIST<FROZEN<cart_item_type>>,
    status        TEXT,      -- "active", "ordered", "abandoned"
    created_at    TIMESTAMP,
    updated_at    TIMESTAMP,
    PRIMARY KEY ((session_id), status, cart_id)
) WITH default_time_to_live = 86400;  -- TTL 24 часа

CREATE TYPE cart_item_type (
    product_id  TEXT,
    quantity    INT
);
```

**Partition key:** `session_id`
**Clustering key:** `status, cart_id`

**Обоснование:**
- `session_id` как partition key — каждая сессия имеет свою партицию, что обеспечивает быстрый доступ к корзине по сессии.
- `status` в clustering key позволяет эффективно фильтровать активные корзины.
- TTL = 24 часа для автоматической очистки неактивных корзин.
- **Равномерное распределение:** Сессии уникальны и генерируются случайно (UUID или random token), что обеспечивает идеальное распределение.
- **Горячие партиции:** Риск отсутствует, т.к. каждая сессия — это один пользователь с одной корзиной.

**Дополнительная таблица для поиска по user_id:**

```sql
CREATE TABLE carts_by_user (
    user_id       TEXT,
    status        TEXT,
    session_id    TEXT,
    cart_id       UUID,
    items         LIST<FROZEN<cart_item_type>>,
    created_at    TIMESTAMP,
    updated_at    TIMESTAMP,
    PRIMARY KEY ((user_id), status, cart_id)
) WITH default_time_to_live = 86400;
```

**Обоснование дублирования:** В Cassandra принято моделировать данные под запросы (query-driven design). Отдельная таблица для поиска по `user_id` позволяет выполнять `SELECT * FROM carts_by_user WHERE user_id = ? AND status = 'active'` без ALLOW FILTERING.

**Пример запросов:**
```sql
-- Получение активной корзины гостя
SELECT * FROM carts_by_session WHERE session_id = 'sess_abc' AND status = 'active';

-- Получение активной корзины пользователя
SELECT * FROM carts_by_user WHERE user_id = 'user123' AND status = 'active';
```

#### Таблица `user_sessions` (пользовательские сессии)

```sql
CREATE TABLE user_sessions (
    session_id    TEXT,
    user_id       TEXT,
    created_at    TIMESTAMP,
    last_active   TIMESTAMP,
    ip_address    TEXT,
    user_agent    TEXT,
    PRIMARY KEY ((session_id))
) WITH default_time_to_live = 3600;  -- TTL 1 час
```

**Partition key:** `session_id`

**Обоснование:**
- Одна сессия = одна партиция. Минимальный размер партиции, быстрый доступ.
- TTL обеспечивает автоматическую очистку.
- Равномерное распределение гарантировано случайными session_id.

### 10.3. Стратегии обеспечения целостности данных

#### Обзор стратегий

| Стратегия              | Механизм                                                | Latency impact | Гарантия согласованности |
|-----------------------|--------------------------------------------------------|----------------|-------------------------|
| **Hinted Handoff**     | Узел-координатор сохраняет hint и доставляет при восстановлении | Минимальный    | Eventual                |
| **Read Repair**        | При чтении обнаруживает расхождения и синхронизирует реплики     | Средний        | При чтении              |
| **Anti-Entropy Repair**| Периодическое фоновое сравнение данных с помощью Merkle Trees   | Нулевой (фон)  | Полная (по расписанию)  |

#### Выбор стратегии по сущностям

**1. Корзины (carts) и Сессии (user_sessions) → Hinted Handoff**

```
Consistency Level: ONE (запись) / ONE (чтение)
```

**Обоснование:**
- **Низкая latency критична:** Пользователь добавляет товар в корзину и ожидает мгновенного отклика. При CL=ONE запись подтверждается одной репликой, что минимизирует latency.
- **Eventual consistency допустима:** Если одна реплика временно недоступна, Hinted Handoff доставит данные при её восстановлении. Небольшая задержка в синхронизации (секунды) не критична для корзин.
- **TTL делает данные эфемерными:** Корзины и сессии живут часы/сутки. Даже если произойдёт кратковременная рассинхронизация, данные скоро удалятся.
- **Компромисс:** Минимальная latency за счёт потенциально короткого окна несогласованности (обычно < 1 секунды).

**2. История заказов (orders_by_user) → Read Repair + Hinted Handoff**

```
Consistency Level: QUORUM (запись) / ONE (чтение)
```

**Обоснование:**
- **Запись с QUORUM:** При создании заказа данные записываются на большинство реплик. Это обеспечивает durability — заказ не потеряется.
- **Чтение с ONE + Read Repair:** При чтении истории заказов данные берутся с ближайшей реплики (низкая latency). Если обнаруживается расхождение (например, реплика отставала), Read Repair синхронизирует реплики.
- **Данные immutable:** Заказы после создания не изменяются (статус обновляется в MongoDB, а в Cassandra хранится только история). Это значит, что конфликтов при Read Repair быть не может.
- **Компромисс:** Первый запрос после сбоя может показать неполные данные, но Read Repair исправит это при следующем чтении.

**3. Все таблицы → Anti-Entropy Repair (фоновый процесс)**

```bash
# Запуск repair для конкретной таблицы (раз в неделю)
nodetool repair somedb orders_by_user

# Инкрементальный repair (быстрее)
nodetool repair -inc somedb carts_by_session

# Полный repair всего keyspace (раз в gc_grace_seconds / 2)
nodetool repair somedb
```

**Обоснование:**
- Anti-Entropy Repair запускается как фоновый процесс по расписанию (рекомендуется не реже чем `gc_grace_seconds / 2`, обычно раз в неделю).
- Сравнивает данные между репликами с помощью Merkle Trees и устраняет расхождения.
- Не влияет на latency запросов, т.к. работает в фоне.
- Гарантирует полную согласованность данных на всех репликах.
- **Особенно важно для корзин:** Т.к. корзины используют CL=ONE, Anti-Entropy Repair — это страховка на случай, если Hinted Handoff не сработал (например, узел был недоступен дольше `max_hint_window`).

#### Сводная таблица

| Сущность            | Hinted Handoff | Read Repair | Anti-Entropy Repair | Write CL | Read CL | Обоснование                                    |
|--------------------|---------------|-------------|--------------------|---------|---------|-------------------------------------------------|
| carts              | Основная       | —           | Фоновая (1/нед)    | ONE     | ONE     | Latency важнее, eventual OK, TTL                |
| user_sessions      | Основная       | —           | Фоновая (1/нед)    | ONE     | ONE     | Latency важнее, TTL, эфемерные данные           |
| orders_by_user     | Дополнительная | Основная    | Фоновая (1/нед)    | QUORUM  | ONE     | Durability записи + быстрое чтение с repair      |

#### Конфигурация репликации

```sql
CREATE KEYSPACE somedb
WITH replication = {
  'class': 'NetworkTopologyStrategy',
  'dc_moscow': 3,
  'dc_ekaterinburg': 3,
  'dc_kaliningrad': 3
};
```

**Обоснование:**
- `NetworkTopologyStrategy` — для геораспределённой системы.
- 3 реплики в каждом ДЦ — обеспечивает доступность при выходе из строя до 1 узла в каждом ДЦ и позволяет использовать `LOCAL_QUORUM` для операций, требующих более сильных гарантий.
