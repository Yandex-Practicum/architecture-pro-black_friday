# Проектирование схем коллекций для шардирования данных

## Схема коллекции orders

```
{
  "_id": ObjectId,
  "clientId": String,
  "dateCreated": Date,
  "items": Product[],
  "status": "pending" | "shipped" | "delivered" | "canceled",
  "totalPrice": Number,
  "geo": {
    "type": "Point",
    "coordinates": [longitude, latitude]
  }
}
```

Основные операции:

- Быстрое создание заказов с одновременным списанием остатков.
- Поиск истории заказов конкретного пользователя.
- Отображение статуса заказа.

```
sh.enableSharding("somedb")
sh.shardCollection("somedb.orders", {clientId: "hashed"})

1. Создание заказа:
db.orders.insertOne({
  clientId: "user123",
  dateCreated: new Date(),
  items: [...],
  status: "pending",
  totalPrice: 100,
  geo: {
    type: "Point",
    coordinates: [longitude, latitude]
  }
})

2. Получение истории заказов пользователя:
db.orders.find({ clientId: "user123" })

3. Получение статуса заказа:
db.orders.findOne({ _id: ObjectId("orderId123") }, { status: 1 })
```

### Стратегия шардирования

Использовать хэшированное шардирование по полю _clientId_.

Для коллекции _orders_ оптимально использовать шардирование по полю _clientId_, так как это позволит распределять данные одного и того же клиента на одном шарде, обеспечивая высокую производительность при выполнении запросов, связанных с историей заказов конкретного пользователя.

## Схема коллекции products

```
{
  "_id": ObjectId,
  "name": String,
  "category": String,
  "price": Number,
  "stock": [
    {
      "warehouseId": String,
      "count": Number
    }
  ],
  "attributes": [
    {
      "key": String,
      "value": String
    }
  ]
}
```

Основные операции:

- Частые обновления остатков при покупках.
- Поиск товаров по категориям и фильтрация по диапазону цен.
- Описание товара на странице продукта.

```
sh.enableSharding("somedb")
sh.shardCollection("somedb.products", {price: 1})

1. Обновление остатков при покупке:
db.products.updateOne(
  { _id: ObjectId("productId123"), "stock.warehouseId": "warehouse123" },
  { $inc: { "stock.$.count": -1 } }
)

2. Поиск товаров по категории и фильтрация по цене:
db.products.find({ category: "electronics", price: { $gte: 100, $lte: 500 } })

3. Получение описания товара:
db.products.findOne({ _id: ObjectId("productId123") })

```

### Стратегия шардирования

Использовать диапазонное шардирование по полю _price_.

Для коллекции _products_ оптимально использовать диапазонное шардирование по полю _price_, это позволит производить более быстрый поиск товаров по определенной цене, так как значения цен будут лежать в одном диапазоне на одном шарде.

## Схема коллекции carts

```
{
  "_id": ObjectId,
  "clientId": String,
  "sessionId": String,
  "items": [
    {
      "productId": ObjectId,
      "quantity": Number
    }
  ],
  "status": "active" | "ordered" | "abandoned",
  "createdAt": Date,
  "updatedAt": Date,
  "expireAt": Date
}
```

Основные операции:

- Создание корзины, когда заходит гость или новый пользователь.
- Получение текущей корзины по фильтру { sessionId, status:"active" } или { clientId, status:"active" }.
- Добавление или замена товара в корзине.
- Удаление товара из корзины.
- Слияние гостевой корзины в пользовательскую, если пользователь залогинится:
  - прочитать гостевую { sessionId, status:"active" };
  - добавить её items в корзину { clientId, status:"active" };
  - отметить гостевую как abandoned.
- Отметка корзины как заказанной.

```
sh.enableSharding("somedb")
sh.shardCollection("somedb.carts", {_id: "hashed"})

1. Создание корзины:
db.carts.insertOne({
  clientId: "user123",
  sessionId: "session123",
  items: [],
  status: "active",
  createdAt: new Date(),
  updatedAt: new Date(),
  expireAt: new Date(Date.now() + 24*60*60*1000) // истекает через 24 часа
})

2. Получение текущей корзины:
db.carts.findOne({ sessionId: "session123", status: "active" })
или
db.carts.findOne({ clientId: "user123", status: "active" })

3. Добавление товара в корзину:
db.carts.updateOne(
  { _id: ObjectId("cartId123"), "items.productId": ObjectId("productId123") },
  { $inc: { "items.$.quantity": 1 } }
)
или если товара нет в корзине:
db.carts.updateOne(
  { _id: ObjectId("cartId123") },
  { $push: { items: { productId: ObjectId("productId123"), quantity: 1 } } }
)

4. Удаление товара из корзины:
db.carts.updateOne(
  { _id: ObjectId("cartId123") },
  { $pull: { items: { productId: ObjectId("productId123") } } }
)

5. Слияние гостевой корзины в пользовательскую:
const guestCart = db.carts.findOne({ sessionId: "session123", status: "active" });
db.carts.updateOne(
  { clientId: "user123", status: "active" },
  { $push: { items: { $each: guestCart.items } } }
);
db.carts.updateOne(
  { sessionId: "session123", status: "active" },
  { $set: { status: "abandoned" } }
)

6. Отметка корзины как заказанной:
db.carts.updateOne(
  { _id: ObjectId("cartId123") },
  { $set: { status: "ordered" } }
)
```

### Стратегия шардирования

Использовать хэшированное шардирование по полю _\_id_.

Для коллекции _carts_ оптимально использовать хэшированное шардирование по полю _\_id_, так как поле не изменяется после создания и присутствует всегда, это обеспечит равномерное распределение данных по всем шардам.

# Выявление и устранение «горячих» шардов

Для определения горячих шардов и отслеживания их состояния можно использовать следующие метрики:

- `sh.status()` - для получения информации о распределении данных и загрузке каждого шарда.
- `db.collection.stats()` - для получения статистики по коллекции.
- `db.setProfilingLevel(2)` - для включения профилирования запросов и выявления медленных запросов.

Если товары из популярных категорий начинают создавать непропорциональную нагрузку то можно:

- Использовать балансир Mongodb для перераспределения данных между шардами.

```
sh.startBalancer()
```

- Сделать reshard коллекции, выбрав другое поле для шардирования, например `_id`

```
db.adminCommand({ reshardCollection: "somedb.products", key: { _id: "hashed" } })
```

- Добавить составной ключ для шардирования, например `{ category: 1, _id: "hashed" }`, чтобы распределить нагрузку более равномерно по категориям.

```
db.adminCommand({ reshardCollection: "somedb.products", key: { category: 1, _id: "hashed" } })
```

# Настройка чтения с реплик и консистентность

| Операции                                    | Чтение с Primary | Чтение с Secondary | Описание                                                                                     |
| ------------------------------------------- | ---------------- | ------------------ | -------------------------------------------------------------------------------------------- |
| Создание заказа                             | Да               | Нет                | Гарантирует, что заказ будет актуальным и корректным.                                        |
| Получение истории заказов                   | Да               | Да                 | Можно читать с реплик, так как данные не критичны к моменту чтения.                          |
| Получение статуса заказа                    | Да               | Нет                | Для получения актуального статуса заказа лучше читать с основного шарда.                     |
| Обновление остатков                         | Да               | Нет                | Есть риск оформить отгрузку с некорректными данными, поэтому лучше читать с основного шарда. |
| Поиск товаров по категории и цене           | Да               | Да                 | Можно читать с реплик, так как это часто выполняемые операции и данные не обновляются часто  |
| Получение описания товара                   | Да               | Да                 | Можно читать с реплик, так как данные не критичны к моменту чтения.                          |
| Создание корзины                            | Да               | Нет                | Гарантирует, что корзина будет создана с актуальными данными.                                |
| Получение текущей корзины                   | Да               | Нет                | Лучше читать с основного шарда, чтобы получить актуальные данные.                            |
| Добавление товара в корзину                 | Да               | Нет                | Данные должны быть актуальными, поэтому лучше читать с основного шарда.                      |
| Удаление товара из корзины                  | Да               | Нет                | Данные должны быть актуальными, поэтому лучше читать с основного шарда.                      |
| Слияние гостевой корзины в пользовательскую | Да               | Нет                | Чтобы не потерять корзину нового пользователя, лучше читать с основного шарда.               |
| Отметка корзины как заказанной              | Да               | Нет                | Актуальный статус заказа важен, поэтому лучше читать с основного шарда.                      |

Допустимая задержка репликации для чтения с реплик может быть до 1 секунды, так как большинство операций не требуют мгновенной консистентности.

# Миграция на Cassandra

Критически важные данные:

- Заказы (orders)
- Корзины (carts)
- Остатки товаров (stock)

Для них имеет смысл использовать Cassandra, так как эти данные требуют высокой доступности и масштабируемости.

Они являются критически важными, так как от них зависит корректная работа системы и удовлетворенность клиентов. Меделенная скорость обработки заказов и некорректные данные о корзинах могут привести к потере клиентов и снижению продаж.
Некорретные данные об остатках товаров могут привести к оформлению заказов с недоступными товарами, что также негативно скажется на опыте клиентов.

Можно также рассмотреть использование Cassandra для хранения информации о товарах (products), если требуется высокая производительность при чтении и масштабируемость, особенно если количество товаров очень большое.

## Модель данных в Cassandra

Для заказов (orders):

```sql
CREATE TABLE orders (
  order_id UUID PRIMARY KEY,
  client_id TEXT,
  date_created TIMESTAMP,
  items LIST<FROZEN<item>>,
  status TEXT,
  total_price DECIMAL,
  geo FROZEN<geo_point>
);

CREATE TYPE item (
  product_id UUID,
  quantity INT
);

CREATE TYPE geo_point (
  longitude DOUBLE,
  latitude DOUBLE
);

-- Для оптимизации запросов по истории заказов конкретного пользователя можно создать дополнительную таблицу с шардированием по client_id:
CREATE TABLE orders_by_client (
  client_id TEXT,
  order_id UUID,
  date_created TIMESTAMP,
  items LIST<FROZEN<item>>,
  status TEXT,
  total_price DECIMAL,
  geo FROZEN<geo_point>,
  PRIMARY KEY (client_id, order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);

```

### orders_by_client

- Patrition key: client_id. Горячих партиций не будет, так как клиенты будут распределены равномерно.
- Clustering key: order_id. Позволяет быстро получать историю заказов конкретного пользователя

Для корзин (carts):

```sql
CREATE TABLE carts (
  cart_id UUID PRIMARY KEY,
  client_id TEXT,
  session_id TEXT,
  items LIST<FROZEN<cart_item>>,
  status TEXT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  expire_at TIMESTAMP
);

CREATE TYPE cart_item (
  product_id UUID,
  quantity INT
);

-- Для оптимизации запросов по текущей корзине можно создать дополнительную таблицу с шардированием по session_id:
CREATE TABLE carts_by_session (
  session_id TEXT,
  cart_id UUID,
  client_id TEXT,
  items LIST<FROZEN<cart_item>>,
  status TEXT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  expire_at TIMESTAMP,
  PRIMARY KEY (session_id, cart_id)
) WITH CLUSTERING ORDER BY (cart_id DESC);
```

### carts_by_session

- Partition key: session_id. Горячих партиций не будет, так как сессии будут распределены равномерно.
- Clustering key: cart_id. Позволяет быстро получать текущую корзину по session_id.

Для товаров (products):

```sql
CREATE TABLE products (
  product_id UUID PRIMARY KEY,
  name TEXT,
  category TEXT,
  price DECIMAL,
  stock LIST<FROZEN<stock_info>>,
  attributes LIST<FROZEN<attribute>>
);

CREATE TYPE attribute (
  key TEXT,
  value TEXT
);

CREATE TYPE stock_info (
  warehouse_id TEXT,
  count INT
);

-- Для оптимизации запросов по остаткам товаров можно создать дополнительную таблицу с шардированием по product_id:
CREATE TABLE stock_by_product (
  product_id UUID,
  warehouse_id TEXT,
  count INT,
  PRIMARY KEY (product_id, warehouse_id)
) WITH CLUSTERING ORDER BY (warehouse_id ASC);


-- Для оптимизации запросов по категории и цене можно создать дополнительную таблицу с шардированием по category. Чтобы избежать горячих партиций для популярных категорий, можно добавить bucket:
CREATE TABLE products_by_category (
  category TEXT,
  product_id UUID,
  bucket INT,
  name TEXT,
  price DECIMAL,
  stock LIST<FROZEN<stock_info>>,
  attributes LIST<FROZEN<attribute>>,
  PRIMARY KEY ((category, bucket), product_id)
) WITH CLUSTERING ORDER BY (product_id DESC);

-- Для оптимизации запросов по цене можно создать дополнительную таблицу с шардированием по price:
CREATE TABLE products_by_price (
  price DECIMAL,
  product_id UUID,
  name TEXT,
  category TEXT,
  stock LIST<FROZEN<stock_info>>,
  attributes LIST<FROZEN<attribute>>,
  PRIMARY KEY (price, product_id)
) WITH CLUSTERING ORDER BY (product_id DESC);
```

### products_by_category

- Partition key: category. Горячих партиций не будет, так как категории будут распределены равномерно.
- Clustering key: product_id. Позволяет быстро получать товары по категории.

## Целостность данных

Для обеспечения целостности данных в Cassandra можно использовать следующие механизмы:
- Для orders и carts использовать Read Repair, так как эти данные часто читаются и обновляются, это поможет поддерживать их актуальность.
- Для products можно использовать Hinted Handoff, так как эти данные не обновляются так часто, и это поможет обеспечить их доступность даже при временных сбоях.
- Anti-Entropy Repair стоит использовать при проблемах обнволения остатков товаров и заказов, так как это критически важные данные для бизнеса.
