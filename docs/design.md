## Задание 7. Проектирование схем коллекций для шардирования данных


### Коллекция `orders`

Коллекция orders включает заказы клиентов и содержит атрибуты:
- Уникальный идентификатор заказа (`_id`);
- Идентификатор клиента (`customer_id`);
- Дату и время оформления заказа (`created_at`);
- Список заказанных товаров и их цену (`items`);
- Статус заказа (`status`);
- Общую сумму заказа (`total`);
- Геозону заказа (`geo_zone`).

Каждый документ - один заказ.


### Структура документа

```json
{
  "_id": ObjectId("..."),
  "customer_id": "cust_8f14e45f",
  "created_at": ISODate("2026-04-04T12:34:56.789Z"),
  "items": [
    {
      "product_id": ObjectId("..."),
      "name": "PlayStation 5",
      "category": "Электроника",
      "price": Decimal128("54999.00"),
      "quantity": 1
    },
    {
      "product_id": ObjectId("..."),
      "name": "DualSense Controller",
      "category": "Электроника",
      "price": Decimal128("6999.00"),
      "quantity": 2
    },
    {
      "product_id": ObjectId("..."),
      "name": "«Мастер и Маргарита»",
      "category": "Книги",
      "price": Decimal128("890.00"),
      "quantity": 1
    }
  ],
  "status": "pending",
  "total": Decimal128("68997.00"),
  "geo_zone": "RU-MOW"
}
```

### Поля документа

| Поле                 | Тип          | Описание                                                                     |
| :------------------- | :----------- | :--------------------------------------------------------------------------- |
| `_id`                | `ObjectId`   | Уникальный идентификатор заказа                                              |
| `customer_id`        | `string`     | Идентификатор клиента                                                        |
| `created_at`         | `ISODate`    | Дата и время оформления заказа                                               |
| `items`              | `array`      | Список заказанных товаров                                                    |
| `items[].product_id` | `ObjectId`   | Идентификатор товара                                                         |
| `items[].name`       | `string`     | Название товара                                                              |
| `items[].category`   | `string`     | Категория товара (`Электроника`, `Книги`)                                    |
| `items[].price`      | `Decimal128` | Цена за единицу                                                              |
| `items[].quantity`   | `int`        | Количество                                                                   |
| `status`             | `string`     | Статус заказа (`pending`, `processing`, `shipped`, `delivered`, `cancelled`) |
| `total`              | `Decimal128` | Общая сумма заказа                                                           |
| `geo_zone`           | `string`     | Геозона заказа                                                               |

### Основные операции:

- Быстрое создание заказов с одновременным списанием остатков.
- Поиск истории заказов конкретного пользователя.
- Отображение статуса заказа.

Считаю, что с учетом основных операций, следует использовать хэшированное шардирование. В качестве ключа предлагаю использовать `customer_id`.

Обоснование:

- customer_id имеет высокую кардинальность, что подходит для равномерного распределения данных;
- запросы истории заказов пользователя по customer_id будут targeted и не потребуют scatter-gather;
- при условии, что API во все операции чтения статуса и изменения заказа добавляет customer_id, запросы также можно направлять targeted на один shard.

### Создание коллекции и индексов

```javascript
use somedb

db.createCollection("orders")

db.orders.createIndex({ customer_id: 1, created_at: -1 })

sh.shardCollection("somedb.orders", { customer_id: "hashed" })
```

## Коллекция `products`

Коллекция products хранит сведения о товарах и включает такие атрибуты:
- Уникальный идентификатор товара (`_id`);
- Наименование (`name`);
- Категорию товара (`category`);
- Цену (`price`);
- Остаток товара в каждой геозоне (`stock`);
- Дополнительные атрибуты - цвет, размер (`attributes`).

Каждый документ - один товар.

### Структура документа

```json
{
  "_id": ObjectId("..."),
  "name": "Смартфон X",
  "category": "Электроника",
  "price": Decimal128("89990.00"),
  "stock": {
    "RU-SVE": 50,
    "RU-KGD": 30,
    "RU-MOW": 120,
    "RU-SPB": 75
  },
  "attributes": {
    "color": "чёрный",
    "size": null,
    "ram": "8 ГБ",
    "storage": "256 ГБ"
  }
}
```

### Поля документа

| Поле               | Тип          | Описание                                  |
| :----------------- | :----------- | :---------------------------------------- |
| `_id`              | `ObjectId`   | Уникальный идентификатор товара           |
| `name`             | `string`     | Наименование товара                       |
| `category`         | `string`     | Категория товара (`Электроника`, `Книги`) |
| `price`            | `Decimal128` | Цена товара                               |
| `stock`            | `object`     | Остатки товара по геозонам                |
| `stock.<geo_zone>` | `int`        | Количество единиц в конкретной геозоне    |
| `attributes`       | `object`     | Дополнительные атрибуты товара            |

### Основные операции:

- Частые обновления остатков при покупках.
- Поиск товаров по категориям и фильтрация по диапазону цен.
- Описание товара на странице продукта.

С учетом основных операций, следует использовать хэшированное шардирование. В качестве ключа предлагаю использовать `_id`.

Обоснование:

- высокий cardinality и равномерное распределение документов;
- targeted операции чтения карточки товара по `_id`;
- targeted update остатков конкретного товара по `_id`.

Ограничения:

- scatter-gather + индекс category, price для операции поиска товаров по категориям и фильтрация по диапазону цен.

### Создание коллекции и индексов

```javascript
use somedb

db.createCollection("products")

db.products.createIndex({ category: 1, price: 1 })

sh.shardCollection("somedb.products", { _id: "hashed" })
```

## Коллекция `carts`

Коллекция carts хранит данные о текущих корзинах (как гостевых, так и пользовательских) и включает атрибуты:
- Уникальный идентификатор корзины (`_id`);
- Идентификатор владельца (`owner_id`) - `user_id` для авторизованных, `session_id` для гостей;
- Тип владельца (`owner_type`): `"user"` | `"session"`;
- Список товаров (`items`): массив документов `{ product_id, quantity }`;
- Статус корзины (`status`): `"active"` | `"ordered"` | `"abandoned"`;
- Дату и время создания (`created_at`);
- Дату и время последнего обновления (`updated_at`);
- Время удаления (`expires_at`) - для автоматической очистки старых корзин.

Каждый документ - одна корзина.

### Структура документа

Пользовательская корзина:

```json
{
  "_id": ObjectId("..."),
  "owner_id": "user_abc123",
  "owner_type": "user",
  "items": [
    {
      "product_id": ObjectId("..."),
      "quantity": 2
    },
    {
      "product_id": ObjectId("..."),
      "quantity": 1
    }
  ],
  "status": "active",
  "created_at": ISODate("2026-04-04T10:00:00.000Z"),
  "updated_at": ISODate("2026-04-04T12:30:00.000Z"),
  "expires_at": ISODate("2026-04-05T10:00:00.000Z")
}
```

Гостевая корзина:

```json
{
  "_id": ObjectId("..."),
  "owner_id": "sess_xyz789",
  "owner_type": "session",
  "items": [
    {
      "product_id": ObjectId("..."),
      "quantity": 1
    }
  ],
  "status": "active",
  "created_at": ISODate("2026-04-04T11:00:00.000Z"),
  "updated_at": ISODate("2026-04-04T11:15:00.000Z"),
  "expires_at": ISODate("2026-04-05T11:00:00.000Z")
}
```

### Поля документа

| Поле                 | Тип        | Описание                                             |
| :------------------- | :--------- | :--------------------------------------------------- |
| `_id`                | `ObjectId` | Уникальный идентификатор корзины                     |
| `owner_id`           | `string`   | Идентификатор владельца (`user_id` или `session_id`) |
| `owner_type`         | `string`   | Тип владельца (`user`, `session`)                    |
| `items`              | `array`    | Список товаров в корзине                             |
| `items[].product_id` | `ObjectId` | Идентификатор товара                                 |
| `items[].quantity`   | `int`      | Количество                                           |
| `status`             | `string`   | Статус корзины (`active`, `ordered`, `abandoned`)    |
| `created_at`         | `ISODate`  | Дата и время создания                                |
| `updated_at`         | `ISODate`  | Дата и время последнего обновления                   |
| `expires_at`         | `ISODate`  | Время удаления корзины (TTL)                         |

### Основные операции

- Создание корзины, когда заходит гость или новый пользователь.
- Получение текущей корзины по фильтру `{ owner_id, status: "active" }`.
- Добавление или замена товара в корзине.
- Удаление товара из корзины.
- Слияние гостевой корзины в пользовательскую при логине:
  - прочитать гостевую `{ owner_id: session_id, status: "active" }`;
  - добавить её `items` в корзину `{ owner_id: user_id, status: "active" }`;
  - отметить гостевую как `abandoned`.
- Отметка корзины как заказанной.

### Создание коллекции и индексов

```javascript
use somedb

db.createCollection("carts")

db.carts.createIndex({ owner_id: 1, status: 1 })
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })

sh.shardCollection("somedb.carts", { owner_id: "hashed" })
```

### Стратегия шардирования

С учетом основных операций, следует использовать хешированное шардирование. В качестве ключа предлагаю использовать `owner_id` по следующим причинам:

- Равномерное распределение корзин по шардам (как гостевых, так и пользовательских).
- Получение текущей корзины `{ owner_id, status: "active" }` - targeted запрос на один шард.
- Обновление, удаление товара, отметка как заказанная - все операции по `owner_id`, targeted.
- Слияние гостевой корзины при логине - два targeted-запроса (по `owner_id` гостя и по `owner_id` пользователя).


## Задание 8. Выявление и устранение «горячих» шардов

### Проблема
Коллекция products шардирована по ключу: `{ _id: "hashed" }`. Такой подход обеспечивает равномерное распределение документов между шардами, но не гарантирует равномерное распределение нагрузки. Если значительная часть запросов выполняется по полям, не входящим в ключ шардирования, например по category и price, mongos будет маршрутизировать такие запросы на несколько шардов, что приводит к scatter-gather и росту задержек. MongoDB рекомендует выбирать ключ шардирования с учетом cardinality, frequency, monotonicity и реальных паттернов чтения/записи.

### Метрики мониторинга

Для своевременного выявления «горячих» шардов необходимо отслеживать следующие метрики:

1. Доля targeted и scatter-gather запросов
  На mongos используется статистика маршрутизации запросов: `db.serverStatus().shardingStatistics.numHostsTargeted`. 
  Метрика показывает, сколько запросов было направлено на один шард, на несколько шардов или на все шарды. Рост manyShards и allShards означает увеличение scatter-gather и ухудшение масштабируемости.

2. Распределение read/write нагрузки по shard key
    Для оценки пригодности текущего shard key используется query sampling и команда analyzeShardKey():

    ```javascript
    use somedb

    db.products.configureQueryAnalyzer({
      mode: "full",
      samplesPerSecond: 5
    })

    db.products.analyzeShardKey(
      { _id: "hashed" },
      {
        keyCharacteristics: true,
        readWriteDistribution: true
      }
    )
    ```
    Эта команда позволяет оценить кардинальность ключа, частоту значений и распределение чтений и записей по ключу шардирования. Если нагрузка распределяется неравномерно, значит текущий ключ не соответствует реальному профилю запросов.

3. Баланс чанков между шардами.
   Для контроля равномерности размещения данных используется агрегирование по config.chunks:

    ```javascript
    db.getSiblingDB("config").chunks.aggregate([
      {
        $group: {
          _id: { ns: "$ns", shard: "$shard" },
          chunkCount: { $sum: 1 }
        }
      },
      { $sort: { "_id.ns": 1, chunkCount: -1 } }
    ])
    ```

    Также необходимо контролировать состояние балансировщика:

    ```javascript
      sh.getBalancerState()
      sh.balancerStatus()
    ```

    Если число чанков на шардах сильно отличается, это говорит о перекосе по данным. Если чанки распределены равномерно, а нагрузка все равно неравномерна, проблема связана не с объемом данных, а с workload.

4. Latency и throughput по шардам

    Для throughput используются opcounters:

    ```javascript
    var s1 = db.serverStatus().opcounters;
    sleep(60000);
    var s2 = db.serverStatus().opcounters;

    printjson({
      insert_per_sec: (s2.insert - s1.insert) / 60,
      query_per_sec: (s2.query - s1.query) / 60,
      update_per_sec: (s2.update - s1.update) / 60,
      delete_per_sec: (s2.delete - s1.delete) / 60,
      command_per_sec: (s2.command - s1.command) / 60
    })
    ```

    Для latency используются opLatencies:

    ```javascript
    var s = db.serverStatus().opLatencies;
    printjson({
      read_avg_ms: s.reads.ops > 0 ? s.reads.latency / s.reads.ops / 1000 : 0,
      write_avg_ms: s.writes.ops > 0 ? s.writes.latency / s.writes.ops / 1000 : 0,
      command_avg_ms: s.commands.ops > 0 ? s.commands.latency / s.commands.ops / 1000 : 0
    })
    ```

    Для p95/p99 latency в production рекомендуется использовать внешний мониторинг или APM, так как serverStatus() в self-managed MongoDB в основном дает aggregate latency и счетчики операций.


5. Queue depth и connection pressure на mongos
    На mongos нужно отслеживать очереди и запас соединений:

    ```javascript
    var s = db.serverStatus();
    printjson({
      queue_total: s.globalLock.currentQueue.total,
      queue_readers: s.globalLock.currentQueue.readers,
      queue_writers: s.globalLock.currentQueue.writers,
      connections_current: s.connections.current,
      connections_available: s.connections.available
    })
    ```

    Рост очереди и уменьшение числа доступных соединений указывают на saturation mongos или downstream шардов. 

6. Replication lag на replica set каждого shard
    Для контроля lag secondary-реплик используется:

    ```javascript
    rs.printSecondaryReplicationInfo()
    ```

    или скрипт на основе rs.status():

    ```javascript
    var st = rs.status();
    var primary = st.members.find(m => m.stateStr === "PRIMARY");

    st.members
      .filter(m => m.stateStr === "SECONDARY")
      .forEach(function(m) {
        printjson({
          member: m.name,
          lag_sec: (primary.optimeDate - m.optimeDate) / 1000
        })
      })
    ```

    Рост replication lag означает, что shard replica set не успевает за write-нагрузкой и устойчивость к failover снижается.

    ### **Пороговые значения**

    Для определения горячих шардов предлагается использовать следующие стартовые пороги:

    - manyShards + allShards > 20–30% запросов;
    - ops/sec на одном shard > 1.5x среднего по shard’ам;
    - p99 latency > 300–500 ms;
    - globalLock.currentQueue.total стабильно > 0 в течение нескольких минут;
    - replication lag > 10 s;
    - отклонение chunkCount > 20–25%.

    ### **Механизмы устранения дисбаланса**

    1. Balancer 
      Balancer должен быть включен постоянно, так как он автоматически выравнивает распределение чанков между шардами:
      ```javascript
      sh.getBalancerState()
      sh.setBalancerState(true)
      sh.balancerStatus()
      ```
    2. Регулярный анализ ключа шардирования
      Для коллекции `products` необходимо периодически включать query sampling и анализировать текущий ключ шардирования. Если analyzeShardKey() показывает выраженный перекос read/write distribution, значит ключ не соответствует реальной нагрузке.
    3. Ручной moveChunk
      Если один шард перегружен, допускается точечное перемещение chunk’ов:
      ```javascript
      sh.moveChunk("somedb.products", { _id: ObjectId("...") }, "shard2")
      ```
    4. Resharding
      Если основная нагрузка приходится на запросы по категории, долгосрочным решением становится смена shard key на составной:
      ```javascript
      { category: 1, _id: "hashed" }
      ```
      Такой ключ лучше учитывает реальные запросы каталога и одновременно сохраняет высокую кардинальность за счет _id. MongoDB рекомендует использовать compound shard key, если одно из полей имеет высокую frequency.


    ### План действий при hot shard

      1. Зафиксировать рост ops/sec, latency, replication lag или scatter-gather.
      2. Проверить numHostsTargeted, opcounters, opLatencies, chunkCount и состояние balancer.
      3. При необходимости выполнить moveChunk() для перегруженного диапазона.
      4. Запустить analyzeShardKey() и оценить пригодность текущего shard key.
      5. Если перекос связан с workload, выполнить resharding на более подходящий составной ключ.

    ### Итог

    Для предотвращения hot shard в коллекции products необходимо сочетать постоянный мониторинг маршрутизации запросов, нагрузки на шарды, дага репликации и состояния распределения чанков с регулярным анализом ключа шардирования. В качестве базового механизма перераспределения используется balancer, в аварийных ситуациях - moveChunk(), а в качестве долгосрочного архитектурного решения - resharding коллекции на ключ, соответствующий реальному профилю запросов.

    ## Задание 9. Настройка чтения с реплик и консистентность

    | Коллекция  | Можно читать со secondary                   | Только primary                                                                      | Допустимый lag на secondary                                      |
    | ---------- | ------------------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
    | `products` | каталог, карточка товара, поиск, фильтрация | остатки перед заказом, доступность перед резервированием, чтения внутри checkout    | 1–2 сек, максимум 5 сек для витрины                              |
    | `orders`   | история заказов, архив, аналитика           | актуальный статус после создания/оплаты/отмены, чтения в критичных бизнес-операциях | 2–5 сек, максимум 10 сек для архива                              |
    | `carts`    | только архив/аналитика                      | активная корзина, merge, checkout, все пользовательские операции                    | для active cart secondary нежелателен; фактически только primary |

Обоснование

- `products` обновляется часто, особенно в период распродаж. Для бизнес-логики критично не допустить продажу недоступного товара. Поэтому данные, влияющие на покупку и резервирование, должны читаться только с primary. Secondary допустим только для «витринных» read-heavy сценариев, где небольшая устарелость не критична.
- В `orders` важна не только корректность данных, но и доверие пользователя к статусу заказа. История заказов может быть немного устаревшей, но актуальный статус после оформления, оплаты или отмены должен читаться строго с primary. Частота обновлений ниже, чем у products, но цена ошибки для пользовательского опыта выше именно в точках перехода статуса.
- `carts` требует почти read-after-write consistency. Это коллекция с очень высокой частотой обновлений и максимальной чувствительностью к устареванию данных на уровне пользовательского интерфейса. Поэтому активные корзины должны обслуживаться только с primary. Secondary можно использовать только для фоновых, аналитических и некритичных чтений.

## Задание 10. Миграция на Cassandra: модель данных, стратегии репликации и шардирования
### 10.1

Cassandra целесообразно использовать не для всего интернет-магазина целиком, а прежде всего для масштабируемых, распределённых, write-heavy и read-heavy подсистем с предсказуемыми запросами, где допустима eventual consistency: истории заказов, event log, сессий и витринных read-моделей каталога. Для данных, критичных к мгновенной консистентности - остатков, активных корзин и checkout-операций - Cassandra в данном кейсе не должна быть основным хранилищем.

#### Имеет смысл переносить в Cassandra

- историю заказов;
- журнал событий по заказам;
- пользовательские сессии;
- витринные и денормализованные представления каталога товаров;
- при необходимости - другие read-heavy и append-heavy сущности с предсказуемыми запросами.

Эти данные лучше всего соответствуют сильным сторонам Cassandra: линейному масштабированию, высокой доступности, распределённой репликации и query-driven моделированию. Cassandra специально рассчитана на распределённые кластеры с большим числом узлов, высокой скоростью записи и репликацией между дата-центрами.

#### Не стоит делать первичным кандидатом на миграцию

Cassandra хуже подходит для данных, где критична строгая консистентность "прочитал сразу после записи", сложные межсущностные транзакции и гибкие ad hoc-запросы, потому что её модель проектируется под заранее известные query patterns, без join’ов, с денормализацией данных.

- остатки товаров как источник истины;
- активные корзины;
- критичные этапы оформления заказа;
- актуальный статус заказа в момент пользовательского действия.

### 10.2

1. Сущность `order_history_by_user` (Поиск истории заказов конкретного пользователя.)
    ```sql
    CREATE TABLE order_history_by_user (
        user_id text,
        bucket_ym text,
        created_at timestamp,
        order_id uuid,
        status text,
        total decimal,
        geo_zone text,
        PRIMARY KEY ((user_id, bucket_ym), created_at, order_id)
      ) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
    ```

    Выбор ключей:

   - Partition key: `(user_id, bucket_ym)`
   - Clustering columns: `created_at, order_id`
  
    *Обоснование*

    - `user_id` соответствует основному query pattern: история заказов всегда читается по пользователю.
    - Добавление `bucket_ym` (2026-11, 2026-12) ограничивает размер партиции.
    - `created_at DESC` позволяет быстро читать последние заказы без дополнительной сортировки.
    - `order_id` добавлен для уникальности строк с одинаковым временем.

    Такой дизайн снижает риск hot partition: даже если пользователь очень активен, его история дробится по временным бакетам. Cassandra рекомендует избегать очень больших и постоянно растущих partition, поскольку именно они становятся горячими и дорогими для чтения и обслуживания.

2. Сущность `products_by_category` (Поиск товаров по категориям и фильтрация по диапазону цен. )
   
    ```sql
    CREATE TABLE products_by_category (
        category text,
        bucket_mod int,
        product_id uuid,
        name text,
        price decimal,
        is_active boolean,
        attributes text,
        PRIMARY KEY ((category, bucket_mod), product_id)
    ) WITH CLUSTERING ORDER BY (product_id ASC);

    ```

    Выбор ключей:

    - Partition key: `category, bucket_mod`
    - Clustering columns: `product_id`

    *Обоснование*

    Если использовать только category как ключ партиционирования, популярная категория вроде Электроника станет hot partition. Поэтому в ключ добавляется bucket_mod - искусственный бакет, например `hash(product_id) % 16`. Это даёт:

    - распределение товаров одной категории по нескольким партициям;
    - снижение риска hot partition при популярных категориях;
    - более равномерную нагрузку по токенам кластера.

    Чтение каталога при этом выполняется как fan-out по ограниченному числу бакетов категории, например по 16 партициям вместо одной огромной. Это компромисс между удобством чтения и защитой от перегрева одной партиции. 

3. Сущность `products_by_id`  (Описание товара на странице продукта)
      
    ```sql
    CREATE TABLE products_by_id (
        product_id uuid,
        name text,
        category text,
        price decimal,
        is_active boolean,
        attributes text,
        PRIMARY KEY ((product_id))
    );

    ```
    
    Выбор ключей

    - Partition key: `product_id`
    - Clustering columns: отсутствуют

    *Обоснование*

    Это точечный key-value доступ, который идеально ложится на Cassandra:

    - высокая кардинальность `product_id`;
    - равномерное распределение по кластеру;
    - отсутствие широких партиций;
    - минимальный риск hot partition, кроме случая экстремально популярного одного товара.

    Для карточки товара это нормальная модель, но для inventory как источника истины Cassandra в данном кейсе не рассматривается как основной контур из-за требований к строгой консистентности.

### 10.3

1. Сущность `order_history_by_user` (Поиск истории заказов конкретного пользователя.)
   
    Выбор стратегии
      - Anti-Entropy Repair - основная
      - Hinted Handoff - включён как вспомогательная
      - Read Repair - NONE
     
    Обоснование

      История заказов важна, но не требует мгновенной абсолютной консистентности на каждом чтении. Если новый заказ появится в истории с небольшой задержкой, это допустимо с точки зрения бизнес-логики. При этом таблица будет достаточно большой и read-heavy, поэтому делать каждое чтение дороже из-за BLOCKING read repair невыгодно. Cassandra указывает, что read_repair = NONE снижает влияние на latency чтения, но отказывается от monotonic quorum reads.

2. Сущность `products_by_category` (Поиск товаров по категориям и фильтрация по диапазону цен. )
   
    Выбор стратегии
      - Anti-Entropy Repair — основная
      - Hinted Handoff — включён
      - Read Repair — NONE

    Обоснование

      Списки товаров по категории — это массовый read-heavy сценарий. Здесь приоритет — throughput и низкая latency, а не строгая консистентность каждой строки каталога. Если отдельный товар или цена появятся в выдаче с небольшой задержкой, это не так критично, как падение производительности всего каталога. Поэтому BLOCKING read repair здесь невыгоден: он увеличивает стоимость чтения, а таблица будет обращаться очень часто. Cassandra допускает read_repair = NONE, когда важнее латентность и не требуются monotonic quorum reads.

3. Сущность `products_by_id`  (Описание товара на странице продукта)
   
    Выбор стратегии

      - Hinted Handoff — основная для коротких сбоев
      - Read Repair — BLOCKING
      - Anti-Entropy Repair — регулярный
      - 
    Обоснование

      `products_by_id` используется для чтения карточки товара. Для витринного сценария небольшая устарелость допустима, но карточка товара читается очень часто, и желательно, чтобы после успешного quorum-write пользователь не видел “откат” к старому состоянию. Cassandra указывает, что BLOCKING read repair обеспечивает monotonic quorum reads ценой дополнительной задержки чтения.
