## 10.1 Критически важные данные для переноса в Cassandra

### Анализ сущностей


| Сущность  | Нагрузка                 | Тип операций                                           | Подходит Cassandra? |
| ----------------- | -------------------------------- | ----------------------------------------------------------------- | --------------------------- |
| **orders**        | Высокая в пики       | Запись + история по user_id                        | Да                        |
| **carts**         | Очень высокая        | Чтение/запись + TTL                                   | Да                        |
| **order_history** | Высокая на чтение | Append-only, поиск по user_id                              | Да                        |
| **user_sessions** | Очень высокая        | TTL, key-value                                                    | Да                        |
| **products**      | Средняя                   | Сложные фильтры по категории и цене | Частично            |

> Что касается products — сложные запросы (фильтрация по диапазону цен, поиск по категориям) плохо ложатся на модель Cassandra. Рекомендуется оставить в MongoDB. Для Cassandra переносим только таблицу остатков по геозонам (`product_stock`).

### Обоснование выбора Cassandra

**Cassandra подходит, когда:**

- Нужно линейное горизонтальное масштабирование без даунтайма при ребалансировке
- Преобладает запись (leaderless-репликация без единого primary)
- Паттерн доступа известен заранее (читаем всегда по partition key)
- Нужен TTL на уровне строк (корзины, сессии)
- Требуется геораспределенность (multi-datacenter replication из коробки)

**Проблема MongoDB при пиковой нагрузке (из условия):**

- Range-based sharding + добавление нового шарда = полная миграция чанков, что ведет к просадке latency
- Cassandra использует consistent hashing: новый узел забирает только часть токен-диапазона, без глобального перемещения

## 10.2 Концептуальная модель данных

### Сущность: orders (заказы)

**Паттерн доступа:** история заказов пользователя + статус конкретного заказа

```cql
CREATE TABLE orders_by_user (
    user_id      UUID,
    created_at   TIMESTAMP,
    order_id     UUID,
    geo_zone     TEXT,
    status       TEXT,           -- 'pending'|'paid'|'shipped'|'delivered'|'cancelled'
    total_amount DECIMAL,
    items        LIST<FROZEN<order_item>>,  -- {product_id, quantity, price}
    PRIMARY KEY ((user_id), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC)
  AND default_time_to_live = 0;  -- заказы не удаляем

CREATE TYPE order_item (
    product_id UUID,
    quantity   INT,
    price      DECIMAL
);
```

**Для поиска заказа по ID (без user_id):**

```cql
CREATE TABLE orders_by_id (
    order_id     UUID PRIMARY KEY,
    user_id      UUID,
    created_at   TIMESTAMP,
    status       TEXT,
    total_amount DECIMAL
);
```


| Ключ       | Значение            | Обоснование                                                    |
| -------------- | --------------------------- | ------------------------------------------------------------------------- |
| Partition key  | `user_id`                   | Все заказы пользователя на одном узле     |
| Clustering key | `created_at DESC, order_id` | Сортировка по дате, уникальность строки |

**Риски:**

- Hot partition для VIP-пользователей с тысячами заказов, нужно ограничить список до последних N (pagination)
- Неравномерность: UUID как partition key дает хорошее распределение через murmur3

### Сущность: carts (корзины)

**Паттерн доступа:** активная корзина по `user_id` или `session_id`

```cql
CREATE TABLE carts_by_user (
    user_id     UUID,
    status      TEXT,           -- 'active'|'ordered'|'abandoned'
    cart_id     UUID,
    session_id  TEXT,
    items       LIST<FROZEN<cart_item>>,
    created_at  TIMESTAMP,
    updated_at  TIMESTAMP,
    PRIMARY KEY ((user_id), status, cart_id)
) WITH default_time_to_live = 2592000;  -- TTL 30 дней

-- Для гостевых корзин (по session_id)
CREATE TABLE carts_by_session (
    session_id  TEXT,
    status      TEXT,
    cart_id     UUID,
    items       LIST<FROZEN<cart_item>>,
    created_at  TIMESTAMP,
    updated_at  TIMESTAMP,
    PRIMARY KEY ((session_id), status, cart_id)
) WITH default_time_to_live = 86400;  -- TTL 1 день для гостей

CREATE TYPE cart_item (
    product_id UUID,
    quantity   INT
);
```


| Ключ       | Значение         | Обоснование                                                          |
| -------------- | ------------------------ | ------------------------------------------------------------------------------- |
| Partition key  | `user_id` / `session_id` | Быстрый доступ к корзине конкретного юзера |
| Clustering key | `status, cart_id`        | Фильтрация`WHERE status='active'`                                     |
| TTL            | 30 дней / 1 день | Автоматическая очистка без cron-задач              |

**Риски:**

- Пользователь с историей множества корзин - partition растет. Решение - archiving старых корзин

---

### Сущность: product_stock (остатки товаров по геозонам)

> Полная информация о товарах (описание, фильтры) остается в MongoDB. В Cassandra — только горячие данные об остатках.

```cql
CREATE TABLE product_stock (
    product_id  UUID,
    geo_zone    TEXT,
    stock       INT,
    updated_at  TIMESTAMP,
    PRIMARY KEY ((product_id), geo_zone)
);

-- Для поиска всех товаров с остатком в геозоне:
CREATE TABLE stock_by_geozone (
    geo_zone    TEXT,
    product_id  UUID,
    stock       INT,
    PRIMARY KEY ((geo_zone), product_id)
);
```


| Ключ       | Значение | Обоснование                                               |
| -------------- | ---------------- | -------------------------------------------------------------------- |
| Partition key  | `product_id`     | Все геозоны товара на одном узле          |
| Clustering key | `geo_zone`       | Обновление остатка в конкретной зоне |

**Риски:**

- **Hot partition** для хитовых товаров (iPhone, акции) — все читают один partition key
- Решение: Cassandra lightweight transactions (LWT) для атомарного списания остатков:

```cql
UPDATE product_stock SET stock = stock - 1
WHERE product_id = ? AND geo_zone = ?
IF stock > 0;
```

### Сущность: user_sessions (сессии)

```cql
CREATE TABLE user_sessions (
    session_id  TEXT PRIMARY KEY,
    user_id     UUID,
    created_at  TIMESTAMP,
    data        TEXT            -- JSON с данными сессии
) WITH default_time_to_live = 3600;  -- TTL 1 час
```

Простой key-value паттерн — идеально для Cassandra.

## 10.3 Стратегии восстановления целостности данных

### Обзор стратегий


| Стратегия      | Когда срабатывает                          | Механизм                                                                                                                             |
| ----------------------- | ---------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **Hinted Handoff**      | Узел недоступен во время записи | Координатор хранит подсказку и доставляет данные после восстановления узла |
| **Read Repair**         | При чтении данных                           | Координатор сравнивает ответы реплик, исправляет устаревшие в фоне                 |
| **Anti-Entropy Repair** | По расписанию (`nodetool repair`)              | Merkle tree сравнение между узлами, синхронизация расхождений                                    |

### Выбор стратегий по сущностям

#### orders — строгая консистентность

**Критичность:** потеря заказа = потеря денег и доверия клиента

```cql
-- Запись с QUORUM: большинство реплик должны подтвердить
INSERT INTO orders_by_user (...) VALUES (...)
USING CONSISTENCY QUORUM;

-- Чтение с QUORUM
SELECT * FROM orders_by_user WHERE user_id = ?
CONSISTENCY QUORUM;
```

**Стратегии:**

- Hinted Handoff — обязательно. Если нода упала, заказ не теряется, доставится при восстановлении
- Read Repair (с `read_repair_chance = 1.0`) — каждое чтение проверяет консистентность реплик
- Anti-Entropy Repair — еженедельно (`nodetool repair orders_by_user`)

**Допустимая задержка репликации:** < 1 секунды (QUORUM гарантирует это без дополнительных мер)

---

#### carts — eventual consistency

**Критичность:** устаревшая корзина на 1-2 секунды приемлема, пользователь переживет проблему

```cql
-- Запись с ONE: быстро, без ожидания всех реплик
INSERT INTO carts_by_user (...) VALUES (...)
USING CONSISTENCY ONE;

-- Чтение с ONE
SELECT * FROM carts_by_user WHERE user_id = ? AND status = 'active'
CONSISTENCY ONE;
```

**Стратегии:**

- Hinted Handoff — корзина не теряется при падении узла
- Read Repair (`read_repair_chance = 0.1`) — 10% запросов на чтение триггерят фоновое исправление
- Anti-Entropy Repair — достаточно ежемесячно (данные короткоживущие из-за TTL)

**Допустимая задержка репликации:** до 5 секунд — некритично

---

#### product_stock — компромисс скорость или консистентность

**Критичность:** продать товар которого нет = операционная проблема, но не катастрофа (можно отменить)

```cql
-- Запись с LOCAL_QUORUM (в рамках одного датацентра)
UPDATE product_stock SET stock = stock - 1
WHERE product_id = ? AND geo_zone = ?
CONSISTENCY LOCAL_QUORUM;

-- Чтение с LOCAL_ONE для каталога (отображение)
SELECT stock FROM product_stock WHERE product_id = ? AND geo_zone = ?
CONSISTENCY LOCAL_ONE;
```

**Стратегии:**

- Hinted Handoff — обновления остатков не теряются
- Read Repair (`read_repair_chance = 0.5`) — 50% запросов синхронизируют реплики
- Anti-Entropy Repair — ежедневно (остатки часто меняются)

**Допустимая задержка репликации:** до 2 секунд для отображения; для LWT при списании — синхронно

---

#### user_sessions — только скорость

**Критичность:** потеря сессии = повторный логин, неудобство, не катастрофа

```cql
INSERT INTO user_sessions (...) VALUES (...)
USING CONSISTENCY ONE AND TTL 3600;
```

**Стратегии:**

- Hinted Handoff — достаточно
- Read Repair — избыточно для TTL-данных
- Anti-Entropy Repair — данные протухают сами по TTL

**Допустимая задержка репликации:** до 30 секунд — сессия все равно живет 1 час

---

### Сводная таблица стратегий


| Сущность | Consistency Level | Hinted Handoff         | Read Repair | Anti-Entropy Repair    | Макс. задержка репл. |
| ---------------- | ----------------- | ---------------------- | ----------- | ---------------------- | ------------------------------------ |
| orders           | QUORUM            | Обязательно | 100%        | Еженедельно | < 1 сек                           |
| carts            | ONE               | Обязательно | 10%         | Ежемесячно   | < 5 сек                           |
| product_stock    | LOCAL_QUORUM      | Обязательно | 50%         | Ежедневно     | < 2 сек                           |
| user_sessions    | ONE               | Достаточно   | --          | --                     | < 30 сек                          |

## Сравнение MongoDB vs Cassandra

```
MongoDB (Range Sharding)           Cassandra (Consistent Hashing)
─────────────────────────          ──────────────────────────────
Добавление шарда:                  Добавление узла:
  - Полный ребаланс чанков           - Токен-диапазон делится
  - Просадка latency в пике          - Только ~1/N данных переезжает
  - Блокировки на запись             - Zero downtime
  - Время: часы                      - Время: минуты

Запись:                            Запись:
  - Single primary                   - Любая нода = координатор
  - Bottleneck при 50k RPS           - Линейный рост с узлами
```

## Метрики мониторинга

### Ключевые метрики Cassandra


| Метрика                       | Инструмент  | Пороговое значение | Действие                                                    |
| ------------------------------------ | --------------------- | ----------------------------------- | ------------------------------------------------------------------- |
| `cassandra.reads.latency_p99`        | JMX / Prometheus      | > 10ms                              | Проверить hot partition, добавить узлы         |
| `cassandra.writes.latency_p99`       | JMX / Prometheus      | > 5ms                               | Проверить commit log, disk I/O                             |
| `cassandra.compaction.pending_tasks` | nodetool              | > 100                               | Увеличить ресурсы, настроить compaction    |
| `cassandra.hints.hints_in_progress`  | JMX                   | > 1000                              | Нода долго недоступна, проверить        |
| `cassandra.dropped_messages`         | JMX                   | > 0                                 | Перегрузка, масштабировать                  |
| `cassandra.read_repair.background`   | JMX                   | мониторинг                | Высокое значение = данные расходятся |
| Partition size                       | `nodetool tablestats` | > 100MB                             | Ревизия partition key                                        |

### Диагностика hot partition

```bash
# Найти большие партиции
nodetool tablestats mobile_world.product_stock

# Топ partition по размеру (Cassandra 4.x)
SELECT keyspace_name, table_name, partitions_count
FROM system.size_estimates
WHERE keyspace_name = 'mobile_world';

# Профилирование конкретного запроса
TRACING ON;
SELECT * FROM orders_by_user WHERE user_id = <uuid>;
TRACING OFF;
```
