### Задание 8. Выявление и устранение «горячих» шардов

1. Метрики для отслеживания:
  - Chunk distribution: Количество чанков на каждом шарде. Резкие различия сигнализируют о неравномерной нагрузке.
  - Balancer activity: Частота и продолжительность миграций чанков. Регулярные миграции - признак балансировки, но слишком частые - возможный дисбаланс.
  - Jumbo chunks: Чанки, которые не могут быть разделены из-за ограничения размера. Могут стать причиной перегрузки.
  - Split operations: Скорость разделения чанков. Высокая активность может указывать на рост нагрузки на конкретный диапазон данных.
  - Query targeting: Сколько запросов попадает на каждый шард. Если большинство запросов идут в один шард - это «горячий» узел.
  - Resource usage: CPU, память, дисковый I/O, сетевой I/O каждого шарда. Постоянно высокие значения - индикатор перегрузки.
  - Disk I/O и latency: Скорость чтения/записи и задержки диска. Сильный рост - признак нагрузки на конкретный узел.

2. Меры устранения дисбаланса:
  - Автоматический балансировщик: MongoDB автоматически мигрирует чанки между шардами при обнаружении дисбаланса (настройки по умолчанию).
  - Настройка размера чанка: Можно изменить размер чанка, чтобы чаще/реже происходила балансировка. Меньший размер - более частые миграции, но более равномерное распределение.
  ```js
  db.settings.save({ _id: "chunksize", value: 64 }); // значение в MB
  ```
  - Настройка окна активности балансировщика: Можно ограничить время, когда балансировщик активен (например, только ночью), чтобы не мешать основной нагрузке.
  ```js
  sh.startBalancer();
  sh.stopBalancer();
  ```
  - Мониторинг и ручная корректировка: При обнаружении дисбаланса можно вручную инициировать миграцию чанков через команды moveChunk.
  ```js
  sh.moveChunk(namespace, query, destination);
  ```
