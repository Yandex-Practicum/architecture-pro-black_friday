## 1. Введение

Произошел инцидент перегрузки одного из шардов из-за категории «Электроника» (70% запросов).
Документ описывает метрики для раннего обнаружения дисбаланса и стратегии устранения проблем с распределением нагрузки в кластере MongoDB.

---

## 2. Метрики мониторинга (Observability)

Для предотвращения подобных ситуаций необходимо настроить мониторинг на трех уровнях: инфраструктура, база данных и балансировщик.

### 2.1. Инфраструктурные метрики (System Level)

Необходимо отслеживать эти показатели **в разрезе каждого шарда** отдельно.

| Метрика | Порог тревоги (Alert) | Что означает |
| :--- | :--- | :--- |
| **CPU Usage** | > 80% на одном шарде | Если один шард имеет 90% CPU, а другие 10% — это явный признак Hotspot. |
| **Disk I/O (IOPS)** | Saturation (близко к лимиту) | Неравномерность записи/чтения на диск. |
| **Network In/Out** | Аномальный всплеск | Если трафик на один шард кратно превышает трафик на другие. |

### 2.2. Метрики MongoDB (Database Level)

Эти метрики собираются через `db.serverStatus()` или системы мониторинга (Prometheus + MongoDB Exporter, Percona Monitoring).

| Метрика | Описание | Команда проверки |
| :--- | :--- | :--- |
| **opcounters (Insert/Update/Query)** | Количество операций в секунду. Дисбаланс > 30% между шардами требует внимания. | `db.serverStatus().opcounters` |
| **currentQueue.readers / writers** | Длина очереди операций. Рост очереди на конкретном шарде указывает на его неспособность справиться с нагрузкой. | `db.serverStatus().globalLock.currentQueue` |
| **connections.current** | Количество активных соединений. Резкий рост на одном узле. | `db.serverStatus().connections` |
| **lock.time** | Время блокировок. Высокий % блокировок на запись указывает на конкуренцию за одни и те же документы. | `db.serverStatus().locks` |

### 2.3. Метрики данных (Data Distribution)

| Метрика | Описание                                                                                                          |
| :--- |:------------------------------------------------------------------------------------------------------------------|
| **Chunk Count per Shard** | Разница в количестве чанков. MongoDB Balancer старается выровнять это число, но он инертен.                       |
| **Jumbo Chunks Count** | Количество неделимых чанков, превысивших размер  (64 или 128 MB). Это критическая метрика для Range-шардирования. |
| **Data Size per Shard** | Объем данных на диске.                                                                                            |

## 3. Механизмы устранения дисбаланса

В зависимости от причины «горячего» шарда применяются разные стратегии.

### Сценарий А: Проблема Jumbo Chunk (как в случае с «Электроникой»)

**Симптомы:** Огромное количество запросов к одним и тем же данным, Balancer не может переместить данные, так как чанк слишком велик или ключ имеет низкую кардинальность (все товары «Электроника» имеют одно значение ключа).

**Решение 1: Refine Shard Key (Уточнение ключа)**

Если текущий ключ `{ category: 1 }`, мы не можем разделить «Электронику». Нужно добавить суффикс для гранулярности.
*Добавляем `_id` к ключу шардирования.*

```javascript
sh.refineCollectionShardKey("mobile_world.products", { category: 1, _id: 1 });
```

**Решение 2: Resharding (Полная смена ключа)**

Если Range-шардирование не оправдало себя, переходим на Hashed Sharding. Это радикальное решение, доступное "на лету" в MongoDB 5.0+.

```javascript
sh.reshardCollection("mobile_world.products", { _id: "hashed" });
```

### Сценарий Б: Дисбаланс данных (Data Imbalance)

**Симптомы:** Один шард заполнен данными больше других, но Jumbo Chunks нет.

**Решение:** Настройка автоматического Балансировщика.

Убедиться, что Balancer включен:

```javascript
sh.getBalancerState()
```

Настройка окна работы : Чтобы миграция данных не тормозила базу в часы пик.

```javascript
sh.setBalancerState(true)
db.settings.update(
   { _id: "balancer" },
   { $set: { activeWindow : { start : "02:00", stop : "06:00" } } },
   { upsert: true }
)
```

### Сценарий В: Ручное разбиение (Emergency Splitting)

Если автоматика не справляется, можно вручную разбить чанк.

```javascript
sh.splitFind("mobile_world.products", { category: "Electronics" })
```

## 4. Алгоритм автоматического реагирования

Предлагается внедрить скрипт (например, на Python/Go), запускаемый по cron или триггеру мониторинга, который выполняет следующие шаги:

1.  **Сбор статистики**: Раз в 5 минут опрос `db.adminCommand({ getShardDistribution: "mobile_world.products" })`.
2.  **Анализ дисперсии**: Если `(Max_Data_Size - Min_Data_Size) / Avg_Data_Size > 0.2` (дисбаланс более 20%), инициировать алерт.
3.  **Проверка Jumbo**: Поиск документов в `config.chunks` с флагом `jumbo: true`.
4.  **Действие**: Отправка уведомления администратору (автоматическое разбиение Jumbo опасно без анализа ключа).
5.  **Агрессивный балансинг (Опционально)**: Если нагрузка (CPU) на шард критическая (>90%), временно переместить новые записи на другие шарды с помощью Zone Sharding (Tag specific ranges), изолировав горячий шард только на чтение.

## 5. Резюме

Для решения проблемы с категорией «Электроника» и предотвращения рецидивов:

-   **Немедленно**: Выполнить `sh.refineCollectionShardKey` (добавить `_id` или `sku`) или `sh.reshardCollection` (перейти на Hashed). Это позволит разбить массив данных «Электроника» на мелкие чанки.
-   **Настроить дашборд**: Вывести `opcounters` каждого шарда на один график для визуализации перекосов.
-   **Политика**: Использовать Hashed Sharding для всех коллекций с равномерным доступом (products, orders), избегая Ranged Sharding по полям с низкой кардинальностью.
