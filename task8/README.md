## Горячие шарды в MongoDB: как находить и лечить

### 0) Что было (исходные коллекции, см. /task7)

```text
carts  — шард по owner_key
orders — шард по customer_id
products — шард по category   ← тут и случилась беда (из-за категории «Электроника» произошла перегрузка одного из шардов)
```

### 1) Проблема

Категория **“Электроника”** даёт ~70% запросов, появился «горячий шард».

---

# 2) Решение: разрезаем каждую категорию на 60 чанков.

## 2.1 Что добавляем в `products`

Добавляем поле:

* `category_bucket: int` (значения **0..59**)

Считаем всегда одинаково:

* `category_bucket = hash(_id) % 60`

То есть “Электроника” становится не одной кучей, а 60 куч:
Почему именно 60? Потому что 60 делится на 2, 3, 4, 5, 6, 8. 
Т.е. можем равномерно распределить категорию по стольким шардам (на всякий). 

* `Электроника+bucket=0`, `Электроника+bucket=1`, …, `bucket=59`

## 2.2 Новый shard key для `products`

Шард-ключ становится составным:

```js
{ category: 1, category_bucket: 1 }
```

**Как это помогает:**

* раньше “Электроника” могла жить почти в одном месте;
* теперь “Электроника” заранее разбита на 60 частей, и эти части можно **раскидать по разным шардам**.

---

# 3) Обновлённая модель `products` (как в задании)

```text
class products {
   _id: string <<PK>>
   --
   category: string <<SHARD, IDX>>
   category_bucket: int <<SHARD, IDX>>   // 0..59 = hash(_id) % 60
   --
   price: int32 <<IDX>>
   name: string
   --
   created_at: isodate
   updated_at: isodate
   --
   stocks: list
   stocks.geo_zone: string <<IDX>>
   stocks.qty: int32
   --
   attrs: object
   attrs.color: string
   attrs.memory: string
}
```

---

# 4) Как данные начнут распределяться

### Вставка/обновление товара

Приложение **всегда** пишет `category_bucket`.

### Запросы по категории

Чтобы реально получить распределение по шардам, запросы “по категории” делаем так:

* не один запрос `category="Электроника"`,
* а **параллельно по бакетам** (0..59), потом склеиваем результат.

(Это больше запросов, но они маленькие и распределяются по шардам — и главное: **один шард больше не умирает**.)

# 5) Миграция (как внедрить)

## 5.1 Добавить поле и пересчитать старые документы (backfill)

(идея: заполнить тем, у кого поля нет)

```js
const N = 60;

db.products.updateMany(
  { category_bucket: { $exists: false } },
  [{ $set: { category_bucket: hash(_id) % 60 } }]
);
```
В реальном проекте удобнее: посчитать bucket в приложении/скрипте и батчами обновить.

## 5.2 Включить шардирование и задать shard key

```js
sh.enableSharding("shop")
sh.shardCollection("shop.products", { category: 1, category_bucket: 1})
```

## 5.3 Балансер должен быть включён

Проверяем состояние балансера:

```js
sh.getBalancerState()
sh.isBalancerRunning()
```

Если надо включить:

```js
sh.setBalancerState(true)
```

# 6) Метрики мониторинга (что собирать и зачем)

| Метрика                                | Зачем                                                      |
| -------------------------------------- | ---------------------------------------------------------- |
| QPS по шардам (read/write)             | видим, что один шард получает слишком много запросов       |
| p95/p99 latency по шардам              | видим, что один шард начал тормозить                       |
| CPU/RAM/Disk IO/Network по шардам      | понимаем, что упёрлись в железо                            |
| Кол-во chunks по шардам для `products` | если у шарда много чанков “Электроники” — он будет горячим |
| Топ категорий по доле запросов         | понимаем, что именно “Электроника” причина                 |

Команда для базовой статистики на шарде:

```js
db.adminCommand({ serverStatus: 1 })
```

Распределение чанков:

```js
use config
db.chunks.aggregate([
  { $match: { ns: "shop.products" } },
  { $group: { _id: "$shard", chunks: { $sum: 1 } } },
  { $sort: { chunks: -1 } }
])
```

**Простой алерт**:

* если один шард > **2х** среднего по QPS **и** p95 latency выше нормы → “горячий шард”.

# 7) Механизмы автоматического перераспределения

## 7.1 Автоматика №1 (самая простая): полагаться на балансер

Мы **держим балансер включённым**, и он переносит чанки между шардами.

## 7.2 Автоматика №2 (управляемая): “закрепляем бакеты за шардами через зоны”

Если нужно прям **жёстко разнести Электронику** по 2/3/4/5/6/8 шардам, используем **zones**:

1. Добавляем шарды в зоны (пример):

```js
sh.addShardToZone("shard001", "HOT1")
sh.addShardToZone("shard002", "HOT2")
sh.addShardToZone("shard003", "HOT3")
sh.addShardToZone("shard004", "HOT4")
```

2. Делим bucket-диапазоны. Например для 4 шардов:

* HOT1: bucket 0–14
* HOT2: bucket 15–29
* HOT3: bucket 30–44
* HOT4: bucket 45–59

И задаём диапазоны через `updateZoneKeyRange` (делается на mongos)

```js
sh.updateZoneKeyRange(
  "shop.products",
  { category: "Электроника", category_bucket: 0,  _id: MinKey },
  { category: "Электроника", category_bucket: 15, _id: MinKey },
  "HOT1"
)

sh.updateZoneKeyRange(
  "shop.products",
  { category: "Электроника", category_bucket: 15, _id: MinKey },
  { category: "Электроника", category_bucket: 30, _id: MinKey },
  "HOT2"
)

// ... и так далее до 60
```

> `MinKey` тут просто означает “любой `_id`” в рамках бакета — это удобно для задания границ.

После этого балансер сам будет стараться держать эти диапазоны на нужных шардах.

## 7.3 Если нужно руками: split/moveChunk (аварийный режим)

* нарезать чанк: `sh.splitFind()` или `sh.splitAt()`
* перенести чанк: `moveChunk`

> Важно: `moveChunk` двигает **один конкретный чанк**, а не “всю Электронику сразу”.

---

# 8) Итог

* Мы добавили `category_bucket` (0..59) и новый shard key для `products`.
* Теперь горячая категория **размазана по 60 кускам**, и их можно распределять между шардами.
* Мониторим QPS/latency/ресурсы + распределение чанков.
* Для автоматики используем балансер, а для “точного контроля” — зоны по бакетам.
