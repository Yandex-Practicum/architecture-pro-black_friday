# Задача 7: Проектирование схем и стратегии шардирования MongoDB для проекта «Мобильный мир»

В рамках масштабирования бэкенда «Мобильного мира» были спроектированы схемы для коллекций `orders`, `products` и `carts`, а также выбраны оптимальные ключи и стратегии шардирования.

## 1. Коллекция `orders` (Заказы)

### Схема документов
```json
{
  "_id": "ObjectId",
  "order_id": "string",
  "user_id": "string",
  "created_at": "ISODate",
  "items": [
    {
      "product_id": "string",
      "price": "number",
      "quantity": "number"
    }
  ],
  "status": "string",
  "total_amount": "number",
  "geozone": "string"
}
```

### Выбор шард-ключа и стратегии
- **Шард-ключ:** `{ "user_id": "hashed" }`
- **Стратегия шардирования:** Hashed Sharding (Хэшированное шардирование).
- **Обоснование:** 
  - **Быстрое создание заказов:** Хэширование `user_id` обеспечит равномерное распределение новых заказов (записей) по всем шардам кластера, исключая появление горячих узлов (hot shards).
  - **Поиск истории заказов:** Основная выборка заказов происходит в разрезе конкретного пользователя. Запрос вида `db.orders.find({ user_id: "..." })` будет направлен точно в один конкретный шард (Targeted Query), что значительно эффективнее, чем Scatter-Gather (опрос всех шардов).
  - Если необходимо получать статус конкретного заказа *без* привязки к пользователю, приложение может передавать `user_id` вместе с `order_id` в запросе, чтобы он тоже маршрутизировался на один шард.

---

## 2. Коллекция `products` (Товары)

### Схема документов
```json
{
  "_id": "ObjectId",
  "product_id": "string",
  "name": "string",
  "category": "string",
  "price": "number",
  "stock": {
    "moscow": 50,
    "kaliningrad": 30
  },
  "attributes": {
    "color": "black",
    "size": "L"
  }
}
```
*(Остатки по складам хранятся во вложенном объекте словаря, что позволяет точечно обновлять поля вида `stock.moscow`)*

### Выбор шард-ключа и стратегии
- **Шард-ключ:** `{ "category": 1, "product_id": 1 }`
- **Стратегия шардирования:** Ranged Sharding (Шардирование по диапазонам).
- **Обоснование:**
  - **Поиск товаров и фильтрация:** Самые частые поисковые (Read) запросы идут по категориям и диапазонам цен. Составной ключ, начинающийся с `category`, гарантирует, что товары одной категории локализованы на одном (или нескольких соседних) шардах. Запрос `db.products.find({ category: "Смартфоны", price: {$gte: 10000} })` будет направлен только в нужные шарды (узкий scatter-gather).
  - **Избежание гигантских чанков (Jumbo Chunks):** Использование `category` в чистом виде привело бы к появлению неделимых чанков-гигантов. Добавление `product_id` вторым полем обеспечивает идеальную кардинальность (уникальность).
  - **Обновление остатков:** Точечные апдейты остатков (Write) будут высокопроизводительными, если со стороны кода вместе с `product_id` передавать `category`.

## 3. Коллекция `carts` (Корзины)

### Схема документов
```json
{
  "_id": "ObjectId",
  "owner_id": "string", // Синтетическое поле: user_id (если авторизован) ИЛИ session_id (для гостя)
  "user_id": "string",  // Опционально сохраняем для истории
  "session_id": "string", // Опционально сохраняем для истории
  "items": [
    {
      "product_id": "string",
      "quantity": "number"
    }
  ],
  "status": "string", // "active" | "ordered" | "abandoned"
  "created_at": "ISODate",
  "updated_at": "ISODate",
  "expires_at": "ISODate"
}
```

### Выбор шард-ключа и стратегии
- **Шард-ключ:** `{ "owner_id": "hashed" }`
- **Стратегия шардирования:** Hashed Sharding (Хэшированное шардирование).
- **Обоснование:**
  - **Проблема Null-значений:** Если бы мы использовали непосредственно `user_id`, то все "гостевые" корзины (`user_id: null`) попали бы в один шард, что быстро сделало бы его глобальным узким местом (Bottleneck) из-за огромного количества гостей. 
  - **Синтетическое поле (`owner_id`):** Введение поля `owner_id`, которое *всегда* заполнено уникальным идентификатором, элегантно решает проблему распределения.
  - **Равномерность записи:** Хэшированный ключ равномерно размажет корзины по кластеру. Множественные операции создания корзин и частые обновления `items` будут использовать мощности всех шардов.
  - **Точечные запросы:** Все операции бизнес-логики (получить корзину, добавить товар, слияние) оперируют идентификаторами пользователя или сессии. Используя `owner_id` в запросах, мы обеспечиваем направленные запросы (`Targeted Queries`). 
  - TTL-механизм очистки (по полю `expires_at`) выполняется локальным фоновым потоком на каждом отдельном шарде и не создает межшардового трафика.

# Задача 8: Выявление и устранение «горячих» шардов

Несмотря на корректно подобранные ключи шардирования (например, составной ключ `{ category: 1, product_id: 1 }` для товаров), распределение *количества* документов не гарантирует равномерного распределения *нагрузки* (запросов). Если из 10ГБ данных 7ГБ — это непопулярные аксессуары на Шарде 1, а 3ГБ — популярная «Электроника» на Шарде 2, то балансировщик (Balancer) MongoDB будет "считать", что всё в порядке (разница по объему чанков невелика), но Шард 2 будет "гореть" от 70% всех read/write запросов.

## Разработка метрик для мониторинга
Для выявления таких ситуаций недостаточно полагаться только на количество чанков. Необходим мониторинг производительности базы данных и инфраструктуры. 

Основные метрики для сбора (например, через Prometheus + MongoDB Exporter в Grafana):

1. **Метрики утилизации ресурсов хоста (на каждый шард):**
   - **CPU Usage:** Если на одном узле 90%, а на другом 10% — это явный признак горячего шарда.
   - **Disk I/O (IOPS / Latency):** Узкое место при множественных записях (покупках).
   - **Network TX/RX:** Объем проходящего сетевого трафика.

2. **Метрики базы данных (показатели MongoDB):**
   - **`opcounters` (`query`, `insert`, `update`, `delete`):** Ключевая метрика. Сравнение количества операций в секунду (OPS) между шардами.
   - **`active clients` / `currentQueue`:** Количество текущих подключений и длина очереди ожидающих выполнения запросов. Растущая очередь `readers`/`writers` — верный признак перегрузки.
   - **`ticketReplicas` (WiredTiger):** Количество доступных Read/Write "билетов". Если они исчерпаны, шард захлебывается.
   - Распознавание проблемных индексов или медленных запросов: Мониторинг логов медленных запросов (`system.profile`).

## Механизмы автоматического перераспределения
MongoDB по умолчанию балансирует нагрузку **только по объему данных (количеству чанков)**, а не по частоте обращений к ним. Поэтому штатный балансировщик не решит проблему 70% кликов на 10% данных.

### Стратегия 1: Равномерное "размазывание" (Hashed Sharding)
Если паттерны чтения/записи для коллекции `products` показывают, что горячие зоны (категории) возникают постоянно и непредсказуемо, самым радикальным инфраструктурным решением будет отказ от Ranged Sharding.
- **Изменение шард-ключа:** Переход на чистый хэш: `{ "product_id": "hashed" }`.
- **Плюсы:** Идеальное равномерное распределение запросов записи (апдейт остатков) и чтения по ID товара. Физически невозможно создать "горячий" узел для группы товаров.
- **Минусы (Scatter-Gather):** Поиск всех товаров в категории "Электроника" будет опрашивать *все* шарды. Если кластер небольшой (2-4 шарда), это приемлемо. Для огромных кластеров (20+ шардов) это "убьет" сеть.

### Стратегия 2: Искусственное дробление и ручная миграция
Если мы сохраняем ключи `{ category: 1, product_id: 1 }`, но видим, что "Электроника" убивает конкретный сервер:

1. **Splitting (Дробление):** Мы можем принудительно разбить диапазоны (чанки) категории "Электроника" на более мелкие.
   ```javascript
   // Пример принудительного разделения, если Монго сама не делит
   sh.splitFind("somedb.products", { "category": "Электроника", "product_id": "midpoint_id_here" })
   ```
2. **Move Chunk (Миграция):** Вручную перекинуть часть разбитой "Электроники" на простаивающие шарды. Нагрузка (70% запросов) размажется на несколько серверов.
   ```javascript
   sh.moveChunk("somedb.products", { "category": "Электроника", "product_id": "midpoint_id_here" }, "shard0003")
   ```

### Стратегия 3: Выделение ресурсоемкого диапазона (Zone Sharding)
MongoDB поддерживает механизм **Zones (Тагирование шардов)**. Если "Электроника" требует огромных ресурсов, мы можем выделить под неё самые мощные физические серверы.
1. Присваиваем мощному серверу тег `high_perf`:
   ```javascript
   sh.addShardTag("shard_powerful_1", "high_perf")
   ```
2. Привязываем диапазон "Электроника" к этому тегу:
   ```javascript
   sh.addTagRange(
      "somedb.products",
      { category: "Электроника", product_id: MinKey },
      { category: "Электроника", product_id: MaxKey },
      "high_perf"
   )
   ```
В этом случае балансировщик стянет все чанки "Электроники" на мощные серверы, изолировав остальной сайт от просадок производительности.

### Стратегия 4: Кэширование на уровне приложения (Архитектурная)
Чаще всего (90% случаев) проблема "горячих чтений" каталога решается не на уровне БД, а перед ней.
- Внедрение Redis / Memcached для хранения каталога товаров и категорий. Электроника будет отдаваться из оперативной памяти кэша, а обновления остатков будут собираться в очереди (Kafka/RabbitMQ) или асинхронно писаться в MongoDB, не блокируя UI магазина.

# Задача 9: Настройка чтения с реплик и консистентность

В MongoDB Replica Set чтение по умолчанию всегда идет с Primary узла (обеспечивая строгую консистентность). Однако для масштабирования читающей нагрузки часть некритичных запросов (Read Preference) можно перенаправить на Secondary узлы (получая eventual consistency — итоговую согласованность).

## Таблица операций чтения и выбор узла

| Коллекция      | Операция                                     | Чтение с узла (Read Preference)      | Допустимый лаг репликации          | Обоснование                                                                                                                                                                                                                                                     |
| :------------- | :------------------------------------------- | :----------------------------------- | :--------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`products`** | Просмотр каталога, поиск, фильтрация товаров | **Secondary** (`secondaryPreferred`) | **Высокий** (от секунд до минут)   | Каталог меняется редко, а риск того, что пользователь увидит старую цену или описание в течение пары минут, некритичен для бизнеса.                                                                                                                             |
| **`products`** | Проверка остатков **перед** списанием        | **Primary** (`primary`)              | **0 мс** (Строгая консистентность) | Проверка наличия и последующее списание должны происходить на актуальных данных. Списание по устаревшей реплике приведет к перепродаже отсутствующего товара (overselling).                                                                                     |
| **`products`** | Отображение плашки "Осталось мало шт." в UI  | **Secondary** (`secondaryPreferred`) | **Средний** (до 5-10 секунд)       | Это информативная UI-фича. Если там написано "В наличии 5", а по факту 4 — это не сломает заказ (финальная проверка будет на `Primary`).                                                                                                                        |
| **`orders`**   | Вывод "Истории заказов" в ЛК                 | **Secondary** (`secondaryPreferred`) | **Средний** (до 5-10 секунд)       | Исторические данные неизменны. Задержка в несколько секунд после оплаты вполне допустима (пользователю можно показывать локальный кэш "Заказ принят").                                                                                                          |
| **`orders`**   | Проверка статуса оплаты платежным шлюзом     | **Primary** (`primary`)              | **0 мс**                           | Бизнес-критичная логика перевода заказа в статус "Оплачено" не должна опираться на устаревшие данные.                                                                                                                                                           |
| **`carts`**    | Все операции с корзиной (get, add, remove)   | **Primary** (`primary`)              | **0 мс**                           | Корзина — самый динамичный и важный сценарий пользователя (он кликает "Добавить" и сразу "Открыть корзину"). Если чтение пойдет на отставший Secondary, пользователь увидит пустую корзину после добавления товара (что вызовет панику и дублирование покупок). |

## Обоснование выбора и бизнес-логика

1. **Коллекция `carts` (Корзины):**
   - **Риски:** Пользовательский опыт. Когда пользователь добавляет товар в корзину и переходит на страницу оформления (или просто обновляет страницу), он должен **гарантированно** видеть добавленный товар (Read-Your-Own-Writes консистентность). 
   - **Решение:** Чтение только с `Primary`. Реплики для `carts` используются исключительно для отказоустойчивости (Failover), а не для масштабирования чтения.

2. **Коллекция `products` (Товары):**
   - **Риски:** 90% нагрузки интернет-магазина — это чтение каталога (категории, описания, цены). Грузить этими запросами `Primary` расточительно. 
   - **Решение:** Поиск и вывод каталога направляются на `Secondary`. 
   - **Обработка остатков:** Когда клиент нажимает "Купить", транзакция (списание) должна опираться только на `Primary`. При этом плашка "в наличии Х" на витрине может читаться с `Secondary` (отставание в пару секунд здесь не проблема).

3. **Коллекция `orders` (Заказы):**
   - **Риски:** Балансовые операции и отправка в доставку.
   - **Решение:** Внутренняя бизнес-логика (работа со статусами) строго на `Primary`. А вот тяжелые запросы из аналитики, бухгалтерии (экспорт всех заказов за сутки) или клиентского кабинета (отображение истории за год) должны уходить на `Secondary`, чтобы не стопорить создание новых заказов при пиковых нагрузках (Black Friday).

# Задача 10: Миграция на Apache Cassandra: модель данных и стратегии отказоустойчивости

При нагрузке в 50 000 RPS архитектура MongoDB с централизованным управлением шардами перестает быть эффективной. Переход на Apache Cassandra позволяет использовать кольцевую топологию и механизм согласованного хеширования. Это устраняет проблему лишнего трафика при расширении кластера, но требует проектирования таблиц под конкретные запросы приложения.

## 10.1. Распределение данных по типам хранилищ
Для обеспечения производительности данные магазина разделяются на две группы в зависимости от паттернов их использования.

**Сущности для переноса в Cassandra:**
- **История заказов (`orders`):** Данные часто создаются, но очень редко меняются. Структура запросов ограничена выборкой по идентификатору пользователя с сортировкой по времени. Это позволяет эффективно распределять записи по узлам.
- **Пользовательские корзины (`carts`):** Данные часто читаются и обновляются. При этом TTL ограничен, что означает, что можно использовать встроенный механизм Cassandra для удаления записей.

**Сущности, которые остаются в текущем хранилище:**
- **Каталог товаров и остатки (`products`):** При создании заказов мы должны быть уверены в согласованности и консистентности данных остатков, поэтому правильнее работать через мастер-реплику.

---

## 10.2. Разработка модели данных
Проектирование схем в Cassandra выполняется на основе приоритетных запросов.

### Таблица 1: История заказов пользователя (`orders_by_user`)
Для исключения риска появления слишком объемных партиций у активных клиентов применяется секционирование по годам.

**Фрагмент кода:**
```cql
CREATE TABLE orders_by_user (
    user_id uuid,
    year int,
    created_at timestamp,
    order_id timeuuid,
    total_amount decimal,
    status text,
    items map<uuid, int>,
    PRIMARY KEY ((user_id, year), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

- **Partition Key (`user_id`, `year`):** Обеспечивает равномерное распределение нагрузки и ограничивает размер данных в одной группе.
- **Clustering Keys:** Гарантируют физическую сортировку заказов от новых к старым.
- **Тип данных `map`:** Позволяет хранить список товаров и их количество, обеспечивая доступ к конкретным позициям без перезаписи всей строки.

### Таблица 2: Активные корзины (`carts_by_owner`)
Для эффективной работы с временными данными используется стратегия уплотнения, оптимизированная под TTL.

**Фрагмент кода:**
```cql
CREATE TABLE carts_by_owner (
    owner_id uuid,
    product_id uuid,
    quantity int,
    added_at timestamp,
    PRIMARY KEY ((owner_id), product_id)
) WITH default_time_to_live = 604800
  AND compaction = {
    'class': 'TimeWindowCompactionStrategy',
    'compaction_window_unit': 'DAYS',
    'compaction_window_size': 1
  };
```

Применение `TimeWindowCompactionStrategy` позволяет системе удалять устаревшие файлы данных целиком. Это исключает накопление пустых записей и снижает нагрузку на диски.

## 10.3. Стратегии репликации и отказоустойчивость
В Cassandra можно устанавливать сразу несколько механизмов синхронизации, поэтому чаще они будут включены все. Главный вопрос скорее в необходимом уровне согласованности. На операциях с корзиной он более важен, чтобы уменьшить вероятность пользователя не увидеть добавленный товар и добавить его еще раз.

**Механизмы синхронизации:**
- **Уровни согласованности (Consistency Levels):** Для операций с корзинами устанавливается уровень `LOCAL_QUORUM`. Это гарантирует, что ответ будет получен только после подтверждения записи большинством узлов в локальном дата-центре.
- **Hinted Handoff:** Если один из узлов временно недоступен, соседние серверы сохраняют информацию о пропущенных записях. Сразу после возвращения узла в сеть данные передаются на него в приоритетном порядке.
- **Read Repair:** Синхронизация выполняется непосредственно в момент чтения. Если система обнаруживает расхождение в версиях данных на разных репликах, она автоматически записывает самую актуальную версию на все узлы.
- **Anti-Entropy Repair:** Для поддержания полной идентичности данных на всех серверах запускается регламентный процесс восстановления. Для истории заказов процедура выполняется раз в неделю, что гарантирует сохранность архивов даже при возникновении скрытых ошибок на дисках.
