# Задание 9. Чтение с реплик: primary / secondary и задержка

Предпосылка: шард = replica set, у каждого `mongod` есть `primary` и `secondary`. В драйвере: `readPreference` (`primary`, `primaryPreferred`, `secondary`, `secondaryPreferred`, `nearest`) и при чтении с `secondary` — `maxStalenessSeconds` (в MongoDB 4.4+), `readConcern` (`local`, `available`, `majority`, `linearizable` — по сценарию). Ниже только чтения; запись — на `primary` (oplog → secondaries).

## 1. Сводная таблица: куда направлять чтение

| Коллекция | Операция чтения | Нода | Допустимая задержка репликации (lag) | Краткое обоснование |
|-----------|-----------------|------|--------------------------------------|---------------------|
| `products` | Каталог: список по категории, фильтр по цене, пагинация | secondary (или `secondaryPreferred`) | 30–60 с (калибровка под SLA витрины) | Лёгкое отставание по цене/витрине на фоне высокой частоты read обычно ОК. |
| `products` | Карточка товара: описание, «базовая» цена (без оформления) | secondary | 15–30 с | Как каталог, контент редко должен быть субсекундным. |
| `products` | Остаток `stock[зона]` в цепочке оформления/оплаты/списания, чтение сразу перед `write` | primary (или сессия/транзакция с write) | 0 от «своей» записи | Риск перепродажи: на secondary остаток ещё не обновился. |
| `products` | «В корзину» (проверка доступности) | primary, при нехватке ёмкости — secondary + `maxStalenessSeconds` | при secondary: не больше 1–5 с | Пик безопаснее с primary. |
| `orders` | История заказов (список) | secondary | 5–30 с | Небольшой лаг появления новой строки в списке терпим. |
| `orders` | Деталь/статус сразу после оплаты/создания | primary | 0 (read-your-writes) | Пользователь ожидает актуальный статус, не снимок с secondary. |
| `orders` | Деталь/статус «старых» заказов, не в момент смены статуса | secondary | 10–60 с | Редкие смены статуса, минута отставания редко критична. |
| `orders` | Поиск по `order_no` / саппорт | primary, при низком приоритете — secondary | 0–10 с | Споры: предпочтение свежей копии. |
| `carts` | Активная корзина: get по `user_id`/`session_id` + `active` | primary | 0 | После add/remove/merge на secondary — устаревшее тело. |
| `carts` | Аналитика, фоновый джоб без влияния на UI | secondary | 30 с – минуты | Offload read. |
| `carts` | Merge: чтение гостевой и пользовательской перед слиянием | primary (оба в одной логике) | 0 | Согласованные снимки, иначе дубли/потери. |

Дополнение: `nearest` / `primaryPreferred` — низкая задержка сети; `nearest` может дать `secondary` и нарушить сценарии «только primary» — критичные пути с явным `readPreference: primary`.

## 2. Допустимая задержка: как читать цифры

- 0 — `primary` или сессия с `readConcern`/read-your-writes после записи в ней, либо транзакция.
- 1–5 с — `maxStalenessSeconds` у secondary, где риск согласован (не пик, не списание).
- 5–60 с — витрина, список заказов, несрочный статус; мониторить `replication` lag p95/p99, алерт если lag > верхнего коридора.

Практика: oplog-lag per secondary; при росте — сужать коридор или временно переводить критичные read на `primary`.

## 3. Обоснование

Консистентность. Запись `writeConcern: majority` на `primary`; чтение с secondary — eventual. После «своей» записи читать со свежим снимком: `primary` или read-after-write в сессии.

Частота обновлений. `products.stock` и `carts` на пике обновляются часто; `orders.status` реже, но в момент смены нужен свежий read. Витрина читается очень часто — смысл отдавать на secondary с умеренным lag, разгружая primary.

Бизнес-логика.  
- `stock` с отстающей реплики + списание на primary → риск oversell; у границы — `primary` или одна транзакция.  
- Статус заказа сразу после оплаты — пользователь смотрит в экран, stale на secondary плохо; `primary` или read-your-writes в сессии.  
- Активная корзина и merge — `primary`, иначе путаница строк и двойных действий.


Точные поля (`maxStalenessSeconds` только с `maxStaleness` и для replica set с достаточным oplog) — в [документации](https://www.mongodb.com/docs/manual/core/read-preference/) к версии кластера.

## 5. Краткое резюме

- Secondary: каталог, спокойная карточка, список/старые детали заказов, фон; верхняя граница lag + мониторинг p95 `replication` lag.  
- Primary: цепочка списания `stock`, критичная корзина, merge, заказ/статус сразу после оплаты, спор по `order_no`.  
- Задержка — в таблице; при деградации репликации сужать коридор или откат read на `primary`.
