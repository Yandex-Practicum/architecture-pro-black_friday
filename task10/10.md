## 1. Введение

В связи с проблемами масштабируемости MongoDB при пиковых нагрузках (Black Friday, 50k RPS) 
и высокими накладными расходами на перебалансировку данных (Range-Based Sharding),
принято решение о миграции критических подсистем на Apache Cassandra.

**Цели миграции:**
* Обеспечение линейной масштабируемости записи.
* Устранение простоев и просадок latency при добавлении новых узлов (использование Consistent Hashing).
* High Availability (HA) через Masterless-архитектуру.

## 2. (Задание 10.1) Анализ данных и выбор сущностей для миграции


### Выбор сущностей

| Сущность | Решение | Обоснование (Почему Cassandra?) |
| :--- | :--- | :--- |
| **Корзины (Carts)** | **Мигрируем** | **Идеальный кандидат.** Высочайшая интенсивность записи (добавление товаров). Данные временные (TTL). Доступ всегда идет по ключу (`user_id` или `session_id`). Требуется максимальная доступность (AP-система), чтобы клиент не потерял корзину при сбое узла. |
| **История заказов (Order History)** | **Мигрируем** | **Отличный кандидат.** Модель "Write Once, Read Many". Заказы неизменны после завершения. Основной сценарий — "показать пользователю его заказы по дате". Cassandra эффективно хранит временные ряды внутри широких строк. |
| **Логи / Активность (User Activity)** | **Мигрируем** | Интенсивная запись потока событий ("Black Friday" кликстрим). Cassandra обеспечивает огромную пропускную способность на запись. |
| **Товары (Products)** | *Оставляем в MongoDB / Elasticsearch* | Не подходит. Для товаров нужны сложные фильтры (цена, категория, атрибуты). Cassandra требует `ALLOW FILTERING` для таких запросов, что убивает производительность. |

## 3. (Задание 10.2) Концептуальная модель данных

### Сущность: Активная Корзина (Carts)

**Запрос:** Получить содержимое корзины пользователя. Добавить/удалить товар.

```sql
CREATE TABLE cart_items (
    user_id uuid,
    product_id uuid,
    quantity int,
    added_at timestamp,
    product_name text,
    price decimal,
    PRIMARY KEY ((user_id), product_id)
) WITH default_time_to_live = 604800;
```

*   **Partition Key :** `user_id`. Все товары одного пользователя лежат на одной ноде. Обеспечивает чтение корзины за один запрос к диску.
*   **Clustering Key :** `product_id`. Позволяет сортировать товары внутри партиции и уникально идентифицировать строку для UPDATE/DELETE конкретного товара.

**Обоснование:**
*   **Равномерность:** Пользователи распределяются по кольцу (Consistent Hashing) случайно. Нет "горячих диапазонов" (как было в Mongo с датами или категориями).
*   **Hot Partition:** Риск минимален, так как в корзине редко бывает >1000 товаров.

### Сущность: История заказов (Order History)

**Запрос:** Показать список заказов пользователя, начиная с новых.

```sql
CREATE TABLE orders_by_user (
    user_id uuid,
    order_id timeuuid,
    status text,
    total_amount decimal,
    created_at timestamp,
    PRIMARY KEY ((user_id), order_id)
) WITH CLUSTERING ORDER BY (order_id DESC);
```

*   **Partition Key:** `user_id`. Группируем заказы по клиенту.
*   **Clustering Key:** `order_id` (TimeUUID). Данные физически хранятся на диске отсортированными по времени. Выборка последних 10 заказов происходит мгновенно.

**Проблема Hot Partition:** Если у нас есть B2B-клиенты с 100 000 заказов, партиция станет слишком большой.
**Решение (Bucketing):** Для крупных клиентов можно добавить месяц или год в Partition Key: `PRIMARY KEY ((user_id, year), order_id)`.

### Сущность: Детали заказа (Order Details)

**Запрос:** Показать состав конкретного заказа.

```sql
CREATE TABLE order_details (
    order_id timeuuid,
    product_id uuid,
    product_name text,
    quantity int,
    price decimal,
    PRIMARY KEY ((order_id), product_id)
);
```

*   **Partition Key:** `order_id`. Обеспечивает чтение всех позиций заказа за один запрос.
*   **Clustering Key:** `product_id`. Позволяет обращаться к конкретной позиции.


## 4. (Задание 10.3) Стратегии целостности и репликации

Выбор стратегии зависит от баланса между скоростью (Latency) и надежностью данных.

### Стратегии восстановления (Repair Strategies)

| Стратегия | Механизм                                                                                                                                                                            | Применение                                                                                                                                                                                                   |
| :--- |:------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|:-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Hinted Handoff | Если узел-реплика недоступен (сбой сети или GC пауза), узел-координатор сохраняет hint локально. Когда целевой узел вернется в строй, координатор отправит ему пропущенные данные.  | Для всех (Carts, Orders). Критично для "Black Friday". Если узел кратковременно завис от нагрузки, запись не отпадет с ошибкой, а сохранится в hint. Это обеспечивает низкую задержку записи.                |
| Read Repair | При чтении координатор опрашивает реплики. Если обнаруживается расхождение данных (разные timestamp), он возвращает клиенту последние данные и в фоне обновляет устаревшие реплики. | Для Истории Заказов (Orders). Заказы читаются чаще, чем пишутся. Важно, чтобы клиент видел корректный статус заказа. Мы можем пожертвовать долями миллисекунд на чтение ради консистентности.                |
| Anti-Entropy Repair (Nodetool repair) | Фоновый процесс синхронизации всех реплик (сравнение Merkle Trees). Тяжелая операция.                                                                                               | Обязательно для всего кластера. Запускается по расписанию (например, раз в неделю или каждую ночь в часы минимальной нагрузки) для устранения рассинхрона, которую не исправили Hints и Read Repair.         |

### Выбор уровня консистентности (Consistency Level - CL)

#### 1. Корзины (Carts)
* Приоритет: Максимальная скорость записи и доступность.
* Проблема: Если мы будем ждать подтверждения от всех реплик (ALL) или большинства (QUORUM), один медленный узел затормозит весь сайт. Потеря одного товара в корзине при падении дата-центра — допустимый риск ради того, чтобы сайт работал быстро.
* Write CL: ONE (или LOCAL_ONE для гео-распределения). Запись считается успешной, если записана хотя бы на один диск.
* Read CL: ONE.
* Стратегия: Упор на Hinted Handoff для защиты от кратковременных сбоев.

#### 2. Заказы (Orders)
* Приоритет: Строгая целостность данных. Потеря заказа недопустима.
* Write CL: QUORUM (или LOCAL_QUORUM). Запись должна подтвердиться на большинстве реплик (например, 2 из 3). Это гарантирует, что данные не пропадут при отказе одного узла.
* Read CL: QUORUM. Гарантирует чтение актуальных данных (Strong Consistency).
* Стратегия: Активное использование Read Repair (параметр read_repair_chance в схеме таблицы).

### Итоговая таблица настроек

| Сущность | Partition Key | Write CL | Read CL | Ключевая стратегия Repair |
| :--- | :--- | :--- | :--- | :--- |
| Carts | user_id | ONE | ONE | Hinted Handoff (Защита от сбоев узлов) |
| Order History | user_id | QUORUM | QUORUM | Read Repair (Актуализация при просмотре) |
| Activity Logs | session_id | ANY / ONE | ONE | Anti-Entropy (Фоновое лечение) |

## 4. Резюме

Миграция на Cassandra решает проблему масштабируемости записи для корзин и истории заказов.
Отсутствие ACID-транзакций компенсируется правильным моделированием данных (денормализация, идемпотентность) и гибридным подходом (оставление процессинга в MongoDB/Postgres).
Основной выигрыш — предсказуемая задержка (latency) при любом росте нагрузки.
