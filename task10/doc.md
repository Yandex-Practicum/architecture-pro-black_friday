### Задание 10. Миграция на Cassandra: модель данных, стратегии репликации и шардирования

1. Данные критически важные с точки зрения целостности и скорости обработки:
  - Корзины (carts): Частые обновления, временные данные, высокая нагрузка.
  - Пользовательские сессии: Высокая частота обновлений, временные данные, нужна высокая доступность.
  - Товары (products): Частые обновления остатков, запросы по категории/цене, нужна высокая доступность.

2. Что не подходит для переноса в Cassandra:
  - Заказы (orders) и оплаты (payments): Для них Cassandra не подходит, если требуется строгая консистентность (например, ACID для транзакций).

4. Partition key и кластерные ключи:
  - carts (корзины):
    ```
    CREATE TABLE carts (
      cart_id uuid PRIMARY KEY,
      user_id uuid,
      session_id text,
      products tuple<uuid, int, decimal>,
      created_at timestamp,
      updated_at timestamp,
    );
    ```
    Partition key:  cart_id  — уникальный ключ, равномерное распределение корзин по узлам.

  - sessions (сессии):
    ```
    CREATE TABLE sessions (
      session_id text PRIMARY KEY,
      user_id uuid,
      created_at timestamp,
      expires_at timestamp,
    ) WITH default_time_to_live = 86400;
    ```
    Partition key:  session_id  — уникальный ключ, равномерное распределение.

  - products (товары):
    ```
    CREATE TABLE products (
      product_id uuid PRIMARY KEY,
      category text,
      name text,
      price decimal,
      stock_by_geo map<text, int>,
      attributes map<text, text>,
    );
    ```
    Partition key:  product_id  — уникальный ключ, равномерное распределение.

5. Обоснование модели:
  - Равномерное распределение: Все ключи — уникальные с высокой кардинальностью, избегаем «горячих» партиций.
  - Минимизация решардинга: При добавлении новых узлов данные перераспределяются по хэшу ключа, но не требуется полный перенос всех данных, как в MongoDB с диапазонным шардированием.

  Кластерные ключи нигде не используем. Можно было использовать в корзине, если карзину организовать не как одну строчку, а несколько (если в корзине несколько товаров).

6. Стратегии целостности данных:
  - Hinted Handoff: Сессии, корзины.
    Автоматическое восстановление при временных сбоях, минимизирует потерю данных, временная потеря записи нежелательна, но не критична.
  - Read Repair: Товары.
    Быстрое восстановление при чтении (товары часто запрашиваются), быстрая согласованность важнее latency.
  - Anti-Entropy Repair: Все сущности, периодически.
    Глобальное восстановление согласованности, запускается периодически, чтобы не нагружать кластер.
