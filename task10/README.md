# Задание 10.1
> Определите, какие части данных интернет-магазина (например, заказы, товары, корзины, история заказов, пользовательские сессии) являются критически важными с точки зрения целостности и скорости обработки.
>
> Обоснуйте, для каких именно данных применение Cassandra имеет смысл (учтите требования к масштабируемости, геораспределённости, скорости записи/чтения и целостности данных). 

---

**В schema.puml схема таблиц.**

### 1) **Критично по целостности**

* **Заказы (orders)**: статус, сумма, состав, доставка, платежи/резервы — только строгая консистентность.
* **Остатки/доступность товара (inventory)**: чтобы не продать товар, которого не существует.
* **Платежи/транзакции** (если отдельно): идемпотентность, защита от дублей.

### 2) **Критично по скорости (медленно = падение конверсии), но можно чуть мягче по консистентности**

* **Каталог товаров (products)**: карточки, цены, наличие (если не “последний товар” — можно небольшую задержку).
* **Корзины (carts)**: быстрые чтения/записи, UX страдает мгновенно; консистентность важна, но допускается “eventual” для вторичных витрин.

### 3) **Умеренно критично**

* **История заказов / трекинг статусов**: пользователю ок видеть статус с задержкой.
* **Отзывы, рейтинги, рекомендации, просмотры**: важны для бизнеса, но не для финансовой целостности.

### 4) **Некритично по целостности, но важно для производительности**

* **Пользовательские сессии/токены/корзина гостя**: лучше быстро и дешево (Redis), потеря неприятна, но не фатальна.

# Задание 10.2

## Концептуальная модель (что и зачем в Cassandra)

Cassandra тут — не “основная БД”, а **скоростная витрина под самые частые запросы**:

* **Корзина**: быстрый доступ к активной корзине по владельцу (`owner_key`)
* **Заказ**: быстро открыть заказ по `order_id`
* **История заказов пользователя**: быстро “Мои заказы” с пагинацией
* **Листинг каталога**: категория → список товаров (и сортировка по цене), без горячих партиций

---

## 1) `carts_by_owner`

### Ключи

* **Partition key:** `owner_key`
* **Clustering keys:** `status`, `updated_at DESC`

### Под какие запросы

* Получить активную корзину пользователя/сессии: `WHERE owner_key=? AND status='active' LIMIT 1`
* Показать последние изменения/историю (если надо): по `updated_at DESC`

### Равномерность и “горячие” партиции

* `owner_key` имеет **высокую кардинальность** → партиции размазаны по кластеру.
* Горячесть локальная: “один пользователь — одна корзина”, это ок.

### Про решардинг/смену топологии

* Партиции **маленькие** (обычно 1 активная + немного истории) → при добавлении нод стриминг лёгкий, не болит.

---

## 2) `order_by_id`

### Ключи

* **Partition key:** `order_id`
* **Clustering keys:** нет

### Под какие запросы

* Открыть заказ мгновенно по id: `WHERE order_id=?`
* Быстро обновить статус/сумму (write-heavy точечный апдейт)

### Равномерность и “горячие” партиции

* `order_id` уникален → **идеальное распределение**.
* Горячая партиция невозможна по определению (1 заказ = 1 партиция).

### Про решардинг/смену топологии

* Много мелких партиций → при расширении кластера данные переезжают равномерно, без гигантских “слонов”.

---

## 3) `orders_by_customer_month`

### Ключи

* **Partition key (составной):** `(customer_id, yyyymm)`
* **Clustering keys:** `created_at DESC`, `order_id`

### Под какие запросы

* “Мои заказы” с пагинацией по времени:
  `WHERE customer_id=? AND yyyymm=? ORDER BY created_at DESC`

### Почему нужен `yyyymm` (защита от горячих/гигантских партиций)

* Если сделать PK = `customer_id`, партиция **растёт годами** → медленно, тяжело при ребалансе.
* Time-bucket `yyyymm` режет историю на **куски предсказуемого размера** → быстрее чтение и стабильнее нагрузка.

### Про решардинг/смену топологии

* Много средних партиций вместо одной огромной → при добавлении нод **миграция быстрее**, меньше риска “долго переезжающей” партиции одного супер-активного клиента.

---

## 4) `products_by_category_bucket`

### Ключи

* **Partition key (составной):** `(category, bucket)`
* **Clustering keys:** `price ASC`, `product_id`

### Под какие запросы

* Листинг категории с сортировкой по цене:
  `WHERE category=? AND bucket IN (...) ORDER BY price ASC`

### Почему обязателен `bucket` (анти-“горячая категория”)

* Если PK = `category`, то “Популярная категория” = **одна сверхгорячая партиция** (боль).
* `bucket = hash(product_id) % N` (например, **N=64**) режет категорию на 64 равномерных партиции:

  * нагрузка распределяется
  * можно параллелить запросы по бакетам

### Про решардинг/смену топологии

* Бакеты фиксированы → партиции стабильного размера.
* При расширении кластера Cassandra переносит ranges; мы не упираемся в одну гигантскую партицию категории.
* (Да, смена N потом — это миграция/dual-write, но для учебки N фиксируем и всё.)

# Задание 10.3

## Как применяем стратегии Hinted Handoff, Read Repair, Anti-Entropy Repair

### 1) `order_by_id` (самое критичное)

**Выбор:**

* **Hinted Handoff** (включён, спасает при кратком падении ноды на записи статуса)
* **Anti-Entropy Repair** регулярно (гарантия, что статусы/сумма не разъедутся)

**Почему:** заказы критичны, но чтение должно быть быстрым. Надёжность добиваем плановым repair, а не “чинкой на чтении”.

---

### 2) `orders_by_customer_month` (витрина “мои заказы”)

**Выбор:**

* **Hinted Handoff**
* **Anti-Entropy Repair** (реже, чем для `order_by_id`)

**Почему:** это “список” — пользователь переживёт редкую задержку появления свежего заказа, но хочется, чтобы витрина со временем сходилась. Лёгкий read repair допустим, но не обязателен.

---

### 3) `carts_by_owner` (быстрое UX, данные краткоживущие)

**Выбор:**

* **Hinted Handoff**

**Почему:** корзина — супер-горячая по RPS и короткая по жизни. Гонять repair ради неё — дорого и бессмысленно. 
Если что, Mongo остаётся source of truth/пересбор.

---

### 4) `products_by_category_bucket` (каталог, кешируемый, не “деньги”)

**Выбор:**

* **Hinted Handoff**
* **Anti-Entropy Repair** периодически (или вообще жить от переиндексации/перезаливки)

**Почему:** небольшая рассинхронизация цены/названия в витрине терпима, важнее latency листинга.
Сходимость обеспечиваем плановым repair или регулярной пересборкой витрины.

---

## Итоговая логика компромисса

* **Где деньги/статусы** (`order_by_id`) → **Hinted Handoff + регулярный Anti-Entropy Repair**, read repair не трогаем (ради latency).
* **Где витрина списков** (`orders_by_customer_month`) → HH + Repair, read repair по желанию.
* **Где ultra-hot и короткоживущее** (`carts_by_owner`) → только HH, остальное не окупается.
* **Где каталог/кеш** (`products_by_category_bucket`) → HH + редкий repair/перезаливка.
