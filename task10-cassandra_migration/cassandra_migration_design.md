# Задание 10. Миграция на Cassandra: модель данных, стратегии репликации и восстановление целостности

## 1. Контекст

Во время «чёрной пятницы» интернет‑магазин столкнулся с проблемами MongoDB при масштабировании: Range-based шардирование вызывало полное перераспределение данных при добавлении новых шардов, что приводило к высокой latency на пике нагрузки.

**Цель перехода на Cassandra:**

- Высокая отказоустойчивость через leaderless‑репликацию.
- Быстрое горизонтальное масштабирование.
- Равномерное распределение данных без глобального reshuffling.
- Высокая скорость записи под экстремальной нагрузкой (50 000 RPS и выше).

---

# 2.1. Какие данные имеет смысл переносить в Cassandra

## Анализ доменных данных

### 1. История заказов и события заказов (`orders`, order_events)
- Огромный объём данных.
- Активные вставки.
- Возможность eventual consistency.
- Частые запросы: последние заказы, история пользователя.

✅ **Отлично подходит для Cassandra**.

---

### 2. Корзины (`carts`)
- Очень частые обновления.
- Хранение временных данных с TTL.
- Требование низкой latency > строгой консистентности.

✅ **Идеальный кандидат для Cassandra**.

---

### 3. Пользовательские сессии (`sessions`)
- Большое количество записей/обновлений.
- Короткий TTL.
- Немного устаревшие данные не критичны.

✅ **Нужно переносить в Cassandra**.

---

### 4. Товары и остатки по складам (`products`, `product_stock`)
- Каталог — можно кэшировать.
- Остатки — требуют высокой скорости обновления.
- Идеален подход “single-row per product+zone”.

✅ **Можно переносить частично (read-модели, остатки)**.

---

### 5. Финансовые транзакции, платежи
- Нужна строгая консистентность.

⛔ **Не стоит переносить в Cassandra**.

---

# 2.2. Концептуальная модель данных Cassandra

Ключевая идея Cassandra: **таблица проектируется под запрос**, а не под универсальную модель.

---

# 2.2.1. Таблица: История заказов пользователя (`orders_by_user`)

## Use cases:
- Получить последние заказы пользователя.
- Хронологический просмотр.

## Модель:

```
CREATE TABLE mobile_store.orders_by_user (
    user_id       uuid,
    year_month    text,       -- bucket для партиций
    order_ts      timeuuid,   -- timestamp заказа
    order_id      uuid,
    status        text,
    total_amount  decimal,
    geo_zone      text,
    items         list<text>,
    PRIMARY KEY ((user_id, year_month), order_ts)
) WITH CLUSTERING ORDER BY (order_ts DESC);
```

### Обоснование ключей

- **Partition key**: `(user_id, year_month)`
  - предотвращает разрастание одной большой партиции,
  - равномерное распределение по кольцу.

- **Clustering key**: `order_ts`,
  - эффективная сортировка,
  - позволяет получать последние заказы без сканов.

---

# 2.2.2. Таблица: Поиск заказов по ID (`orders_by_id`)

```
CREATE TABLE mobile_store.orders_by_id (
    order_id     uuid,
    user_id      uuid,
    order_ts     timeuuid,
    status       text,
    total_amount decimal,
    items         list<text>,
    PRIMARY KEY (order_id)
);
```

Use case: быстрый lookup по уникальному ID.

---

# 2.2.3. Таблица: Корзины (`carts`)

Используем универсальный идентификатор владельца корзины:

```
owner_key = "user:<uuid>"  
owner_key = "session:<uuid>"
```

```
CREATE TABLE mobile_store.carts (
    owner_key   text,
    cart_id     uuid,
    status      text,
    items       map<uuid, int>,
    updated_at  timestamp,
    created_at  timestamp,
    PRIMARY KEY (owner_key)
);
```

### Почему так?
- Одна корзина = одна строка → быстрые обновления.
- Огромная кардинальность ключа → равное распределение.

---

# 2.2.4. Таблица: Сессии пользователей (`sessions`)

```
CREATE TABLE mobile_store.sessions (
    session_id   uuid,
    user_id      uuid,
    created_at   timestamp,
    last_seen_at timestamp,
    ip           text,
    user_agent   text,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400;
```

TTL снижает нагрузку и автоочищает устаревшие записи.

---

# 2.2.5. Таблица: Остатки товара по геозонам (`product_stock_by_geo`)

```
CREATE TABLE mobile_store.product_stock_by_geo (
    product_id uuid,
    geo_zone   text,
    stock      int,
    updated_at timestamp,
    PRIMARY KEY ((product_id, geo_zone))
);
```

- Partition key: `(product_id, geo_zone)`
- Каждая запись маленькая.
- Нет широких партиций.
- Обновления быстрые.

# 2.3. Стратегии целостности данных Cassandra

Cassandra использует комбинацию:

- **Hinted Handoff**
- **Read Repair**
- **Anti-Entropy Repair**
- **Consistency Levels** (CL)

Ниже — стратегия для разных сущностей.

---

## 2.3.1. Заказы (orders)

### Требования:
- Данные должны быть достаточно консистентны.
- Потеря заказа недопустима.

### Выбор:

- **RF = 3**
- **Запись**: `LOCAL_QUORUM`
- **Чтение**: `LOCAL_QUORUM` для критичных операций; `LOCAL_ONE` для истории.
- **Hinted Handoff**: включён.
- **Read Repair**: включён (0.1–0.2).
- **Anti-Entropy Repair**: периодический `repair`.

Почему?  
→ Лучше немного жертвовать latency, чем потерять заказ.

---

## 2.3.2. Корзины (carts)

### Требования:
- latency > консистентность,
- кратковременные несостыковки не критичны.

### Стратегия:

- **RF = 3**
- **Запись**: `LOCAL_ONE`
- **Чтение**: `LOCAL_ONE`
- **Hinted Handoff**: обязательно
- **Read Repair**: выключен (`0.0`)
- **Repair**: периодический, но можно реже

Пояснение:  
Hinted Handoff компенсирует временную недоступность нод, но не влияет на latency.

---

## 2.3.3. Сессии (sessions)

### Требования:
- Максимально низкая задержка.
- Согласованность вторична.

### Стратегия:

- **RF = 3**
- **CL: LOCAL_ONE** на чтение/запись
- **Hinted Handoff** включён
- **Read Repair**: 0.0
- **Repair**: редкий, поскольку данные живут мало (TTL)

---

## 2.3.4. Остатки (`product_stock_by_geo`)

Если SLA мягкий:  
- `LOCAL_ONE` чтение/запись.

Если SLA жёсткий (нельзя oversell):  
- `LOCAL_QUORUM` на запись и финальную проверку.

Типичная комбинация:
- write path → QUORUM  
- read path → ONE + final consistency check при заказе.

# Итог

## Что переносим в Cassandra:

| Данные | Cassandra? | Причина |
|--------|------------|---------|
| История заказов | ✅ | Большой объём, append-only, скоростные чтения |
| Корзины | ✅ | Частые обновления, TTL, масштабируемость |
| Сессии | ✅ | Огромная нагрузка, малый TTL |
| Остатки | ⚠️ Частично | Зависит от SLA, модель оптимальна |
| Критичные финансы | ❌ | Требуется строгая транзакционность |

---

## Ключевые преимущества модели:

- равномерное распределение partition keys;
- отсутствие горячих партиций;
- минимизация влияния решардинга;
- возможность экстремального масштабирования (50k RPS+);
- гибкая балансировка между latency и консистентностью.
