# Миграция на Cassandra: модель данных, стратегии репликации и шардирования

## 10.1 Выбор сущностей

| Сущность                                | Переносим в Cassandra | Обоснование                                                                                                                                                                                                                                                            |
|-----------------------------------------|-----------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Товары (products)                       | **Нет**               | В нашей системе продукты хранятся вместе со складской информацией. Так как мы их не разделяем, то требуется транзакционность и согласованность, иначе есть риск "овербукинга". К тому же по товарам идёт сложный поиск, который будет сложно организовать в Cassandra. |
| Заказы (orders)                         | **Нет**               | Требуется строгая транзакционность и согласованность. При создании заказа нужно списать данные об остатках из products.                                                                                                                                                |
| Корзины (carts)                         | **Да**                | Идеальный вариант для переноса. Требует быстрой записи/обновления, высокой доступности и TTL для устаревших корзин. Модель "ключ-значение" с ключом по ID пользователя/сессии отлично ложится на Cassandra.                                                            |
| Пользовательские сессии (user_sessions) | **Да**                | Один из классических use case для Cassandra. Требует быстрой записи, чтения по ID сессии, высокой доступности и автоматического удаления (TTL). Отлично масштабируется для большого числа активных сессий.                                                             |

Таким образом товары из заказы остаются в mongo, а корзины и пользовательские сессии переезжают в Cassandra. Вынос данных таблиц должен сильно снизить нагрузку на mongo.

## 10.2 Концептуальная модель данных


```CQL
CREATE TABLE carts (
    customer_id   UUID,
    session_id    UUID,
    status        text,
    product_id    UUID,
    quantity      int,
    created_at    timestamp,
    updated_at    timestamp,
    expires_at    timestamp,
    PRIMARY KEY (customer_id, session_id)
) WITH CLUSTERING ORDER BY (product_id);
  AND  default_time_to_live = 2592000; -- TTL = 30 дней
```

**Partition Key**: customer_id, session_id
**Обоснование**: Благодаря такому ключу все позиции в корзине пользователя, либо неавторизованный пользователя (session_id) будут храниться в одной партиции.

**Clustering Keys**: product_id.
**Обоснование**: Уникальная идентификация позиции в корзине и быстрое обновление/удаление конкретного товара

```CQL
CREATE TABLE user_sessions (
    session_id   UUID,
    customer_id   UUID,
    data         text,
    created_at   timestamp,
    updated_at   timestamp,
    expires_at   timestamp,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400; -- TTL = 24 часа
```

**Partition Key**:  session_id

**Обоснование**:
- session_id обеспечивает идеальное распределение
- прямой доступ по ID сессии для проверки авторизации
- Высокая кардинальность, отсутствие "горячих" партиций
- TTL автоматически очищает старые сессии

## 10.3 Обеспечение целостности


### Корзины (carts)

**Стратегия**: Hinted Handoff + Read Repair

**Обоснование**:
- Корзины требуют высокой доступности, при этом слабая согласованность приемлема -> Получаем ускорение записи от Hinted Handoff за счёт обеспечивает доступность при временной недоступности ноды.
- Корзины часто читают -> Read Repair будет хорошо работать.


### Пользовательские сессии (user_sessions)

**Стратегия**: Hinted Handoff

**Обоснование**:
Мы принимаем риск разлогиниться в системе в случае потери данных -> получаем максимальную доступность, при минимальной согласованности.
